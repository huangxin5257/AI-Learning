# LLM 底层原理：工程师实用指南

> 以"你收到我的提示词之后做了什么"为视角，理解大语言模型的核心技术机制。
> 读完之后，你应该能解释 AI 的典型行为从哪里来——而不只是记住几个术语。

## 1. Tokenization

**AI 读的不是字，是"块"**

**在做什么：** 把输入的原始文本切成一段一段的"块"（token），再把每个块换成一个整数 ID，交给后续步骤处理。

**为什么做：** 神经网络只能处理数字，不能直接处理字符串。但如果以单个字符为单位，序列会很长、计算很贵；如果以完整单词为单位，词表会爆炸、生僻词和新词无法覆盖。Token 是在这两个极端之间找的平衡点——用"高频子串"作为基本单位，既控制了词表大小，又能拼出任意文本。具体来说，英文字符约 100 个，英文单词超 50 万个，而 BPE 词表通常在 3 ～ 10 万之间——是一个数量级的折中。

### 技术层面实际发生了什么

主流方案叫 **BPE（Byte-Pair Encoding，字节对编码）**：从单个字符出发，统计语料里所有相邻字符对的出现频率，把最高频的一对合并成一个新 token，反复迭代，直到词表达到预设大小（GPT-4 约 10 万个 token）。

训练结束后，词表固定。推理时，分词器用贪心匹配把输入文本切成词表里已有的 token，每个 token 对应一个整数 ID。模型从头到尾看到的都是整数序列，不是文字。

**切分示例：**

| 输入 | Token 切分 | token 数 |
|------|-----------|---------|
| `tokenization` | `["token", "ization"]` | 2 |
| `人工智能` | `["人工", "智能"]` | 2 |
| `def forward(x):` | `["def", " forward", "(", "x", "):"]` | 5 |
| `Quaternionfrobnicator` | `["Qu","atern","ion","fr","obn","ic","ator"]` | 7 |

> ⚠️ 做了简化：实际分词还涉及字节级处理（BPE 的 byte-level 变体）和语言相关规则，这里只讲核心思路。

### 这个机制直接导致了哪些可观测行为

**AI 数不清字母：** "strawberry 有几个 r？"容易答错——模型操作的是 `["straw", "berry"]`，不是字母。它在 token 层面推理，没有原生的字符级视图。→ 所以需要精确字符操作时（计数、逆序、首字母提取），不要让模型"想"，让它写代码执行。

**代码比自然语言"更贵"：** 缩进、特殊符号、下划线命名在语料里罕见，BPE 合并率低，同等信息量的代码消耗更多 token。一段 50 行的 Python 函数大约消耗 200 ～ 400 token，而同等信息量的自然语言描述可能只需 100 ～ 150 token。→ 所以在 context 紧张时，优先用自然语言描述意图，而不是直接粘贴大段代码。

## 2. Embedding

**语义是怎么变成数字的**

**在做什么：** 把每个 token 的整数 ID 查表换成一个高维浮点数向量（例如 GPT-3 用 12288 维，即每个 token 用 12288 个浮点数表示），同时叠加一个位置编码，告诉模型这个 token 在序列里的位置。

**为什么做：** 整数 ID 只是编号，没有任何语义信息——ID 42 和 ID 43 在数学上差 1，但"猫"和"狗"的语义距离跟编号顺序毫无关系。向量空间才能承载语义：训练会把语义相似的 token 拉到相近的位置，让后续的 Attention 计算能在几何意义上度量"哪些词相关"。

### 技术层面实际发生了什么

模型有一个 **Embedding 矩阵**，本质上就是一个可训练的查找表——和你在工程里用的 lookup table 一样，输入一个 ID，返回一行数据。行数 = 词表大小（约 10 万），列数 = 向量维度（通常 768 ～ 12288）。输入一个 token ID，就取出对应那一行，得到该 token 的语义向量。

这个向量不是人工设计的——训练时，出现在相似上下文里的 token 的向量会被梯度下降反复调整，最终使语义相近的词在空间里聚拢。"国王 − 男人 + 女人 ≈ 女王"这类关系是从数据里自然浮现的几何结构。

Embedding 之后再叠加 **位置编码（Positional Encoding）**：给每个位置生成一个固定或可学习的向量，加到 token 向量上。没有这一步，模型看"猫咬狗"和"狗咬猫"是完全一样的——两个句子的 token 集合相同，顺序不同。

**语义距离示意（简化为 4 维）：**

| 词 | 与"国王"相似度 |
|----|--------------|
| 女王 | 91% |
| 王子 | 84% |
| 权力 | 67% |
| 银行 | 31% |
| 苹果 | 12% |

### 这个机制直接导致了哪些可观测行为

**近义词提问效果相近：** "帮我写"和"帮我生成"的向量在空间里很接近，模型的后续计算趋同，所以两种问法结果差不多。→ 所以与其纠结用词，不如在 prompt 里加上具体的约束条件（格式、长度、示例），这比换一个近义词有效得多。

**AI 对新词表现不稳定：** 训练后词表固定。新词（如新发布的产品名）不在词表里，会被拆成子词 token，这些子词的向量可能没有学到足够的语义，导致模型理解偏差。→ 遇到专有名词（产品名、内部术语），在 prompt 里给出明确定义或上下文示例，帮助模型从子词拼出正确语义。

## 3. Attention 机制

**AI 怎么决定"该关注哪里"**

**在做什么：** 对序列里的每一个 token，计算它和其他所有 token 的相关程度，然后按相关程度加权聚合信息，生成一个融合了上下文的新向量。

**为什么做：** 词义依赖上下文。"苹果"在"苹果手机"和"苹果树"里意思不同，光靠 Embedding 查表得到的静态向量无法区分。Attention 让每个 token 在每一层都能动态地"看"整个序列，按当前任务需要调整自己的表示——这是模型理解语言的核心机制。

### 技术层面实际发生了什么

你可以把 Attention 想象成一个搜索引擎：Q 是搜索词，K 是每个文档的标题，V 是文档正文。先用搜索词和所有标题算相关性排名，再按排名加权混合正文内容。对应到模型里，对每个 token，Attention 把它的向量线性投影成三个角色：

- **Query（Q）：** 我在找什么——当前 token 的"搜索词"
- **Key（K）：** 我能提供什么——每个 token 的"标题标签"
- **Value（V）：** 我的实际内容——每个 token 携带的"正文信息"

用当前 token 的 Q 和序列里所有 token 的 K 做点积，得到相关性得分，归一化后得到权重，再用权重对所有 token 的 V 加权求和，输出融合了上下文的新向量。

```
q = W_Q × embedding(t)
score[s] = dot(q, W_K × embedding(s))   # 对每个 token s
weights  = softmax(scores / √d)
output   = sum(weights[s] × W_V × embedding(s))
```

**为什么要多个头？** 一个 Attention 头只能学会一种关注模式。但理解语言需要同时捕捉多种关系：语法搭配、指代关系、语义关联……所以模型使用 **Multi-Head Attention**——并行运行多个 Attention 头，每个头专注不同类型的关系。

举个例子，在 "The cat sat on the mat because it was tired" 里：一个头在计算 it → cat 的指代关系，另一个头在计算 sat → on 的语法搭配，第三个头在关注 tired → cat 的语义关联。

所有头的输出拼接后，再做一次线性变换（把多个头的结果融合成统一的维度），然后传入下一层。整个模型通常堆叠数十层（GPT-3 有 96 层），每一层都重复这个过程，逐层构建越来越抽象的语义表示。

> ⚠️ 做了简化：实际还有缩放、Mask（防止看到未来 token）等细节，这里只讲核心计算逻辑。

### 这个机制直接导致了哪些可观测行为

**指代消解有时出错：** "it" 指向谁，靠 Attention 在上下文里找相关性最高的候选名词。句子很长、候选词多时，权重分布容易受噪声干扰，AI 会搞混指代。→ 在 prompt 中避免模糊指代，用具体名词重复替代"它"、"这个"、"上述内容"。

**Prompt 位置影响效果：** 实验发现头部和尾部 token 的 attention 权重天然偏高（"Lost in the Middle" 效应）。→ 把最关键的指令放在 system prompt 开头和 user message 末尾，避免埋在中间段落里。

## 4. Context Window

**为什么有长度限制**

**在做什么：** 把当前对话的所有 token（system prompt + 历史消息 + 当前输入）拼成一个序列，作为模型这次推理的完整输入。这个序列的最大长度就是 Context Window。

**为什么做：** Attention 需要让每个 token 和序列里所有其他 token 做点积，计算量是 O(n²)——序列长度翻倍，计算量翻四倍。这是物理约束，不是工程偷懒。必须设上限，否则无法在有限资源内完成推理。Context Window 就是这个上限的名字。

### 技术层面实际发生了什么

每次推理时，模型只能"看到"当前 context window 内的 token。超出窗口的内容会被截断（通常截掉最早的部分），对模型来说那部分根本不存在。

**常见 Context Window 大小对比：**

| 模型世代 | 典型 Context Window |
|---------|-------------------|
| 早期 GPT | 4K token |
| GPT-4 | 8K ～ 128K token |
| Claude 3.x | 200K token |
| Claude 4.x | 200K+ token |

窗口越大，能放进去的历史和文档越多，但每个 token 的生成成本也越高（Attention 的 O(n²) 特性）。这就是为什么超长 context 的模型既贵又慢。

> ⚠️ 做了简化：Flash Attention、Sliding Window 等技术改善了显存效率，但 attention 的根本复杂度仍是约束。

### 这个机制直接导致了哪些可观测行为

**长对话为什么会"忘事"：** 对话超出 context 后，早期消息被截断，模型字面上不再能看到那部分历史。这不是"注意力不集中"，是上下文窗口满了。→ 解决方案：开新会话，或用摘要压缩历史再塞回去。

**质量在 context 末段下降：** 即使没超出限制，context 很长时模型的注意力会更分散（lost-in-the-middle），靠近窗口头尾的信息权重更高，中间段容易被忽略。→ 把最重要的参考资料放在 context 的开头或末尾，而不是堆在中间。

## 5. 解码与采样

**Temperature 在控制什么**

**在做什么：** 把模型最后一层的输出向量转换成词表上的概率分布，然后从这个分布里"抽签"决定下一个 token 是什么。

**为什么做：** 模型的原始输出（logits）只是每个 token 的原始得分，不能直接用。需要先归一化成概率，再决定怎么从中选词。"怎么选"有讲究：每次选最高概率的词（贪心），输出会单调重复；完全随机采样，输出会乱成一团。采样策略就是在确定性和随机性之间调出合适的平衡。

### 技术层面实际发生了什么

模型输出 **logits 向量**（词表大小，如 10 万维），经过 Softmax 变成概率分布。**Temperature** 是在 Softmax 之前做的缩放——用 logits 除以 T：

| Temperature | 效果 | 适用场景 |
|------------|------|---------|
| T → 0 | 分布极度尖锐，最高分 token 概率趋近 1（贪心解码） | SQL 生成、代码补全 |
| T = 1 | 原始分布，按训练数据的自然频率采样 | 通用对话 |
| T > 1 | 分布被"压平"，低概率词的机会增大 | 创意写作、头脑风暴 |

T 越小，分布越尖，越确定；T 越大，分布越平，越随机。

此外还有两个常用的截断策略，在 Temperature 之后再做过滤：

- **Top-k：** 只保留概率最高的 k 个候选，其余归零再归一化
- **Top-p（nucleus sampling）：** 从高到低累积概率，到达 p 后截断，只在这个最小集合里采样

**Temperature 对概率分布的影响（示意）：**

```
上文："The weather today is..."

T = 0.1  → sunny: 98%,  cloudy: 2%,   perfect: 0%
T = 1.0  → sunny: 55%,  cloudy: 25%,  perfect: 12%,  nice: 6%,  ...
T = 2.0  → sunny: 35%,  cloudy: 22%,  perfect: 18%,  nice: 14%, bizarre: 8%, ...
```

### 这个机制直接导致了哪些可观测行为

**同一问题问两次，答案不一样：** Temperature > 0 时每次解码都是随机采样，即使概率分布相同，结果也可能不同。这是设计，不是 bug——增加了多样性，避免了重复套路。→ 如果需要可复现的输出（如回归测试、结果比对），把 Temperature 设为 0。

**创意写作 vs 代码应该用不同参数：** 代码要精确，用低 temperature（0.1 ～ 0.3）；创意写作要新颖，用高 temperature（0.7 ～ 1.2）。大多数 API 默认值在 0.7 ～ 1.0 之间，适合通用场景。

## 6. 自回归生成

**一个 token 一个 token 地输出**

**在做什么：** 每次只生成一个 token，把它追加到输入序列末尾，然后用这个新序列再做一次完整推理，生成下一个 token，如此循环直到结束。

**为什么做：** 模型没有能力"一次性规划整句话"——它的训练目标就是"给定前文，预测下一个 token"。这个目标天然决定了生成方式：一步一步地往前走，每步的输出是下步的输入。这种方式在训练和推理上完全对齐，也是 Transformer 架构在生成任务上工作的基本方式。

### 技术层面实际发生了什么

```
第 1 步：输入 = [prompt]                       → 推理 → 生成 token_1
第 2 步：输入 = [prompt + token_1]             → 推理 → 生成 token_2
第 3 步：输入 = [prompt + token_1 + token_2]   → 推理 → 生成 token_3
……直到生成 <EOS> 或达到 max_tokens 上限
```

每一步都是一次**完整的前向传播**：所有层、所有 Attention 头全部跑一遍，只为决定下一个 token 是什么。以 GPT-3 为例，它有 96 层 Transformer——每生成一个 token 都要依次跑完这 96 层。

关键的工程优化是 **KV Cache**：Attention 计算需要每个 token 的 Key 和 Value 矩阵。之前步骤已经算过的 K/V 可以缓存复用，每步只需计算新 token 的 K/V，避免从头重算整个序列。

以 1000 token 的序列为例：没有 KV Cache，生成第 1000 个 token 需要重新计算前 999 个 token 的 K/V，总计算量是 O(n²)；有了 KV Cache，只需计算第 1000 个 token 自己的 K/V 并与缓存拼接，计算量降到 O(n)。代价是显存随 context 增长持续占用。

生成终止的条件：模型输出了 `<EOS>`（end-of-sequence）token，或者达到了 `max_tokens` 上限。

### 这个机制直接导致了哪些可观测行为

**生成无法"反悔"：** token 一旦采样出来就追加进输入，后续所有生成都在此基础上条件化。模型没有全局规划——它不知道后面要说什么，只看当前最高概率的路。开了一个错误的头就很难自我纠正，这就是为什么有时会"越说越偏"。→ 这就是为什么 Chain-of-Thought 和分步提示有效：让模型在生成最终答案之前先"想一想"，相当于给它更多 token 来修正方向，而不是直接跳到结论。

**流式输出不是假象：** 你看到的文字一个个冒出来，完全对应真实的生成节奏——每推理一次吐出一个 token，没有预生成再分批展示。

**Prompt 越长，每个 token 生成越慢：** 即使有 KV Cache，更长的 context 意味着每步 Attention 要处理更多 K/V，显存带宽压力增大，速度下降。→ 超长 context 的模型不仅贵，还慢。在性能敏感的场景下，优先精简 context 而不是升级模型。

**两次输出长度可能不同：** 生成终止靠 `<EOS>` token 的概率，而采样有随机性。Temperature 高时 `<EOS>` 的权重被压低，模型可能生成更长的回复；Temperature 低时更快收敛到结束符。→ 如果需要严格控制输出长度，同时使用 `max_tokens` 参数和 prompt 里的长度指令。

## 7. 训练 vs 推理

**为什么 AI "不学习"你说的话**

**在做什么：** 训练阶段用海量数据反复更新模型的权重参数；推理阶段用固定的权重处理你的输入，生成输出——权重全程只读，不做任何修改。

**为什么做：** 这是两个目标完全不同的过程。训练的目标是让模型从数据里学到尽可能通用的语言模式，需要大量算力和时间；推理的目标是快速响应，必须在毫秒到秒级内完成。把两件事分开，才能让部署后的模型又快又稳定——如果每次对话都更新权重，不仅极慢，还会因为每个用户的输入互相干扰而彻底崩溃。

### 技术层面实际发生了什么

**训练（Training）** 分两步理解：

第一步——学习过程：把大量文本喂给模型，让它预测"下一个 token"，把预测结果和真实答案做对比得到 loss（损失值，衡量"猜得有多偏"）。

第二步——参数调整：通过**反向传播**把 loss 逐层分配回去。类似于你在调试时从最终输出的错误往回追溯——loss 告诉模型"最终结果偏了多少"，反向传播就是把这个偏差逐层分配到每个参数，告诉它"你该往哪个方向调、调多少"。这个过程在海量数据上重复数百万次，直到模型的预测足够准确。

规模感：GPT-4 级别的训练耗费数千张 GPU 跑数周，参数量达千亿级。训练完毕，权重冻结。

**推理（Inference）：** 你发一条消息，模型把它和历史对话拼成 context，做一次前向传播，输出 token 序列。权重全程只读，不修改。你在对话里"教"给它的任何内容只存在于 context window 里，会话结束即消失。

要让模型真正学会新东西，有两条路：

- **Fine-tuning：** 用新数据继续训练，永久更新权重。需要数据、算力和专业工程，成本高。
- **RAG（Retrieval-Augmented Generation）：** 把新知识在推理时塞进 context，模型当次可以参考，但不会记住。工程上就是：在调 API 之前，先从你的知识库检索相关段落，拼到 system prompt 或 user message 里一起发过去。成本低，灵活，是目前最常用的补充方案。

### 这个机制直接导致了哪些可观测行为

**幻觉（Hallucination）从哪来：** 模型在预测"下一个最可能的 token"，不是在查数据库。当训练数据里某个领域覆盖不足，或问题超出训练截止日期，模型会用"听起来合理"的 token 填充——权重里没有"我不知道"的硬约束，只有概率分布。→ 对事实性要求高的场景，用 RAG 注入可靠来源，并在 prompt 里要求模型"仅基于提供的资料回答，如无相关信息则说明不知道"。

**知识有截止日期：** 训练数据有 cutoff，之后发生的事件权重里不存在。对话里纠正它的错误，下次会话它可能再犯——因为那次纠正没有进入权重，只活在上次的 context 里。→ 持续性的知识更新靠 RAG 或 Fine-tuning，不能靠对话里"告诉"模型。

## 术语速查

| 术语 | 一句话解释 |
|------|-----------|
| Token | 文本被切分后的最小处理单位，介于字符和单词之间 |
| BPE | Byte-Pair Encoding，主流分词算法，反复合并高频字符对 |
| Embedding | 把 token ID 映射成高维浮点向量的过程 |
| Positional Encoding | 给向量叠加位置信息，让模型知道 token 的顺序 |
| Self-Attention | 让每个 token 动态关注序列中其他所有 token 的机制 |
| Q/K/V | Query/Key/Value，Attention 机制的三个投影角色 |
| Multi-Head Attention | 并行运行多个 Attention 头，每个学习不同类型的关系 |
| Context Window | 模型一次能处理的最大 token 数量 |
| Logits | Softmax 之前的原始得分向量 |
| Temperature | 控制采样随机性的缩放参数，越低越确定，越高越随机 |
| Top-p / Top-k | 限制采样范围的策略，过滤低质量候选 token |
| KV Cache | 缓存历史 token 的 Key/Value 矩阵，避免每步重复计算，加速自回归生成 |
| 自回归生成 | 逐 token 循环生成，每步输出一个 token 并追加到输入再做下一次推理 |

| Hallucination | 模型生成"听起来合理但实际错误"的内容 |

| RAG | Retrieval-Augmented Generation，把外部知识注入 context |
| Lost in the Middle | 长 context 中间段信息权重偏低、容易被忽略的现象 |
