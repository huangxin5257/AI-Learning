## LLMOps

| 知识点             | 是什么                                                       | 为什么学                                                    |
| ------------------ | ------------------------------------------------------------ | ----------------------------------------------------------- |
| Tracing 链路追踪   | 记录每次LLM调用的输入输出、耗时、Token用量的完整链路（Langfuse、Phoenix） | 没有可观测性就没有调试能力，生产问题排查全靠猜              |
| Prompt 版本管理    | 像管理代码一样对Prompt进行版本控制和发布管理                 | Prompt是AI应用的核心逻辑，随意修改会引入不可追溯的退化      |
| A/B 测试框架       | 对不同Prompt版本或模型进行分流实验并量化效果差异             | 凭感觉改Prompt不如用数据说话，是从实验到上线的关键环节      |
| 成本监控与优化     | 追踪Token消耗、识别高成本路径、选择合适模型降本              | LLM调用成本在规模化后会快速失控，提前建立监控是必须的       |
| 缓存策略           | 对相同或相似请求复用已有结果，减少重复调用（语义缓存）       | 高频场景下缓存可以将成本和延迟同时降低一个量级              |
| 模型路由           | 根据任务复杂度动态选择不同能力/价格的模型                    | 不是所有请求都需要最强模型，路由策略是成本优化的核心杠杆    |
| 延迟优化           | 流式输出、并行调用、提前终止等降低响应时间的工程手段         | 用户对AI响应速度极其敏感，延迟直接影响产品体验              |
| 错误处理与重试     | 超时、限流、模型拒绝等异常的降级和重试机制设计               | LLM API并不稳定，没有健壮的错误处理生产环境会频繁出故障     |
| 评估流水线自动化   | 将模型/Prompt评估集成进CI/CD，每次变更自动跑eval             | 手动评估无法规模化，自动化eval是LLMOps成熟度的标志          |
| 护栏（Guardrails） | 对输入输出进行合规检测、内容过滤、格式校验的防护层           | 裸调LLM在生产环境是危险的，护栏是保护用户和产品的最后一道门 |
