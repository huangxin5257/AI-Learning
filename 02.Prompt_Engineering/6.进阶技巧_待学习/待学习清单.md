**Meta-prompting**

1. 自动 Prompt 生成 — 给定任务描述，让模型生成候选 Prompt。手工从零写 Prompt 效率低，用模型生成草稿再人工打磨是更快的起点。
2. Prompt 批评与优化 — 让模型指出当前 Prompt 存在的问题。自己审视自己写的 Prompt 容易有盲区，模型能提供一个相对客观的第三方视角。
3. 自动优化循环（APO） — 基于评估反馈让模型迭代改进 Prompt。当 Eval Set 已经建好，可以把"看评分→改 Prompt"这个循环部分自动化，提升迭代速度。
4. Meta-prompt 的局限 — 模型优化自身存在系统性盲区。不了解局限就会过度依赖自动优化结果，在模型改不好的方向上浪费时间。

**任务分解模式**

1. 串行链 — 上一步输出作为下一步输入，将复杂任务拆成有序阶段。单次 Prompt 承载过多子任务时质量会下降，串行链让每一步只专注一件事。
2. 并行分支 — 多个子任务同时执行后合并结果。独立子任务没必要串行等待，并行能缩短整体耗时并提升各分支的专注度。
3. 路由分发 — 根据输入特征选择不同的 Prompt 分支处理。不同类型的输入用同一个通用 Prompt 处理往往效果平庸，路由让每类输入走最适合它的路径。
4. 聚合归纳 — 多个输出汇总为最终答案。多路结果需要有人"拍板"，聚合步骤负责整合、去重、消解矛盾。
5. 何时分解 — 判断任务复杂度是否超过单次 Prompt 的可靠处理范围。分解有成本，不是所有任务都需要，掌握判断标准避免过度工程化。

**鲁棒性设计**

1. 措辞敏感性测试 — 对同一意图用不同措辞改写后测试输出是否稳定。Prompt 对小幅措辞变化过于敏感，意味着它在真实用户输入面前不可靠。
2. 对抗性输入处理 — 预先设计当用户输入包含误导、矛盾或极端内容时的行为。生产环境的输入不受控，不提前设计降级行为就只能被动救火。
3. Prompt 注入防御 — 防止用户输入覆盖或篡改系统指令。System Prompt 的权威性一旦被用户输入突破，整个 Prompt 设计的约束就会失效。
4. 边界条件声明 — 在 Prompt 中预先声明极端或异常输入的处理方式。模型遇到没有预案的边界情况会自行猜测，显式声明比依赖模型默认行为更可控。
5. 降级策略 — 任务无法完成时定义 fallback 行为而非让模型自由发挥。没有降级设计的 Prompt，失败时的输出是不可预测的，有时比直接报错更危险。
