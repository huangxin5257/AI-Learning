# Chain-of-Thought（思维链，CoT）

> **一句话总结**：通过让模型"把推理步骤写出来"，将隐式的中间计算过程显式化，从而大幅提升复杂任务的准确率。

---

## 为什么 CoT 有效：Token 预测视角

LLM 本质上是一个**条件概率模型**：每次只预测下一个 token，依据是它目前"看到"的全部上下文。

关键洞察：**上下文里的内容，就是模型能"思考"的全部工作内存。**

```
没有 CoT：
[题目] → 模型在一次 forward pass 中直接输出答案
          → 多步推导被压缩进单步，中间变量无处存放，容易出错

有 CoT：
[题目] → 生成"步骤1" → 步骤1写入 context
       → 基于[题目+步骤1]生成"步骤2" → ...
       → 基于完整推导链生成最终答案
```

每个中间步骤写入 context 后，就成为后续 token 预测的"外部草稿纸"。这不是魔法——是把本来需要在单次前向传播中完成的多步计算，拆散到多次 token 生成，每次只做一小步。

**类比**：让你心算 `17 × 24` 容易出错；拿纸列竖式，每步一行，就很少错。CoT 就是给模型这张"纸"。

---

## 模块一：Zero-shot CoT

### ① 一句话定义

不提供任何示例，只在 prompt 末尾加触发词（最典型："Let's think step by step"），激活模型的逐步推理模式。

### ② 底层原理

"Let's think step by step" 在预训练语料中大量出现于高质量解题过程的开头。模型学到了：**见到这句话 → 接下来应该输出逐步推理**。触发词本质上是一个"分布切换"信号，把模型从"直答模式"拨到"推理模式"。

其他等效触发词：`"Let's work through this carefully."` / `"First, let me break this down."` / `"Step 1:"` （直接开头）

### ③ 最小可运行示例

**不用 CoT（直接回答）**

```text
系统: 你是一个运维助手。

人类: Kubernetes 集群有 3 个节点，每节点 8 核 CPU，单个 Pod 申请 2 核。
      当前已运行 9 个 Pod，新部署需要再启动 5 个 Pod，集群还能调度吗？

模型: 不能调度，需要扩容。   ← 结论对，但推导不透明，出错时无法定位原因
```

**用 Zero-shot CoT**

```text
系统: 你是一个运维助手。

人类: Kubernetes 集群有 3 个节点，每节点 8 核 CPU，单个 Pod 申请 2 核。
      当前已运行 9 个 Pod，新部署需要再启动 5 个 Pod，集群还能调度吗？
      Let's think step by step.

模型:
第一步：计算集群总 CPU 资源
  3 节点 × 8 核 = 24 核

第二步：计算当前已占用 CPU
  9 个 Pod × 2 核 = 18 核

第三步：计算新增 Pod 所需 CPU
  5 个 Pod × 2 核 = 10 核

第四步：判断是否可调度
  剩余: 24 - 18 = 6 核；需要: 10 核；6 < 10

无法调度，需要扩容节点或降低 Pod 的 CPU 申请量。
```

差异：加了触发词后，模型被迫把中间计算写出来，每一步的输入都能"看到"前一步的结果，算错的概率显著下降。

### ④ 适用 vs 不适用场景

**适用（加 CoT 有收益）**

| 场景 | 例子 | 为什么有效 |
|------|------|-----------|
| 多步数学/逻辑推导 | 应用题、费米估算、代码复杂度分析 | 步骤多，中间变量多，必须显式存储 |
| 需要因果链的判断 | "这段代码会不会死锁？为什么" | 需要追踪多个条件的交互关系 |

**不适用（加 CoT 无益甚至变差）**

| 场景 | 例子 | 为什么无效 |
|------|------|-----------|
| 简单事实检索 | "Python 的 list 是有序的吗？" | 单步回答，强迫推理反而引入噪声 |
| 创意/风格类任务 | "帮我改写这段文案，更有活力" | 没有"正确步骤"，逻辑链条对结果无帮助 |

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 在简单问题上加 CoT | 模型绕了一大圈才给出显而易见的答案，成本上升 | 先判断任务是否需要多步推导，不需要就不加 |
| 触发词位置错误 | 放在 system prompt 里但 user 消息没有呼应，效果打折 | 触发词紧跟问题之后效果最稳定 |
| 期望 CoT 解决知识缺失 | 模型推理链完整但基础事实错了 | CoT 只提升推理准确率，不补充训练数据之外的知识 |

**开发者视角一句话总结**：Zero-shot CoT 是零成本的"思维扩展插件"，对多步推理任务直接加在 user 消息末尾，无需示例，立竿见影。

## 模块二：Few-shot CoT

### ① 一句话定义

在 prompt 中提供 2～5 个带有完整推理链的示例（exemplar），用示例的结构"教"模型如何推理，再让它照此处理新问题。

### ② 底层原理

Few-shot CoT 的核心是**in-context learning（情境学习）**：模型在预训练中学会了"看示例 → 识别模式 → 类比推广"的元能力。提供带推理链的示例时，模型不仅学到了答案格式，还学到了推理的粒度、结构和风格。

Zero-shot CoT 只是触发模型"去推理"，但推理的方式由模型自行决定；Few-shot CoT 则**精确示范推理的方式**，效果更可控。

从 token 预测角度：示例的推理链和你的新问题共享同一个 context 窗口，模型在预测新问题的答案时，会"参照"示例推理链的分布来生成自己的推理链。

### ③ 最小可运行示例

```text
系统: 你是一个后端开发助手。对每个问题，先逐步分析，再给出答案。

人类:
【示例1】
问题: 定时任务每批处理 200 条记录，已处理 600 条，数据库共 1400 条，
      还需要运行几批？
推理:
  - 已处理: 600 条
  - 剩余: 1400 - 600 = 800 条
  - 每批: 200 条
  - 需要批次: 800 ÷ 200 = 4 批
答案: 4 批

【示例2】
问题: API 限流 60次/分钟，用户 10 秒内已调用 12 次，按此速率还剩多少秒触达限流？
推理:
  - 调用速率: 12 次 / 10 秒 = 1.2 次/秒 = 72 次/分钟
  - 72 > 60，已超速率；分钟内剩余配额: 60 - 12 = 48 次
  - 触达限流还需时间: 48 ÷ 1.2 = 40 秒
答案: 40 秒后触达限流

【新问题】
问题: Redis 最大内存 512MB，当前已用 380MB，每个缓存条目平均 2KB，
      还能存多少条？

模型:
推理:
  - 剩余空间: 512 - 380 = 132 MB = 135,168 KB
  - 每条: 2 KB
  - 可存: 135,168 ÷ 2 = 67,584 条
答案: 约 67,584 条
```

关键：示例的推理链格式（标签名、缩进、粒度）直接影响模型输出的格式。示例怎么写，模型就怎么推。

### ④ 适用 vs 不适用场景

**适用**

| 场景 | 例子 | 原因 |
|------|------|------|
| 推理风格需要统一 | 生产环境中批量处理同类问题（合同审查、数据分类） | 示例锁定了推理格式，输出一致性高 |
| 领域特定推理逻辑 | 医疗诊断步骤、法律条款适用判断 | 通用 CoT 不了解领域规则，示例可以注入领域知识 |

**不适用**

| 场景 | 例子 | 原因 |
|------|------|------|
| 问题类型多样、难以预先示例覆盖 | 开放式问答客服 | 写不出有代表性的示例，或示例太多超出 context 限制 |
| 示例本身难以构造 | 高度专业的数学证明 | 构造高质量示例比直接解题还难，性价比低 |

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 示例推理链错误 | 模型照抄错误的推理模式，系统性出错 | 人工验证每个示例的推理链，绝不能包含错误步骤 |
| 示例和新问题分布差距大 | 示例是简单问题，新问题复杂，模型推理深度不足 | 示例的复杂度应与目标问题接近 |
| 示例数量过多 | context 被示例占满，新问题信息被压缩或忽略 | 通常 3～5 个示例足够，过多适得其反 |

**开发者视角一句话总结**：Few-shot CoT = 带推理链的 in-context learning，用 2～5 个精心构造的示例"手把手教"模型推理方式，适合需要稳定输出格式的生产场景。

## 模块三：Self-Consistency（自洽性采样）

### ① 一句话定义

对同一问题用 CoT 采样多条推理路径（通常 5～20 次），再对最终答案做多数投票，用"集体智慧"替代"单次运气"。

### ② 底层原理

LLM 的输出受**采样随机性（temperature > 0）**影响：同一个问题，每次推理链都可能不同，但如果答案是确定的，多数路径会收敛到正确答案，少数路径会偏差到不同的错误答案。

```
推理路径 1: A → B → C → 答案: X   ✓
推理路径 2: A → D → E → 答案: X   ✓
推理路径 3: A → B → F → 答案: Y   ✗  （计算失误）
推理路径 4: A → G → H → 答案: X   ✓
推理路径 5: A → D → I → 答案: X   ✓

多数投票: X（4票）> Y（1票）→ 输出 X
```

Self-consistency 不改变单次推理能力，而是通过**统计聚合降低方差**。本质上是用计算量换准确率。

### ③ 最小可运行示例

```python
import anthropic
from collections import Counter

client = anthropic.Anthropic()

QUESTION = """
一台服务器运行了 3 个 Java 服务，每个服务配置 -Xmx512m，
系统总内存 4GB，OS 和基础进程已占用 1.5GB，
问：还能再部署几个相同规格的服务？
"""

PROMPT = f"{QUESTION}\nLet's think step by step."

# 采样 5 次（生产环境通常 10～20 次）
answers = []
for _ in range(5):
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=500,
        temperature=0.7,   # 保留随机性，让路径多样化
        messages=[{"role": "human", "content": PROMPT}]
    )
    raw = response.content[0].text
    # 简单提取最后一行数字（生产中需更健壮的解析）
    last_line = [l for l in raw.strip().split("\n") if l.strip()][-1]
    answers.append(last_line)

# 多数投票
vote = Counter(answers)
final_answer = vote.most_common(1)[0][0]
print(f"投票结果: {vote}")
print(f"最终答案: {final_answer}")
```

**实现要点**：
- `temperature` 设 0.6～0.8，让每次路径有差异；设 0 则每次输出相同，采样无意义
- 答案提取需要结构化（要求模型在固定位置输出 `答案: XXX`），再做聚合

### ④ 适用 vs 不适用场景

**适用**

| 场景 | 例子 | 原因 |
|------|------|------|
| 高准确率要求的单问题 | 关键数学计算、重要决策的理由归因 | 可以接受多次调用成本来换取准确率 |
| 答案空间有限、可以投票 | 分类、是/否判断、数值答案 | 投票机制需要答案可比较 |

**不适用**

| 场景 | 例子 | 原因 |
|------|------|------|
| 开放式生成任务 | 写文章、生成代码 | 答案无法直接投票，路径多样性无法聚合 |
| 延迟敏感的实时场景 | 在线 chat、API 响应 < 1s | 多次采样使延迟线性倍增 |

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| temperature=0 | 每次输出完全相同，投票无意义，浪费 API 调用 | 采样时必须设 temperature > 0 |
| 答案提取不稳定 | 同一答案因格式不同被当作不同选项，票数分散 | 在 prompt 中约束输出格式（如最后一行固定写"最终答案: X"） |
| 采样次数太少 | 3 次采样，1 次错误就占 33%，可靠性低 | 对高精度需求至少 10 次，成本和准确率做 trade-off |

**开发者视角一句话总结**：Self-consistency = 用多次 CoT 采样 + 多数投票替代单次推理，本质是用 API 调用成本换准确率，适合高价值、低频次的关键推理任务。

## 模块四：Tree of Thoughts（思维树，ToT）

### ① 一句话定义

CoT 是一条线性推理链；ToT 是一棵推理树——在每个推理节点，探索多个分支，用启发式评估剪枝，找到通向正确答案的最优路径。

### ② 和 CoT 的关系

```
CoT:   问题 → 步骤1 → 步骤2 → 步骤3 → 答案
                （单条路径，走错了无法回头）

ToT:   问题 → 步骤1a ─→ 步骤2a ─→ 答案a（评估：差）✗
            ↘ 步骤1b ─→ 步骤2b ─→ 答案b（评估：好）✓
            ↘ 步骤1c ─→ 步骤2c ─→ 步骤3c → 答案c（评估：好）✓
```

ToT 的三个核心机制：
1. **思维生成（Thought Generation）**：在每个节点生成 k 个候选下一步
2. **状态评估（State Evaluation）**：用模型（或外部函数）给每个节点打分
3. **搜索策略（Search Strategy）**：BFS（广度优先）或 DFS（深度优先）+ 剪枝

**ToT 和 Self-consistency 的区别**：Self-consistency 是并行采样完整路径后投票；ToT 是在推理过程中就做评估和剪枝，更节省计算但实现更复杂。

### ③ 适用场景（点到为止）

ToT 适合以下特征的问题：
- **路径依赖强**：早期错误会导致后续全部失效（如规划、博弈、程序合成）
- **有局部评估函数**：能在中途判断"这条路走下去有没有希望"
- **搜索空间结构化**：问题可以被分解为明确的决策节点

典型例子：数独求解、24 点游戏、多步代码调试规划。

**不适合 ToT 的场景**：普通问答、文本生成——这类任务没有明确的中途评估标准，ToT 的额外复杂度得不偿失。

### ④ 实现成本提示

ToT 需要编排层（Orchestration）：
- 调用模型生成候选思维 → 调用模型评估 → 选择继续探索的节点 → 循环
- 实现一个基础版本需要至少 50～100 行代码的外部控制逻辑
- 已有开源实现：`tree-of-thought-llm`（GitHub）

**开发者视角一句话总结**：ToT 是 CoT 的"带回溯的升级版"，专为需要探索和剪枝的结构化问题设计，实现复杂度远高于 CoT，普通任务用 CoT 或 Self-consistency 已足够。

## 综合对比：何时选哪种技法

| 技法 | 实现成本 | API 调用次数 | 最适合的场景 |
|------|---------|------------|------------|
| Zero-shot CoT | 极低（加一句话） | 1 次 | 临时任务、快速验证、多步推理 |
| Few-shot CoT | 低（写示例） | 1 次 | 生产环境、需要格式统一的批量任务 |
| Self-consistency | 中（需聚合逻辑） | N 次（5～20） | 高精度要求、答案空间有限 |
| Tree of Thoughts | 高（需编排框架） | 大量 | 复杂规划、搜索、有局部评估的问题 |

**决策树（快速判断）**：

```
任务需要多步推理？
├── 否 → 不用 CoT，直接回答
└── 是 → 有现成的推理示例吗？
          ├── 有 → Few-shot CoT
          └── 没有 → 对准确率要求极高吗？
                    ├── 是 → Self-consistency
                    └── 否 → Zero-shot CoT
                              问题需要探索+回溯吗？
                              └── 是 → Tree of Thoughts（慎用，评估实现成本）
```
