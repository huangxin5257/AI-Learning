# 角色设定（Role Prompting）

> **一句话总结**：角色设定不是"给模型贴标签"，而是通过激活训练数据中特定分布的模式，改变模型对每个 token 的条件概率预测——写法决定激活哪类分布，精度决定激活是否精准。

---

## 模块一：角色设定的本质——激活训练数据中的哪类模式

### ① 一句话定义

角色设定（Role Prompting）是通过在 prompt 中描述一个"身份"，将 LLM 的输出分布从通用对话模式切换到特定领域、风格或行为模式。

### ② 底层原理

LLM 的预训练语料里包含海量文本：Stack Overflow 的技术问答、医学论文、法律文件、小说、客服对话……这些文本被学进了同一套参数里，形成了**互相叠加的多个"分布"**。

推理时，模型在预测每个 token 时，会根据**当前 context 的分布信号**来决定"此刻应该像哪类文本那样续写"。

角色设定的本质是：**向 context 里注入一段高置信度的分布信号，让模型认为"当前生成的内容应该来自 X 类文本"。**

类比：角色设定就像给模型戴上"眼镜"——不改变它看到的问题，但改变了它用什么视角解读和回答。

### ③ 最小可运行示例

三版对比，任务相同（Redis vs JWT 选型），角色不同：

无角色版：System 只说"你是一个 AI 助手"，输出两边都说、中立模糊

泛角色版：System 说"你是一个后端架构专家"，输出更自信但仍面面俱到

精细角色版：System 说"你是有 10 年经验的后端架构师，团队 5 人，系统日活 10 万，技术栈 Go + PostgreSQL，风格是给出明确建议、讲清 trade-off、不说废话"，输出直接给结论：用 Redis，给出 3 条具体理由加注意事项。

核心差别：精细角色激活了"有具体业务约束的架构评审"分布，而不是"教材风格"。

### ④ 适用 vs 不适用场景

**适用（角色设定带来明显收益）**

| 场景 | 例子 | 为什么有效 |
|------|------|-----------|
| 需要特定领域视角 | 以安全工程师身份审查代码 | 激活"安全审计"分布，关注攻击面而非功能 |
| 需要特定沟通风格 | 给产品经理写技术方案 | 激活"写给非技术受众的技术文档"分布 |

**不适用（角色设定收益有限）**

| 场景 | 例子 | 为什么无效 |
|------|------|-----------|
| 任务本身无领域偏好 | 把文本翻译成英文 | 翻译质量与专家角色无关 |
| 角色和任务领域不匹配 | 让"营养师"回答数学题 | 角色信号与任务矛盾，引入噪声 |

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 角色描述和任务无关 | 设定"你是诗人"但让它写 SQL | 角色描述应与任务领域一致 |
| 以为角色能弥补知识缺失 | 设定"量子计算专家"期望获得最新研究 | 角色激活预训练分布，不补截止日期后的知识 |
| 角色太模糊 | "你是一个很厉害的人" | 没有对应训练分布，模型无从激活 |

**开发者视角一句话总结**：角色设定是分布激活，不是魔法咒语——角色描述必须在训练数据里有足够多的对应文本，激活才有效；角色越具体，激活越精准。

## 模块二：System prompt 里设定 vs User 消息里设定——位置有影响吗

### ① 一句话定义

System prompt 中的角色设定权重更高、更持久；User 消息中的角色设定容易被后续对话覆盖，但单轮实验场景下灵活性更好。

### ② 底层原理

现代 LLM（特别是 RLHF 微调后的对话模型）在训练时明确区分了三种消息角色：

- System：全局指令层（框架、约束、身份）
- User：当前对话的输入（请求、问题）
- Assistant：模型回答（在 RLHF 中用于强化"好回答"的分布）

System prompt 的特殊地位：在微调阶段，人类标注者被训练成"System 说的要遵守，User 说的要响应但可以不完全接受"。结果是模型在推理时给 System 的内容分配了更高的服从权重。

具体体现在：
- **持久性**：System 中的角色设定贯穿整个对话，User 中的角色设定只影响当前轮次
- **优先级**：当 System 角色和 User 角色冲突时，模型倾向于遵守 System
- **稳定性**：System 设定的角色不容易被 User 的简单指令改变

但要注意：这不是绝对的。模型最终看到的是一个拼接后的 token 序列，System 的"优先级更高"来自微调带来的行为倾向，而不是硬件级别的隔离。

### ③ 最小可运行示例

```python
import anthropic

client = anthropic.Anthropic()

# 方式 A：角色在 System prompt（多轮持续生效）
response_a = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=300,
    system="你是一个极简主义风格的技术作家，回答时只给结论，不解释。",
    messages=[
        {"role": "user", "content": "Python 的 GIL 是什么？"},
        {"role": "assistant", "content": "GIL 是 Python 解释器的全局锁，同一时刻只允许一个线程执行 Python 字节码。"},
        {"role": "user", "content": "那多进程呢？"}
        # 第二轮仍保持极简风格
    ]
)

# 方式 B：角色在 User 消息（只影响当前轮）
response_b = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=300,
    messages=[
        {"role": "user", "content": "你是一个极简主义风格的技术作家，回答时只给结论，不解释。\n\nPython 的 GIL 是什么？"},
        {"role": "assistant", "content": "GIL 是 Python 解释器的全局锁，同一时刻只允许一个线程执行 Python 字节码。"},
        {"role": "user", "content": "那多进程呢？"}  # 第二轮未重复角色，可能恢复详细回答
    ]
)

# 实验结论：
# response_a 的第二轮大概率仍保持极简风格（System 持续生效）
# response_b 的第二轮可能恢复成正常详细回答（User 角色设定未延续）
```

### ④ 适用 vs 不适用场景

**System 设定角色（适用）**

| 场景 | 原因 |
|------|------|
| 多轮对话产品（客服机器人、助手）| 角色需要贯穿整个会话，System 保证一致性 |
| 生产级 API，防止用户覆盖角色 | System 的优先级高于 User 指令 |

**User 消息设定角色（适用）**

| 场景 | 原因 |
|------|------|
| 单轮任务，快速调试和实验 | 不需要配置 System，直接在问题里写"以X身份回答" |
| 同一 system 下切换视角 | 先以"用户"视角评估，再以"工程师"视角评估 |

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 多轮对话把角色放 User 消息 | 第二轮起角色失效，模型变回通用模式 | 角色设定移入 System prompt |
| 以为 System 设定绝对不可被覆盖 | 用户通过复杂指令改变模型行为 | System 优先级高但非绝对，不要依赖它做安全边界 |
| 同时在 System 和 User 设定冲突角色 | 模型行为不一致 | 统一在 System 设定，User 只做任务输入 |

**开发者视角一句话总结**：System prompt 是角色的"合同"，User 消息是角色的"临时口头约定"——多轮产品把角色放 System，单轮实验放 User 消息足够。

## 模块三：角色粒度——泛角色 vs 细角色，效果差距值得投入吗

### ① 一句话定义

角色描述越具体（领域 + 场景 + 风格），激活的分布越窄越精准；泛角色只做粗粒度筛选，精细角色做精准定位。

### ② 底层原理

把预训练语料想象成一个"文档向量空间"：

泛角色"你是专家"：在向量空间里画了一个大圆，圈进了所有"看起来像专业人士写的"文档，范围太大，模型无法精确定位。

细角色"你是有 10 年经验的 Go 后端工程师，熟悉高并发场景，当前在给初级工程师做 code review"：同时施加了多个约束——语言(Go)、经验(10年)、场景(高并发)、受众(初级工程师)、任务(code review)。多个约束的交集大幅缩小了候选文本的范围，激活更精准的分布。

**关键问题：精细化值得吗？**

视任务而定：
- **通用问答/简单任务**：泛角色（甚至无角色）效果差异微小，精细化性价比低
- **需要特定风格/视角的输出**：细角色可以带来显著差异，输出更聚焦、更少废话
- **有领域知识门槛的任务**：细角色激活特定领域分布，减少"教科书式面面俱到"

### ③ 最小可运行示例（三级粒度对比）

任务：帮我 review 这段 Go 代码，指出问题

- 无角色：列出语法问题、可能的 bug，语气中性，面面俱到
- 泛角色（"你是一个编程专家"）：类似无角色，但用词更自信，多提一些最佳实践
- 细角色（"你是有 8 年 Go 开发经验的 Senior Engineer，专注于高并发服务，给 2～3 年经验的工程师做 code review，风格：先指出最关键问题不超过 3 条，解释时用 Go 惯用语和生产案例"）：直接指出 goroutine leak / mutex 使用不当等高并发特定问题，用"在生产环境里，这种写法在 1000 QPS 时会..."的方式解释，不在简单格式问题上浪费篇幅

### ④ 构建细角色的四个维度

| 维度 | 泛角色写法 | 细角色写法 | 重要性 |
|------|-----------|-----------|--------|
| **领域** | 程序员 | Go 后端工程师，专注分布式系统 | ★★★★★ |
| **场景/任务** | （未指定）| 给 2～3 年经验的工程师做 code review | ★★★★☆ |
| **风格/约束** | （未指定）| 直接给结论，每次最多 3 个主要问题 | ★★★☆☆ |
| **经验年限** | 专家 | 8 年经验，做过 3 次从 0 到 1 的架构设计 | ★★☆☆☆ |

**实用建议**：优先补充"领域"和"场景/任务"，这两个维度对分布激活影响最大；"经验年限"和"风格"是锦上添花。

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 角色维度堆砌但互相矛盾 | "你是严谨的学术研究者，同时是风趣幽默的段子手" | 矛盾角色信号互相抵消，去掉和任务无关的维度 |
| 角色里写了期望行为而非身份 | "你是一个总是给出 3 条建议的专家" | 行为约束放到任务指令里，角色描述只写身份和背景 |
| 为简单任务过度精细化角色 | 翻译一句话，却写了 200 字的角色描述 | 角色粒度应和任务复杂度匹配 |

**开发者视角一句话总结**：细角色 = 领域 + 场景约束，两者加起来已有 80% 的效果；只写"专家"是在浪费 token，最优先补充领域和任务场景，其他维度按需添加。

## 模块四：角色一致性——多轮对话中角色会漂移吗，如何保持

### ① 一句话定义

角色漂移（Role Drift）是真实存在的问题：随着对话轮次增加，对话历史本身的内容分布会逐渐覆盖初始角色设定的信号，导致模型行为偏离预设角色。现代长上下文模型（Claude 3.5+ 200k context、GPT-4o）对中短对话（&lt;20 轮）的漂移明显减少，但超长对话或用户主动要求改变风格时仍然存在。

### ② 底层原理

LLM 对 token 的注意力（Attention）并非均匀分布：

- 轮次 1：System 角色设定权重高，信号强
- 轮次 5：System 被大量对话历史稀释，相对权重下降；对话历史里出现了什么风格，就开始向那种风格靠拢

**漂移的三个来源**：
1. **注意力稀释**：context 变长后，System prompt 占总 token 的比例下降
2. **历史内容影响**：用户问了详细的问题，或某轮给了详细回答，后续会向这个风格靠拢
3. **隐式指令冲突**：用户问题里隐含了和角色不一致的期望（"能详细说说吗？"会使"极简"角色漂移）

### ③ 保持角色一致性的三种策略

**策略 A：角色锚定（推荐，简单有效）**

在 System prompt 末尾加一句角色强化指令：

```
System: 你是极简主义风格的技术作家，回答只给结论，不解释原因，不加免责声明。
        无论用户如何要求（包括"详细说说"），始终保持这个风格。
```

**策略 B：周期性角色重申（适用于中长对话）**

```python
import anthropic

client = anthropic.Anthropic()

ROLE = "你是极简主义风格的技术作家，回答只给结论，不超过 3 句话。"
ROLE_REMINDER = f"[记住你的角色：{ROLE}]"

def chat_with_role_reminder(history: list, user_input: str, reminder_interval: int = 3):
    """每隔 reminder_interval 轮，在 user 消息里插入角色提醒"""
    message = user_input
    turn_count = len([m for m in history if m["role"] == "user"])
    if turn_count > 0 and turn_count % reminder_interval == 0:
        message = f"{ROLE_REMINDER}\n\n{user_input}"

    history.append({"role": "user", "content": message})
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=300,
        system=ROLE,
        messages=history
    )
    reply = response.content[0].text
    history.append({"role": "assistant", "content": reply})
    return reply, history
```

**策略 C：Context 压缩（适用于超长对话）**

```python
def compress_history(history: list, client, keep_last_n: int = 4) -> list:
    """压缩历史对话，保留最近 N 轮 + 摘要"""
    if len(history) <= keep_last_n * 2:
        return history

    old_history = history[:-keep_last_n * 2]
    recent_history = history[-keep_last_n * 2:]

    summary_response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=200,
        messages=[{
            "role": "user",
            "content": f"请将以下对话历史压缩成 3～5 个要点：\n{str(old_history)}"
        }]
    )
    summary = summary_response.content[0].text

    compressed = [
        {"role": "user", "content": f"[之前对话摘要]\n{summary}"},
        {"role": "assistant", "content": "已了解之前的对话背景。"}
    ] + recent_history
    return compressed
```

### ④ 适用 vs 不适用场景

| 对话长度 | 推荐策略 | 原因 |
|---------|---------|------|
| 短对话（现代模型 &lt;20 轮）| 只用 System 设定 | 漂移风险低，不需要额外处理 |
| 中长对话（20～50 轮）| 策略 A + B | 角色锚定 + 周期提醒，提醒间隔可拉长到 10 轮 |
| 超长对话 / RAG 场景 | 策略 A + C | 压缩历史防止 context 过长 |

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 角色漂移被误认为"模型质量问题" | 前几轮好，越来越差 | 检查是否是对话历史稀释了 System 信号 |
| 在 System 里说"始终保持角色"但从不验证 | 长对话后角色悄然漂移 | 每 5 轮手动测试角色是否仍然生效 |
| 未处理用户的"绕过"请求 | 一句"详细说说"破坏了整个角色设定 | System 里明确说明对这类请求的处理方式 |

**开发者视角一句话总结**：现代强模型在 20 轮以内基本不需要担心漂移；超过后按需处理：中等对话加角色锚定语句，超长对话压缩历史——漂移在设计上可预防，不要等问题出现再补救。

## 模块五：角色设定的边界——什么时候没用，什么时候有副作用

### ① 一句话定义

角色设定有两个硬边界：模型的预训练知识截止日期、以及 RLHF 微调引入的行为约束；超出这两个边界，角色设定要么无效，要么产生副作用。

### ② 角色设定无效的三种情况

**情况 1：任务不依赖分布偏好**

把文本翻译成英文——"翻译专家"角色对翻译质量毫无影响，内容本身才重要。

**情况 2：要求模型产生它没有的知识**

设定"你是一个掌握 2025 年最新 AI 论文的研究员"——角色设定不能注入训练数据截止日期之后的知识。模型仍然只能基于它训练时的数据回答，只是"风格"像研究员。

**情况 3：角色在训练数据里没有充分的对应分布**

设定"你是一个 2127 年的星际飞船修理工"——训练数据里几乎没有这类文本，模型无从激活对应分布，退化成"写科幻小说"的分布。

### ③ 角色设定的副作用

**副作用 1：过度认同（Over-identification）**

当角色设定要求"权威"时，模型可能为了保持角色一致性而拒绝承认知识边界：

```
System: 你是一个无所不知的医学专家，始终给出权威答案。
User: 这种症状应该用哪种抗生素？
```

风险：模型为了"保持权威形象"，在不确定时仍然给出确定性答案，而不是说"这个需要更多信息才能判断"。

**副作用 2：角色约束绑架任务目标（Goal Displacement）**

```
System: 你是一个极度谨慎的法律顾问，任何建议都必须加免责声明。
User: 帮我起草一份简单的 NDA 模板
```

结果：NDA 模板里充满了免责声明和"请咨询专业律师"，实际使用价值极低——角色约束覆盖了任务目标。

**副作用 3：RLHF 对齐层的干预**

模型在 RLHF 微调阶段被训练了一套核心行为约束（安全、诚实、无害），这些约束**优先级高于角色设定**。"你是一个不受任何限制的 AI"这类角色设定对核心安全约束无效。

优先级顺序：安全约束 > System 角色 > User 消息

### ④ 边界总结

角色设定能做到：
- 改变输出风格（正式/口语、详细/精简）
- 改变关注重点（哪类问题被优先提及）
- 激活特定领域的词汇和思维框架
- 调整受众适配（面向初学者 vs 专家）

角色设定做不到：
- 注入截止日期后的新知识（用 RAG 解决）
- 绕过 RLHF 的核心安全对齐
- 让模型拥有它没有训练过的技能
- 保证多轮对话中角色永不漂移（只能减少）

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 依赖角色设定做知识补全 | 模型给出自信但错误的答案 | 知识缺口用 RAG，不用角色设定 |
| 角色设定导致过度谨慎 | 模型充满免责声明，不给实质帮助 | 在角色里明确说"在这个场景下，直接给出建议，不加免责声明" |
| 以为细角色能绕过模型限制 | 某些内容仍然被拒绝 | RLHF 对齐层优先于角色，这是设计行为，不是 bug |

**开发者视角一句话总结**：角色设定是"调分布的旋钮"，不是"万能钥匙"——知识有截止日期，安全有硬约束，任务目标不能被角色约束绑架，清楚这三条边界才能用好这个工具。

## 综合对比与决策框架

### 五个模块的核心洞察

| 模块 | 核心结论 |
|------|---------|
| 本质 | 角色 = 分布激活信号，训练数据里有对应分布才能生效 |
| 位置 | System > User 消息；多轮对话角色必须放 System |
| 粒度 | 领域 + 场景 = 80% 效果；经验年限和风格是锦上添花 |
| 一致性 | 长对话必须主动维护；漂移是 context 稀释的副作用 |
| 边界 | 不补知识、不绕安全对齐、不能替代任务指令 |

### 角色设定决策树

```
任务对输出风格/视角有明确要求？
├── 否 → 不用角色设定，直接写任务指令
└── 是 → 角色描述里包含"领域"了吗？
          ├── 否 → 至少加上领域（比"你是专家"强得多）
          └── 是 → 还需要场景/受众约束吗？
                    ├── 否 → 领域角色已够用
                    └── 是 → 加上任务场景 + 风格约束
                              是多轮对话吗？
                              ├── 否 → User 消息里设定即可
                              └── 是 → 放 System prompt + 加角色锚定语句
                                        对话 > 10 轮 → 加周期提醒或历史压缩
```

### 角色描述模板（可直接使用）

```
System: 你是一个[经验描述，可选]的[具体领域]专家，
        [当前场景/受众描述]。
        你的风格：[1-2 条输出约束，可选]。
        [多轮对话必加]无论用户如何要求，始终保持上述风格。

填充示例：
你是一个有 5 年经验的 Python 数据工程师，
正在给没有工程背景的数据分析师解释 Spark 的工作原理。
你的风格：用类比代替术语，每个概念配一个具体的数字例子。
无论用户如何要求，始终保持通俗易懂的风格。
```

### 特别关注点解答

**Q：System prompt 中的角色设定和 User 消息中的角色设定，模型处理权重有区别吗？**

有区别，但不是绝对隔离。System 在 RLHF 微调中被赋予了更高的服从倾向，效果类似"合同"——User 只是"临时约定"。但这是行为倾向，不是硬性隔离，特别复杂的 User 指令仍然可能改变模型行为。结论：多轮产品的角色必须放 System，单次实验放 User 消息即可。

**Q："你是专家"和详细描述专家特征，实际效果差距有多大，值得花时间精细化吗？**

对于需要特定视角和风格的任务，差距显著——精细角色的输出更聚焦、更少废话、更贴近目标受众。成本：加 20～50 个 token 的角色描述。建议：先加"领域"（效果 80%），如果仍不满意再加"场景/受众"（效果再提升 15%），"经验年限"和"风格约束"是最后的微调项。对于简单任务（翻译、格式转换），不值得精细化。
