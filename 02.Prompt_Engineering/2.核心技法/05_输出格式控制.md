# 输出格式控制

> **一句话总结**：格式控制的本质是在"概率分布"上施加偏置——通过 prompt 写法、few-shot 示范和 API 级约束，把模型的输出分布从"可能合法"推向"几乎必然合法"。

---

## 模块一：格式控制的本质——指令遵循 vs 模式匹配

### ① 一句话定义

模型遵循格式指令，不是执行规则，而是在"这段上下文之后，什么 token 概率最高"的问题上做出选择——指令遵循（instruction following）和模式匹配（pattern matching）同时在起作用。

### ② 底层原理

LLM 没有"规则解析器"，每次都在做同一件事：**预测下一个 token 的概率分布**。

格式遵循有两条路径并行运作：

```
路径 A：指令遵循（Instruction Following）
  "请输出 JSON" → 模型学过"这类指令后该输出什么样的 token 序列"
  → 训练时的 RLHF / 指令微调让这条路径变强

路径 B：模式匹配（Pattern Matching）
  上下文里出现了 {" → 模型"见过"无数 JSON，接下来应该是什么有强先验
  → 预训练语料里的分布直接生效
```

**关键洞察：两条路径会相互增强，也会相互冲突。**

- 写了 `输出 JSON` + 提供了 JSON 示例 → 双路加持，最稳定
- 只写 `输出 JSON`，system prompt 里大量自然语言段落 → 路径 B 在拉向自然语言，格式漂移
- 写了 `不要输出 Markdown`，但对话历史全是 Markdown 回答 → 路径 B 的历史分布对抗路径 A 的指令

**为什么格式遵循是概率性而非确定性的：**

模型没有 `if format == "json": enforce_json()` 这样的执行分支。格式遵循度 = 当前上下文下输出合法格式 token 序列的概率。temperature、采样策略、上下文长度、指令位置都会改变这个概率。

### ③ 最小可运行示例

**不控制格式**

```
[System]
你是一个信息提取助手。

[User]
从以下文本中提取姓名和职位：
"张伟是公司的首席技术官，李娜担任产品总监。"

[Assistant]
文中提到了两个人：张伟担任首席技术官，李娜担任产品总监。
← 自然语言段落，下游代码无法直接 parse
```

**控制格式（指令 + 模式双激活）**

```
[System]
你是一个信息提取助手。只输出 JSON 数组，不要任何额外文字。

[User]
从以下文本中提取姓名和职位，格式如下示例：
示例输出：[{"name": "某人", "title": "某职位"}]

现在处理：
"张伟是公司的首席技术官，李娜担任产品总监。"

[Assistant]
[{"name": "张伟", "title": "首席技术官"}, {"name": "李娜", "title": "产品总监"}]
← 合法 JSON，直接 json.loads() 可用
```

关键改动：① system prompt 明确格式 + 禁止多余文字；② user prompt 提供了格式示例（激活路径 B）。

### ④ 适用 vs 不适用场景

**适用：格式控制有明显收益**

| 场景 | 例子 | 为什么有效 |
|------|------|-----------|
| 下游有 parse 代码 | API 返回数据给前端渲染 | 格式错误直接导致程序崩溃，收益高 |
| 批量处理需一致性 | 从 1000 条评论中提取情绪标签 | 格式不一致会污染整个数据集 |

**不适用：格式约束反而有害**

| 场景 | 例子 | 为什么不适合 |
|------|------|------------|
| 创意生成 | 写一首诗、头脑风暴 | 强格式压缩输出空间，限制创意涌现 |
| 复杂推理 | 多步分析、诊断报告 | 过度格式化导致模型为"填表"而非"思考" |

### ⑤ 常见误用和排查

**误用：把格式指令放到 user prompt 末尾**

```
[User]
帮我分析这段代码的问题。请输出 JSON。  ← 格式指令在最后，权重低
```

排查方法：把核心格式约束移到 system prompt，并在 user prompt 开头重申。

**误用：指令模糊（"结构化输出"≠"JSON"）**

```
[System]
请以结构化方式回答。  ← 模型可能输出 Markdown 表格、编号列表或 JSON
```

排查方法：永远指定具体格式名称 + 示例，不用模糊词。

> **开发者视角一句话总结**：格式控制不是"告诉模型规则"，而是同时调高两条路径的信号强度——指令说清楚，示例做示范，system prompt 是主战场。

## 模块二：JSON 输出——prompt 最佳实践与 tool use 的选型

### ① 一句话定义

现代强模型（Claude 3.5+、GPT-4o）在正确 prompt 下输出合法 JSON 的可靠性已大幅提升；生产环境的确定性需求应优先使用 tool use / structured output API，prompt 层是快速验证和简单场景的方案。

### ② 底层原理

JSON 输出在较弱模型（GPT-3 系、早期 Claude）上存在明显的"两种竞争压力"问题：

```
压力 A：格式正确性
  → 需要精确匹配括号、引号、逗号等 token 序列
  → JSON 语法本质上是一种"低熵"分布，任何偏差都是 invalid

压力 B：内容表达欲
  → 模型被训练成"有帮助"，想解释、想加注释、想提示用户
  → 在 JSON 之前加 "好的，这是结果："，在之后加 "希望这对你有帮助"
  → 内容长时想截断但括号没闭合 → invalid JSON
```

**三类典型失效模式**（现代强模型概率明显降低，但 prompt 不当时仍会出现）：

1. **前缀污染**：`好的，以下是 JSON：\n{"key": ...}` → json.loads() 直接报错
2. **截断**：内容过长，模型在 max_tokens 耗尽前没闭合括号 → 不完整 JSON
3. **格式崩溃**：嵌套超过 3 层或字段超过 10 个时，模型混入自然语言

### ③ 最小可运行示例

**不稳定写法（三个常见错误都犯了）**

```
[System]
你是一个数据提取助手，请用 JSON 回答。

[User]
提取以下用户反馈的情绪和关键词：
"这个产品真的太好用了，界面简洁，速度很快，就是价格有点贵。"
```

可能输出：
```
当然，以下是分析结果：

{
  "sentiment": "positive",
  "keywords": ["好用", "界面简洁", "速度快", "价格贵"]
}

希望这个分析对你有帮助！
```

这段输出无法直接 `json.loads()`，需要额外的字符串处理。

**稳定写法（prompt 层）**

```
[System]
你是数据提取 API。
规则：
1. 只输出 JSON 对象，不输出任何其他文字
2. 不要在 JSON 前后加任何解释、问候或注释
3. 严格遵守以下 schema：
   {"sentiment": "positive|negative|neutral", "keywords": ["string"]}

[User]
提取情绪和关键词：
"这个产品真的太好用了，界面简洁，速度很快，就是价格有点贵。"

[Assistant（期望输出）]
{"sentiment": "positive", "keywords": ["好用", "界面简洁", "速度快", "价格贵"]}
```

**更稳定写法（API 层，使用 response_format）**

```python
import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system="""你是数据提取 API。只输出 JSON 对象，不输出任何其他文字。
Schema: {"sentiment": "positive|negative|neutral", "keywords": ["string"]}""",
    messages=[{
        "role": "user",
        "content": '提取情绪和关键词："这个产品真的太好用了，界面简洁，速度很快，就是价格有点贵。"'
    }]
)

import json
result = json.loads(response.content[0].text)
```

注：OpenAI 有 `response_format={"type": "json_object"}` / Structured Outputs；Claude 通过 tool use + `tool_choice` 强制调用同样实现确定性输出（见模块四的代码示例），两者均已在标准 API 可用，不需要 Bedrock/Vertex 特殊版本。

### ④ 适用 vs 不适用场景

**适用：JSON 输出收益高**

| 场景 | 例子 | 说明 |
|------|------|------|
| 结构化信息提取 | 从简历中提取姓名、技能、工作经历 | 字段明确，schema 固定，parse 代码简单 |
| 分类 + 标注流水线 | 批量给文章打标签并返回置信度 | 批量处理必须格式一致 |

**不适用：JSON 会增加复杂度**

| 场景 | 例子 | 说明 |
|------|------|------|
| 自由文本生成 | 写营销文案、总结报告 | 没有下游 parse 需求，强加 JSON 只增加约束 |
| 长内容输出 | 生成 2000 字分析文章 | 长内容放进 JSON 字符串反而容易截断和转义错误 |

### ⑤ 常见误用和系统性排查路径

**JSON 不稳定排查路径（从内到外）：**

```
Step 1：检查 prompt 写法
  □ system prompt 有无明确说"只输出 JSON，不要额外文字"？
  □ 有无提供 schema 示例？
  □ 指令是否在 system prompt 开头（而非埋在中间）？

Step 2：检查 schema 复杂度
  □ 嵌套层级是否超过 3 层？→ 拆分成多次调用
  □ 字段数量是否超过 10 个？→ 考虑分批提取

Step 3：检查 API 参数
  □ temperature 是否 > 0.3？→ JSON 提取任务建议 temperature=0
  □ max_tokens 是否足够容纳完整 JSON？→ 估算 token 数，留 20% buffer
  □ 是否可以用 tool use / function calling 代替？

Step 4：检查输出处理
  □ 是否有 json.loads() 的 try/except + 重试逻辑？
  □ 对于前缀污染，可以用正则 r'\{.*\}' 或 r'\[.*\]' 先提取
```

> **开发者视角一句话总结**：现代强模型 + 正确 prompt 可以稳定输出 JSON；生产环境若需要确定性保证，直接用 tool use，不要在 prompt 层打补丁——排查顺序：先换 tool use，若必须用 prompt，再查 system prompt → schema 复杂度 → API 参数 → 兜底重试。

## 模块三：Markdown 控制——什么时候该用，什么时候该禁用

### ① 一句话定义

Markdown 是模型的"默认输出习惯"——它在预训练时见过大量 Markdown，遇到"解释/列举/总结"类任务会自动倾向于输出标题、加粗、列表；你需要主动控制"开"或"关"。

### ② 底层原理

模型的 Markdown 偏好来自预训练语料分布：

```
技术文档、README、Stack Overflow 回答、教程博客
→ 大量 Markdown 格式
→ 模型学到：解释类任务 → 用 ## 标题 / **加粗** / - 列表

聊天记录、小说、新闻
→ 纯文本
→ 模型学到：对话类任务 → 用自然段落
```

当任务类型介于两者之间（例如"解释一个概念"），模型会根据当前上下文的格式信号做判断。如果 system prompt 本身是纯文本，模型更可能输出纯文本；如果 system prompt 里有 Markdown，模型会跟着用 Markdown。

**Markdown 失控的两个方向：**

- **不该用时用了**：API 返回的文字被直接渲染到移动 App，`**加粗**` 变成字面 `**` 符号
- **该用时没用**：在 Web 聊天界面输出长篇纯文本，可读性差

### ③ 最小可运行示例

**场景：API 返回内容给移动 App**

**有问题（输出了 Markdown 符号）**

```
[System]
你是客服助手，回答用户的产品问题。

[User]
这款耳机支持主动降噪吗？

[Assistant]
**是的**，这款耳机支持**主动降噪（ANC）**功能，具体特性如下：
- 三档降噪强度可调
- 续航约 **30 小时**（开启降噪）
← App 直接展示文字，用户看到的是 "**是的**，这款..." 带星号
```

**正确控制（禁用 Markdown）**

```
[System]
你是客服助手，回答用户的产品问题。
重要：只输出纯文本，不使用任何 Markdown 语法（不用 **加粗**、不用 # 标题、不用 - 列表）。
用自然段落和顿号来组织内容。

[User]
这款耳机支持主动降噪吗？

[Assistant]
是的，这款耳机支持主动降噪（ANC）功能，提供三档降噪强度可调，开启降噪后续航约 30 小时。
← 干净的纯文本，App 可直接展示
```

### ④ 适用 vs 不适用场景

**应该启用 Markdown（默认行为，无需额外控制）**

| 场景 | 例子 | 说明 |
|------|------|------|
| Web 聊天界面 | ChatGPT 式对话产品 | 前端有 Markdown 渲染器，标题/列表提升可读性 |
| 文档/报告生成 | 技术方案、周报 | 用户期望有结构，Markdown 直接复制到编辑器 |

**应该禁用 Markdown**

| 场景 | 例子 | 说明 |
|------|------|------|
| 移动 App / 原生 UI | 客服机器人、语音合成 | 没有 Markdown 渲染器，符号会原样显示 |
| 下游文本处理 | 情感分析、内容审核 | 多余符号干扰 NLP 处理 |

### ⑤ 常见误用和排查

**误用：用"不要用 Markdown"后模型仍然输出**

根本原因：指令太弱（模式匹配路径 B 的惯性大于路径 A 的指令）。

排查 + 修复方法：

```
方法 1：在 system prompt 开头，不是末尾
  "你是客服助手。重要规则：只输出纯文本。[以下才是任务描述]"

方法 2：把禁用指令拆细，逐一列举
  "不使用 ** 加粗、不使用 # 标题、不使用 - 或 * 列表、不使用 > 引用块"

方法 3：用 few-shot 示范纯文本风格
  [User] 问题1
  [Assistant] 纯文本回答示例（不含任何 Markdown 符号）
```

> **开发者视角一句话总结**：Markdown 是模型的默认"礼貌输出"，在不渲染的场景必须主动关掉——指令要细化到具体语法（**、#、-），越具体越有效。

## 模块四：结构化 schema 约束——function calling / structured output vs prompt 控制

### ① 一句话定义

Prompt 控制是概率性的（"模型尽力遵守"），function calling / structured output 是确定性的（"API 强制合法格式"）——两者解决同一个问题，但在可靠性和灵活性上存在本质差异。

### ② 底层原理

**Prompt 控制的工作方式：**

```
Prompt 指令 → 调高"输出合法 JSON"的概率
→ 但模型仍可能输出非法格式（概率低但不为零）
→ 你的代码需要 try/except + 重试逻辑
```

**Function calling / Structured Output 的工作方式：**

```
API 接收到 tools/schema 定义
→ 解码阶段强制 token 采样只从"合法下一 token"中选取
  （通过 constrained decoding / grammar-based sampling 实现）
→ 输出一定是合法结构（除非 max_tokens 截断）
→ 你的代码可以直接 parse，无需重试
```

这是**解码层**的约束，不是模型层面的理解——即使模型"不知道"正确答案，它也无法输出语法非法的 JSON。

**三种方案对比：**

| 方案 | 可靠性 | 灵活性 | 适用模型 | 典型用途 |
|------|--------|--------|---------|---------|
| Prompt 控制 | 中（95%+） | 高 | 所有模型 | 快速原型、简单结构 |
| Function calling | 高（99%+） | 中（需定义 schema） | GPT-4/Claude w/ tools | 工具调用、agent |
| Structured output | 极高（近 100%） | 中 | OpenAI + 部分 Claude API | 批量提取、数据管道 |

### ③ 最小可运行示例

**方案 A：Prompt 控制（灵活但有概率失败）**

```
[System]
只输出 JSON，schema：{"name": string, "age": number, "city": string}
不输出任何额外文字。

[User]
提取：张伟，32岁，来自北京。
```

**方案 B：Function calling（Claude tool use 实现强制 schema）**

```python
import anthropic, json

client = anthropic.Anthropic()

tools = [{
    "name": "extract_person",
    "description": "从文本中提取人物信息",
    "input_schema": {
        "type": "object",
        "properties": {
            "name":  {"type": "string", "description": "姓名"},
            "age":   {"type": "integer", "description": "年龄"},
            "city":  {"type": "string", "description": "城市"}
        },
        "required": ["name", "age", "city"]
    }
}]

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=256,
    tools=tools,
    tool_choice={"type": "tool", "name": "extract_person"},  # 强制调用此工具
    messages=[{
        "role": "user",
        "content": "提取：张伟，32岁，来自北京。"
    }]
)

# 输出一定是合法的 tool_use block
result = response.content[0].input
# {"name": "张伟", "age": 32, "city": "北京"}
```

`tool_choice={"type": "tool", "name": "..."}` 强制模型调用指定工具，输出通过 constrained decoding 保证合法。

### ④ 适用 vs 不适用场景

**使用 function calling / structured output**

| 场景 | 原因 |
|------|------|
| 生产环境数据管道 | 格式错误会导致下游系统崩溃，需要极高可靠性 |
| 复杂嵌套 schema | Prompt 控制在嵌套 > 3 层时成功率明显下降 |

**使用 Prompt 控制**

| 场景 | 原因 |
|------|------|
| 快速原型验证 | 无需定义 schema，速度快 |
| schema 在运行时动态变化 | 动态 schema 不适合预定义 tools |

### ⑤ 常见误用和排查

**误用：schema 过于复杂导致 tool use 也失败**

tool use 通过 constrained decoding 保证语法合法，但无法保证**语义正确**（字段值是否合理）。如果 schema 字段超过 20 个，模型会开始乱填。

排查方法：把大 schema 拆成多次 tool call，每次专注 3-5 个字段。

**误用：在创意/对话场景强用 tool use**

```python
# 错误：用 tool use 来"控制"对话回复的风格
tool_choice={"type": "tool", "name": "respond_to_user"}
# 结果：模型在 tool call 里生成回复，但没了自然的对话感
```

排查方法：tool use 只用于"提取信息/调用功能"，不用于"生成内容"。

> **开发者视角一句话总结**：Prompt 控制是"拜托模型遵守"，tool use 是"API 强制执行"——生产环境的数据提取管道必须用 tool use，原型阶段 prompt 控制够用。

## 模块五：输出长度控制——为什么"不超过 100 字"经常失效

### ① 一句话定义

长度指令失效是因为模型的"结束"决策依赖的是内容完整感而非字数计数——正确做法是从结构上限制内容，而不是给模型贴一个"字数表"。

### ② 底层原理

模型不会在生成过程中计数 token 或汉字。"不超过 100 字"这个指令的工作方式是：

```
训练中：见过"请简短回答"→短回答的配对数据
→ 模型学到：这类指令后应该输出更短的内容
→ 但"更短"是相对的，不是精确的 100 字
```

**两个核心问题：**

1. **模型不计数**：生成时没有一个内部计数器在倒计时，"100 字"只是一个模糊的"短"的信号
2. **内容完整感 > 长度约束**：当任务本身感觉需要 300 字才能"说清楚"，模型会优先保证内容完整，长度约束被覆盖

**长度由什么决定：**

```
主要因素（按影响权重）：
1. 任务复杂度感知    ← 最重要，模型认为这个问题需要多少信息
2. Few-shot 示例长度 ← 如果示例都是长回答，模型会倾向于长回答
3. 长度指令          ← 有效，但精确度低
4. max_tokens        ← API 层硬截断，不推荐靠它控制长度
```

### ③ 最小可运行示例

**失效写法**

```
[System]
你是一个助手，回答问题时不超过 100 字。

[User]
解释一下什么是 Transformer 架构？

[Assistant]
Transformer 是一种深度学习架构，主要用于自然语言处理任务。它于 2017 年由 Google 在论文"Attention is All You Need"中提出...
（继续输出 300+ 字）
← 任务复杂度感知压倒了长度指令
```

**有效写法（从结构上限制）**

```
[System]
你是一个技术概念速查助手。
每个回答必须严格遵守以下结构，不多不少：
- 一句话定义（1句）
- 核心组件（最多3个，每个不超过10字）
- 典型应用（1个例子）

[User]
解释一下什么是 Transformer 架构？

[Assistant]
一句话定义：用注意力机制替代 RNN，并行处理序列的神经网络架构。
核心组件：自注意力层、前馈网络、位置编码
典型应用：GPT、BERT 等大语言模型的基础架构
← 结构本身限制了长度，不需要字数约束
```

**Few-shot 校准法（最可靠）**

```
[System]
你是技术概念速查助手。

[User]
解释：神经网络

[Assistant]
由互联连接的节点（神经元）组成的计算模型，模拟大脑处理信息的方式。核心：前向传播 + 反向传播更新权重。应用：图像识别、语音识别。

[User]
解释：Transformer 架构

[Assistant]
← 模型会参照示例长度生成，比字数指令更准确
```

### ④ 适用 vs 不适用场景

**长度控制有效的场景**

| 场景 | 推荐做法 | 说明 |
|------|---------|------|
| 固定格式表单填写 | 指定每个字段的字数上限 | 结构化字段容易控制 |
| 标题/摘要生成 | 提供 few-shot 示例 | 示例长度直接校准输出 |

**长度控制容易失效的场景**

| 场景 | 原因 | 替代方案 |
|------|------|---------|
| 复杂技术解释 | 任务复杂度感知高，长度约束被覆盖 | 从结构限制，而非字数限制 |
| 开放式问题 | 没有自然的"完成感"边界 | 限制维度数量，不限字数 |

### ⑤ 常见误用和排查

**误用：用 max_tokens 当长度控制工具**

```python
response = client.messages.create(
    max_tokens=50,  # 错误：想控制长度，所以把 max_tokens 设很低
    ...
)
# 结果：模型在句子中间被截断，输出不完整内容
```

`max_tokens` 是硬截断，不是"生成到这个长度后自然结束"。用它控制长度会产生半截回答。

排查方法：

```
如果要控制长度：
  → 首选：从结构约束（限制条目数量、字段数量）
  → 次选：提供目标长度的 few-shot 示例
  → 最后：在 prompt 里明确说"不超过 X 字，超出时删减次要信息"
    （加上"如何处理超长"的指令，比只说"不超过"有效）

如果输出被截断（怀疑是 max_tokens）：
  → 检查 stop_reason 是否为 "max_tokens"（而非 "end_turn"）
  → 增大 max_tokens，而不是减小
```

**误用：长度指令和内容要求冲突**

```
[System]
请用 50 字以内解释机器学习、深度学习和强化学习三者的区别。
← 三个概念对比 50 字根本说不清楚
```

排查方法：在写长度约束前，先估算任务本身需要多少信息量。如果任务需要 300 字，50 字的约束会让模型在"完整性"和"简洁性"之间挣扎，输出质量下降。

> **开发者视角一句话总结**：字数指令是暗示，不是命令——用结构约束（限制条目数/字段数）替代字数约束，用 few-shot 示例校准长度，是最可靠的两种方法。

## 附录：强制格式 vs 内容质量的张力

> **核心问题**：格式约束太死，会不会损失内容质量？

**答案：会——在特定条件下。**

以下是格式约束损害内容的三种机制：

**1. 模型"填表"而非"思考"**

当 schema 过于具体（例如强制要求每个分析结论必须包含 confidence_score 字段），模型会优先想"这个字段填什么合理"，而不是"这个问题的本质是什么"。

**2. 格式压缩推理空间**

要求输出 JSON 时，模型倾向于直接输出结果，跳过中间推理。CoT（让模型先推理再输出）和严格 JSON 格式天然冲突。

解决方案：把推理和输出分开
```python
# 第一步：让模型自由推理
reasoning_response = client.messages.create(
    messages=[{"role": "user", "content": f"请分析以下内容并给出你的推理：{content}"}]
)

# 第二步：基于推理结果，提取结构化输出
extraction_response = client.messages.create(
    tools=tools,
    messages=[
        {"role": "user", "content": f"基于以下分析，提取结构化数据：{reasoning_response.content[0].text}"}
    ]
)
```

**3. 字段粒度太细导致信息失真**

把"产品的综合评价"强行拆成 `ease_of_use: 0-10, design: 0-10, value: 0-10` 等字段，模型为了填满字段会捏造精确度，原本模糊的判断变成了虚假的精确数字。

**实践原则：**

| 场景 | 建议 |
|------|------|
| 提取已知存在的信息 | 强格式没问题，信息本来就在那里 |
| 生成/分析/评价 | 格式要留有余地，不要强制量化主观判断 |
| 复杂推理 | 先自由推理，再结构化提取，两步走 |
