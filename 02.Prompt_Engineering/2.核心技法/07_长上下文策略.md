# 长上下文策略（Long Context Strategy）

> **一句话总结**：context window 大不等于模型能均匀利用所有内容——在 20k+ token 的上下文中，信息的位置决定注意力权重，关键内容放开头或结尾，多文档要结构化，长对话要主动压缩。

---

## 为什么长上下文需要单独研究

现代 LLM 的 context window 动辄 200k token（Claude 3.5+），让很多开发者误以为"把所有内容塞进去"就行了。实际情况是：**大 context window 是上限，不是均匀利用的保证**。

研究（Liu et al. 2023，"Lost in the Middle"）和工程实践都表明：模型对长 context 中不同位置的信息有不均匀的注意力分布。忽视这个特性，会导致"明明把答案放进去了，模型却没用到"的困惑。

---

## 模块一：长上下文的注意力分布——"Lost in the Middle"现象

### ① 一句话定义

即使在能处理 200k token 的模型上，位于 context 中间位置的信息被有效利用的概率，也显著低于位于开头或结尾的信息——这是 Transformer 注意力机制在长序列上的固有特性。

### ② 底层原理

Liu et al. 2023（"Lost in the Middle: How Language Models Use Long Contexts"）的研究揭示了一个反直觉的现象：

```
context 利用率（示意）：

位置     开头     中间     结尾
权重     ████    ██      ████████

→ 开头较高（首因效应）
→ 中间最低（注意力洼地）
→ 结尾最高（近因效应，紧邻生成位置）
```

**为什么会这样？**

Transformer 注意力机制在长序列上存在两个偏向：

1. **近因效应（Recency Bias）**：距离当前生成位置越近的 token，注意力权重越高。结尾的内容"最近"，权重最大。

2. **首因效应（Primacy Bias）**：开头的内容在 KV cache 中被所有后续 token 的注意力访问，积累了相对稳定的高权重。

3. **中间"注意力洼地"**：既不近也不在开头的中间位置，受两端竞争，相对权重最低。

**对不同 context 长度的影响**：

| context 长度 | 中间位置丢失风险 |
|-------------|---------------|
| < 8k token | 低，整体在有效注意力范围内 |
| 8k～50k | 中，重要信息应避免放中间 |
| 50k+ | 高，中间位置信息利用率显著下降 |

**重要说明**：现代长上下文模型（Claude 3.5+）对这个问题有所改善，但并未消除——工程实践中仍然需要有意识地管理信息位置。

### ③ 最小可运行示例

**测试关键文档位置对回答质量的影响**

```python
import anthropic

client = anthropic.Anthropic()

def test_position_effect(
    question: str,
    key_doc: str,
    filler_docs: list[str],
    position: str  # "start" | "middle" | "end"
) -> str:
    """
    测试关键文档放在不同位置时的回答质量
    """
    if position == "start":
        all_docs = [key_doc] + filler_docs
    elif position == "middle":
        mid = len(filler_docs) // 2
        all_docs = filler_docs[:mid] + [key_doc] + filler_docs[mid:]
    else:  # end（推荐）
        all_docs = filler_docs + [key_doc]

    doc_blocks = "\n\n".join([
        f"<document id='{i+1}'>\n{doc}\n</document>"
        for i, doc in enumerate(all_docs)
    ])

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        messages=[{
            "role": "user",
            "content": f"<documents>\n{doc_blocks}\n</documents>\n\n{question}"
        }]
    )
    return response.content[0].text

# 预期结论：key_doc 在 end 时，回答准确率最高；在 middle 时最低
```

### ④ 适用 vs 不适用场景

**需要关注位置效应的场景**

| 场景 | context 特征 | 建议 |
|------|------------|------|
| RAG 检索结果注入 | 多文档，总 token 超过 20k | 最相关文档放在结尾（紧邻 query） |
| 长文档摘要 | 单文档，50k+ token | 先压缩再处理，或分块处理 |
| 多轮对话 | 历史轮次多 | 主动压缩早期对话历史 |

**位置效应可以忽略的场景**

| 场景 | 原因 |
|------|------|
| 短文档（< 8k token）| 整个 context 在有效注意力范围内 |
| 单一信息来源（无需"找"信息）| 不存在位置竞争 |

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 把最关键的文档放在一批文档的中间 | 模型回答时忽略了关键文档的内容 | 调整顺序，关键文档放最后（紧邻 query） |
| 以为 200k context 就可以无脑塞内容 | 长 context 下回答质量不稳定 | 评估关键信息的 token 位置，确保不在"洼地" |
| 不区分"context 支持"和"context 有效利用" | 模型能接受 200k token，但利用率低 | 建立评估集，主动测试不同位置的回答质量差异 |

**开发者视角一句话总结**：大 context window 是上限，不是保证——"Lost in the Middle"效应在工程实践中真实存在，关键信息放开头或结尾，不要放中间。

## 模块二：信息位置策略——什么放哪里

### ① 一句话定义

在长 context 中，不同类型的内容有不同的推荐位置：**查询/任务在结尾**（近因效应最强），**最相关参考内容紧邻查询之前**，**背景和次要信息在中间**，**核心约束在开头**（首因效应）。

### ② 底层原理

根据注意力分布特性，长 context 的推荐布局：

```
┌──────────────────────────────────┐
│  [开头] 系统指令 / 核心约束        │  ← 首因效应，贯穿整个推理
│                                  │
│  [中间] 次要背景文档               │  ← 注意力洼地，放支撑材料
│         大量填充内容               │
│                                  │
│  [结尾前] 最相关的参考文档         │  ← 近因效应，影响最大
│  [结尾] 当前任务 / 用户查询        │  ← 近因效应最强位置
└──────────────────────────────────┘
```

**具体布局规则**：

| 内容类型 | 推荐位置 | 原因 |
|---------|---------|------|
| 系统指令 / 核心约束 | 最开头 | 首因效应，贯穿整个推理过程 |
| 次要背景文档（支撑材料） | 中间区域 | 模型会部分利用，允许放这里 |
| 最相关的参考内容 | 结尾前（紧邻 query） | 近因效应，对回答影响最大 |
| 当前任务 / 用户查询 | 最结尾 | 近因效应最强位置 |

这和 Claude 官方推荐一致：对于 20k+ 的长文档场景，将数据放在查询之前（data at top above query），query 放最后。

### ③ 最小可运行示例

**RAG 场景：按相关性排序注入文档**

```python
import anthropic
from dataclasses import dataclass

@dataclass
class Document:
    id: str
    content: str
    relevance_score: float  # 0-1，越高越相关

def build_rag_prompt(query: str, documents: list[Document]) -> str:
    """
    构建长上下文 RAG prompt，最相关文档放最后，利用近因效应
    """
    # 按相关性升序排序（最相关的放最后，紧邻 query）
    sorted_docs = sorted(documents, key=lambda d: d.relevance_score)

    doc_blocks = []
    for doc in sorted_docs:
        doc_blocks.append(
            f"<document id='{doc.id}' relevance='{doc.relevance_score:.2f}'>\n"
            f"{doc.content}\n"
            f"</document>"
        )

    return f"""<documents>
{chr(10).join(doc_blocks)}
</documents>

<query>
{query}
</query>

请根据 documents 中的内容回答 query。
优先使用相关性更高的文档；引用时注明文档 id。"""


client = anthropic.Anthropic()

def answer_with_rag(query: str, documents: list[Document]) -> str:
    prompt = build_rag_prompt(query, documents)
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": prompt}]
    )
    return response.content[0].text
```

关键设计：`sorted(..., key=lambda d: d.relevance_score)` 升序排列，最相关文档放在最后，紧邻 query，充分利用近因效应。

### ④ 适用 vs 不适用场景

**位置策略价值高的场景**

| 场景 | 关键操作 |
|------|---------|
| 多文档 RAG（10+ 篇） | 最相关文档排最后 |
| 长 system prompt（> 1k token） | 核心约束在开头，示例在结尾 |
| 文档检索 + 回答生成 | 文档在前，query 在最后 |

**位置策略影响不大的场景**

| 场景 | 原因 |
|------|------|
| 少量文档（1-3 篇，总量 < 8k）| 整体在有效注意力范围内 |
| 纯代码生成（无外部文档）| 没有"找信息"的需求 |

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| query 放在文档前面 | 模型回答遗漏部分文档的信息 | 把 query 移到所有文档之后 |
| 文档按检索顺序直接注入（未按相关性排序）| 最相关文档可能在中间，效果不稳定 | 按相关性评分排序，最相关放最后 |
| system prompt 里的关键约束在中间 | 约束遵循率低 | 核心约束放 system prompt 开头或结尾 |

**开发者视角一句话总结**：query 最后，最相关内容紧邻 query，背景信息在前——在长 context 里，位置决定注意力权重，内容排版是工程决策，不是美观选择。

## 模块三：多文档结构化——XML + 编号的最佳实践

### ① 一句话定义

多文档场景（如 RAG 注入 5-20 篇文档）需要两层结构：**XML 标签提供边界**，**id 属性提供精确引用锚点**——两者结合让模型既知道每篇文档的范围，又能在输出中精确归因。

### ② 底层原理

多文档处理时，模型面临两个挑战：
1. **文档边界识别**：哪里是一篇文档的开始和结束
2. **跨文档归因**：回答时能精确指明信息来自哪篇文档

XML 标签 + id 属性同时解决这两个问题：

```xml
<documents>
  <document id="1" title="API 限流设计">
    [文档内容]
  </document>
  <document id="2" title="熔断器模式">
    [文档内容]
  </document>
</documents>
```

任务指令说"引用文档时注明 id"，模型输出中会出现"根据文档 2 的熔断器模式……"这样的精确归因。

**可选的元数据属性**：

| 属性 | 例子 | 用途 |
|------|------|------|
| `id` | `id="1"` | 唯一标识，支持精确引用（必选）|
| `title` | `title="API 设计规范"` | 让模型理解文档主题 |
| `source` | `source="官方文档 v2.3"` | 引用时标注来源 |
| `date` | `date="2024-01"` | 时效性判断 |

### ③ 最小可运行示例

**多文档 Q&A 系统**

```python
import anthropic
from dataclasses import dataclass, field

@dataclass
class Doc:
    id: int
    title: str
    content: str
    source: str = ""

def multi_doc_qa(question: str, docs: list[Doc]) -> str:
    # 构建结构化文档块
    def fmt_doc(doc: Doc) -> str:
        attrs = f'id="{doc.id}" title="{doc.title}"'
        if doc.source:
            attrs += f' source="{doc.source}"'
        return f"<document {attrs}>\n{doc.content}\n</document>"

    doc_blocks = "\n\n".join(fmt_doc(doc) for doc in docs)

    prompt = f"""<documents>
{doc_blocks}
</documents>

<question>
{question}
</question>

请根据 documents 中的内容回答 question。
- 每个关键论点注明来源文档 id（格式：[文档N]）
- 如果不同文档有冲突，明确指出冲突所在
- 如果 documents 中没有足够信息，直接说明"""

    client = anthropic.Anthropic()
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": prompt}]
    )
    return response.content[0].text
```

### ④ 文档数量与结构策略

| 文档数量 | 推荐结构 |
|---------|---------|
| 1-2 篇 | XML 标签即可，可不加 `<documents>` 容器 |
| 3-10 篇 | `<documents>` 容器 + `<document id="n">` |
| 10-30 篇 | 同上 + title/source 属性，按相关性排序（最相关在最后）|
| 30+ 篇 | 先筛选/压缩再注入，避免超出有效注意力范围 |

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 文档之间无边界标记 | 模型混淆来自不同文档的信息 | 加 XML 标签，每篇文档有明确开闭标签 |
| 没有 id，任务指令说"根据第3个文档" | 模型无法可靠地"数第几个" | 用 `id="3"` 属性，任务指令引用 `id` 而非位置 |
| 属性堆砌过多 | 标签复杂，增加 token 消耗 | 只加实际需要引用的属性（`id` 必选，其余按需）|

**开发者视角一句话总结**：多文档 = `<documents>` 容器 + `<document id="n">` + 任务指令里引用 id——这三件套解决边界识别和精确归因，不要省略任何一个。

## 模块四：长对话的历史管理

### ① 一句话定义

多轮对话中，随着历史增长，早期信息被持续稀释；主动压缩对话历史（截断、摘要、结构化状态）是保持长对话质量的工程手段，而不是等 context 爆了再补救。

### ② 底层原理

长对话的 context 结构：

```
[System Prompt]
[轮次 1：user + assistant]
[轮次 2：user + assistant]
...
[轮次 N：user + assistant]  ← 近因效应最强
[当前 user 消息]             ← 当前生成位置
```

随着轮次增加：
- 早期轮次离当前位置越来越远，近因效应越来越弱
- System Prompt 的 token 占比下降，首因效应被稀释
- 模型风格可能随历史内容漂移（详见 02_角色设定 的角色漂移分析）

**三种压缩策略对比**：

| 策略 | 适用对话长度 | 实现复杂度 | 信息保留度 |
|------|------------|----------|----------|
| 截断（只保留最近 N 轮）| 任意 | 低 | 低（早期信息丢失）|
| 摘要替换（老轮次 → 摘要）| 中长 | 中 | 中（语义保留）|
| 结构化状态文件 | 超长/跨会话 | 高 | 高（关键决策持久化）|

### ③ 最小可运行示例

**策略 A：滑动窗口截断（最简单）**

```python
import anthropic

client = anthropic.Anthropic()

def chat_with_sliding_window(
    history: list[dict],
    user_input: str,
    system: str,
    max_turns: int = 10
) -> tuple[str, list[dict]]:
    """保留最近 max_turns 轮对话"""
    # 截断历史，只保留最近 N 轮（每轮 2 条消息：user + assistant）
    trimmed = history[-(max_turns * 2):] if len(history) > max_turns * 2 else history
    trimmed = trimmed + [{"role": "user", "content": user_input}]

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system=system,
        messages=trimmed
    )
    reply = response.content[0].text
    trimmed.append({"role": "assistant", "content": reply})
    return reply, trimmed
```

**策略 B：摘要压缩（保留语义，减少 token）**

```python
def compress_history(
    history: list[dict],
    keep_recent: int = 4
) -> list[dict]:
    """压缩早期历史为摘要，保留最近 keep_recent 轮"""
    if len(history) <= keep_recent * 2:
        return history  # 不够长，不需要压缩

    old_turns = history[:-(keep_recent * 2)]
    recent_turns = history[-(keep_recent * 2):]

    # 生成结构化摘要
    old_text = "\n".join([
        f"{m['role'].upper()}: {m['content']}"
        for m in old_turns
    ])
    summary_resp = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=400,
        messages=[{
            "role": "user",
            "content": (
                "将以下对话历史压缩成结构化摘要，保留：\n"
                "- 已确认的关键事实\n"
                "- 已做出的决策\n"
                "- 尚未解决的问题\n"
                "忽略礼貌用语和重复内容。\n\n"
                f"{old_text}"
            )
        }]
    )
    summary = summary_resp.content[0].text

    compressed = [
        {"role": "user", "content": f"[之前对话摘要]\n{summary}"},
        {"role": "assistant", "content": "已了解之前的对话背景。"}
    ] + recent_turns

    return compressed
```

**策略 C：结构化状态文件（超长/跨会话）**

```python
import json

def save_session_state(
    key_facts: list[str],
    decisions: list[str],
    recent_history: list[dict],
    filepath: str
) -> None:
    """将关键对话状态持久化"""
    state = {
        "key_facts": key_facts,
        "decisions": decisions,
        "recent_turns": recent_history[-6:]  # 最近 3 轮原文
    }
    with open(filepath, "w", encoding="utf-8") as f:
        json.dump(state, f, ensure_ascii=False, indent=2)

def load_session_context(filepath: str) -> str:
    """加载状态文件，构建 context 注入字符串"""
    with open(filepath, encoding="utf-8") as f:
        state = json.load(f)

    facts = "\n".join(f"- {f}" for f in state["key_facts"])
    decisions = "\n".join(f"- {d}" for d in state["decisions"])

    return f"""<session_context>
<key_facts>
{facts}
</key_facts>
<decisions>
{decisions}
</decisions>
</session_context>"""
```

### ④ 适用 vs 不适用场景

**压缩策略选型**

| 场景 | 推荐策略 | 原因 |
|------|---------|------|
| 简单 Q&A 对话（< 20 轮）| 无需压缩，或滑动窗口 | 成本低，对话通常不复杂 |
| 任务导向的持续对话（20-50 轮）| 摘要压缩 | 保留语义，显著减少 token |
| 多日持续的复杂项目 | 结构化状态文件 | 跨会话持久化，随时注入 |

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 无限积累历史不压缩 | 对话越长越慢，早期内容被忽略 | 设置历史长度阈值，超过后自动压缩 |
| 压缩时间点太晚（context 已超限）| API 报 context length 错误 | 在达到 80% context 限制时主动触发压缩 |
| 摘要质量差导致关键信息丢失 | 压缩后模型"忘记"重要决策 | 用结构化摘要格式（关键事实/决策/待办），验证关键信息是否保留 |

**开发者视角一句话总结**：对话历史是会膨胀的资源——短对话放任自流，中等对话摘要压缩，超长对话结构化状态文件；在问题出现之前设好阈值，不要等到 context 爆了再补救。

## 模块五：长上下文的失效模式与排查

### ① 一句话定义

长上下文有四种主要失效模式：**遗忘**（中间内容丢失）、**混淆**（文档间信息混淆）、**稀释**（角色/约束被覆盖）、**截断**（输出被 max_tokens 硬切）——每种有不同的工程解法。

### ② 四种失效模式详解

**失效 1：关键信息遗忘（Lost in the Middle）**

症状：模型回答时明显忽略了某个文档或某条指令的内容，但这个内容确实在 context 里。

根本原因：该内容处于 context 的中间位置，近因和首因效应都未有效覆盖。

```
排查：计算该内容的 token 位置（约第几 token / 总 token）
      如果在 30%～70% 区间，移到结尾前
```

**失效 2：跨文档信息混淆**

症状：模型把文档 A 的信息归因到文档 B，或把不同文档的内容混在一起。

根本原因：多文档之间没有清晰的边界，模型无法可靠区分文档来源。

```
排查：检查文档之间是否有 XML 标签边界
      检查是否有 id 属性支持精确归因
      → 缺失 → 添加 <document id="n"> 结构
```

**失效 3：角色/约束被历史内容稀释**

症状：多轮对话中，早期设定的角色或规则在后续轮次被忽略。

根本原因：对话历史变长后，System prompt 的首因效应被稀释（详见 02_角色设定 模块四）。

```
排查：计算 system prompt token 占总 context 的比例
      如果 < 5%，在 user 消息中周期性重申关键约束
```

**失效 4：输出被截断（max_tokens 不足）**

症状：输出在句子中间突然结束。

根本原因：max_tokens 设置不足以容纳完整输出。

```python
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[...]
)

# 必须检查 stop_reason
if response.stop_reason == "max_tokens":
    # 输出被截断，增大 max_tokens 或改为分批处理
    pass
elif response.stop_reason == "end_turn":
    # 正常结束
    pass
```

### ③ 长上下文调试清单

```
长上下文输出质量差，逐步排查：

Step 1：定位关键信息的位置
  □ 关键信息是否在 context 的 30%～70% 区间？
    → 是 → 移到结尾前（紧邻 query）

Step 2：检查文档结构
  □ 多文档是否有 XML 标签边界？
  □ 是否有 id 属性支持精确引用？
    → 缺失 → 添加 <document id="n"> 结构

Step 3：检查 context 长度和比例
  □ 总 token 是否超过 50k？
  □ System prompt 占比是否 < 5%？
  □ 是否有压缩历史的机制？
    → 是 → 压缩历史 or 增加角色锚定语句

Step 4：检查输出截断
  □ stop_reason 是否为 "max_tokens"？
    → 是 → 增大 max_tokens 或分批生成
```

### ④ 适用 vs 不适用场景

**需要主动应对这些失效模式的场景**

| 场景 | 主要风险 | 预防措施 |
|------|---------|---------|
| 10+ 文档的 RAG 系统 | 跨文档混淆 + 关键文档遗忘 | XML 边界 + 相关性排序 |
| 50+ 轮的持续对话 | 角色稀释 + 早期信息遗忘 | 历史压缩 + 角色锚定 |
| 长文档（50k+ token）分析 | 中间内容遗忘 | 分块处理 or 先摘要再分析 |

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 以为"放进去了就能用" | 长 context 下结果不稳定 | 建立位置感知评估：测试关键信息在不同位置的命中率 |
| 不监控 stop_reason | 截断输出被当作完整输出处理 | 所有长输出场景必须检查 stop_reason |
| 历史不压缩导致 context 膨胀 | API 调用越来越慢，成本线性增长 | 设置 token 预算上限，超出自动触发压缩 |

**开发者视角一句话总结**：长上下文失效有四种——遗忘（位置问题）、混淆（边界问题）、稀释（比例问题）、截断（容量问题）——每种都有对应的工程解法，按诊断清单逐一排查。

## 综合对比：长上下文场景决策速查

| 场景 | 关键策略 | 核心工具 |
|------|---------|---------|
| RAG 多文档注入 | 相关性排序，最相关在最后；XML 边界 + id | `<document id="n">` 结构 |
| 长对话历史管理 | 定期压缩；关键约束周期性重申 | 摘要压缩 / 结构化状态文件 |
| 超长单文档分析 | 分块处理 + 摘要聚合 | 分批 API 调用 |
| 长 system prompt | 核心约束在开头，示例在结尾 | 位置策略 |

**长上下文 prompt 布局模板**：

```
[System]
核心约束（放开头，首因效应）

[User / Documents]
次要背景文档（放中间）
...
最相关文档（放结尾前，近因效应）

当前 query（最后，近因效应最强）
```
