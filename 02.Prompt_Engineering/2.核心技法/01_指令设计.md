# 指令设计（Instruction Design）

> **一句话总结**：指令设计是把你的意图精确翻译成模型能稳定执行的语言——结构、极性、位置、粒度四个维度决定了模型"听不听话"。

---

## 为什么指令设计值得单独研究

模型"没按要求来"，通常不是模型能力不足，而是指令本身有歧义、冲突，或被注意力机制降权了。

LLM 在推理时做的事本质上是：**在当前 context 的概率分布下，选择最符合训练分布的 token 序列**。你的指令是这个概率分布的条件之一，但不是唯一条件——模型同时受预训练惯性、context 中其他内容、指令的相对位置和权重共同影响。

理解这一点，才能把"为什么模型不听话"从玄学变成可排查的工程问题。

---

## 模块一：指令的基本结构

### ① 一句话定义

把一条 prompt 里的信息拆成三层——**任务（Task）、约束（Constraint）、输出规格（Output Spec）**——并在结构上明确区分，是让模型稳定执行的最小前提。

### ② 底层原理

模型没有"解析语法树"的过程，它把 token 序列整体作为条件预测输出。三层信息混写时，模型需要从散乱句子中"猜"哪部分是核心任务、哪部分是限制、哪部分描述期望格式——猜错的概率随复杂度上升。

三层分离后，每层在 context 中形成局部的语义聚类（semantic cluster），模型更容易正确归因：

    混写（让模型猜结构）：
    "帮我写一篇关于 RAG 的文章，不要太长，面向工程师，
     要有代码示例，避免学术语气，500字以内，用中文。"

    分层（结构明确）：
    任务：  写一篇介绍 RAG 技术的文章
    约束：  面向工程师读者；避免学术语气；用中文
    输出：  500 字以内；包含至少一个代码示例；Markdown 格式

分层后，模型不会把"500字"误解成主题的一部分，也不会漏掉代码示例要求。

### ③ 最小可运行示例

**模糊指令（结构混写）**

    [System]
    你是一个写作助手。

    [User]
    帮我写个关于 API 限流的介绍，技术一点，简短，有例子。

    [Assistant]
    → 输出长度不定（"简短"无具体标准）
    → 例子可能是文字描述而非代码（没指定"代码示例"）
    → 语气和格式由模型自由发挥

**精确指令（三层分离）**

    [System]
    你是一个面向后端工程师的技术写作助手。

    [User]
    任务：写一段 API 限流（Rate Limiting）的技术介绍

    约束：
    - 读者是有 2 年经验的后端工程师，熟悉 HTTP 协议
    - 不解释基础 HTTP 概念，直接进入限流原理

    输出规格：
    - 200 字以内的散文介绍
    - 附一个 Python 代码示例，演示令牌桶算法（Token Bucket）
    - 使用 Markdown 格式，代码用代码块包裹

    [Assistant]
    → 长度可预期（200字约束）
    → 代码示例类型明确（Python + 令牌桶）
    → 读者假设清晰，不会出现"HTTP 是一种协议……"这类废话

### ④ 适用 vs 不适用场景

**适用（三层分离有显著收益）**

| 场景 | 例子 | 原因 |
|------|------|------|
| 生产环境的 system prompt | 客服机器人、代码审查助手 | 复杂指令混写导致长期输出不稳定，分层后可维护性大幅提升 |
| 多人协作维护的 prompt | 团队共享的 prompt 模板 | 分层结构让他人一眼看出每部分的职责，便于修改和扩展 |

**不适用（分层反而过重）**

| 场景 | 例子 | 原因 |
|------|------|------|
| 简单的单轮对话 | "帮我翻译这句话：…" | 任务本身已足够清晰，分层是形式主义，增加 token 消耗 |
| 快速原型验证 | 探索模型是否具备某个能力 | 此阶段目标是快速试探，精确分层是优化阶段的事 |

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 任务和输出规格混写 | 模型输出格式随机，有时是列表有时是散文 | 检查 prompt 里"要什么"和"输出成什么样"是否分开写 |
| 约束写成任务 | "不要用 markdown"写在任务开头，模型有时忽略 | 约束和输出规格独立成段，不要嵌入任务描述句 |
| 输出规格太隐式 | "简洁一点"——模型无法量化 | 把形容词换成数字：200字以内、3条以内、一个代码块 |

**开发者视角一句话总结**：把 prompt 当函数签名写——任务是函数名，约束是前置条件，输出规格是返回值类型；签名清晰，函数体才能稳定执行。

## 模块二：正向指令 vs 负向指令

### ① 一句话定义

**正向指令（Positive Instruction）** 告诉模型"要做什么"；**负向指令（Negative Instruction）** 告诉模型"不要做什么"。两者不等价——模型执行正向指令比负向指令更稳定，因为正向指令直接收窄了概率空间，负向指令只是排除部分选项。

### ② 底层原理

模型生成 token 时，每次都在一个高维概率分布上采样。正向和负向指令对这个分布的影响方式不同：

    正向指令："用要点列表输出"
    → 模型的注意力被引导到"列表格式的 token 分布"
    → 采样时列表相关 token（如 "- "、"1."）概率显著升高
    → 效果：分布被主动收窄到目标区域

    负向指令："不要用散文格式"
    → 模型需要先激活"散文格式"的表征，再抑制它
    → 抑制效果依赖模型对"散文"的理解是否与你一致
    → 效果：从大分布中切除一块，但剩余空间仍然宽泛

**关键区别**：正向指令是"指路"，负向指令是"堵路"。堵路之后模型会走哪条路，仍然不确定。

此外，负向指令存在一个类似认知心理学中"白熊效应（White Bear Effect）"的现象：当指令中出现"不要……"时，模型需要先在内部"激活"被禁止的概念才能抑制它，反而提高了那个行为的概率。"不要道歉"这条指令，有时反而让模型更容易输出道歉。

### ③ 最小可运行示例

**同一需求的两种写法**

需求：让模型给出简洁的错误原因分析，不需要解决方案。

    ── 负向写法 ──
    [System]
    你是一个代码审查助手。

    [User]
    分析这段 Python 代码为什么报错。不要给出修复方案，不要解释如何修改，
    不要提供替代实现。

    代码：
    def divide(a, b):
        return a / b
    result = divide(10, 0)

    [Assistant]
    → 常见结果：模型分析错误原因后，仍然会"顺手"给出修复方案
      （因为训练数据中"报错分析"和"修复方案"高度共现）

    ── 正向写法 ──
    [System]
    你是一个代码审查助手。

    [User]
    分析这段 Python 代码的报错原因。

    输出格式：
    - 只输出错误原因（一句话）
    - 只输出错误发生的具体位置（行号和表达式）
    - 停止，不输出任何其他内容

    代码：
    def divide(a, b):
        return a / b
    result = divide(10, 0)

    [Assistant]
    → 模型被明确告知"输出两条信息后停止"，修复方案没有了出现空间

### ④ 适用 vs 不适用场景

**正向指令更优（绝大多数情况）**

| 场景 | 负向写法（不稳定） | 正向写法（稳定） |
|------|-----------------|----------------|
| 控制输出长度 | "不要太长" | "输出不超过 3 句话" |
| 控制输出格式 | "不要用 markdown" | "纯文本输出，无任何标记符号" |
| 控制内容范围 | "不要讨论竞争对手产品" | "只讨论我们自己的产品功能" |

**负向指令仍有价值的场景**

| 场景 | 例子 | 原因 |
|------|------|------|
| 明确排除歧义选项 | "这里的'苹果'指水果，不是指苹果公司" | 正向描述"苹果是水果"不如负向排除更直接 |
| 安全/合规边界 | "不要生成包含个人身份信息的内容" | 正向写法"生成匿名内容"无法覆盖所有 PII 类型 |

**实验对比结论**（基于实践观察）：

    同一需求，纯正向写法 vs 纯负向写法 vs 混合写法 的稳定性：

    正向为主，负向补充  ████████████  最稳定
    纯正向              ██████████    稳定
    混合（正负混用）    ████████      中等
    纯负向              █████         不稳定

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 用负向指令堆叠禁止项 | prompt 里有 5 条"不要……"，模型仍然违反其中几条 | 把每条"不要X"翻译成"要Y"，用正向指令替代 |
| 负向指令反激活问题 | "不要道歉"却更频繁道歉 | 改成"对话风格：直接、自信，直接进入回答" |
| 误以为负向指令更安全 | 以为"不要泄露系统提示"能防止提示词泄露 | 负向指令对对抗性输入（prompt injection）几乎无效，需要架构层面的隔离 |

**开发者视角一句话总结**：把"不要做X"改成"只做Y"——正向指令直接告诉模型目标状态，不需要模型先想象X再抑制X，执行更可靠。

## 模块三：指令冲突

### ① 一句话定义

**指令冲突（Instruction Conflict）** 是指 prompt 中多条指令在某些输入上无法同时满足，模型必须"取舍"时的状态。冲突分显性（两条指令直接矛盾）和隐性（单独看各自合理，组合后在特定输入下互斥）两类。

### ② 底层原理

模型面对冲突指令时，不会报错，而是按照内部的"优先级启发式"做取舍：

    优先级（从高到低，经验规律，非官方标准）：
    1. 越近（context 末尾）的指令 → 权重更高（注意力机制的近因效应）
    2. 越具体（有数字、有示例）的指令 → 比模糊指令优先
    3. 用强化语言标注的指令（"必须"、"Always"、"Never"）→ 比平淡陈述优先
    4. system prompt 中的指令 → 在多数模型中比 user 消息优先
       （但这不是硬性规则，足够强的 user 指令仍能覆盖 system 指令）

问题在于：**这个优先级是隐式的、不可靠的，且随 context 长度变化**。当冲突发生时，模型不会告诉你它选择了哪条指令，也不会提示你存在冲突。

**隐性冲突的典型来源**：

    场景：客服机器人
    指令A（system）："始终以友好、热情的语气回应用户"
    指令B（system）："对于超出产品范围的问题，简短拒绝，不做展开"

    单独看：两条都合理
    冲突触发：用户问了一个超出范围但情绪激动的问题
    → "友好热情"要求详细安慰，"简短拒绝"要求不展开
    → 模型会在两者之间摇摆，输出不稳定

### ③ 最小可运行示例

**显性冲突**

    [System]
    回答问题时，始终用中文输出。
    Always respond in the user's language.

    [User]
    What is RAG?

    [Assistant]
    → 两条指令在英文输入下直接冲突
    → 模型可能用英文（遵循第二条），可能用中文（遵循第一条）
    → 行为不可预测，且每次可能不同

**隐性冲突**

    [System]
    你是一个代码审查助手。
    规则1：给出全面、详细的代码问题分析
    规则2：回复必须在 100 字以内

    [User]
    审查这个 200 行的函数，找出所有问题。

    [Assistant]
    → 对小代码段两条规则可以共存
    → 对复杂代码，"全面详细"和"100字以内"在特定输入下互斥
    → 这类冲突在测试阶段用简单输入发现不了，上线后才暴露

**冲突预防写法**

    [System]
    你是一个代码审查助手。

    审查策略：
    - 如果代码少于 30 行：给出每个问题的详细分析
    - 如果代码超过 30 行：给出问题列表（每条一行），不展开细节

    → 把冲突条件显式化，模型根据输入状态选择规则，不需要"取舍"

### ④ 指令冲突的系统性排查

这是最容易被忽视的技能，下面给出一个可操作的排查流程：

**Step 1：列出所有指令**

把 system prompt 里的每一条要求提取出来，编号。不要跳过看起来"显而易见"的指令。

**Step 2：两两检查**

对每对指令问：
> "是否存在某种用户输入，使得同时满足 A 和 B 在逻辑上不可能？"

    示例：
    A = "用中文回复"
    B = "用用户的语言回复"
    → 当用户用英文输入时，A 和 B 冲突 ✓（显性冲突）

    A = "回答要详细"
    B = "控制在 200 字以内"
    → 当用户问复杂问题时冲突 ✓（隐性冲突，取决于输入复杂度）

    A = "始终友好"
    B = "如果用户提问超出范围，简短拒绝"
    → 当用户带情绪提出超范围问题时冲突 ✓（隐性冲突，取决于用户状态）

**Step 3：对冲突添加条件分支**

不要删掉任何一条指令，而是加条件让两条规则在不同情境下分别生效：

    冲突：详细 vs 简短
    解决：
    - 简单问题（事实查询、单步操作）：一句话回答
    - 复杂问题（涉及多个概念或多步骤）：结构化回答，不超过 400 字

**Step 4：用边界输入验证**

构造专门触发冲突的测试用例：
- 极端长度的输入
- 混合语言的输入
- 情绪化 + 超范围的输入
- 需要大量细节才能准确的简单问题

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 依赖模型"自己判断"来解决冲突 | 同类问题输出风格随机变化 | 找出变化背后的冲突指令，加条件分支 |
| 多次迭代 prompt 后指令互相矛盾 | 后加的指令和早期指令冲突，形成"历史债务" | 每次修改前重新列出所有指令，执行两两检查 |
| 不测试边界输入 | 日常输入下 prompt 正常，极端输入下崩溃 | 建立边界测试集，包含能触发各类冲突的用例 |

**开发者视角一句话总结**：指令冲突是 prompt 的隐性 bug——模型不会报错，只会静默地选边站，系统性排查方法是把所有指令列出来，两两问"是否存在让它们互斥的输入"。

## 模块四：指令位置

### ① 一句话定义

同一条指令放在 prompt 的开头、结尾或多处重复，对模型的遵循率有可测量的差异——这由 Transformer 注意力机制的**近因效应（Recency Bias）** 和**首因效应（Primacy Bias）** 共同决定。

### ② 底层原理

Transformer 的注意力机制在处理长序列时，对不同位置的 token 的权重分配不均匀。研究（Lost in the Middle, 2023）和工程实践都观察到一个共同规律：

    注意力权重分布（示意）：

    [开头 ████████] [中间 ██] [结尾 ████████]

    → 开头和结尾的信息权重最高，中间最低
    → 长 prompt 中，放在中间的指令被模型"遗忘"的概率最高

这意味着：

- **关键约束放开头**：在模型开始处理之前就建立"操作框架"，后续生成都在这个框架内进行
- **关键约束放结尾**：利用近因效应，在模型即将生成回复前"刷新"注意力
- **中间区域**：适合放背景信息、上下文、示例——这些内容被"遗忘"一部分影响不大

**system prompt vs user message 的位置效应**：

    大多数模型的 context 结构：
    [System Prompt] → [历史对话] → [当前 User Message]

    当前 User Message 紧邻生成位置，近因效应最强。
    → 如果 system prompt 里有一条重要约束，在 user message 结尾重复一次，遵循率更高。

### ③ 最小可运行示例

**约束放中间（容易被忽视）**

    [System]
    你是一个财务数据分析助手。

    你的职责是分析财务报表数据，识别异常趋势，生成分析报告。

    重要：所有数字必须四舍五入到小数点后两位。        ← 放在中间

    回答时要结合历史趋势，提供数据支撑，并给出具体的
    改善建议。语气要专业、客观，避免过于绝对的判断。

    [User]
    分析一下这份数据：收入 1234567.891，成本 876543.219

    [Assistant]
    → 数字格式约束位于中间，长文本时经常被模型忽略，输出原始数字

**约束放首尾（遵循率更高）**

    [System]
    【输出规范】所有数字四舍五入到小数点后两位。        ← 开头明确

    你是一个财务数据分析助手，负责分析财务报表，识别
    异常趋势，生成分析报告。结合历史趋势，提供数据支撑，
    给出具体改善建议。语气专业、客观。

    [User]
    分析一下这份数据：收入 1234567.891，成本 876543.219
    请四舍五入到两位小数后输出。                        ← 结尾重复

    [Assistant]
    → 开头建立框架，结尾刷新注意力，数字格式约束遵循率显著提升

### ④ 适用 vs 不适用场景

**位置策略有显著效果的场景**

| 场景 | 推荐策略 | 原因 |
|------|---------|------|
| 长 system prompt（>500 token） | 关键约束放开头 + 结尾各一次 | 中间内容被近因/首因以外的位置降权，关键信息需要锚定 |
| 输出格式严格要求（如 JSON） | user message 结尾重复格式要求 | 模型在生成前最后"看到"的是格式要求，近因效应最强 |

**位置优化收益有限的场景**

| 场景 | 原因 |
|------|------|
| 短 prompt（< 200 token） | 整个 prompt 都在有效注意力范围内，位置差异可忽略 |
| 指令本身已经足够强和具体 | 强指令（含示例、含数字约束）不需要位置加持 |

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 把所有指令堆在 system prompt 中间 | 指令遵循率低且不稳定，但单独测试时正常 | 把最关键的 1-2 条指令移到 system prompt 开头或 user message 结尾 |
| 过度依赖"重复指令"解决问题 | 同一指令重复 3 次以上，但问题未根本解决 | 重复是症状缓解，根本问题可能是指令表达本身模糊，需要重写 |
| 忽略 user message 的位置杠杆 | 只在 system prompt 里写约束，user 消息什么都没有 | 对于关键格式/范围约束，在 user 消息末尾加一行简短重申 |

**开发者视角一句话总结**：重要约束要么开头占位、要么结尾压轴——中间的内容在长 prompt 里是注意力洼地，关键指令别埋在那里。

## 模块五：指令粒度

### ① 一句话定义

**指令粒度（Instruction Granularity）** 是指描述一项任务时的细化程度——过细的指令限制了模型的语言能力，过粗的指令导致结果不可控，找到平衡点的方法是**只约束结果，不约束过程**。

### ② 底层原理

LLM 的核心能力是**语言模式的泛化**：它在海量文本上学到了"好的写作/代码/分析是什么样的"，这是它最强的地方。过细的指令会干扰这个泛化能力：

    过细指令的问题：

    "第一段写背景，第二段写问题，第三段写原因，
     第四段写解决方案，每段恰好 3 句话，第一句用
     主动语态，第二句用数据支撑，第三句做总结。"

    → 模型把认知资源用在满足格式限制上
    → 内容质量（逻辑、表达）反而下降
    → 结果是"格式达标但内容空洞"

**粒度原则**：约束"输出是什么"，不约束"模型怎么到达输出"。

    结果约束（好）：  "输出一份 3 段的分析，每段 2-4 句，有数据支撑"
    过程约束（差）：  "先想背景，再想问题，再想原因，然后……"

一个实用的判断标准：
> 如果你的指令在描述"模型应该怎么思考"，而不是"输出应该是什么"，那这条指令的粒度可能已经过细了。

### ③ 最小可运行示例

**过细指令（过程约束）**

    [System]
    你是一个技术博客写手。

    [User]
    写一篇关于 Redis 缓存失效的文章。

    要求：
    - 第一部分：先用 2 句话介绍缓存的概念
    - 第二部分：用 3 个小标题描述三种失效策略（TTL、LRU、主动失效），
      每个小标题下恰好 2 句话，第一句解释原理，第二句给使用场景
    - 第三部分：写 1 段 50 字的总结
    - 不能使用任何类比
    - 所有句子必须是主动语态

    [Assistant]
    → 模型把精力放在数句子、检查语态上
    → 输出机械、生硬，技术内容被"格式合规性"压缩
    → "不能使用类比"删掉了最有助于理解的表达方式

**适度粒度（结果约束）**

    [System]
    你是一个面向工程师的技术博客写手，擅长用具体例子解释抽象概念。

    [User]
    写一篇 Redis 缓存失效策略的技术文章。

    读者：有 1-3 年经验的后端工程师，了解 Redis 基础用法

    输出规格：
    - 覆盖 TTL、LRU、主动失效三种策略
    - 每种策略给出一个真实的使用场景举例
    - 文章总长 400-600 字
    - Markdown 格式，有小标题

    [Assistant]
    → 模型自由决定如何组织每个策略的讲解方式
    → 对 TTL 可能用类比，对 LRU 可能用图解，灵活运用语言能力
    → 输出质量更高，同时关键结果（三种策略 + 场景 + 长度）有保证

### ④ 适用 vs 不适用场景

**需要更细粒度指令的场景**

| 场景 | 例子 | 为什么需要细 |
|------|------|------------|
| 严格合规输出（如法律文本、医疗报告） | "诊断报告必须包含：主诉、体征、诊断结论，顺序不可变" | 格式本身是合规要求，结构必须精确 |
| 结构化数据提取 | "从文本中提取：姓名、日期、金额，输出 JSON，key 名固定" | 下游系统依赖固定格式，不能让模型自由发挥 |

**应该放粗粒度的场景**

| 场景 | 例子 | 为什么要粗 |
|------|------|----------|
| 创意写作和内容生成 | 文案、技术博客、解释性文章 | 模型的语言能力是核心价值，过度约束是负优化 |
| 开放式推理和分析 | 代码审查、方案评估 | 推理路径是模型的强项，约束过程会降低分析质量 |

### ⑤ 常见误用和排查

| 误用 | 症状 | 排查方法 |
|------|------|---------|
| 指令太细导致内容空洞 | 格式完全符合，但读起来生硬、缺乏洞察 | 逐条检查：这条指令是在约束"输出是什么"还是"怎么到达输出"？删掉过程指令 |
| 指令太粗导致结果发散 | 每次运行结果结构差异很大，难以复用 | 找出"什么是不可缺少的结果要素"，把这些要素加入输出规格 |
| 用粒度掩盖任务不清晰的问题 | 加了很多细指令，但效果仍然不稳定 | 先写清楚任务是什么（模块一），再讨论粒度；任务不清时增加粒度只是在错误方向上用力 |

**粒度自检问题**：

> 1. 我的约束是针对"输出的样子"还是"模型的思考过程"？
> 2. 删掉这条指令后，结果会不可接受地变差吗？（如果不会，删掉它）
> 3. 这条指令是为了当前用例，还是我在为假想中的未来用例防御？

**开发者视角一句话总结**：指令粒度的黄金法则——约束输出（what），释放过程（how）；把模型当高级工程师用，告诉他交付什么，不要告诉他怎么想。

## 综合对比：四个维度的决策速查

| 维度 | 核心原则 | 最常见的坑 |
|------|---------|-----------|
| **结构**（模块一） | 任务/约束/输出规格三层分离 | 混写导致模型猜结构，长期输出不稳定 |
| **极性**（模块二） | 正向指令为主，负向指令补充边界 | 用"不要X"堆叠禁止项，不如直接说"只做Y" |
| **冲突**（模块三） | 两两检查指令，加条件分支消除冲突 | 隐性冲突只在边界输入时触发，日常测试发现不了 |
| **位置**（模块四） | 关键约束放开头或结尾，避免中间 | 长 prompt 中，中间区域是注意力洼地 |
| **粒度**（模块五） | 约束输出（what），释放过程（how） | 过细指令让模型把精力花在满足格式上，内容质量下降 |

**Prompt 调试的快速诊断流程**：

    模型没按要求输出？
    ↓
    是否存在冲突指令？（两两检查）
      → 有冲突 → 加条件分支
    ↓
    关键指令是否在中间位置？
      → 是 → 移到开头或结尾
    ↓
    是否用了大量负向指令？
      → 是 → 翻译成正向写法
    ↓
    是否有过细的过程约束？
      → 是 → 改成结果约束
    ↓
    三层结构是否清晰？
      → 混写 → 任务/约束/输出规格分段写
