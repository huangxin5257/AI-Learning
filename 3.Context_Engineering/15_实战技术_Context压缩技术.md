# Context 压缩技术

**一句话：** Context 窗口有限、token 要花钱、信息太多反而干扰模型——压缩技术的目标是用更少的 token 保留更多的有效信息。

## 是什么

Context 压缩是指在不丢失（或尽量少丢失）关键信息的前提下，减少注入 context 的 token 数量。

为什么需要压缩？三个现实约束同时存在：

1. **窗口有限**——即使是 128k 甚至 1M token 的模型，当你要塞入商品目录、对话历史、用户画像、工具调用结果时，总量很容易接近上限。
2. **成本正比于 token 数**——API 按 token 计费，输入越长，每次调用越贵。客服系统日调用 10,000 次，每次多 500 token，一个月下来多出的费用非常可观。
3. **信息过多会降低质量**——这是很多人忽略的关键点。即使窗口装得下，大量无关或冗余信息会稀释模型对关键内容的注意力（参见"信息衰减与位置效应"），导致输出质量下降。

压缩不是目的，压缩是为了**在有限的 token 预算内，最大化信息密度和信噪比**。

### 压缩技术全景：从简单到高级

下面按复杂度从低到高，介绍五种压缩手段。

### 1. 手动裁剪（Manual Pruning）

最原始也最直接的方法——人工审查注入 context 的每一段内容，删除不相关、不重要或冗余的部分。

**示例：鲜果优选的商品描述压缩**

原始商品描述（约 180 token）：

```
丹东红颜草莓是鲜果优选平台2024年冬季推出的明星产品。精选自辽宁丹东东港市
的露天草莓种植基地，采用传统有机种植方式，不使用化学农药和化肥。红颜草莓
又称"红颊"，是日本引进的优良品种，果形呈圆锥形，色泽鲜红有光泽，果肉细
腻多汁，甜度高达15-18度。每颗草莓均经过人工筛选，确保果径在3cm以上，坏果
率低于1%。本产品采用顺丰冷链配送，泡沫箱加冰袋保温，确保48小时内从田间到
餐桌。建议收到后冷藏保存，不要提前清洗，保质期3-5天。如有质量问题请在签收
后24小时内联系客服。
```

客服助手需要回答的典型问题：保质期、保存方法、退换货政策。大量营销描述（"明星产品""精选自"等）对回答这些问题没有帮助。

手动裁剪后（约 80 token）：

```
丹东红颜草莓
- 产地：辽宁丹东东港市
- 种植：有机种植，无化学农药
- 规格：果径≥3cm，坏果率<1%
- 配送：顺丰冷链，48小时内送达
- 保存：冷藏，不要提前清洗，保质期3-5天
- 售后：签收后24小时内联系客服
```

token 减少约 55%，但客服需要的所有关键信息全部保留。

**手动裁剪的适用场景：** 内容量不大且相对稳定（如产品规格、公司政策），值得花人工一次性精炼。

### 2. 摘要压缩（Summarization）

用 LLM 把长文本压缩成短摘要，再把摘要注入 context。这是"用 AI 压缩 AI 的输入"。

**最典型的应用场景：对话历史压缩。**

鲜果优选客服助手与用户已经聊了 15 轮对话。如果把完整的 15 轮对话全部塞入 context：

```
完整 15 轮对话：约 2,000-3,000 token
```

用摘要压缩前 12 轮，只保留最近 3 轮的原文：

```
【对话摘要】用户询问了草莓的价格（39.9元/盒）和保存方法（冷藏3-5天），
确认了配送地址（北京朝阳区），已下单 #67890。随后反馈第二天收到的草莓
有3颗压坏了，客服已登记售后，承诺48小时内处理退款。
---
[最近3轮完整对话]
用户：退款什么时候到账？
助手：您好！退款会在审核通过后1-3个工作日内原路退回。您的售后单已提交，
      预计明天会完成审核。
用户：好的，那压坏的草莓我需要退回去吗？
```

压缩后总量约 500-800 token，节省了 60-75% 的 token，同时保留了对话的完整脉络。

**摘要压缩的代码实现思路：**

```python
def compress_conversation(full_history, recent_turns=3):
    """将长对话压缩为：摘要 + 最近N轮原文"""

    # 分割：早期历史 vs 最近对话
    early_history = full_history[:-recent_turns]
    recent_history = full_history[-recent_turns:]

    # 用 LLM 摘要早期历史
    summary_prompt = f"""请将以下客服对话历史压缩为一段简洁摘要。
保留：用户身份、已确认的订单信息、已解决和未解决的问题、客服做出的承诺。
删除：寒暄、重复确认、过渡语句。

对话历史：
{format_messages(early_history)}"""

    summary = call_llm(summary_prompt)

    # 拼接：摘要 + 最近原文
    return f"【对话摘要】{summary}\n---\n{format_messages(recent_history)}"
```

**注意：** 摘要本身会引入一次额外的 LLM 调用（增加延迟和成本）。适合对话已经足够长、直接注入全部历史不划算的场景。常见的触发策略是"对话超过 N 轮时启动摘要压缩"。

### 3. 关键信息提取（Extraction）

和摘要不同，提取不是"把文本说短"，而是"从文本中抽出结构化的关键事实"。

**示例：从用户邮件中提取关键信息**

用户发来一封长邮件投诉：

```
你好，我是你们鲜果优选的老客户了，从去年开始基本每个月都会下单买水果。
上周三我在你们平台买了两盒丹东草莓和一盒智利车厘子，订单号是 #12345。
本来是打算周末给孩子带去学校的。结果周五收到快递打开一看，草莓有一盒底
部全压坏了，至少有七八颗是烂的，车厘子倒还好。我当时就拍照了，照片在手
机里还没来得及传。我联系了快递说不关他们的事，让我找你们。我觉得这个包装
确实有问题，就一层薄薄的塑料托，根本保护不了。希望你们能给我退款或者重
新补发一盒草莓。另外能不能看看包装的问题，不然下次还是一样。
```

提取后：

```
客户类型：老客户（一年以上）
订单号：#12345
问题商品：丹东草莓×1盒（另1盒正常，车厘子正常）
问题描述：底部压坏，约7-8颗变质
证据：已拍照，未上传
诉求：退款或补发草莓
附加反馈：包装保护不足
```

原文约 200 token，提取后约 60 token。而且提取后的格式对模型更友好——每个关键事实独立一行，模型处理起来比从一段叙述性文字中找信息要高效得多。

**提取 vs 摘要的区别：**

```
摘要：把一段长文变成一段短文（仍然是散文形式）
提取：从长文中抽出若干个关键字段（结构化形式）

摘要适合：需要保留叙事脉络的场景（如对话历史）
提取适合：只需要关键事实的场景（如工单信息、用户需求）
```

### 4. 算法压缩（Algorithmic Compression）

用专门的算法自动识别并移除文本中的"低信息量 token"，保留高信息量的部分。这是学术界近两年的研究热点。

代表性工具：

- **LLMLingua / LongLLMLingua**（微软）：使用一个小语言模型计算每个 token 的困惑度（perplexity），移除困惑度低的 token（即"在上下文中可预测的、信息量低的"token）。
- **Selective Context**：类似思路，选择性保留信息密度高的句子。

```
原文：
"我们鲜果优选平台一直致力于为广大消费者提供最新鲜、最优质的水果产品。
我们的丹东红颜草莓精选自辽宁丹东东港市的优质种植基地。"

LLMLingua 压缩后（约50%压缩率）：
"鲜果优选提供新鲜优质水果。丹东红颜草莓选自辽宁丹东东港种植基地。"
```

**注意事项：**

- 算法压缩的效果高度依赖具体文本和任务。在某些任务上压缩到 50% 后性能几乎不下降，但在另一些任务上 20% 的压缩就会导致关键信息丢失。
- 这些工具需要额外部署和运行小模型，增加系统复杂度。
- 目前更适合研究和高级优化场景，不是入门首选。

### 5. 结构化重写（Structured Representation）

把散文形式的信息转换成结构化形式（表格、键值对、列表），用更少的 token 表达同样的信息。

**示例：鲜果优选退货政策**

散文版本（约 200 token）：

```
关于退货政策，如果您收到的商品存在质量问题，例如水果变质、损坏或者与描述
严重不符，您可以在签收后24小时内联系我们的客服团队申请退货退款。我们会在
收到您的退货申请后48小时内进行审核。审核通过后，退款将在1-3个工作日内按照
原支付方式退回到您的账户。如果是非质量问题的退货，例如您改变了主意不想要了，
由于生鲜产品的特殊性，我们很抱歉无法接受退货，但您可以联系客服看是否有其他
解决方案。
```

结构化版本（约 90 token）：

```
# 退货政策

| 场景 | 条件 | 时限 | 退款方式 |
|------|------|------|----------|
| 质量问题（变质/损坏/描述不符） | 签收后联系客服 | 24小时内申请 | 原路退回，1-3工作日 |
| 非质量问题（改变主意等） | 不接受退货 | — | 联系客服协商其他方案 |

审核周期：收到申请后48小时内完成
```

信息完全一致，token 减少约 55%，而且模型从表格中检索特定规则比从散文中提取要容易得多。

## 为什么重要

### 一笔经济账

鲜果优选客服助手的一次典型调用 context 构成：

```
组件                    未压缩        压缩后        压缩手段
──────────            ─────        ─────        ──────
System Prompt          300 token    300 token    （不压缩）
商品信息（相关3个）      540 token    240 token    手动裁剪 + 结构化
退货/配送政策           400 token    180 token    结构化重写
对话历史（12轮）        2,400 token  600 token    摘要 + 最近3轮
用户当前消息            100 token    100 token    （不压缩）
──────────            ─────        ─────
总计                   3,740 token  1,420 token
```

压缩率约 62%。如果用 GPT-4o，按输入 $2.50/M token 计算：

```
日均 10,000 次调用：
  未压缩：3,740 × 10,000 / 1,000,000 × $2.50 = $93.50/天
  压缩后：1,420 × 10,000 / 1,000,000 × $2.50 = $35.50/天

  每天节省 $58，每月节省约 $1,740
```

而且这还只是输入成本。更短的输入通常意味着更快的响应速度（首 token 延迟更低），以及更好的输出质量（信噪比更高）。

### 质量提升的实证

考虑一个具体场景：用户问"草莓怎么保存"，context 中同时包含了草莓、蓝莓、车厘子、芒果、榴莲的完整商品描述。

模型需要从 5 个商品的信息海洋中找到草莓的保存方法。大量无关商品信息不仅浪费 token，还可能导致模型"串台"——把车厘子的保存方法混入草莓的回答中。

如果通过"选择 + 压缩"只注入草莓的关键信息，模型的任务从"大海捞针"变成"桌上取物"，准确率自然更高。

## 设计启示

### 压缩 vs 选择：先选再压

压缩和选择是两个不同的操作，但经常被混淆：

```
选择（Selection）：决定哪些信息需要进入 context
压缩（Compression）：把选中的信息用更少的 token 表达

正确的顺序：先选择，再压缩。
```

先选择意味着：用户问草莓的问题，就只检索草莓相关的文档（选择），然后对这些文档做裁剪和结构化（压缩）。不要把所有商品文档都压缩后塞进去——这是用压缩来弥补选择的失败，效果不好。

### 不同内容类型的最佳压缩策略

```
内容类型            推荐策略                    不推荐
──────            ──────                    ──────
商品/知识条目        手动裁剪 + 结构化重写         摘要（会丢失精确数据）
对话历史            摘要（早期）+ 保留（近期）      直接截断前面的
用户输入            提取关键信息                  直接压缩原文（会丢语气和意图）
政策/规则文档        结构化重写（表格/列表）         删除"看起来不重要"的条款
代码上下文          只保留相关函数签名和注释        摘要（代码摘要经常出错）
```

### 压缩质量的验证方法

压缩后你需要验证：压缩版本是否仍然包含完成任务所需的全部信息？

一个简单的验证方法是**对比测试**：

```
1. 准备 20 个典型用户问题
2. 分别用"未压缩 context"和"压缩后 context"跑一遍
3. 人工对比两组输出：
   - 答案正确性是否一致？
   - 有没有压缩后遗漏的关键信息？
   - 输出质量有没有明显下降？
4. 如果某个问题在压缩后答错了，回溯是哪条信息在压缩中丢失了
```

这个过程不需要自动化，20 个问题人工看一遍只需要半小时，但能帮你建立对压缩边界的直觉。

### 渐进式压缩策略

不需要一步到位把所有内容都压缩到极致。建议的渐进路径：

```
第一步：结构化重写（成本最低，效果最显著）
        把散文形式的知识库条目改写为表格/列表/键值对形式

第二步：手动裁剪（配合结构化重写）
        删除营销用语、冗余描述、对任务无用的信息

第三步：对话摘要（当对话超过一定长度时启用）
        保留最近 N 轮原文，早期内容用摘要替代

第四步：算法压缩（高级优化）
        当前三步已经做到位，但仍需进一步压缩时考虑
```

大多数应用做好前三步就已经能把 token 用量压缩 50-70%，不需要引入算法压缩的复杂度。

## 常见误区

**误区一："压缩一定会丢失重要信息"**

这取决于你怎么压缩。很多文本包含大量冗余——客气话、过渡语句、重复表述——移除它们不会丢失任何有效信息。结构化重写（散文→表格）在很多场景下是"零损失压缩"，因为信息完全没变，只是表达方式变了。

真正会丢失信息的是粗暴的截断和不审慎的摘要。关键是区分"冗余"和"信息"——营销话术是冗余，"保质期 3-5 天"是信息。

**误区二："直接截断最前面的内容就好了"**

这是最常见也最危险的"压缩"方法。直接从头截断对话历史意味着：

- System prompt 可能被截掉（模型丢失身份和规则）
- 对话的起因和背景被截掉（模型不知道用户一开始在问什么）
- 关键的早期承诺被截掉（"我们承诺 48 小时内处理"没了）

如果必须缩短对话历史，应该用摘要压缩早期内容，而不是简单截断。而且 system prompt 永远不能被截断——它是 context 的"宪法"。

**误区三："LLM 摘要是无损的"**

用 LLM 做摘要本身就是一次信息压缩，必然有信息损失。常见的摘要失误：

- **数字变模糊**：原文说"签收后24小时内"，摘要变成"尽快联系"
- **条件被简化**：原文说"质量问题可退，非质量问题不可退"，摘要变成"支持退货"
- **关键细节被省略**：原文提到了特定的例外情况，摘要把它当作"不重要的细节"跳过了

对策：在摘要 prompt 中明确指出哪些信息必须保留（如数字、时限、条件、承诺），并对摘要结果做抽样核查。

```python
summary_prompt = """请压缩以下对话历史为一段摘要。

【必须保留的信息】
- 所有订单号、金额、日期等具体数字
- 客服做出的任何承诺（时限、处理方式）
- 尚未解决的问题

【可以省略的信息】
- 寒暄和客气话
- 重复的确认对话
- 已经完全解决且不再相关的问题
"""
```

**误区四："压缩做一次就够了"**

不同阶段的对话需要不同程度的压缩。对话刚开始时可能不需要压缩，10 轮后需要摘要前 7 轮，30 轮后可能需要对摘要再做摘要（递归摘要）。压缩策略应该是动态的，随着 context 增长逐步加强，而不是一开始就设定一个固定的压缩方案。
