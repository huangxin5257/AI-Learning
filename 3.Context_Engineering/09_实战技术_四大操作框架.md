# 四大操作框架：Write / Select / Compress / Isolate

**一句话：** Context Engineering 的所有具体技术，都可以归入四种基本操作——Write（手写）、Select（选取）、Compress（压缩）、Isolate（隔离），它们共同决定了模型在每次调用时"看到什么"。

## 是什么

当你构建一个 LLM 应用时，你面对的核心问题始终是：**这一次调用，模型的 context 里应该放什么？**

这个问题可以拆解为四个子问题，对应四种操作：

```
┌─────────────────────────────────────────────────────────┐
│                  Context 设计的四种操作                    │
├──────────┬──────────────────────────────────────────────┤
│  Write   │  手工编写放入 context 的内容                    │
│          │  （system prompt、指令、静态 few-shot 示例）    │
├──────────┼──────────────────────────────────────────────┤
│  Select  │  从大量候选信息中动态选取相关的子集              │
│          │  （RAG 检索、对话历史筛选、动态 few-shot）      │
├──────────┼──────────────────────────────────────────────┤
│  Compress│  在保留关键信息的前提下缩减 context 体积        │
│          │  （摘要、提取、token 级压缩）                   │
├──────────┼──────────────────────────────────────────────┤
│  Isolate │  将复杂任务拆分到独立的 context 空间中处理       │
│          │  （多 Agent 分工、工具沙箱、子任务拆解）        │
└──────────┴──────────────────────────────────────────────┘
```

### Write — 手写 context

**含义：** 你亲手编写的、直接写入 context 的所有内容。

这是最基础的操作。你在编辑器里敲的每一个字，都是在执行 Write 操作。包括：

- **System prompt** 的每一层（角色、规则、知识、示例）
- **静态指令**（"请用 JSON 格式回答"、"回答限制在 100 字以内"）
- **静态 few-shot 示例**（硬编码在 prompt 中的输入→输出对）
- **提示词模板**（prompt template）中的固定部分

Write 的特点是**完全可控但无法适应变化**——你写什么就是什么，不会根据用户输入或外部数据自动调整。

**鲜果优选示例：**

```markdown
# 你是「鲜果优选」的智能客服助手"果小蜜"
# 以下回答模板是手工编写并固定在 system prompt 中的

## 退货流程标准回答模板
当用户询问退货时，按以下步骤回答：
1. 确认订单号和商品名称
2. 告知退货政策（签收24小时内，需提供照片）
3. 引导用户在 APP 内提交退货申请
4. 告知预计退款时间（审核通过后3-5个工作日）
```

这段文字是你写好的，每个用户问退货都会看到同样的模板。它是 Write 操作的产物。

### Select — 动态选取

**含义：** 从一个大的信息池中，根据当前请求的特征，选出最相关的子集放入 context。

这是让 LLM 应用从"玩具"变成"产品"的关键操作。因为真实场景中，相关信息往往远超 context window 的容量——你不可能把整个知识库、全部对话历史、所有历史案例都塞进去。你需要选。

Select 操作的常见形式：

- **RAG（检索增强生成）：** 用户提问 → 在知识库中检索相关文档片段 → 将检索结果注入 context
- **对话历史选择：** 从过去 50 轮对话中选出与当前问题最相关的 5 轮
- **动态 few-shot：** 根据当前问题的类型/相似度，从示例库中挑选最匹配的几个示例
- **用户画像选择：** 根据用户标签/历史行为，决定注入哪些个性化信息

Select 的核心挑战是**相关性判断**——选错了比不选更糟，因为不相关的信息会干扰模型。

**鲜果优选示例：**

```python
# 用户问："你们的车厘子是智利的还是塔斯马尼亚的？口感有什么区别？"

# Select 操作 1：从商品知识库中检索相关文档
retrieved_docs = knowledge_base.search("车厘子 产地 智利 塔斯马尼亚 口感")
# 返回：智利车厘子产品详情、塔斯马尼亚车厘子产品详情、车厘子选购指南

# Select 操作 2：从历史问答中检索相似问题
similar_qas = qa_history.search("车厘子产地区别", top_k=2)
# 返回：之前其他用户问过的类似问题和客服回答

# 将选取的信息注入 context
context = f"""
{system_prompt}

<retrieved_knowledge>
{retrieved_docs}
</retrieved_knowledge>

<similar_cases>
{similar_qas}
</similar_cases>

用户问题：你们的车厘子是智利的还是塔斯马尼亚的？口感有什么区别？
"""
```

### Compress — 压缩

**含义：** 将已有信息在保留核心语义的前提下缩减体积，以腾出 context 空间。

当 context window 快满了，或者你需要在有限空间里塞入更多信息时，就需要 Compress。

常见的压缩手段：

- **摘要（Summarization）：** 把一段长文本压缩成几句话的摘要
- **提取（Extraction）：** 从长文本中只提取与当前任务相关的关键字段
- **对话历史压缩：** 把前 20 轮对话摘要为一段背景描述
- **Token 级压缩：** 用专门的工具（如 LLMLingua）在 token 级别移除冗余

Compress 的核心挑战是**信息损失控制**——压缩必然丢失信息，关键是确保丢失的是不重要的信息。

**鲜果优选示例：**

```
# 原始：20 轮对话历史（约 2000 token）
用户：我想买草莓
果小蜜：推荐丹东99草莓...
用户：多少钱？
果小蜜：1斤装 39.9 元...
用户：贵了
果小蜜：目前有满 99 减 20 的活动...
用户：那来两斤装的吧
果小蜜：好的，2斤装 69.9 元...
用户：下单了，订单号 #67890
果小蜜：已查到您的订单...
（中间还有 14 轮关于配送时间、保鲜方式的讨论）
用户：我的草莓到了，但是有几颗烂的

# Compress 后的摘要（约 200 token）
<conversation_summary>
用户购买了丹东99草莓 2斤装（订单 #67890，69.9元），使用了满99减20优惠。
讨论过配送时效（48小时内）和保鲜方式（冷链+气泡柱）。
当前问题：用户收到的草莓有部分损坏，需要售后处理。
</conversation_summary>
```

压缩后，模型依然知道用户的完整背景，但 context 占用从 2000 token 降到了 200 token。

### Isolate — 隔离

**含义：** 将一个复杂任务拆分到多个独立的 context 空间中处理，每个空间只看到与其子任务相关的信息。

当一个任务涉及多个差异很大的子任务时，把所有信息塞到一个 context 里会导致：
- Context 过长，超出窗口或降低质量
- 不同子任务的指令互相干扰
- 安全敏感信息暴露给不需要它的环节

Isolate 通过拆分来解决这些问题：

- **多 Agent 架构：** 不同 Agent 负责不同子任务，各自有独立的 system prompt 和 context
- **工具沙箱：** 把工具调用的输入输出隔离，不让原始数据污染对话 context
- **子任务拆解：** 先用一个 LLM 调用做信息提取，再用另一个调用做回答生成

**鲜果优选示例：**

```
用户问题："帮我看看上次买的车厘子订单退款进度，顺便推荐一下现在有什么当季水果"

这个问题包含两个性质完全不同的子任务：
1. 查询退款进度 → 需要访问订单系统、退款记录（敏感数据）
2. 推荐当季水果 → 需要访问商品目录、促销信息（公开数据）

Isolate 方案：

┌────────────────┐     ┌────────────────────┐
│  Router Agent   │────→│  订单查询 Agent     │
│  理解用户意图    │     │  context:           │
│  拆分子任务      │     │  - 订单系统权限     │
│                 │     │  - 退款流程规则     │
│                 │     │  - 用户订单数据     │
│                 │     └────────────────────┘
│                 │
│                 │     ┌────────────────────┐
│                 │────→│  商品推荐 Agent     │
│                 │     │  context:           │
│                 │     │  - 商品目录         │
│                 │     │  - 当季推荐策略     │
│                 │     │  - 促销活动信息     │
│                 │     └────────────────────┘
└────────────────┘
         │
         ↓ 汇总两个 Agent 的结果，生成最终回答
```

通过 Isolate，订单查询 Agent 不需要知道商品推荐的逻辑，商品推荐 Agent 也接触不到用户的订单敏感数据。每个 Agent 的 context 更小、更聚焦、更安全。

## 为什么重要

### 场景：只靠 Write，系统走不远

假设你只用 Write 操作构建鲜果优选客服——把所有商品信息、所有退换货政策、所有常见 QA 都写进 system prompt。

刚开始 10 个 SKU 时还行，但当商品扩展到 500 个 SKU 时：
- System prompt 膨胀到几万 token，超出 context window
- 大量不相关的商品信息干扰模型回答（用户问草莓，模型翻了半天榴莲的信息）
- 商品信息更新时需要手动改 prompt，容易遗漏

这时你需要 Select（用 RAG 动态检索相关商品）。当对话变长后，你需要 Compress（压缩历史）。当任务变复杂后，你需要 Isolate（拆分子任务）。

**四种操作不是替代关系，而是协作关系。** 一个成熟的 LLM 应用会同时使用全部四种操作。

### 场景：理解操作类型帮助诊断问题

当你的 LLM 应用输出不理想时，四大操作框架给你一个清晰的诊断路径：

| 症状 | 可能的操作缺陷 | 改进方向 |
|------|---------------|----------|
| 模型不遵守规则 | Write 问题：规则不清晰或有冲突 | 重新组织 system prompt 结构 |
| 模型回答基于过时信息 | Select 问题：没有动态检索最新数据 | 引入 RAG 或实时数据注入 |
| 长对话后模型"忘记"早期信息 | Compress 问题：没有做对话历史压缩 | 引入摘要/滑动窗口策略 |
| 复杂任务回答质量差 | Isolate 问题：所有子任务挤在一个 context | 拆分为多 Agent 或多步调用 |

## 设计启示

### 1. 四种操作是并行的，不是串行的

一个常见的误解是：先 Write 好 prompt，再加 Select（RAG），再做 Compress，最后考虑 Isolate。实际上，这四种操作从设计之初就应该同时考虑：

```
用户请求到来
    │
    ├── Write：system prompt 提供角色和规则（预先编写好的）
    ├── Select：RAG 检索相关知识，选取相关对话历史（实时动态的）
    ├── Compress：压缩过长的对话历史（实时动态的）
    └── Isolate：判断是否需要拆分子任务（实时动态的）
    │
    ↓
  组装最终 context → 调用模型
```

每次请求到来时，这四种操作可能同时发生。它们的组合决定了最终 context 的内容。

### 2. 四种操作对应 M2 的各个专题

这个框架不是孤立的理论，它直接映射到你将要学习的具体技术：

| 操作 | 对应的 M2 专题 | 核心技术 |
|------|---------------|----------|
| Write | System Prompt 架构设计 | 四层结构、Markdown/XML 标记 |
| Write | Few-shot（静态部分） | 硬编码示例、格式示范 |
| Write | 输出格式约束 | JSON Schema、回答模板 |
| Select | RAG 的 Context 注入 | 向量检索、重排序、chunk 策略 |
| Select | Few-shot（动态部分） | 相似度匹配、示例库检索 |
| Select | 多轮对话 Context 管理 | 历史筛选、相关轮次提取 |
| Compress | Context 压缩技术 | 摘要、提取、LLMLingua |
| Compress | 多轮对话 Context 管理 | 对话摘要、滑动窗口 |
| Isolate | Tool Use Context 设计 | 工具沙箱、结果格式化 |
| Isolate | 多 Agent 架构（进阶） | 任务分发、context 隔离 |

**当你后续学习每个专题时，先想一想它属于哪种操作，就能理解它在整体架构中的位置。**

### 3. 从 Write 开始，按需引入其他操作

虽然四种操作是并行的，但在实际开发中有一个自然的递进顺序：

1. **先做好 Write**——把 system prompt 写好，用静态 few-shot 验证基本行为。这是成本最低、反馈最快的起点。
2. **发现 Write 的局限后引入 Select**——当静态知识不够用、需要动态信息时，加入 RAG 或动态示例选择。
3. **发现 context 太长后引入 Compress**——当对话历史或检索结果撑爆 context window 时，加入压缩策略。
4. **发现单一 context 太拥挤后引入 Isolate**——当一个 context 里塞了太多异质任务时，拆分为多 Agent 或多步调用。

这个顺序不是教条，而是复杂度递增的自然路径。每引入一种操作都增加系统复杂度，所以只在确实需要时才引入。

### 4. 每种操作都有成本

| 操作 | 主要成本 |
|------|---------|
| Write | 人力成本：需要人工编写和维护 |
| Select | 基础设施成本：需要检索系统、向量数据库等 |
| Compress | 质量成本：压缩必然丢失信息，可能丢失关键细节 |
| Isolate | 复杂度成本：多 Agent 架构的编排、调试、一致性维护 |

不要为了"技术完整性"而使用所有操作。如果你的场景用 Write + Select 就够了，就不要为了"看起来先进"而强行加入 Compress 和 Isolate。

## 常见误区

**误区一："我只需要 Write —— 把 prompt 写好就够了"**

这是最常见的初学者思维。当你的应用场景很简单（比如翻译一段话、写一封邮件）时，确实只需要 Write。但几乎所有生产级应用都需要至少两种操作的组合。

判断标准：如果你的应用需要访问外部知识、需要处理长对话、或者需要执行多步任务——那就不可能只靠 Write。

**误区二："四大操作是四个阶段，要按顺序执行"**

Write → Select → Compress → Isolate 看起来像一个流水线，但实际上它们是四个维度，在每次请求中可能同时发生。

比如一次客服请求的处理过程中：
- Write 提供的 system prompt 始终在 context 中（持续生效）
- Select 根据用户问题检索了相关商品信息（本次请求触发）
- Compress 把前 30 轮对话压缩成摘要（本次请求触发）
- Isolate 把"查物流"和"推荐商品"拆给了两个子 Agent（本次请求触发）

这四件事是并发的，不是串行的。

**误区三："Select 就是 RAG"**

RAG 是 Select 操作最知名的实现方式，但 Select 远不止 RAG。对话历史的筛选、动态 few-shot 示例的选取、用户画像信息的注入——这些都是 Select 操作。它们的共同特征是"从大量候选中选出相关子集"，不一定涉及向量检索。

**误区四："Compress 只在 token 不够用时才需要"**

即使 context window 没有满，Compress 也可能有价值。因为 context 越长，模型的注意力越分散（参考 M1 中学到的信息衰减效应）。适当压缩可以提高信息密度，让模型更聚焦于关键内容。

一个极端的例子：把 10 页的产品手册原文塞进 context，vs 提取出与用户问题相关的 5 个关键段落——后者的 context 更短，但模型回答质量往往更高。这就是 Compress（或者说 Compress + Select 的组合）的价值。
