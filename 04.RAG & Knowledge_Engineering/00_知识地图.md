# 00 · RAG 知识地图总览

---

## 开篇定位

**RAG 解决什么问题**
LLM 的"知识边界"问题：模型只知道训练数据里有的东西，但企业的产品手册、内部文档、最新故障记录它从没见过——RAG 让模型在回答时能看到这些私有知识。

**与 Prompt / Context Engineering 的边界**
你已经会的那套技术，核心是**设计送进 LLM 的内容**；RAG 要解决的是**这些内容从哪里来**——它是 context 的上游供给系统，两者是流水线上的前后工序。

**学完 RAG 之后会去哪里**
RAG 是取水的管道；Knowledge Engineering 是保证水质的水源治理。RAG 告诉你怎么把文档检索出来送进 LLM；Knowledge Engineering 告诉你这些文档本身应该怎么组织、结构化、维护，才能让 RAG 检索得更准。学完 RAG 有了工程感知再看，会自然理解它的每一个设计决策。

---

## 系统流程全景

RAG 是一条流水线，每个模块解决流水线上的一个具体问题：

```
原始文档（PDF / Word / HTML / Markdown）
       ↓
  [B] 文档处理与分块
       把文档切成可检索的片段，并打上元数据标签
       → 解决"文档怎么进系统"的问题
       ↓
  [A] 向量化（Embedding）
       把每个片段转成数字向量，存入向量数据库
       → 解决"怎么把文字变成可以计算相似度的形式"的问题
       ↓
  [C] 检索
       用户提问 → 转成向量 → 在库里找最相近的片段
       → 解决"怎么从库里找到相关内容"的问题
       ↓
  [D] Context 构建与注入
       把检索结果组装成 Prompt，送给 LLM
       → 解决"怎么把检索结果交给 LLM 用"的问题
       ↓
  [生成结果] LLM 基于注入的内容生成回答
```

模块 E（评估）横跨整条流水线，用于验证每一步是否工作正常。
模块 F/G 是在这条主链路跑通之后，针对具体瓶颈的进阶优化。

## RAG 知识地图

### 模块 A · 核心流程与基本概念 — Must Know

**RAG 整体流程**
知识点：理解"检索 → 注入 → 生成"三步闭环；没有这个全局视角，后续每个模块都是孤立的零件，不知道自己在做什么。
→ 术语注解：RAG = Retrieval-Augmented Generation（检索增强生成），大白话就是"先去仓库找资料，再让 LLM 用这些资料答题"。

**向量化（Embedding）**
知识点：把文字变成数字向量，才能做语义检索；缺少它，系统只能关键词匹配，"API 调用失败"和"接口报错"这两个意思相同的句子会被当成无关内容。
→ 术语注解：Embedding（嵌入），把一段文字映射成一串浮点数，语义相近的文字在这个数字空间里"住得近"。

**向量数据库**
知识点：存储和快速检索向量的专用数据库；没有它，每次提问都要把全部文档重新计算一遍相似度，百份文档就卡死了。
→ 术语注解：向量数据库（Vector DB），专门为"找最近邻居"设计的数据库，代表产品有 Chroma、Qdrant、Pinecone。

**相似度检索**
知识点：用提问的向量去向量库里找最相近的文档片段；这是整个 RAG 能"找到对的内容"的核心机制。
→ 术语注解：ANN 搜索（Approximate Nearest Neighbor，近似最近邻），不是精确找最像的，而是"足够快地找到足够像的"。

→ [**模块 A · 核心流程与基本概念→**](01_核心流程.md)

### 模块 B · 文档处理与分块 — Must Know

> **与你已有基础的关联**：分块粒度的设计，和你在 Context Engineering 里做的"信息粒度控制"是同一个问题——放进 context 的信息块应该多大？太大有噪声，太小失去上下文。区别在于 RAG 里这个决策在入库时就要做好，改起来代价更高。

**文档加载（Document Loading）**
知识点：把 PDF、Word、HTML、Markdown 等格式的文件统一转成可处理的纯文本；缺少它，产品手册进不了系统，一切无从谈起。

**文档分块（Chunking）**
知识点：把长文档切成合适大小的片段再 Embedding；分块策略直接影响检索质量——块太大会引入噪声，块太小会丢失上下文，故障排查步骤被切断一半，LLM 根本看不懂。
→ 术语注解：Chunk（块），文档被切分后的最小检索单元，类比 context window 里一条消息的粒度。
→ **本模块评估指标**：人工检查 Top-K 召回片段是否包含完整的语义单元；对照问题集看关键段落是否被切断。

**元数据管理（Metadata）**
知识点：给每个 chunk 打上来源文件名、章节、更新日期等标签；没有元数据，检索到内容后不知道是哪份手册说的，无法引用来源，也无法按条件过滤（比如"只查 2024 年之后的文档"）。

→ [**模块 B · 文档处理与分块 →**](02_文档切片.md)

### 模块 C · 检索策略 — Must Know

> **与你已有基础的关联**：你在 Context Engineering 里做"信息选择"——从多个候选内容里决定放哪些进 context；检索策略解决的是同一件事的上游版本——从文档库里决定候选集是什么。选择逻辑的好坏，都直接决定 LLM 最终看到什么。

**稠密检索 vs 稀疏检索**
知识点：语义向量检索（稠密）擅长意思相近的模糊匹配，BM25 等关键词检索（稀疏）擅长精确词命中；只用一种，要么找不到精确术语，要么找不到同义表达。
→ 术语注解：BM25，一种经典关键词排序算法，大白话是"看你问的词在文档里出现多少次、占比多少"。

**混合检索（Hybrid Search）**
知识点：把稠密和稀疏检索结果融合后再排序；在技术文档场景中，员工既可能用口语提问也可能直接搜函数名，混合检索两头都能接住。

**重排序（Reranking）**
知识点：检索到候选片段后，用更精准的模型对结果二次打分排序；没有 Reranking，向量检索返回的 Top-10 结果里相关性参差不齐，LLM 收到的 context 质量无法保证。
→ 术语注解：Reranker，二次排序模型，比向量检索慢但更精准，只用于对已经召回的少量候选做精排。
→ **本模块评估指标**：Recall@K（前 K 个结果里有几个真正相关）；MRR（最相关结果排在第几位）。

### 模块 D · Context 构建与注入 — Must Know（你的已有基础直接切入点）

**检索结果组装成 Prompt**
知识点：把检索到的 chunks 格式化、排序后拼进 Prompt 的正确位置；这正是你已有 Context Engineering 能力的直接应用场景——同样的检索结果，组装方式不同，LLM 回答质量可以天差地别。

**上下文窗口管理**
知识点：检索到的片段总长度可能超出模型 context window 限制，需要截断或压缩策略；不处理会直接报错，或者把最重要的内容挤出窗口。
→ 术语注解：Context Window（上下文窗口），你已经熟悉的概念，这里只是多了一个"动态填充"的上游供给问题。

**引用与溯源**
知识点：在回答中附上内容来自哪份文档哪个章节；技术文档场景里这是信任度的关键，没有来源员工不会相信机器人的答案。

→ [**进入模块 D · Context 构建与注入 →**](04_上下文注入.md)

### 模块 E · 评估 — Must Know，贯穿 B/C/D

> 评估不是学完所有模块后才做的事——每学一个模块，就应该知道怎么验证它是否工作正常。模块 E 的三个工具，分别对应流水线的三层。

**检索质量评估**
知识点：衡量"找到的内容是否真的相关"，独立于生成质量；检索坏了，LLM 再强也没用，必须能单独诊断。
→ 术语注解：Recall@K，在返回的前 K 个结果里，有几个是真正相关的，大白话是"找对了多少"。
→ **何时用**：学完 B/C 后立即建立，每次调整分块或检索策略都要重新跑。

**生成质量评估**
知识点：衡量 LLM 的回答是否忠实于检索到的内容，有没有幻觉；有了这个才知道问题出在检索层还是生成层。
→ 术语注解：Faithfulness（忠实度），回答里的每句话是否都能在检索到的 context 里找到依据。
→ **何时用**：学完 D 后建立，用于区分"检索没找对"和"LLM 没用好检索结果"这两类问题。

**端到端评估框架**
知识点：用测试集自动化评估整个 RAG 流水线；没有评估框架，每次改动只能靠人工感受，无法系统性迭代。
→ 术语注解：RAGAS，一个专门评估 RAG 系统的开源评估框架，能同时测检索和生成两个维度。
→ **何时用**：系统初步跑通后建立，之后每次优化都依赖这个基线数据做对比。

→ [**进入模块 E · 评估体系 →**](05_评估体系.md)

### 模块 F · 高级检索模式 — Nice to Know

> **依赖**：需要先掌握模块 C（知道基础检索的局限）和模块 E（有评估手段才能验证高级策略是否真的有效）。

**查询改写（Query Rewriting）**
知识点：用 LLM 把用户原始问题改写成更适合检索的形式；员工问"那个接口为什么老报 500"，直接检索效果很差，改写成"API 返回 500 错误原因"效果好得多。
→ **触发场景**：评估发现 Recall@K 低，但文档库里确实有相关内容——说明是问题表达方式和文档用语之间有落差。

**HyDE（假设文档嵌入）**
知识点：先让 LLM 虚构一个"理想答案"，用这个虚构答案去检索，而不是用原始问题；反直觉但效果通常更好。
→ 术语注解：HyDE = Hypothetical Document Embeddings，大白话是"先猜答案长什么样，用猜的答案去找真答案"。
→ **触发场景**：问题较短、意图模糊，直接检索效果差；尤其适合开放式的"为什么"类问题。

**多跳检索（Multi-hop Retrieval）**
知识点：需要跨多个文档拼接信息才能回答的问题；比如"产品 A 的认证流程"需要先找到认证标准文档，再找到产品 A 的技术规格，两份文档结合才能回答。
→ 术语注解：Multi-hop（多跳），每一跳检索一个文档，下一跳的问题由上一跳的结果动态生成，像侦探破案一样逐步推进。
→ **触发场景**：单次检索返回的内容不足以回答问题，LLM 回答时明显缺少中间推理步骤。

→ [**进入模块 F · 高级检索 →**](06_高级检索.md)

### 模块 G · 系统架构与工程化 — Nice to Know

> **依赖**：需要先跑通 A→B→C→D 的完整链路，有了真实系统才能理解这些工程问题的价值。

**索引更新策略**
知识点：文档更新后如何增量同步到向量库，不用全量重建；产品手册每周都在更新，全量重建不仅太慢，还会有一段时间窗口系统回答的是旧版本内容，员工拿着过期步骤去排查故障。
→ **触发场景**：文档库有频繁更新需求，或者发现系统在用已被废弃的旧文档内容回答。

**缓存机制**
知识点：对高频相似问题缓存检索结果或回答，降低延迟和成本；员工高峰期一起查同一个故障手册，每次都重新检索和调用 LLM，成本线性增长，响应也会变慢。
→ 术语注解：语义缓存（Semantic Cache），不是精确匹配问题字符串，而是"语义足够相近的问题复用同一个结果"。
→ **触发场景**：系统上线后发现大量重复问题，或延迟、成本超出预算。

**异步流水线与监控**
知识点：生产环境下将文档入库、检索、生成等步骤解耦为异步队列，并建立日志和可观测性；没有它，任何一步卡住都会阻塞整个请求，出了问题也找不到是哪步出的错。
→ **触发场景**：系统从原型进入生产，并发请求增加，或开始有 SLA 要求。

→ [**进入模块 G · 工程实践 →**](07_工程实践.md)

### 模块 H · GraphRAG 与结构化知识 — 暂时跳过

图数据库 + RAG 的组合，适合实体关系复杂的场景——比如"这个组件会影响哪些下游系统"这类需要沿关系边推理的问题。

**什么时候回来看**：当你遇到以下情况时，Multi-hop 检索已经不够用了——问题涉及"A 影响 B，B 影响 C"这样的链式关系，或者文档库里实体之间的关联比文档内容本身更重要。

## 学习顺序建议

```
A（核心流程）→ B（文档处理）→ C（检索策略）→ D（Context 构建）→ F（高级检索）→ G（工程化）
                                                 ↑
                              E（评估）贯穿 B/C/D，不是最后才学
```

**前置依赖关系**

- B 依赖 A：不理解整体流程，分块的"为什么这样切"无从判断
- C 依赖 B：没有好的分块，检索策略再好也是在垃圾里找垃圾
- D 依赖 A + C：你已有的 Context Engineering 能力在这里直接复用，但需要先理解检索结果长什么样
- E 贯穿 B/C/D：学 B 时建立分块评估，学 C 时建立检索评估，学 D 后建立端到端评估
- F 依赖 C + E：高级检索策略需要先知道基础检索的瓶颈在哪，且需要有评估手段验证改进效果
- G 依赖 A→D 全链路：工程化问题只有在系统真正跑起来之后才有意义

**你的已有基础直接切入点**

模块 D 是你进入 RAG 最无摩擦的入口——检索结果拼进 Prompt 这件事，和你已经在做的 context 组装本质相同，只是内容来源从手写变成了自动检索。建议学完 A/B/C 的基础概念后，在 D 这里放慢节奏深入打磨，这是 RAG 质量差异最容易出现的地方。

---

## 知识要点总结

- RAG 是 context 的上游供给系统：解决"内容从哪来"，不是"内容怎么写"
- 质量在流水线里逐层传导：分块差 → 检索差 → 注入差 → 答案差，问题总在下游暴露，根因在上游
- 故障诊断要逐层定位：答案不对不等于 Prompt 写得不好，先排查检索层
- 评估要贯穿学习过程：不是最后才做，而是每学一个模块就建立对应的评估视角
- 模块 D 是你的已有基础切入点：Context Engineering 的能力在这里直接复用
