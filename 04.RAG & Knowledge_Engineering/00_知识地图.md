# 00 · RAG 知识地图

---

## 开篇定位

**先从一个你绕不过去的问题开始**

你已经会用 Prompt Engineering 和 Context Engineering 让 LLM 给出高质量回答。但有一类需求，这套技术碰壁了：

```
用户："我们产品的 v3 API 鉴权方式改成什么了？"
用户："上周那个数据库连接超时是什么原因？"
用户："新员工入职需要申请哪些权限？"
```

这些答案在你的内部文档里，不在 LLM 的训练数据里。你不可能把几百份文档全部粘进每条 prompt——超 token、太贵、大量内容和当前问题无关。

RAG 是这个问题的工程解：**在 LLM 回答之前，自动从文档库里检索出相关内容，注入进 context。** 你熟悉的 Context Engineering 那一套还在用，只是 context 里的内容从手写变成了系统自动填充。

---

## 系统流程全景

要实现"按需检索注入"，需要两条运行时机不同的流水线。**离线流水线**在文档入库时运行，把原始文档处理成可检索的形式提前存好；**在线流水线**在用户每次提问时实时触发，检索相关片段并注入 context。用户感知到的只有在线阶段的延迟，离线阶段的处理成本在提问之前就已摊销完毕。

```
── 离线流水线（文档更新时运行）──────────────────────────

  原始文档（PDF / Word / HTML / Markdown）
         ↓
    [模块 02] 文档处理与分块
         把文档切成可检索的片段，并打上元数据标签
         ↓
    [模块 01] 向量化（Embedding）→ 存入向量数据库

── 在线流水线（用户每次提问时触发）──────────────────────

  用户提问
         ↓
    [模块 03] 检索
         问题向量化 → 在向量库里找最相近的片段
         ↓
    [模块 04] Context 构建与注入
         把检索结果组装成 Prompt，送给 LLM
         ↓
  LLM 基于注入的内容生成回答
```

模块 05（评估与调优）横跨两条流水线，用于验证每一步是否工作正常。
模块 06 / 07 是主链路跑通之后，针对具体瓶颈的进阶优化。

---

## 贯穿始终的核心问题

学习 RAG 的过程，本质上是在反复回答同一组问题，只是在流水线的不同节点上：

> **切多大？** — 分块粒度的选择，是 RAG 里影响最大、又最容易被忽视的决策。没有标准答案，只有权衡。
>
> **找什么？** — 语义相近不等于真正相关。检索到的内容能不能帮 LLM 答出好答案，是两件不同的事。
>
> **怎么用？** — 检索结果的排列方式、截断策略、引用格式，都会影响 LLM 最终的利用率。

带着这三个问题读每个模块，会比只记住概念有用得多。

---

## RAG 的两个常见误解

**误解一：RAG = 向量数据库**
向量库只是存储层，RAG 是一条完整的工程流水线。瓶颈往往不在向量库，而在分块策略和 Prompt 组装。

**误解二：答案不准 = Prompt 写得不好**
在 RAG 系统里，答案质量差可能出在流水线的任意一层：文档切坏了、检索没找对、LLM 没用好检索结果——三种根因，修法完全不同。评估体系就是用来区分这三种情况的。
