# 03 · 检索策略

> 模块 C — Must Know

---

## 本模块解决什么工程问题

文档已经分块存进向量库了。现在员工每次提问，系统要从几千个块里找出最相关的几个。

这件事比看起来难。两个典型场景说明问题：

**场景一：** 员工问"getAccessToken 这个方法的参数是什么"。这是精确查询，函数名就是答案的定位符，但向量语义检索可能把"获取令牌的方式"相关的块排在函数文档前面。

**场景二：** 员工问"连接数据库老是断开怎么办"。这是语义查询，文档里可能写的是"数据库连接超时问题排查"，没有一个词和提问完全重叠，关键词检索找不到。

**本模块的核心问题：单一检索方式有盲区，需要组合策略，以及在召回之后进一步提升精准度。**

---

## 知识点 1：稠密检索（Dense Retrieval）

### 它在做什么

就是模块 A 里介绍的向量相似度检索：把问题向量化，去向量库找余弦相似度最高的 K 个块。"稠密"指的是向量的每一个维度都有值（1536 维全部非零）。

**在贯穿案例里的优势：** 语义理解。员工问"接口认证怎么做"，能找到写着"Bearer Token 授权机制"的文档块，即使字面上没有重叠词。

**盲区：**
- 精确专有名词：`getAccessToken`、`HTTP 401`、产品型号 `XT-2000`——这些词在向量空间里没有特别的近邻，语义检索不可靠
- 罕见词汇：训练数据里很少出现的内部术语，Embedding 模型对它们的表达能力弱

---

## 知识点 2：稀疏检索（Sparse Retrieval）

### 它在做什么

基于词频统计的关键词检索。"稀疏"指的是每篇文档用一个巨大的词表向量表示，大多数维度是 0（这个词没出现），只有出现过的词的维度有值。

最经典的算法是 **BM25**。

→ **术语注解：BM25（Best Match 25）**，一种关键词排序算法。给定查询词，它综合考虑：该词在文档里出现几次（词频 TF）、这个词在所有文档里多常见（逆文档频率 IDF）、文档长度归一化。大白话：一个词在某篇文档里出现很多次，但在其他文档里很少出现，这篇文档就很可能是答案。

**在贯穿案例里的优势：** 精确词命中。员工搜 `getAccessToken`，BM25 直接找到包含这个函数名的文档块，不会被"获取令牌"的语义近邻干扰。

**盲区：**
- 语义理解为零：搜"连接数据库断开"找不到"数据库连接超时"，因为"断开"≠"超时"
- 同义词盲区：搜"报错"找不到只写了"异常"的文档

### BM25 评分公式（理解原理，不需要记住）

```
score(Q, d) = Σ IDF(qi) × TF(qi, d) × (k1 + 1) / (TF(qi, d) + k1 × (1 - b + b × |d|/avgdl))

其中：
- qi：查询里的每个词
- IDF：词的稀有度（越稀有分越高）
- TF：词在文档里的出现次数
- |d|/avgdl：文档长度归一化（避免长文档自动占优）
- k1, b：可调参数
```

## 知识点 3：混合检索（Hybrid Search）

### 它在做什么

同时运行稠密检索和稀疏检索，把两组结果合并后统一排序，取最终的 Top-K。

**在贯穿案例里：** 员工既可能用口语提问（语义查询），也可能直接输入函数名或错误码（精确查询）。混合检索让系统对两种提问方式都有良好表现。

### 结果融合：RRF 算法

两组检索结果的评分不在同一个量纲上（余弦相似度 vs BM25 分数），不能直接加权平均。常用 **RRF（Reciprocal Rank Fusion）** 来融合。

→ **术语注解：RRF（Reciprocal Rank Fusion，倒数排名融合）**，大白话是"只看每个结果的排名位置，不看原始分数"。在稠密结果里排第 3，在稀疏结果里排第 1，各自换算成一个倒数分再加总，综合排名。

```python
# 伪代码：RRF 融合
def rrf_merge(dense_results, sparse_results, k=60):
    scores = {}
    
    for rank, doc_id in enumerate(dense_results):
        scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)
    
    for rank, doc_id in enumerate(sparse_results):
        scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)
    
    # 按总分排序
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)
```

**k=60** 是经验值，控制排名靠后的结果对融合的影响程度。

### 实现混合检索的两种路径

**路径一：向量库原生支持**
Qdrant、Weaviate、Elasticsearch 都内置了混合检索，直接调用 API。

**路径二：手动实现**
分别查询，用 RRF 合并。适合向量库不支持混合检索的情况。

```python
# 伪代码
dense_results = vector_db.search(query_vector, top_k=20)
sparse_results = bm25_index.search(query_text, top_k=20)
final_results = rrf_merge(dense_results, sparse_results)[:5]
```

**在贯穿案例里的建议：** 技术文档场景强烈推荐混合检索。API 文档里有大量精确术语（函数名、参数名、错误码），必须有稀疏检索兜底；同时员工提问往往口语化，需要语义理解。两者缺一不可。

## 知识点 4：重排序（Reranking）

### 它在做什么

向量检索（或混合检索）返回的是基于向量相似度的粗排结果。相似度高不等于真正相关，Top-K 里可能有"语义相近但答非所问"的块。

Reranking 是第二道过滤：用一个更精准（但更慢）的模型，对粗排结果重新打分排序。

**两阶段检索流程：**

```
用户问题
    ↓
[粗排] 向量检索 / 混合检索，召回 Top-20 候选
    ↓
[精排] Reranker 模型对这 20 个候选重新打分
    ↓
取 Top-5，送入 Context 构建
```

→ **术语注解：召回（Recall）和精排（Rerank）**，这是信息检索的两阶段术语。召回阶段目标是"宁可多捞，不要漏掉"，精排阶段目标是"从候选里找最准的"。两阶段分工让系统兼顾效率和精度。

### 为什么 Reranker 更准

向量检索用的是双塔模型：问题和文档分别独立编码，再计算相似度。问题和文档从未"见过对方"。

Reranker 用的是交叉编码器（Cross-encoder）：把问题和文档**拼在一起**输入模型，模型能看到两者的细粒度交互，判断相关性更精准。

→ **术语注解：双塔模型（Bi-encoder）vs 交叉编码器（Cross-encoder）**
- 双塔：问题和文档分别编码 → 速度快，适合大规模检索
- 交叉编码器：问题+文档一起编码 → 精度高，但只能处理少量候选（因为慢）
- 所以：先用双塔大范围召回，再用交叉编码器精排少量候选，是标准组合。

**在贯穿案例里：**

员工问"怎么处理 API 返回 401"，向量检索 Top-5 可能返回：
1. [相似度0.88] "HTTP 状态码说明：401 表示未认证"
2. [相似度0.85] "认证流程：获取 access token 后在请求头..."
3. [相似度0.82] "错误处理：捕获 401 后自动刷新 token 的实现方式"  ← 真正有用
4. [相似度0.80] "API 版本说明：v2 与 v1 的认证差异"
5. [相似度0.78] "常见错误码汇总表"

Reranker 重排后：块3 因为直接回答"怎么处理"被排到第一，块1 虽然相似度最高但只是定义，被降序。

### 常用 Reranker 模型

| 模型 | 特点 |
|------|------|
| Cohere Rerank | 云 API，效果好，有中文支持 |
| BGE-Reranker（BAAI） | 开源，中文效果优秀，可本地部署 |
| cross-encoder/ms-marco | HuggingFace 上的经典模型，英文场景常用 |

**在贯穿案例里（中文文档场景）：** 优先考虑 BGE-Reranker，开源且中文表现强。

```python
# 伪代码：接入 Reranker
candidates = hybrid_search(query, top_k=20)   # 粗排召回 20 个

reranked = reranker.rerank(
    query=query,
    documents=[c.text for c in candidates],
    top_n=5                                    # 精排后取 Top-5
)

# reranked 是最终送入 Context 的内容
```

## 知识点 5：检索参数调优

### Top-K 怎么选

召回多少个候选，直接影响后续处理：

```
Top-K 太小（如 K=3）：
→ 可能漏掉真正相关的块
→ 尤其是问题需要多个文档片段综合回答时

Top-K 太大（如 K=30）：
→ 塞进 Prompt 会超出 context window
→ 即使有 Reranker，精排也会引入更多噪声
→ LLM 处理过多无关内容会"迷失"

推荐做法：
- 没有 Reranker：召回 K=5，直接送 Context
- 有 Reranker：粗排 K=20，精排后取 top_n=5
```

### 相似度阈值过滤

设置一个最低相似度阈值，低于阈值的块即使在 Top-K 里也丢弃，防止"凑数"的不相关结果被送进 Prompt。

```python
# 伪代码
results = vector_db.search(query_vector, top_k=10)
filtered = [r for r in results if r.score > 0.75]   # 过滤低相关结果

if len(filtered) == 0:
    return "未在文档库中找到相关内容"   # 兜底处理，避免 LLM 胡编
```

**在贯穿案例里这个兜底非常重要：** 员工问了一个超出文档覆盖范围的问题（比如问竞品信息），如果没有阈值过滤，系统会把最相关的几个块（其实不相关）送给 LLM，LLM 可能根据这些内容编造一个看似有根据的错误答案。

## 模块小结

检索策略的核心设计原则：**粗排求全，精排求准**。

| 阶段 | 工具 | 目标 |
|------|------|------|
| 粗排 | 混合检索（向量 + BM25） | 不漏掉相关内容 |
| 精排 | Reranker | 把真正有用的推到最前面 |
| 兜底 | 相似度阈值过滤 | 防止无关内容被送给 LLM |

**和下一个模块（D：Context 构建与注入）的衔接：**

Reranker 输出了最终的 Top-5 块。这些块是"原材料"，接下来的问题是：怎么把它们组装成高质量的 Prompt？

这里才是你的 Context Engineering 经验最直接发力的地方：这 5 个块的排列顺序重要吗？要不要加 system prompt 约束 LLM 的行为？怎么让 LLM 在回答时标注引用来源？这些决策的质量，决定了"找到了好内容"最终能不能转化成"生成了好回答"。
