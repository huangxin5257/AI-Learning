# 05 · 评估与调优

> 模块 E — Must Know

---

## 本模块解决什么工程问题

RAG 系统上线后，你做了一个改动——把 chunk size 从 512 改成了 256。系统"感觉"更准了，但你不确定。然后你又换了 Reranker 模型，再次"感觉"不错。

两周后，一个之前能答好的问题开始答错了。你不知道是哪个改动导致的，也不知道从哪里开始排查。

**这就是没有评估体系的代价。**

本模块解决的核心问题：**如何用数字衡量 RAG 系统的质量，以及当系统表现不好时，如何定位是检索层的问题还是生成层的问题。**

---

## 知识点 1：评估的两个维度

RAG 系统有两个独立的子系统，需要分别评估：

```
[检索系统]  ←→  [生成系统]
    ↓                ↓
检索了对的内容吗？   基于检索内容回答准确吗？
```

**为什么必须分开评估：**

假设系统回答了一个错误答案，可能有两种原因：
- 检索到了错误的文档块（检索层问题）
- 检索内容正确，但 LLM 误读或产生了幻觉（生成层问题）

两种原因的修复方式完全不同：前者需要优化检索策略或分块，后者需要调整 Prompt 或换模型。不分开评估，无法定位问题。

---

## 知识点 2：检索质量评估

### 核心指标：Recall@K 和 Precision@K

给定一个问题，人工标注哪些文档块是"真正相关的"（Ground Truth），然后看系统检索到的 Top-K 里有多少命中了。

→ **术语注解：Ground Truth（标准答案集）**，人工标注的"正确答案"，用于和系统输出对比评分。RAG 评估的地基，没有 Ground Truth 评估只能靠主观感受。

**Recall@K（召回率）：**
```
Recall@K = 命中的相关块数量 / 全部相关块总数量

示例：
问题"怎么处理 API 401 错误"，人工标注有 3 个相关块（块A、块B、块C）
系统 Top-5 检索到：块A、块X、块B、块Y、块Z（命中A和B，漏了C）
Recall@5 = 2/3 = 0.67
```

**Precision@K（精确率）：**
```
Precision@K = 命中的相关块数量 / 检索返回的 K 个块总数

同上示例：
Precision@5 = 2/5 = 0.40（5个里只有2个真正相关）
```

**两个指标的取舍：**
- 提高 K（召回更多）→ Recall 升，Precision 降
- 降低 K（召回更少）→ Precision 升，Recall 降
- 技术文档场景优先保证 Recall（不漏掉关键信息），用 Reranker 来保 Precision

### MRR：关心"最相关的块排第几"

→ **术语注解：MRR（Mean Reciprocal Rank，平均倒数排名）**，不只看命中多少，还看命中的排名。如果最相关的块排在第1位，得1分；排在第3位，得1/3分。大白话是"你有多快找到了最好的那个块"。

```
MRR = 平均（1 / 最相关块的排名位置）

问题1：最相关块排第2 → 1/2
问题2：最相关块排第1 → 1/1
问题3：最相关块排第4 → 1/4
MRR = (0.5 + 1 + 0.25) / 3 = 0.58
```

**在贯穿案例里：** MRR 对于技术文档场景很有意义——如果最相关的块被排到了 Top-5 的末尾（第5位），它在 Lost-in-Middle 效应下会被 LLM 忽视，等同于没找到。

## 知识点 3：生成质量评估

### 三个核心指标

#### Faithfulness（忠实度）

回答里的每个陈述，是否都能在检索到的文档块里找到依据？

```
问题：怎么获取 access_token？
检索内容：POST /auth/token 接口，需要传 client_id 和 client_secret
LLM 回答：调用 POST /auth/token，传入 client_id、client_secret 和 scope 参数

评估："scope 参数"在检索内容里没有提到 → 该句 Faithfulness = 0（幻觉）
总体 Faithfulness = 命中检索内容的陈述数 / 全部陈述数
```

**这是 RAG 最关键的指标**，因为 Faithfulness 低意味着系统在产生幻觉，对用户有误导性。

#### Answer Relevance（答案相关性）

LLM 的回答是否真正回答了用户的问题？（回答可能忠于文档，但答非所问）

```
问题：Token 过期了怎么办？
检索内容：Token 有效期 3600 秒；refresh_token 可以刷新 token
LLM 回答：Token 的有效期是 3600 秒，即一小时后会过期。

分析：回答忠于文档（Faithfulness=1），但没有回答"过期了怎么办"（Relevance 低）
```

#### Context Recall（上下文召回率）

LLM 生成回答时，是否利用了检索到的所有相关信息？（避免检索到了正确内容，但 LLM 没用上）

## 知识点 4：端到端评估框架——RAGAS

手动评估每个指标太耗人力。RAGAS 是一个自动化评估 RAG 系统的开源框架，用 LLM 来评估 LLM。

→ **术语注解：RAGAS（RAG Assessment）**，专为 RAG 系统设计的评估框架，核心思路是"用 LLM 作为评判者"来打分，不需要大量人工标注。

### RAGAS 的评估流程

```
输入：
  - question：用户的问题
  - answer：RAG 系统生成的回答
  - contexts：检索到的文档块列表
  - ground_truth：人工标注的标准答案（可选，用于部分指标）

RAGAS 输出：
  - faithfulness：0-1，LLM 作为裁判检验回答是否忠于检索内容
  - answer_relevancy：0-1，LLM 评估回答是否切题
  - context_recall：0-1，需要 ground_truth
  - context_precision：0-1，检索内容里有多少是真正有用的
```

```python
# 伪代码
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_recall

dataset = [
    {
        "question": "怎么处理 API 401 错误",
        "answer": system_answer,          # RAG 系统的输出
        "contexts": retrieved_chunks,     # 检索到的块
        "ground_truth": "应检查 token 是否有效，使用 refresh_token 刷新..."
    },
    ...  # 更多测试问题
]

result = evaluate(dataset, metrics=[faithfulness, answer_relevancy, context_recall])
print(result)
# Output: {'faithfulness': 0.85, 'answer_relevancy': 0.79, 'context_recall': 0.72}
```

### 构建测试集

RAGAS 能自动评分，但需要一批测试问题。在贯穿案例里构建测试集的方法：

**方法一：让 LLM 基于文档生成测试问题**
```python
# 伪代码
for chunk in sample_chunks:
    questions = llm.generate(f"""
    基于以下文档内容，生成3个员工可能会问的技术问题：
    {chunk.text}
    只输出问题，每行一个。
    """)
```

**方法二：收集真实提问**
系统上线后把员工的真实提问收集起来，人工标注标准答案，形成持续更新的测试集。

**在贯穿案例里建议：** 先用方法一快速生成 50-100 个测试问题，覆盖主要文档类型（API 文档、故障排查、产品手册），建立评估基线。系统上线后逐步用方法二替换。

## 知识点 5：问题定位与调优流程

### 指标组合诊断

```
case 1: Faithfulness 低，Answer Relevance 高
→ 检索内容对的，但 LLM 没有忠实于文档，在胡编
→ 修复方向：加强 System Prompt 约束；换更听话的模型

case 2: Faithfulness 高，Answer Relevance 低
→ LLM 忠实于文档，但检索到的内容不是问题真正需要的
→ 修复方向：检索策略有问题，回到模块 C 优化

case 3: Context Recall 低
→ 相关内容存在于文档库，但检索没找到
→ 修复方向：分块策略（模块 B）或检索策略（模块 C）有问题

case 4: Context Precision 低
→ 检索到的内容里大量无关内容，稀释了真正有用的信息
→ 修复方向：Reranking 不够强，或 Top-K 设置太大
```

### 系统性调优流程

```
建立基线测试集（50-100个问题）
          ↓
运行 RAGAS，得到各指标的基线分数
          ↓
做一个改动（比如：chunk_size 512→256）
          ↓
再次运行 RAGAS，对比前后数字
          ↓
只有数字变好了，才确认保留这个改动
          ↓
记录改动日志，持续迭代
```

这个流程的关键是：**每次只改一个变量**，否则无法判断是哪个改动起了作用。

→ **术语注解：A/B 测试（A/B Testing）**，同时运行两个版本（A = 旧策略，B = 新策略），对比指标。在 RAG 调优中，可以用同一批测试集分别跑两个版本，不需要真实流量分流。

## 知识点 6：人工评估的位置

自动化评估（RAGAS）高效，但有局限：用 LLM 评估 LLM，存在系统性偏差。对于以下情况，仍需人工评估：

- **专业术语准确性**：LLM 评判者可能不懂你们公司的内部术语，给出错误的 Faithfulness 分
- **回答的可用性**：技术上正确但格式混乱、难以阅读的回答，自动评估给高分但用户体验差
- **边缘 case**：文档覆盖不到的问题，LLM 回答"文档中未找到"是对的，但自动评估可能打低分

**在贯穿案例里的建议：** 每次迭代用 RAGAS 跑自动评估（快，每次改动后 10 分钟内有结果），每两周做一次人工评审（找 2-3 名工程师，抽查 20 个问题的回答质量）。两者结合。

## 模块小结

评估不是 RAG 的终点，是持续改进的引擎。

**最小可行评估体系：**
1. 建一个 50 题的测试集（覆盖主要文档类型）
2. 配置 RAGAS，跑出 Faithfulness + Answer Relevance 两个核心指标
3. 每次改动前后对比数字，建立改动日志

**和下一个模块（F：高级检索模式）的衔接：**

有了评估体系，你会发现一类问题：某些问题检索不准，不是因为文档没有内容，而是因为**用户的提问方式和文档里的写法差异太大**。

比如员工问"那个认证的东西要怎么用"，这么模糊的问题直接去检索几乎找不到好内容。

模块 F 的高级检索技术——查询改写、HyDE、多跳检索——就是专门解决这类"提问到检索之间的鸿沟"问题的。
