## 来自模块 01：核心流程与基本概念

### 知识点 1：RAG 的两条流水线

- [ ] **离线流水线（Indexing Pipeline）**
  → 提前把所有文档加载、切块、向量化，存入向量库。在用户提问之前完成，只在文档更新时重跑。缺少它，系统只能在用户提问时临时处理文档，延迟不可接受。

- [ ] **在线流水线（Query Pipeline）**
  → 用户每次提问时实时触发：问题向量化 → 向量检索 → 检索结果拼入 Prompt → LLM 生成回答。这条流水线的最后两步（拼 Prompt + LLM）就是 Context Engineering。

---

### 知识点 2：Embedding（向量化）

- [ ] **Embedding 的作用**
  → 把文字转成固定长度的数字数组（向量），语义相近的文字在向量空间里位置相近，使得"语义检索"成为可能。没有 Embedding，只能做关键词匹配。

- [ ] **离线和在线必须用同一个 Embedding 模型**
  → 不同模型的向量空间不同，混用会导致相似度计算完全失去意义，检索结果随机乱序。

---

### 知识点 3：向量数据库

- [ ] **向量数据库的作用**
  → 存储大量向量，并能快速回答"给定查询向量，找最相似的 K 个"（最近邻搜索）。普通数据库需要逐一比较，数万条记录时慢到不可用。

- [ ] **ANN vs 精确搜索**
  → 向量库用近似最近邻（ANN）算法（如 HNSW），牺牲极少精度换取数百倍速度提升。工程上足够用。

- [ ] **工具选型**
  → 原型用 Chroma（轻量本地）；生产环境用 Qdrant（开源自部署，支持元数据过滤）；已有 PostgreSQL 的团队可用 pgvector。

---

### 知识点 4：相似度度量

- [ ] **余弦相似度**
  → 衡量两个向量方向是否一致（不管长度），是 Embedding 场景最常用的度量方式。用错度量（如用欧氏距离但模型推荐余弦）会明显降低检索质量。按 Embedding 模型文档推荐的度量方式用即可。

---

### 知识点 5：在线流程的三个易出问题的节点

- [ ] **三个关键节点**
  → ①问题向量化（问题太短/口语化则向量代表性差）；②Top-K 大小（太小漏内容，太大塞不进 context）；③Context 排列（最相关内容放哪个位置影响 LLM 利用率）。

---

## 来自模块 02：文档处理与分块

### 知识点 1：文档加载（Document Loading）

- [ ] **文档加载的作用**
  → 把 PDF、Word、HTML、Markdown 等不同格式统一转成纯文本，才能进行后续向量化。没有这一步，非纯文本文档无法进入知识库。

- [ ] **各格式的核心坑**
  → PDF 扫描件需 OCR；Word 表格容易丢失；HTML 需过滤导航栏/页脚等 boilerplate；Markdown 最干净可直接处理。

- [ ] **工程实践建议**
  → LangChain 和 LlamaIndex 都内置了各种格式的 Document Loader，原型阶段直接用，不需要自己写解析器。

---

### 知识点 2：文档分块（Chunking）

- [ ] **分块的本质目的**
  → 把长文档切成适合检索的最小单元。块太大：一个向量代表不了全部内容，检索失真；块太小：单句无完整语义，LLM 看到残缺信息。

- [ ] **策略一：固定长度分块（Fixed-size Chunking）**
  → 按字符数/token 数硬切，加 overlap 防止知识单元被切断。实现最简单，但完全不考虑语义边界。

- [ ] **策略二：语义分块（Semantic Chunking）**
  → 按自然语言边界切（句子/段落/章节标题）。块内语义更完整，但段落长短不一，块大小不均匀。

- [ ] **策略三：递归分块（Recursive Chunking）**
  → 先按大分隔符（章节）切，切出的块还太大再按小分隔符（段落→句子）继续切。兼顾语义完整性和大小可控，是目前最常用的通用策略。LangChain 的 `RecursiveCharacterTextSplitter` 即此策略。

- [ ] **策略四：结构感知分块（Structure-aware Chunking）**
  → 利用文档固有结构切分：Markdown 按标题层级、代码按函数边界、HTML 按标签层级。每个块是一个结构完整的单元（如一个完整 API 端点说明）。

---

### 知识点 3：Chunk 大小的工程决策

- [ ] **Chunk Size 的选取原则**
  → 没有通用最优值，需结合文档类型和下游使用方式决定：叙述性文档 500-800 tokens；结构化文档按边界切；FAQ 一问一答 100-300 tokens。

- [ ] **Overlap 的作用**
  → 相邻块共享一段内容，防止连续的知识单元（如操作步骤 1-5）被切断后，检索到的块只包含后半部分，LLM 回答不完整。

---

### 知识点 4：元数据管理（Metadata）

- [ ] **元数据的作用**
  → 给每个 chunk 附加描述性标签（来源文件、章节、类型、更新时间等），检索时用于过滤，回答时用于引用来源，文档更新时用于精准删除。

- [ ] **Pre-filtering vs Post-filtering**
  → Pre-filtering：先用元数据筛掉不符合条件的向量再做相似度搜索（效率高，推荐）。Post-filtering：搜完再筛，可能导致返回数量不足。

---

### 知识点 5：父子块策略（Parent-Child Chunking）

- [ ] **父子块策略的核心思路**
  → 用小块（100-200 tokens）做向量检索（相关性更精准），检索命中后送给 LLM 的是对应的大块（500-1000 tokens，上下文更完整）。解决"检索精准"和"生成完整"的两难。

---

### 知识点 6：工程实践之多策略切块

- [ ] **决策一：双级主切点（H2 → H3）**
  → 以 H2 作为章节边界主切点；H2 块超过大小上限时，再以 H3 为次切点细切。保证沿文档结构走，不硬截。

- [ ] **决策二：代码块保护**
  → 处于代码块（` ``` `）内时跳过标题检测，避免代码注释中的 `##` 或 ASCII 图里的 `---` 触发错误切点。

- [ ] **决策三：面包屑上下文注入（Breadcrumb Injection）**
  → 在每个 chunk 文本开头注入层级路径（如 `[产品手册 > 第3章 > 3.2节]`），使 chunk 脱离原文件后仍有完整定位信息，LLM 知道这段内容属于哪个产品的哪个功能。

- [ ] **决策四：双粒度并存索引**
  → 同一文档同时生成粗粒度（H2 整体）和细粒度（H3 子节）两套 chunks，分别打标签存入向量库。粗粒度适合"整体流程"查询，细粒度适合"具体细节"查询。

- [ ] **决策五：丰富元数据（h1/h2/h3/breadcrumb 字段）**
  → 每个 chunk 存储 h1/h2/h3/breadcrumb/granularity 等结构字段，支持按模块过滤检索、自动生成引用定位、精准删除某章节的所有 chunks。

---

## 来自模块 03：检索策略

### 知识点 1：稠密检索（Dense Retrieval）

- [ ] **稠密检索的原理与盲区**
  → 向量相似度检索，能理解语义（"接口认证"能匹配"Bearer Token 授权机制"）。盲区：精确专有名词（函数名、错误码、型号）在向量空间没有特殊近邻，语义检索不可靠。

---

### 知识点 2：稀疏检索（Sparse Retrieval / BM25）

- [ ] **BM25 的原理与盲区**
  → 基于词频统计的关键词检索，能精确命中函数名、错误码等专有名词。盲区：语义理解为零，"断开"找不到"超时"，同义词完全无效。

---

### 知识点 3：混合检索（Hybrid Search）

- [ ] **混合检索的必要性**
  → 同时运行稠密检索和稀疏检索，合并结果。技术文档场景必须两者结合：口语提问需要语义理解，函数名/错误码需要精确匹配。

- [ ] **RRF 融合算法**
  → 两组结果的分数量纲不同，不能直接加权平均。RRF（倒数排名融合）只看排名位置不看原始分数，把两组排名换算成倒数后相加，统一排序。

---

### 知识点 4：重排序（Reranking）

- [ ] **Reranking 的两阶段流程**
  → 粗排（向量/混合检索）先召回 Top-20 候选，精排（Reranker 模型）重新打分取 Top-5。粗排求全，精排求准。

- [ ] **为什么 Reranker 比向量检索更准**
  → 向量检索用双塔模型，问题和文档独立编码；Reranker 用交叉编码器，把问题和文档拼在一起输入模型，能看到细粒度交互，判断相关性更精准，但速度慢，只适合精排少量候选。

- [ ] **中文场景的 Reranker 选型**
  → BGE-Reranker（BAAI 开源）中文效果优秀，可本地部署；Cohere Rerank 云 API，也支持中文。

---

### 知识点 5：检索参数调优

- [ ] **Top-K 的设置建议**
  → 没有 Reranker 时召回 K=5 直接用；有 Reranker 时粗排 K=20，精排后取 top_n=5。

- [ ] **相似度阈值过滤**
  → 低于阈值的块即使在 Top-K 里也丢弃，并返回"未找到相关内容"。防止无关内容被送给 LLM 后产生凭空编造的错误回答。

---

## 来自模块 04：Context 构建与注入

### 知识点 1：RAG Prompt 的基本结构

- [ ] **四个组成部分**
  → [System Prompt（角色+约束）] + [检索到的文档块（动态注入）] + [对话历史] + [用户当前问题]。其中文档块部分由检索系统自动填充，其余和普通 Prompt 设计一致。

- [ ] **System Prompt 的核心约束**
  → 必须明确要求"只基于文档内容回答，文档没有就说没有"。没有这个约束，LLM 会在文档覆盖不到的问题上用通用知识填充，产生幻觉但看起来有根据。

---

### 知识点 2：文档块的排列策略

- [ ] **Lost in the Middle 现象**
  → LLM 对 context 里首尾的内容利用率最高，中间位置的内容容易被忽视。

- [ ] **排列建议**
  → 最相关的块放最后（紧贴用户问题，利用近因效应）；次相关的块放最前；相关性一般的块放中间。实践上优先试相关性升序排列（最相关在最后）。

---

### 知识点 3：Context Window 管理

- [ ] **Token 预算分配**
  → 可用于文档块的空间 = context window - system_prompt - 对话历史 - 问题 - 预留回答空间。文档块是最大的动态变量，超限时按相关性从低到高截断。

- [ ] **三种超限处理策略**
  → ①按相关性截断（简单但损失信息）；②块内容压缩（用摘要模型先压缩每个块）；③动态调整 Top-K（根据当前对话长度反算能放几个块）。

---

### 知识点 4：引用与溯源

- [ ] **三种引用实现方式**
  → ①Prompt 约束引用（最简单，在 System Prompt 要求标注来源）；②结构化输出引用（要求 LLM 输出 JSON，把引用和内容分开，方便程序处理）；③后处理高亮（前端把回答句子和检索块做匹配，高亮可溯源部分）。

- [ ] **引用溯源在技术文档场景的价值**
  → 工程师不会盲信机器人；有来源标注时，回答出错可以快速定位是检索问题（找到了错误内容）还是生成问题（LLM 幻觉），修复方向完全不同。

---

### 知识点 5：多轮对话的 Context 管理

- [ ] **多轮对话的两个核心问题**
  → ①检索用什么问题：追问往往缺乏上下文，需要结合历史改写成完整的独立检索查询；②历史保留多少：建议最近 3-5 轮，更早的历史价值有限且挤占文档块空间。

---

## 来自模块 05：评估与调优

### 知识点 1：评估的两个维度

- [ ] **检索层和生成层必须分开评估**
  → 回答错误的原因可能是检索到了错误内容（修复方向：优化检索/分块），也可能是检索内容正确但 LLM 幻觉（修复方向：调 Prompt/换模型）。混在一起无法定位问题。

---

### 知识点 2：检索质量评估指标

- [ ] **Recall@K（召回率）**
  → 系统 Top-K 结果中命中了多少个"真正相关的块"（相对于人工标注的全集）。技术文档场景优先保 Recall，不能漏掉关键信息。

- [ ] **Precision@K（精确率）**
  → Top-K 结果里有多少是真正相关的。Recall 和 Precision 存在权衡：K 越大 Recall 升 Precision 降，用 Reranker 来在高召回基础上保精确。

- [ ] **MRR（平均倒数排名）**
  → 衡量最相关的块排在第几位。技术文档场景中，最相关块排到 Top-5 末尾会因 Lost-in-Middle 效应被 LLM 忽视，等同于没找到。

---

### 知识点 3：生成质量评估指标

- [ ] **Faithfulness（忠实度）**
  → 回答里的每个陈述是否都能在检索文档里找到依据。这是 RAG 最关键的指标，低 Faithfulness 意味着系统在产生幻觉，对用户有误导性。

- [ ] **Answer Relevance（答案相关性）**
  → 回答是否真正回答了用户的问题。回答可能忠于文档（Faithfulness 高）但答非所问（Relevance 低）。

- [ ] **Context Recall（上下文召回率）**
  → LLM 生成回答时是否利用了检索到的所有相关信息，检验"检索到了但 LLM 没用上"的情况。

---

### 知识点 4：RAGAS 自动评估框架

- [ ] **RAGAS 的核心思路**
  → 用 LLM 作为评判者，自动对 Faithfulness、Answer Relevance、Context Recall 等指标打分。输入：问题 + 系统回答 + 检索块 + 标准答案（可选）。

- [ ] **测试集构建方法**
  → ①用 LLM 基于文档块自动生成测试问题（快速冷启动）；②收集真实用户提问并人工标注（持续迭代）。建议先用方法①生成 50-100 题建立评估基线。

- [ ] **RAGAS 的冷启动成本**
  → RAGAS 本身调用 LLM API 打分，跑一次 100 题测试集大约消耗数千 tokens（约 $0.1-0.5）。可以用更便宜的模型（如 GPT-4o-mini）做评判者降低成本，效果略差但够用。

---

### 知识点 5：问题定位与调优

- [ ] **指标组合诊断**
  → Faithfulness 低 → LLM 幻觉，调 Prompt 或换模型；Answer Relevance 低 → 检索内容不对，回到检索策略；Context Recall 低 → 相关内容没被检索到，看分块或检索；Context Precision 低 → 检索结果噪声多，加强 Reranking。

- [ ] **调优的基本原则**
  → 每次只改一个变量，改前后分别跑 RAGAS 对比数字，只有数字变好才确认保留改动并记录日志。

---

## 来自模块 06：高级检索模式

### 知识点 1：查询改写（Query Rewriting）

- [ ] **问题补全（Query Expansion）**
  → 用 LLM 把口语化、模糊的用户问题改写成标准技术查询，去除噪声、补全术语，再去检索。投入产出比最高的高级检索优化，应优先做。

- [ ] **多查询生成（Multi-Query）**
  → 一个问题生成多个不同角度的查询，分别检索后合并去重。覆盖文档里不同的表达方式，扩大召回范围。

- [ ] **对话历史感知改写**
  → 多轮对话中，把当前追问结合历史改写成一个不依赖上下文就能独立理解的完整查询，再去检索。

---

### 知识点 2：HyDE（假设文档嵌入）

- [ ] **HyDE 的核心思路**
  → 不用问题去检索，而是先让 LLM 生成一个"假设答案"，用假设答案的向量去检索。原因：假设答案的向量风格和文档块更接近，相似度更高，检索更准。

- [ ] **HyDE 的适用范围**
  → 适合抽象/口语化问题；不适合精确查询（函数名、错误码），假设答案可能包含错误术语反而干扰检索。可与普通检索并行，用 RRF 合并结果。

---

### 知识点 3：多跳检索（Multi-hop Retrieval）

- [ ] **多跳检索解决的问题**
  → 需要跨多个文档片段推理才能回答的问题（如"v3 和 v2 认证流程有什么变化"）。每跳检索后让 LLM 判断信息是否充足，不够则生成下一跳的查询。

- [ ] **多跳检索的代价**
  → 延迟高（每跳一次 LLM + 检索）、错误累积（第一跳偏了后续越走越偏）、实现复杂。建议先不做，只在评估发现"跨文档综合"问题比例超 20% 时引入。

---

### 知识点 4：查询路由（Query Routing）

- [ ] **查询路由的思路**
  → 用 LLM 对问题分类（精确查询/语义查询/跨文档/超出范围），不同类型路由到不同检索策略。识别"超出范围"直接拒答、精确查询走 BM25，是收益最明显的简单路由。

---

### 知识点 5：Self-RAG

- [ ] **Self-RAG 与标准 RAG 的区别**
  → 标准 RAG 无论问题是否需要检索都强制走完整流程；Self-RAG 让 LLM 自主判断是否需要检索、检索结果是否相关、回答是否有依据。适合"通用知识问题不需要检索文档"的场景，实现复杂度较高。

---

## 来自模块 07：系统架构与工程化

### 知识点 1：索引更新策略

- [ ] **增量更新 vs 全量重建**
  → 全量重建耗时长、期间不可用、重复计算大量未变更文档。增量更新只处理新增/修改/删除的文档，前提是每个 chunk 元数据里有 `source_file` 字段，才能按来源精准删除旧块。

- [ ] **文档变更检测**
  → 方法一：对每个文件计算哈希，和上次存储的哈希对比；方法二：接入 Confluence/SharePoint 等平台的 Webhook 监听变更事件，比定时轮询更高效。

---

### 知识点 2：缓存机制

- [ ] **精确匹配缓存**
  → 对问题文本做 MD5 作为 key，完全相同的问题直接返回缓存结果，绕过检索和 LLM 调用。实现简单，但同义问法不同命中率低。

- [ ] **语义缓存**
  → 对问题做 Embedding，在缓存向量库里找语义相似的历史问题（相似度超阈值即命中）。能覆盖同义问法，GPTCache 是开源实现。建议先上精确缓存，命中率低于 30% 再考虑语义缓存。

---

### 知识点 3：异步处理与并发

- [ ] **异步架构的必要性**
  → 同步处理在并发场景下延迟线性累积。异步架构（如 Celery + Redis）让用户提问后立即返回任务 ID，Worker 池并行处理，用户轮询或 WebSocket 获取结果。

- [ ] **流式输出（Streaming）**
  → LLM 支持流式返回，用 SSE 或 WebSocket 一边生成一边推送，减少用户等待感，不需要等完整回答生成完才显示。

---

### 知识点 4：可观测性与监控

- [ ] **必须记录的日志字段**
  → 原始问题、改写后查询、检索到的 chunks（含来源和分数）、Reranker 重排顺序、LLM 模型和 token 消耗、各阶段延迟分解、最终回答、用户反馈。没有完整链路日志，RAG 出问题无法定位是哪个环节。

- [ ] **关键监控指标**
  → 性能类：P95 延迟（各阶段分解）、token 消耗、缓存命中率；质量类："未找到相关内容"比例（过高说明检索有问题）、用户满意度、定期 RAGAS 评分。

## 好分块的三条标准（模块 02 小结）

- [ ] **自包含** → 单独看这一块能理解，不依赖其他块补充上下文
- [ ] **大小合适** → 不超出 context 预算，又不过于碎片化
- [ ] **有元数据** → 知道来自哪里，可以被过滤和引用
