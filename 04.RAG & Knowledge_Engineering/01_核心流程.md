# 01 · 核心流程与基本概念

> 模块 A — Must Know

---

## 本模块解决什么工程问题

回到贯穿案例：员工问"我们的 OAuth 2.0 接口如何获取 access token"，这个问题的答案藏在某份 API 文档的第 3 章第 2 节里。

问题来了——LLM 没有读过你公司的内部 API 文档。它只有训练时见过的通用知识，无法准确回答基于私有文档的具体问题。

RAG 的核心流程就是解决这一个问题：**在 LLM 回答之前，自动找到文档里的相关内容，作为 context 送进去**。

本模块的目标是建立对整个流水线的全局视角。不求细节，求的是"每个零件在整台机器里做什么"。

---

## 知识点 1：RAG 的两条流水线

RAG 系统有两条独立的流水线，分别在不同时间运行，必须先分清楚。

### 离线流水线（Indexing Pipeline）

**它在做什么：** 把所有文档提前处理好，存进一个可以快速检索的仓库。这一步在员工提问之前就已经完成。

**在贯穿案例里：** 把数百份产品手册、API 文档、故障排查记录，全部切分、向量化，存进向量数据库。这个过程可能跑一次需要几分钟到几小时，但之后不会重跑（除非文档更新）。

**缺少它会怎样：** 员工第一次提问时才开始处理文档，等待时间不可接受，系统无法使用。

```
[文档文件] → 加载解析 → 切分成块 → Embedding → 存入向量库
   PDF/Word                  chunks        向量         Vector DB
```

→ **术语注解：Index（索引）**，这里不是数据库的索引，而是"为检索做的预处理产物"。把文档处理好存进向量库这个动作，叫做"建索引"。

---

### 在线流水线（Query Pipeline）

**它在做什么：** 员工每次提问时实时触发。把问题向量化，去仓库里找最相关的内容，拼进 Prompt，送给 LLM 生成回答。

**在贯穿案例里：** 员工问"OAuth 接口怎么用"，系统在 1-2 秒内找到 API 文档里相关的 3 个段落，拼进 Prompt，LLM 基于这 3 个段落生成回答。

**缺少它会怎样：** LLM 没有原材料，只能靠训练数据里的通用知识猜答案，大概率不准确或者虚构细节。

```
[员工提问] → 问题向量化 → 向量检索 → 检索结果 → 拼 Prompt → LLM → [回答]
                  ↑                        ↑
              Embedding模型           向量数据库
```

**和你的已有经验衔接：** 在线流水线的最后两步——"拼 Prompt → LLM"——就是你已经在做的 Context Engineering。RAG 只是在这之前多了一个"自动取材料"的过程。你可以理解为：RAG = 自动取材料系统 + 你已会的 Context 组装技巧。

## 知识点 2：Embedding（向量化）

### 它在做什么

把一段文字转换成一个固定长度的数字数组（比如 1536 个浮点数）。这个数组在数学空间里有一个位置，**语义相近的文字，它们的数组在空间里位置相近**。

**在贯穿案例里：**
- "如何获取 access token" → [0.23, -0.81, 0.44, ...]（1536 个数字）
- "OAuth 认证流程" → [0.21, -0.79, 0.47, ...]（位置非常接近）
- "产品外观设计规范" → [0.91, 0.12, -0.63, ...]（位置很远）

检索时，用问题的向量去仓库里找"位置最近"的文档块，就找到了语义相关的内容。

**缺少它会怎样：** 退回到关键词匹配。员工问"接口认证怎么做"，关键词检索可能找不到写着"OAuth 2.0 授权流程"的文档段落——因为字面上没有重叠的词。

→ **术语注解：Embedding Model（嵌入模型）**，专门负责把文字转成向量的模型，和负责生成回答的 LLM 是两个不同的模型。常见的有 OpenAI 的 `text-embedding-3-small`、开源的 `bge-m3` 等。

### 关键实现思路

离线阶段和在线阶段必须用**同一个** Embedding 模型，否则向量空间对不上，检索结果会完全混乱。

```
离线：文档块 → Embedding模型A → 向量 → 存库
在线：问题   → Embedding模型A → 向量 → 去库里找

❌ 错误：
离线用模型A，在线用模型B → 两批向量在不同的空间，相似度计算没有意义
```

## 知识点 3：向量数据库

### 它在做什么

存储大量向量，并能快速回答"给我一个向量 Q，找出库里最相似的 K 个向量"这个查询。这个操作叫 **最近邻搜索**。

**在贯穿案例里：** 向量库里存了数百份文档切成的数千个块，每个块一个向量。员工提问时，用问题的向量去库里执行一次最近邻搜索，几十毫秒内返回最相似的 5 个块。

**缺少它会怎样：** 用普通数据库或内存存向量，每次查询都要把问题向量和库里的每一个向量逐一计算相似度——几千个块还勉强，十万个块就慢到不可用。

→ **术语注解：ANN（Approximate Nearest Neighbor，近似最近邻）**，向量数据库不是精确找最近的，而是用特殊的索引结构（比如 HNSW）快速找到"足够近"的。牺牲一点点精度，换来成百上千倍的速度提升。

→ **术语注解：HNSW（Hierarchical Navigable Small World）**，一种常见的向量索引算法，大白话是"建了一个分层的'社交网络'，找邻居时从高层跳到低层，越来越精确"。你不需要实现它，知道它是向量库快速检索的底层机制就够了。

### 主流选择对比

| 工具 | 定位 | 适合场景 |
|------|------|---------|
| Chroma | 轻量本地 | 原型开发、小规模 |
| Qdrant | 开源自部署 | 生产环境，需要控制数据 |
| Pinecone | 云服务 | 不想管基础设施 |
| pgvector | PostgreSQL 插件 | 已有 PG 数据库的团队 |

**在贯穿案例里的建议：** 原型阶段用 Chroma 跑通流程，生产部署考虑 Qdrant（开源、支持元数据过滤、性能好）。

## 知识点 4：相似度度量

### 它在做什么

定义两个向量"有多近"的计算方式。向量库在做最近邻搜索时，需要一个数学公式来打分。

**在贯穿案例里：** 每次员工提问，系统要给库里几千个块打分，取得分最高的 Top-K 个。打分用的就是相似度公式。

**最常用的：余弦相似度**——只看两个向量的方向是否一致，不管长度。对于 Embedding 来说，这通常是最合适的，因为文字语义的相近程度体现在方向上而不是长度上。

```
cosine_similarity(问题向量, 文档块向量) = 两向量夹角的余弦值
值域：[-1, 1]，越接近 1 表示越相似
```

→ **术语注解：余弦相似度（Cosine Similarity）**，大白话是"两个向量指向差不多的方向吗"。两支箭头都朝右，角度接近，相似度高；一个朝右一个朝上，相似度低。

**缺少正确的相似度度量会怎样：** 大多数情况下 Embedding 模型的文档会直接告诉你用哪种度量，按推荐用就行。用错了（比如用欧氏距离但模型推荐余弦），检索质量会明显下降。

## 知识点 5：完整的在线流程串联

把以上概念串成一次完整的问答过程：

```
员工输入："我们的 API 鉴权方式是什么？"
                    ↓
          [Embedding 模型]
          把问题转成向量 Q
                    ↓
          [向量数据库]
          用余弦相似度找最近的 K=5 个块
          返回：
            - chunk_id: 42 (API文档第3章, 相似度0.89)
            - chunk_id: 18 (FAQ第2条, 相似度0.85)
            - chunk_id: 91 (故障排查记录, 相似度0.71)
            - ...
                    ↓
          [Context 构建] ← 你已有的 Context Engineering 在这里发力
          把 5 个块拼成 Prompt：
          "根据以下文档内容回答问题：
           [文档1内容...]
           [文档2内容...]
           问题：我们的 API 鉴权方式是什么？"
                    ↓
          [LLM]
          生成回答，附带来源标注
                    ↓
          "根据 API 文档第3章，我们使用 Bearer Token 方式鉴权..."
```

**这条流程里最容易出问题的节点：**

1. **问题 → 向量化**：如果问题太短或太口语化，向量代表性差，后续检索就偏了
2. **向量检索 → Top-K**：K 取多少？太小可能漏掉相关内容，太大 context 塞不下
3. **Context 构建**：多个块怎么排序？重要的块放哪个位置？（这是模块 D 的核心）

这三个节点各对应后续的一个模块（F 的查询改写、C 的检索策略、D 的 Context 构建）。

## 模块小结

本模块建立的是**整体认知框架**：两条流水线（离线建索引 + 在线检索生成），四个核心组件（Embedding、向量库、相似度检索、Context 组装）。

**和下一个模块（B：文档处理与分块）的衔接：**

离线流水线里有一步被我们一笔带过了——"切分成块"。这一步看起来简单，实际上是 RAG 系统里最影响最终效果、又最容易被忽视的环节。

问题是：文档不能整篇塞进去，必须切小。但怎么切？按换行切？按句子切？按段落切？切多大？相邻块要不要重叠？

这些决策会直接决定：检索的时候能不能找到正确的内容，以及找到之后 LLM 能不能看懂。模块 B 就是回答这些问题的。
