学完理论之后，真正动手时往往卡在两个地方：一是不知道从哪个文件开始写；二是知道概念，但遇到具体决策点（比如"切块要不要注入面包屑？"）仍然没有判断依据。

## 完整实现流水线

══════════════════════════════════════════════════════════════════

**三个文件、两个阶段、一条数据流**。离线阶段的产物是 Qdrant 向量库；在线阶段每次提问都经过"召回 → 精排 → 组装 → 生成"四步。

## 系统概览

整个系统由三个文件组成，各自负责 RAG 流水线的一个阶段：

```
indexer.py          →  离线阶段：读文档 → 切块 → 生成 Embedding → 存入向量库
retriever.py        →  检索阶段：向量召回 → CrossEncoder 精排
main.py             →  在线阶段：组装 Prompt → 调用 LLM → 管理多轮对话
```

**开发顺序即阅读顺序**：先把文档索引进向量库（indexer），再写检索接口（retriever），最后接上对话主程序（main）。每一步都可以独立验证，不用等整个系统跑通才能看到结果。

| 文件 | 对应模块 | 可独立验证的方式 |
|------|---------|----------------|
| indexer.py | 模块 01、02 | 运行后看 chunk 数量和向量库大小 |
| retriever.py | 模块 03 | 直接 `python retriever.py "你的问题"` 看召回结果 |
| main.py | 模块 04 | 交互式问答，`--debug` 可看完整检索链路 |

## 第一步：建立索引（indexer.py）

### 读取文档

```python
def read_md_files(docs_dir: Path) -> list[tuple[Path, str]]:
    results = []
    for md_file in sorted(docs_dir.rglob("*.md")):
        text = md_file.read_text(encoding="utf-8")
        results.append((md_file, text))
    return results
```

**关键决策：`rglob` 递归扫描 + `sorted` 保证顺序**。文档库往往有子目录分模块存放，`rglob` 一次扫完；`sorted` 让每次重建索引的 chunk_index 顺序稳定，便于排查问题。

### 切块：四个关键决策

模块 02 讲了分块策略的理论，这里是它落地时真正需要做的四个具体决策。

#### 决策一：代码块保护

```python
in_code_block = False
for line in lines:
    # 检测 ``` 开关，避免把代码块内的 # 误判为标题
    if line.strip().startswith("```"):
        in_code_block = not in_code_block

    if not in_code_block and line.startswith(prefix):
        # 遇到真正的标题：切点
        ...
```

**问题背景**：技术文档里的代码示例经常包含 `# 注释` 或者 Markdown 风格的标题行。不加保护，这些内容会被误识别为切点，把一段完整的代码示例切成两半。TEMS 文档里有大量 API 示例和配置片段，不处理这个问题会产生大量碎片 chunk。

**实现原理**：用一个布尔变量 `in_code_block` 追踪当前是否在代码块内。遇到 ` ``` ` 就取反，只有在代码块外面才判断是否是标题行。

---

#### 决策二：面包屑注入（Breadcrumb Injection）

```python
def _append_chunk(chunks, h1, h2, h3, body):
    # 面包屑：把层级路径拼成一句话，跳过空层级
    crumb_parts = [p for p in [h1, h2, h3] if p]
    breadcrumb = " > ".join(crumb_parts)

    # 在正文前注入标题前缀，让 embedding 包含位置信息
    heading_prefix = f"# {h1}\n## {h2}\n" + (f"### {h3}\n" if h3 else "")
    content = heading_prefix + body
```

**问题背景**：一个 chunk 被切出来之后，它就脱离了原始文档的上下文。想象一段只有 200 字的正文，内容是"支持三种认证方式：API Key、OAuth 2.0、JWT"——这段话脱离文档后，你完全不知道它说的是哪个系统哪个版本的哪个接口。向量模型对这种孤立内容的编码质量很低，检索时也容易错位。

**解决方案**：在每个 chunk 的正文前，注入一段"面包屑"：`Invoice Loader > 执行流程 > 步骤三`。这段文字虽然短，但让向量模型在编码时能感知到这个 chunk 在文档中的位置，显著提升语义准确度。同时 `breadcrumb` 字段也存入元数据，供前端展示引用路径。

---

#### 决策三：双粒度并存（H2 整体 + H3 细粒度）

```python
if len(h3_sections) > 1:
    # 有多个 H3：每个 H3 独立成 chunk（细粒度，适合"具体步骤"类查询）
    for h3_title, h3_content in h3_sections:
        if len(h3_content.strip()) >= MIN_CHUNK_LEN:
            _append_chunk(chunks, h1, h2_title, h3_title, h3_content)

    # 同时保留 H2 整体 chunk（粗粒度，适合"整体流程"类查询）
    _append_chunk(chunks, h1, h2_title, "", h2_content)
else:
    # 无 H3 或只有一个：保持完整，不强制切断
    _append_chunk(chunks, h1, h2_title, "", h2_content)
```

**问题背景**：用户问题有两种粒度。"API 鉴权的整体流程是什么？"——这个问题需要完整的 H2 章节作为答案。"OAuth 2.0 的 token 刷新步骤怎么做？"——这个问题只需要 H2 下某个 H3 小节的内容。如果只切到 H2，细节问题的检索精度不够；如果只切到 H3，整体问题的检索命中率不够。

**解决方案**：两种粒度的 chunk 同时入库。这会让总 chunk 数增加，但向量库的检索本来就是 approximate nearest neighbor，多一些候选反而让精排阶段有更多选择空间，代价可接受。

---

#### 决策四：大内容不强制切断

```python
else:
    # 无 H3（如大表格、ASCII 流程图）：保持完整，不强制切断
    _append_chunk(chunks, h1, h2_title, "", h2_content)
```

**问题背景**：技术文档里的表格（比如接口参数对照表）和 ASCII 流程图（比如系统架构示意图），它们的信息密度很高，而且内部结构无法用普通文本边界来切分。强制切断一张参数表，上半部分说"参数名 / 类型"，下半部分说"是否必填 / 描述"，两段单独来看都是废话。

**解决方案**：当一个 H2 块没有子 H3 时，直接整体保留，哪怕超过 MAX_CHUNK_SIZE 限制。这会产生较大的 chunk，但大 chunk 的检索精度没有小 chunk 高，这个代价是可以接受的——表格和流程图本来就是整体性内容，需要完整看。

### Embedding 与写入向量库

```python
# 参数选择
EMBED_MODEL_NAME = "paraphrase-multilingual-MiniLM-L12-v2"  # 384 维，支持中英混合
EMBED_DIM = 384
BATCH_SIZE = 64

# 全量重建：先删旧 collection，再写入
if client.collection_exists(COLLECTION_NAME):
    client.delete_collection(COLLECTION_NAME)

client.create_collection(
    collection_name=COLLECTION_NAME,
    vectors_config=VectorParams(size=EMBED_DIM, distance=Distance.COSINE),
)

# 分批 upsert，避免一次性占用过多内存
embeddings = model.encode(texts, batch_size=BATCH_SIZE, show_progress_bar=True)
```

**三个工程决策：**

1. **模型选型**：`paraphrase-multilingual-MiniLM-L12-v2` 支持中英混合，TEMS 文档中文为主但有大量英文技术术语，这个模型在两种语言下的语义理解都够用。384 维向量比 1536 维小很多，检索速度更快，存储开销更小，在文档规模不超过百万 chunk 的场景下是更务实的选择。

2. **全量重建策略**：先删除旧 collection 再重建，而不是增量更新。原因：文档更新时老 chunk 的 ID 可能失效，增量更新需要额外的变更检测逻辑（文件哈希比对或 Webhook），复杂度高。全量重建在文档规模小（百份以内）的情况下够用，也是快速验证阶段的最简做法。大规模场景再考虑增量更新（参见模块 08 工程实践）。

3. **分批 upsert**：`BATCH_SIZE=64` 控制每次写入的数据量，避免大文档集合时内存溢出。

## 第二步：构建检索器（retriever.py）

### 两阶段检索：召回 → 精排

```python
RECALL_K = 10    # 向量召回数量（粗筛）
RERANK_K = 5     # 精排后最终返回数量

# 阶段一：向量召回，快但粗糙
query_embedding = embed_model.encode(query).tolist()
recall_results = client.query_points(
    collection_name=COLLECTION_NAME,
    query=query_embedding,
    limit=RECALL_K,           # 召回 10 个候选
)

# 阶段二：CrossEncoder 精排，慢但准确
pairs = [(query, chunk_text) for chunk_text in texts]
rerank_scores = rerank_model.predict(pairs)  # 逐对计算相关性
ranked = sorted(zip(rerank_scores, candidates), key=lambda x: x[0], reverse=True)[:RERANK_K]
```

**为什么是两阶段而不是直接向量检索取 Top-5？**

向量检索（Bi-encoder）是把问题和文档分别编码成向量，用余弦距离衡量相似度——速度极快，但精度有限，因为问题和文档是独立编码的，没有交叉注意力。

CrossEncoder 精排是把"问题 + 文档"拼在一起送进模型，让模型同时看到两段文字来判断相关性——精度更高，但计算量是 O(n)，不能对全库使用。

两阶段的设计是一个经典的精度/效率权衡：向量检索快速筛出 10 个候选（召回阶段），CrossEncoder 对这 10 个候选精排取 Top-5（精排阶段）。精排只对小候选集运行，总延迟可控。

**RECALL_K=10 这个数字怎么来的？** 它是精排输入数量（10）和最终返回数量（5）之间的放大系数。放大系数越大，精排的选择空间越大，但精排计算量也增加。10 是常用起点，如果发现精排后的 Top-5 质量仍然差，可以把 RECALL_K 调大到 20。

## 第三步：组装问答系统（main.py）

### System Prompt：三种回答模式

```python
SYSTEM_PROMPT = (
    "**模式 A — 文档命中**\n"
    "当参考文档中包含相关信息时：直接基于文档内容作答，末尾标注 [来源: 文档]\n\n"

    "**模式 B — 文档未命中，模型补充**\n"
    "当文档无相关信息，但属于通用技术知识时：基于模型知识作答，"
    "开头声明 [来源: 模型知识，非文档内容，请注意核实]\n\n"

    "**模式 C — 无法回答**\n"
    "当问题超出文档和模型可靠知识范围时：仅返回 [无法回答]\n"
)
```

**为什么要显式定义三种模式，而不是让 LLM 自己决定？**

如果 Prompt 里只说"基于文档回答"，LLM 在文档没命中时会有两种倾向：要么直接用自身知识回答但不说明来源，要么说"文档中没有相关内容"然后停止。这两种行为都不是用户想要的——用户希望模型能做知识补充，但同时需要明确的来源标注来区分"文档依据"和"模型推断"。

三模式设计解决了这个问题：用结构化的行为约束替代模糊的指令，LLM 只需要判断问题属于哪种情况，然后按规则执行。**`[来源: ...]` 标注**是另一个关键细节——它让调用方（或前端）可以用字符串匹配来判断这条回答的置信程度，无需额外的后处理逻辑。

---

### RAG Prompt 组装

```python
def build_user_message(query: str, chunks: list[dict]) -> str:
    context_parts = []
    for i, chunk in enumerate(chunks, start=1):
        context_parts.append(f"[片段{i}，来源：{chunk['source']}]\n{chunk['text']}")
    context = "\n\n".join(context_parts)
    return f"参考内容：\n{context}\n\n用户问题：{query}"
```

**格式设计的两个细节：**

1. **`[片段{i}，来源：{chunk['source']}]` 标注**：每个片段前注明序号和来源文件。LLM 在回答时可以引用"片段 2 显示…"，用户也能直接定位到原始文档。

2. **`参考内容` 和 `用户问题` 分段**：把检索内容和用户问题用语义标签隔开，而不是直接拼接在一起。这让 LLM 更容易识别哪部分是"参考资料"、哪部分是"需要回答的问题"，减少混淆。

### 对话历史管理：只存裸问答

```python
def ask(query, history, debug=False):
    chunks = retrieve(query, top_k=TOP_K)

    # user_message 包含检索内容（很长）
    user_message = build_user_message(query, chunks)

    # messages 组装：system + 历史 + 当前轮（含检索内容）
    messages = [
        {"role": "system", "content": SYSTEM_PROMPT},
        *history,                                          # history 只存裸问答
        {"role": "user", "content": user_message},        # 当前轮含检索内容
    ]

    answer = call_llm(messages)

    # 追加到 history 时，只存裸问题，不存 user_message（不含检索内容）
    history.append({"role": "user", "content": query})
    history.append({"role": "assistant", "content": answer})

    # 超出窗口时丢弃最早的一问一答
    if len(history) > MAX_HISTORY_TURNS * 2:
        del history[:2]
```

**`history` 为什么只存裸问答，不存 `user_message`？**

`user_message` 包含 3 个检索片段，每个片段可能有几百个 token。如果每轮都把完整的 `user_message` 存进 history，5 轮对话后 history 里就积累了大量重复的文档内容——这些内容对后续轮次的 LLM 没有参考价值，但会快速耗尽 context window。

正确做法：只把裸问题和回答存入 history，让 LLM 理解对话上下文；每一轮独立执行检索，把最新的相关文档注入当前轮的 `user_message`。这样 history 的 token 消耗是 O(轮数 × 问答长度)，而不是 O(轮数 × 问答长度 × 检索片段大小)。

**滑动窗口** `del history[:2]` 从列表头部删除最早的一问一答（2 条记录），保持 history 总条数不超过 `MAX_HISTORY_TURNS * 2`。

## Context 组装全貌：一次真实请求长什么样

上面的代码分散在几个函数里，不直观。下面把一次完整问答中发给 LLM 的 `messages` 数组展开，让你看到 System Prompt、对话历史、检索片段、用户问题，四种东西是怎么拼在一起的。

**场景**：用户已经问过一轮，现在问第二个问题"OAuth token 过期了怎么刷新？"，系统检索到 2 个相关片段。

```
messages = [

  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  role: "system"
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  content:
    ## 角色定义
    你是 TEMS 技术文档助手，基于用户提供的参考文档准确回答技术问题。

    ## 回答规则
    模式 A — 文档命中：直接作答，末尾标注 [来源: 文档]
    模式 B — 文档未命中：基于模型知识作答，开头声明 [来源: 模型知识，请注意核实]
    模式 C — 无法回答：仅返回 [无法回答]

    ## 输出约束
    回答简洁结构化，每次必须包含来源标注。

**对照代码看这个结构：**

- `SYSTEM_PROMPT` → 第一条 `system` 消息，每轮不变

## 知识要点总结

**1. 开发顺序决定调试效率**
indexer → retriever → main，每个文件独立可验证。不要等整个系统跑通才看第一个结果。

**2. 切块的四个决策都有代价**
代码块保护避免碎片；面包屑注入提升孤立 chunk 的语义质量；双粒度并存增加 chunk 数但覆盖两类查询；大内容不强制切断牺牲精度换完整性。每个决策都是权衡，不是"最优解"。

**3. 两阶段检索的本质是精度/效率分离**
向量检索（快/粗）→ CrossEncoder 精排（慢/准）。`RECALL_K` 是精排的候选池大小，直接影响精排上限。如果精排结果仍然差，先调大 `RECALL_K`，再考虑换更好的精排模型。

**4. history 存什么决定多轮对话的 token 成本**
`history` 只存裸问答，不存检索内容。RAG 的检索是无状态的，每轮重新检索，不需要把上一轮的文档片段带入下一轮。

**5. System Prompt 的三模式设计替代了模糊指令**
结构化行为约束（A/B/C 三模式 + 强制来源标注）让 LLM 的输出可预测、可解析，而不是让它"自由发挥"然后你再猜它依据的是文档还是模型知识。
