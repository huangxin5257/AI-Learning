# M5 — 质量评估与调试

> **核心问题：怎么判断 context 的质量，出了问题怎么定位和修复**
> **与 M3 的关系：** M3 的可测试性/可调试性是事前设计原则，M5 是事后评估与诊断的具体方法。
> **学习目标：** LLM 输出不符合预期时，能系统性地排查是 context 的问题还是模型的问题，并定位到具体环节。

---

## 5.1 Context 失效的典型症状识别

**一句话：** LLM 输出的每一种"不对劲"，背后几乎都能追溯到 context 中某个具体部分的缺失、冲突或失真——学会从症状反推病因，是 context 调试的第一步。

### 是什么

当你把一段 context（系统提示、用户指令、背景资料、对话历史等）送入 LLM，模型基于这些信息生成回答。如果输出不符合预期，问题的根源只有两大类：要么是 context 本身有问题（该给的没给、给的有矛盾、给的方式不对），要么是模型能力的边界（即使 context 完美，模型也做不到）。实践中，绝大多数问题属于前者。

症状识别，就是建立一张"输出异常 → context 归因"的映射表。就像医生看到发烧不会只说"你发烧了"，而是通过体温、伴随症状来判断是感冒还是感染——我们看到 LLM 输出异常时，也要通过症状特征来判断是 context 的哪个环节出了问题。

以下是一张系统性的症状归因表：

**症状一：幻觉（输出中包含虚构的事实、不存在的数据、编造的引用）**

| 维度 | 说明 |
|------|------|
| 典型表现 | 模型引用了不存在的论文、编造了具体的统计数字、虚构了 API 参数 |
| 最可能的 context 病因 | 1) context 中缺少回答所需的事实性材料，模型被迫"猜测"填充；2) context 中没有明确的"不知道就说不知道"指令；3) 提供的参考资料不够具体，只有模糊描述而非精确数据 |
| 排查方向 | 检查 context 中是否真的包含了回答这个问题所需的全部事实。如果你自己读完 context 也无法回答这个问题，那模型产生幻觉就是预期行为 |

**症状二：指令遗忘（明确写了要求，模型却没遵守）**

| 维度 | 说明 |
|------|------|
| 典型表现 | 你要求"用 JSON 输出"但模型给了纯文本；你要求"不要解释原因"但模型还是解释了一堆；你列了 5 条规则但模型只遵守了前 3 条 |
| 最可能的 context 病因 | 1) 指令淹没在大量文本中，位置不突出（放在了中间而非开头或结尾）；2) 指令之间存在隐性冲突（"详细回答"和"控制在 100 字以内"同时出现）；3) context 太长，指令被截断或落在注意力衰减区；4) 指令表述模糊，存在多种合理解读 |
| 排查方向 | 把所有指令单独提取出来，检查是否有矛盾；检查被遗忘的指令在 context 中的位置；尝试把被遗忘的指令移到 context 的最前面或最后面，看是否改善 |

**症状三：格式错乱（输出结构不符合要求）**

| 维度 | 说明 |
|------|------|
| 典型表现 | 要求 Markdown 表格但输出了列表；要求 JSON 但 JSON 格式不合法；要求分三段但模型分了五段 |
| 最可能的 context 病因 | 1) 格式要求只用自然语言描述而没有给出具体示例；2) context 中包含的示例与格式要求不一致；3) 之前的对话历史中模型用了另一种格式，产生了惯性 |
| 排查方向 | 检查是否提供了明确的输出格式示例（few-shot example）；检查对话历史中是否有格式冲突的先例 |

**症状四：角色混乱（模型在不同身份之间切换或打破角色设定）**

| 维度 | 说明 |
|------|------|
| 典型表现 | 你设定模型扮演客服，但模型突然开始讨论自己是 AI 的事实；角色设定的语气不稳定，一会儿正式一会儿随意 |
| 最可能的 context 病因 | 1) 系统提示中的角色设定过于简短或模糊；2) 用户消息中包含了诱导模型脱离角色的内容，而 context 中没有抵抗这种诱导的指令；3) 角色设定与后续指令存在矛盾（设定为"简洁的技术专家"但又要求"详细解释每个细节"） |
| 排查方向 | 审视系统提示的角色定义是否足够具体；检查是否有显式的"边界规则"（遇到什么情况应该如何应对） |

**症状五：上下文遗失（模型表现得像是"忘记了"之前说过的话）**

| 维度 | 说明 |
|------|------|
| 典型表现 | 多轮对话中，模型对前几轮讨论过的内容不再记得；长文档处理时，模型只回应了文档的后半部分而忽略前半部分 |
| 最可能的 context 病因 | 1) 对话历史被截断，早期对话已经不在当前 context window 里了；2) context 总长度接近或超过模型的上下文窗口限制；3) 关键信息埋在大量无关内容之间，被"稀释"了 |
| 排查方向 | 统计当前 context 的 token 数，确认是否接近上限；检查对话历史管理策略——是全量保留还是有截断/摘要机制 |

**症状六：推理错误（逻辑链条断裂或结论与前提矛盾）**

| 维度 | 说明 |
|------|------|
| 典型表现 | 模型列出了正确的事实，但得出了错误的结论；多步推理中某一步跳跃了逻辑 |
| 最可能的 context 病因 | 1) context 中缺少引导分步推理的指令（如"请先分析A，再分析B，最后综合判断"）；2) 提供的信息量超出了单次推理能处理的合理范围；3) context 中的信息组织方式让模型难以建立正确的逻辑关系（相关信息分散在不同位置） |
| 排查方向 | 检查是否提供了推理步骤的脚手架；尝试把相关信息集中放在一起；尝试把复杂推理拆分为多次调用 |

**症状七：过度安全/拒绝回答（模型不必要地拒绝了合理请求）**

| 维度 | 说明 |
|------|------|
| 典型表现 | 模型说"我无法提供这方面的建议"但其实是一个完全正常的业务问题；模型在技术问题上过度添加免责声明 |
| 最可能的 context 病因 | 1) 系统提示中的安全限制写得过于宽泛；2) 用户问题的表述无意中触发了模型的安全策略；3) context 中缺少清晰的业务场景说明，让模型无法判断请求的合理性 |
| 排查方向 | 在系统提示中添加明确的业务场景说明和合理使用范围；调整用户问题的措辞；检查安全指令是否误伤了正常功能 |

### 为什么重要

假设你正在开发一个内部知识问答系统。产品上线后，用户反馈说"回答经常不准确"。如果你不会做症状归因，你面对的就是一个模糊的"不好用"。你可能会本能地去换更大的模型、加更多的数据，但这些都可能是无效投入。

而如果你学会了症状识别，你会先看具体的"不准确"是什么形式：是编造了不存在的产品功能（幻觉——检索到的文档不匹配）？是回答了用户没问的问题（指令理解偏差——系统提示不清晰）？还是前后回答矛盾（上下文遗失——对话历史管理有 bug）？

不同的症状指向完全不同的修复路径。症状识别的价值不在于分类本身，而在于它能让你的调试从"随机尝试"变成"定向排查"。

### 设计启示

在设计 context 时，可以预防性地考虑每种症状的主要诱因：

- **针对幻觉预防**：确保 context 中包含足够的事实性材料，并加入"如果提供的资料中不包含答案，请明确说明"的兜底指令。
- **针对指令遗忘预防**：关键指令放在 context 的开头或结尾（注意力最强的位置），避免超过 5 条以上的平级规则同时出现。
- **针对格式错乱预防**：永远提供一个具体的输出示例，而不仅仅是文字描述。
- **针对上下文遗失预防**：在设计阶段就规划好 context 的 token 预算（详见 5.2），而不是等超限了再处理。

### 常见误区

- **误区一："输出不好就是模型不行"。** 很多时候换模型不如改 context。一个设计良好的 context 在小模型上的表现，可能优于一个设计粗糙的 context 在大模型上的表现。在归咎模型之前，先穷尽 context 层面的排查。
- **误区二："加更多信息就能解决幻觉"。** 盲目往 context 里塞更多文本，如果信息不相关或组织混乱，反而会加重问题。关键不是"更多"而是"更准确、更相关"。
- **误区三："症状只有一个原因"。** 实际中，一个输出异常往往是多个 context 问题的叠加。比如一个"格式错乱"可能同时是格式示例缺失 + token 超限导致的指令截断。要学会多因素排查。

## 5.2 Token 预算规划与监控

**一句话：** Token 是 context 的硬通货——每个模型的上下文窗口就是你的总预算，怎么分配这些 token 给系统提示、用户输入、检索资料和对话历史，直接决定了系统的表现上限。

### 是什么

每个 LLM 都有一个上下文窗口（context window），通常以 token 数来衡量。Token 是模型处理文本的基本单位，粗略地说，一个中文字大约是 1-2 个 token，一个英文单词大约是 1-1.5 个 token。

以一个 128K token 上下文窗口的模型为例，这 128K 就是你的总预算。你送入模型的所有内容——系统提示（system prompt）、用户当前输入、检索到的参考资料、历史对话记录——加在一起不能超过这个上限。更关键的是，模型生成的输出也要占用一部分预算，所以你能用于输入的 token 数还要再减去预留给输出的空间。

Token 预算规划，就是在系统设计阶段就明确：总预算中，每个组成部分可以分到多少 token，当某个部分超标时如何处理。这和做财务预算是一样的道理——你不会等到月底钱花光了才想该怎么分配。

**一个具体的预算规划示例：**

假设你要建一个客服问答系统，使用的模型上下文窗口是 128K token：

```
总预算：128,000 tokens
├── 输出预留：4,000 tokens（预留给模型生成回答）
├── 实际可用输入预算：124,000 tokens
│   ├── 系统提示（固定）：2,000 tokens
│   │   ├── 角色设定与行为规则：800 tokens
│   │   ├── 输出格式要求：400 tokens
│   │   └── 安全规则与边界：800 tokens
│   ├── 用户当前输入：1,000 tokens（通常够一条消息）
│   ├── 检索资料（动态）：8,000 tokens
│   │   ├── 产品知识库文档：最多 3 篇 × 2,000 tokens
│   │   └── 常见问题匹配：最多 2 条 × 1,000 tokens
│   ├── 对话历史（动态）：6,000 tokens
│   │   └── 最近 N 轮对话，按时间倒序保留
│   └── 安全缓冲：107,000 tokens（未使用的余量）
```

你可能注意到上面有大量的"安全缓冲"。这是因为在大多数应用中，你不需要也不应该填满整个上下文窗口。128K 是上限，不是目标。实际上，很多研究表明模型在处理较短的 context 时表现更好——信息密度越高，模型的注意力越集中。

**那么对于上下文窗口较小的模型或更复杂的场景，预算规划就真正成为一个约束性问题：**

```
紧凑预算示例（8K token 模型）：
总预算：8,000 tokens
├── 输出预留：1,500 tokens
├── 实际可用输入预算：6,500 tokens
│   ├── 系统提示（固定）：800 tokens（必须精简）
│   ├── 用户当前输入：500 tokens
│   ├── 检索资料（动态）：3,000 tokens（最多 1-2 篇精选文档）
│   └── 对话历史（动态）：2,200 tokens（最近 3-5 轮）
```

在这种紧凑预算下，每个部分的分配都需要认真考虑，超标任何一个部分都会挤压其他部分的空间。

### 为什么重要

Token 预算问题之所以关键，是因为它的失败模式是"静默的"——系统不会报错，只是表现变差。

一个真实的场景：你开发了一个文档问答系统，测试阶段表现良好。上线后，随着用户对话轮次增多，对话历史不断膨胀。某天一个用户在第 20 轮对话时发现回答质量骤降——模型开始忽略系统提示中的格式要求，检索到的文档内容也只引用了一半。

排查后发现：对话历史已经占了 80% 的 token 预算。系统提示虽然还在 context 中，但被挤到了一个注意力衰减的位置。检索到的三篇文档，第三篇因为 token 超限被截断了，而关键信息恰好在被截断的部分。

这个问题的根源不是模型能力不够，也不是检索不准确，而是没有做 token 预算规划——没有设定对话历史的上限，没有截断策略，没有监控机制。

### 设计启示

**预算分配的基本原则：**

1. **固定部分优先保障。** 系统提示是你的"宪法"，必须确保它永远完整地出现在 context 中，不被其他部分挤压。先把固定部分的 token 数确定下来，剩余空间再分配给动态部分。

2. **为每个动态部分设定硬上限。** 对话历史最多保留 N 轮或 X 个 token；检索资料最多 K 篇或 Y 个 token。当接近上限时，有明确的截断策略（摘要、丢弃最旧的、只保留最相关的）。

3. **监控实际消耗。** 在代码中加入 token 计数逻辑，记录每次 API 调用中各部分的实际 token 消耗。这可以用 tiktoken（OpenAI 的分词库）等工具来实现。

4. **留出安全余量。** 不要把预算分配到 100%。留出 10-15% 的缓冲空间，应对用户输入异常长、检索结果异常多等边界情况。

**对话历史管理的具体策略：**

- **滑动窗口法：** 只保留最近 N 轮对话。简单直接，但可能丢失重要的早期信息。
- **摘要压缩法：** 当对话历史超过阈值时，用 LLM 把早期对话压缩成一段摘要，放在对话历史的开头。保留信息密度但引入摘要误差。
- **关键信息提取法：** 从历史对话中提取关键实体和决策（如"用户选择了方案 A"、"用户的预算是 5 万"），以结构化格式保留。信息密度最高，但实现复杂度也最高。

**Token 监控的实践做法：**

```
每次 API 调用时记录：
├── 系统提示 token 数（应该是稳定的常数）
├── 用户输入 token 数
├── 检索资料 token 数
├── 对话历史 token 数
├── 总输入 token 数
├── 输出 token 数
└── 总 token 数 / 上限 的比例（利用率）
```

当你有了这些数据，就能建立告警规则：比如当总 token 利用率超过 80% 时触发告警，当对话历史 token 数超过预算的 150% 时自动触发摘要压缩。

### 常见误区

- **误区一："模型支持 128K，那就尽量填满"。** 上下文窗口越大，模型处理的计算量越大（成本更高、延迟更长），而且在超长 context 中模型的注意力分布不均匀，中间部分的信息容易被忽略（"Lost in the Middle"现象）。更大的窗口是保险，不是鼓励你塞满它。
- **误区二："token 计数不需要太精确"。** 粗略估算在日常开发中够用，但在生产系统中，误差可能导致 context 截断。不同模型的分词方式不同，中英混合文本的 token 数尤其难以凭感觉估算，应该用对应模型的分词工具来精确计数。
- **误区三："只关注输入 token，忽略输出预留"。** 如果你把 128K 的预算全部分配给输入，模型没有空间生成足够长的回答，输出会被截断。必须为输出预留合理的空间。不同任务需要的输出长度差异很大——简单问答可能只需要 200 token，生成长文可能需要 4000+。

## 5.3 Context 对输出质量的因果归因方法

**一句话：** 当 LLM 输出不符合预期时，通过"只改变 context 的一个部分、保持其他一切不变"来精确定位是哪个部分导致了问题——这就是控制变量法在 context 调试中的应用。

### 是什么

因果归因方法是一套系统的排查流程，用来回答一个具体问题：**"输出的这个问题，是 context 的哪个部分造成的？"**

核心思路借鉴了科学实验中的控制变量法：每次只改变一个变量，观察输出的变化。如果改变了某个部分后问题消失了，那这个部分大概率就是病因。

具体操作分为两大策略：

**策略一：消融法（Ablation）——逐步移除**

从完整的 context 开始，每次移除一个组成部分，观察输出变化：

```
实验 0（基线）：完整 context → 观察输出（有问题）
实验 1：移除检索资料 → 输出变好了？→ 问题出在检索资料
实验 2：移除对话历史 → 输出变好了？→ 问题出在对话历史
实验 3：简化系统提示 → 输出变好了？→ 问题出在系统提示的某些规则
实验 4：移除 few-shot 示例 → 输出变好了？→ 问题出在示例
```

每次实验只改变一个部分。如果多个实验都让问题消失，说明问题来源于这些部分的交互（冲突）而非某个单独部分。

**策略二：累加法（Accumulation）——逐步构建**

从最小的 context 开始，每次添加一个组成部分，观察什么时候问题开始出现：

```
实验 0（最小集）：只有核心指令 → 输出正常
实验 1：+ 系统提示 → 输出还正常
实验 2：+ 检索资料 → 输出开始出问题！→ 问题与检索资料有关
实验 3：换一批检索资料 → 如果问题消失 → 定位到具体的某篇资料
```

累加法特别适合在"一切正常"到"出问题"之间找到那个临界点。

**策略三：替换法（Substitution）——等价替换**

不增不减，只是把某个部分替换成另一个版本：

```
实验 0：原始 context → 问题输出
实验 1：把系统提示换成另一种措辞 → 观察变化
实验 2：把检索到的文档 A 换成文档 B → 观察变化
实验 3：把 few-shot 示例从 3 个减到 1 个 → 观察变化
```

替换法特别适合定位"不是有没有的问题，而是质量/措辞/顺序的问题"。

### 为什么重要

假设你在做一个合同审查助手。用户上传一份合同，系统提取关键条款并标出风险点。测试时你发现，模型偶尔会遗漏"违约责任"条款的风险提示。

如果不用控制变量法，你可能会同时修改系统提示、调整检索逻辑、增加示例——然后问题确实消失了，但你不知道是哪个改动起了作用。下次遇到类似问题，你又回到了"随机尝试"的状态。

而如果你用消融法排查：先移除检索资料，发现模型依然能识别违约责任条款（说明不是检索的问题）；再检查系统提示中的指令，发现"请关注以下风险类型"的列表中没有包含"违约责任"——问题定位了，修复也就是加一行字的事。

因果归因的价值不仅在于解决当前问题，更在于积累可复用的调试经验：你知道了"指令列表不完整"是一种常见的病因模式，以后遇到类似症状会直接检查这个方向。

### 设计启示

**建立调试的标准流程：**

1. **复现问题**：先确认问题是稳定复现的，还是随机出现的。如果是随机的（比如 10 次里出现 3 次），需要多次实验才能判断某个改动是否有效。把模型的 temperature 设为 0 可以减少随机性。
2. **记录基线**：把当前的完整 context 和对应的输出保存下来，作为对照组。
3. **单变量实验**：每次只改一个地方，记录改动内容和输出变化。
4. **交叉验证**：找到可疑的部分后，在其他问题上验证——如果这个部分确实有问题，那不同问题上应该都能看到影响。

**在代码中支持调试：**

设计 context 组装逻辑时，应该让每个组成部分都可以独立开关或替换。比如用配置项控制是否包含检索资料、是否保留对话历史、使用哪个版本的系统提示。这样做消融实验时不需要改代码，只需要改配置。

### 常见误区

- **误区一："同时改多个变量，节省时间"。** 这是最常见的错误。如果你同时改了系统提示和检索资料，问题消失了，你无法知道是哪个改动起了作用——可能两个都需要改，也可能只需要改其中一个，甚至可能你改的两个地方互相抵消了。同时改多个变量不是节省时间，是浪费时间（因为你获得的信息量为零）。
- **误区二："问题消失就是修好了"。** 模型的输出有随机性。一次实验中问题消失不代表真的修好了，可能只是恰好这次没出现。对于重要的修复，至少要验证 5-10 次才能有信心。
- **误区三："只关注 context 内容，忽略 context 结构"。** 有时候问题不在于"写了什么"，而在于"怎么组织的"。同样的信息，放在 context 开头和放在中间，效果可能完全不同。归因时不要只关注内容变化，也要考虑位置、顺序、格式等结构性因素。

## 5.4 A/B 测试 Context 变体的实验框架

**一句话：** 当你有了一个改进方案但不确定是否真的更好时，A/B 测试框架让你用数据说话——在足够多的样本上同时对比两个 context 版本，用统计方法判断差异是否显著。

### 是什么

5.3 的控制变量法适用于单个问题的调试。但在生产环境中，你面对的往往是一个更宏观的问题："我重写了系统提示，整体质量是变好了还是变差了？"这种问题不能靠看几个案例来回答，需要在大量请求上进行系统性对比。

A/B 测试 Context 变体的实验框架，就是把互联网行业成熟的 A/B 测试方法论应用到 context 工程中。核心流程如下：

**第一步：定义评估指标**

在做任何实验之前，先明确你要度量什么。常见的评估指标分为三类：

| 指标类别 | 具体指标 | 度量方式 |
|----------|----------|----------|
| 准确性 | 回答与标准答案的匹配度 | 人工评分（1-5 分）或自动化评分（用另一个 LLM 打分） |
| 指令遵循 | 格式是否正确、规则是否遵守 | 规则检查脚本（正则匹配、JSON 合法性检查等） |
| 完整性 | 是否覆盖了所有要求的要点 | 清单式检查（列出必须包含的要素，逐一核对） |
| 安全性 | 是否出现不当内容、信息泄露 | 关键词过滤 + 人工审查 |
| 用户体验 | 回答是否自然、有用 | 用户反馈评分、对话完成率 |

**第二步：准备测试集**

测试集是一组预先准备好的请求（问题/任务），每个请求最好有一个期望的输出或评估标准。测试集的质量直接决定了实验结论的可靠性。

好的测试集应该满足：
- **覆盖性**：涵盖你关心的各种场景（常见问题、边界情况、容易出错的类型）。
- **代表性**：分布应该接近真实流量。如果 80% 的真实请求是简单查询，你的测试集也应该以简单查询为主，而不是全堆边界情况。
- **稳定性**：测试集在实验期间保持不变，不要一边实验一边改测试集。

**第三步：设计实验**

```
实验结构：
├── 对照组（Control）：使用当前版本的 context（版本 A）
├── 实验组（Treatment）：使用改进后的 context（版本 B）
├── 每组样本量：至少 50-100 个请求（越多越可靠）
├── 分配方式：随机分配（确保两组面对的问题分布一致）
└── 控制变量：除了被测试的 context 部分，其他一切保持一致
     ├── 同一个模型版本
     ├── 同样的 temperature 和其他参数
     ├── 同一批测试问题
     └── 同样的评估方法
```

**第四步：执行与分析**

跑完所有测试后，对比两组的评估指标。需要注意的是，不能只看平均分——如果版本 A 平均 4.1 分，版本 B 平均 4.2 分，这 0.1 分的差异可能只是随机波动。需要用统计检验（比如 t 检验或 Mann-Whitney U 检验）来判断差异是否"统计显著"。

一个简化的判断方法：如果版本 B 在大多数指标上都优于版本 A，且在最关心的核心指标上优势明显（比如准确性从 75% 提升到 85%），那可以有信心地采纳版本 B。

### 为什么重要

假设你的客服机器人收到了用户反馈："回答太啰嗦了"。你决定修改系统提示，加上"请简洁回答，控制在 3 句话以内"。修改之后，你试了几个问题，确实简洁了很多。上线！

一周后另一批用户反馈："回答太简短了，不够详细"。原来，你的修改虽然解决了啰嗦问题，但在需要详细解释的复杂问题上也变简短了，信息量不够。

如果你在上线前做了 A/B 测试，在测试集中同时包含了简单问题和复杂问题，你会在数据中看到：简洁性指标提升了，但复杂问题的完整性指标下降了。这样你就能在上线前发现这个权衡，做出更好的决策——比如只对简单问题启用简洁模式，或者用更精细的措辞来平衡简洁性和完整性。

### 设计启示

**轻量级实验框架的搭建：**

不需要搭建复杂的 A/B 测试平台，一个最小可用的框架只需要：

1. **一个测试集文件**（比如 JSON 或 CSV），包含测试问题和期望输出。
2. **一个评估脚本**，能自动跑完所有测试问题，调用 LLM API，收集输出。
3. **一个评分机制**——最简单的方式是用另一个 LLM 来当评判，给输出打分（"LLM-as-a-Judge"）。

```
最小实验框架：

test_cases.json        → 测试用例（问题 + 期望输出/评估标准）
context_v1.txt         → Context 版本 A
context_v2.txt         → Context 版本 B
run_experiment.py      → 遍历测试用例，分别用两个 context 调 API
evaluate.py            → 对比两组输出，计算指标
results.csv            → 结果记录
```

**"LLM-as-a-Judge"评估法：**

让另一个 LLM（最好是能力更强的模型）来评价输出质量。给评判模型的 context 包含：原始问题、期望标准、被评价的输出，然后让它按你定义的维度打分。这种方法虽然不如人工评估准确，但成本低、速度快，非常适合在迭代过程中快速获得反馈。

**实验记录的习惯：**

每次实验都应该记录：改了什么、为什么改、测试集是什么、各项指标的变化、最终决策。这些记录本身就是有价值的知识资产——三个月后你想知道"为什么当初选了这个版本的系统提示"，记录可以告诉你。

### 常见误区

- **误区一："试了三个例子都变好了，可以上线"。** 三个例子的样本量完全不够得出任何可靠结论。人的注意力天然倾向于确认自己的预期——你可能无意识地选了三个最容易体现改进效果的例子。至少用 50 个覆盖不同场景的测试用例。
- **误区二："只看平均分"。** 平均分可能掩盖了严重的分布问题。比如版本 B 的平均分和版本 A 一样，但版本 B 在简单问题上满分、在复杂问题上零分。一定要看分数的分布，特别关注最差的那些案例（尾部风险）。
- **误区三："一次大改，一次测试"。** 如果你同时修改了系统提示、检索策略和对话历史管理，测试结果变好了，你不知道是哪个改动的功劳。最好一次只改一个方面（或明确关联的一组改动），测试确认有效后再改下一个。这和 5.3 的控制变量原则是一致的。

## 5.5 Context 版本管理与 Diff

**一句话：** Context 是代码——它决定了系统的行为，每次修改都可能引入回归问题，因此需要像管理代码一样管理 context 的变更历史、做 diff 对比、支持回滚。

### 是什么

在 LLM 应用中，context（尤其是系统提示）本质上就是"用自然语言写的程序"。它定义了模型的行为：该做什么、不该做什么、以什么格式输出、遇到边界情况怎么处理。这和传统代码的作用完全一样——只是语言从 Python/Java 变成了中文/英文。

既然是"程序"，就应该有程序的管理规范：

**版本控制的基本做法：**

最直接的方式是把 context（系统提示、prompt 模板等）作为文本文件纳入 Git 版本控制。每次修改 context 就是一次 commit，commit message 说明改了什么、为什么改。

```
项目结构示例：
prompts/
├── system_prompt_v1.txt         ← 历史版本，保留参考
├── system_prompt_v2.txt         ← 历史版本
├── system_prompt_current.txt    ← 当前生产版本
├── few_shot_examples.json       ← Few-shot 示例集
└── prompt_changelog.md          ← 变更记录
```

或者更推荐的做法是直接用 Git 分支管理，而不是手动标版本号：

```
推荐的 Git 原生管理方式：
prompts/
├── system_prompt.txt           ← 唯一文件，通过 Git 历史追溯变更
├── few_shot_examples.json
└── retrieval_template.txt
```

用 `git log prompts/system_prompt.txt` 查看变更历史，用 `git diff` 对比版本差异。

**Diff 的价值：**

当输出质量突然下降时，`git diff` 可以告诉你最近一次 context 变更改了什么。自然语言的 diff 比代码的 diff 更需要细心阅读——改一个词可能就改变了整条指令的含义。

例如：

```diff
- 当用户询问竞品信息时，礼貌地说明你只能回答关于本公司产品的问题。
+ 当用户询问竞品信息时，可以提供客观的对比信息。
```

这一行 diff 看起来只是措辞调整，但它实质性地改变了系统的信息边界——从"拒绝回答竞品"变成了"可以讨论竞品"。如果没有版本管理和 diff 机制，你可能不会注意到这个变更的影响范围。

### 为什么重要

一个团队在开发 AI 写作助手。三个月里，系统提示被不同团队成员修改了二十多次。某天产品经理发现，模型的输出风格和早期版本完全不同了——变得过于正式，失去了原来的亲和感。

问题是：没有人能说清楚是哪次修改导致了风格的变化。每个人都只改了"一点点"，但二十次"一点点"叠加起来，系统提示已经和最初的版本面目全非了。

如果有版本管理，可以用二分查找法快速定位：对比 v1 和 v20 确认有差异，然后对比 v10 看问题是前十次还是后十次引入的，以此类推，几次 diff 就能锁定是哪次修改引入了风格偏移。

另一个常见场景：线上出了问题，需要紧急回滚。如果 context 在版本控制中，`git revert` 一条命令就能回到上一个稳定版本。如果 context 是直接硬编码在代码里或者存在数据库中没有版本记录，回滚就变成了"谁还记得之前写的是什么？"

### 设计启示

**Context 即配置，配置即代码：**

把 context 从业务代码中分离出来，作为独立的配置文件管理。这样修改 context 不需要改代码、不需要重新部署，但修改过程仍然有 Git 的完整追踪。

```
推荐的管理流程：
1. 修改 prompt 文件
2. 本地测试（用 5.3 的方法验证改动效果）
3. 提交 commit（附带清晰的 commit message）
4. 代码审查（是的，prompt 修改也需要 review）
5. A/B 测试（用 5.4 的框架对比新旧版本）
6. 合并上线
7. 监控（观察上线后的指标变化）
```

**Prompt 的代码审查（Code Review）：**

这是很多团队忽略的一步。系统提示的每次修改都应该有另一个人审查，关注点包括：
- 新指令是否和现有指令冲突？
- 措辞是否有歧义？
- 修改的影响范围是否被充分考虑？（改了一条规则，可能影响十种场景）
- 是否有配套的测试来验证改动效果？

**结构化的变更记录：**

除了 Git 的 commit history，可以维护一个更高层次的变更记录，记录每次重要修改的背景和决策：

```
日期：2025-03-15
修改内容：系统提示中添加了"回答长度不超过 200 字"的约束
修改原因：用户反馈回答过长，A/B 测试显示加入长度约束后用户满意度提升 12%
影响范围：所有对话场景
回滚方案：删除该约束即可
```

这种记录对于团队协作尤其重要——新加入的成员可以通过变更记录快速理解"为什么系统提示长成这样"，每一条规则背后都有它的理由。

**环境隔离：**

和代码一样，context 也应该有开发环境、测试环境和生产环境的区分：

```
environments/
├── dev/
│   └── system_prompt.txt      ← 开发者自由实验
├── staging/
│   └── system_prompt.txt      ← 上线前验证
└── prod/
    └── system_prompt.txt      ← 生产环境，变更需审批
```

开发环境可以自由实验，但推到生产环境之前必须经过测试和审查。

### 常见误区

- **误区一："prompt 就是一句话的事，不需要版本管理"。** 在项目早期可能确实只有一句话。但随着功能迭代，系统提示会不断增长——规则越来越多、边界情况越来越细。没有版本管理的 prompt 最终会变成一团无人敢动的"祖传代码"。越早建立版本管理习惯，后期维护成本越低。
- **误区二："把 prompt 写死在代码里"。** 这样做的结果是每次调 prompt 都需要改代码、走完整的部署流程。把 prompt 分离为配置文件后，可以独立迭代、快速调整，同时仍然享受 Git 版本控制的所有好处。
- **误区三："只管理系统提示，忽略其他 context 组件"。** Few-shot 示例、检索模板（retrieval prompt template）、摘要指令等也是 context 的重要组成部分，同样需要版本管理。任何会影响 LLM 行为的文本配置，都应该被纳入管理范围。
