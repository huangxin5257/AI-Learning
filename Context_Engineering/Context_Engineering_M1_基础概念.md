# M1 — 基础概念与设计思想

> **核心问题：Context Engineering 是什么，和 Prompt Engineering 什么关系，受什么约束**
> **学习目标：** 建立"Context 是一个需要被工程化管理的运行时输入"的认知，而不是"写好 prompt 就行"。

---

## 1.1 Context 的定义：LLM 推理时能"看到"的一切信息的总和

**一句话：** Context 不是你写的那段 prompt，而是模型在生成回答的那一刻，能"看到"的所有信息的总和。

### 是什么

当你调用一个 LLM 时，模型接收到的输入远不止你手写的那段提示词。一次完整的模型调用，模型实际"看到"的信息通常包括：

- **System Prompt** — 你（或系统）预设的角色定义、行为规则
- **用户的当前输入** — 用户这一轮说的话
- **对话历史** — 之前几轮的问答记录
- **检索到的外部知识** — 比如从文档库、数据库中检索出来注入的内容（RAG）
- **工具调用的结果** — 模型调用了搜索、计算器等工具后返回的数据
- **结构化元数据** — 当前时间、用户身份、环境变量等

这些信息拼接在一起，构成了一个完整的输入序列，发送给模型。这个序列的全部内容，就是 **Context**。

一个关键认知：**模型没有"记忆"，它只有 context。** 每次调用都是一次独立的推理过程，模型看到的全部信息就是你这次传入的 context。它不记得上一次调用发生了什么——除非你把上一次的对话历史显式地放进了这一次的 context 里。所谓的"多轮对话"，本质上是你每一轮都把之前的对话历史拼接进 context 重新发送。

### 为什么重要

假设你在做一个客服机器人，用户问"我上次的订单到哪了"。你的 system prompt 写得很好，模型也很聪明，但它回答不了——因为 context 里根本没有这个用户的订单信息。

这个问题不是 prompt 写得不好，而是 **context 里缺了关键信息**。

再看一个反面场景：你把整个产品文档（几万字）塞进 context，希望模型能回答任何产品问题。结果模型有时候表现很好，有时候答非所问，尤其是当答案藏在文档中间位置的时候。这也不是 prompt 的问题，而是 **context 的量和结构出了问题**。

这两个场景说明同一件事：**模型输出的质量上限，不取决于模型多聪明，而取决于 context 里有没有正确的信息、以正确的方式呈现。**

### 设计启示

当 LLM 的输出不符合预期时，第一反应不应该是"换个模型"或"改改 prompt 措辞"，而是：

1. **先检查 context 里有没有模型需要的信息** — 缺了就补，这是最常见的问题
2. **再检查信息是不是以模型容易理解的方式组织的** — 有了但结构混乱，模型也提取不出来
3. **最后才考虑调整 prompt 措辞** — 措辞是微调，不是根因

### 常见误区

**误区："我只需要关注 prompt 写得好不好。"**

很多人把注意力放在 prompt 的措辞上——"用中文还是英文""要不要说 please""角色设定的措辞怎么更精确"——但 prompt 只是 context 的一个组成部分。如果你的 context 里缺少关键事实、塞了太多噪音、或者组织结构混乱，无论 prompt 措辞多精致，模型表现都不会好。

**措辞上的微调能带来的提升通常在 5%-15%，而 context 结构和内容的调整能带来质的差异。**

## 1.2 Context Engineering vs Prompt Engineering — 范畴差异与演进关系

**一句话：** Prompt Engineering 关注"怎么写指令"，Context Engineering 关注"模型在推理时应该看到什么信息、以什么结构呈现、在什么时机注入"。前者是后者的子集。

### 是什么

**Prompt Engineering** 解决的问题是：给定一个输入框，怎么写出更好的指令让模型表现更好。它的核心手段包括：

- 角色设定（"你是一个资深律师"）
- 指令优化（"请一步步思考"）
- 输出格式要求（"用 JSON 格式返回"）
- Few-shot 示例（给几个输入输出的例子）

这些技巧是有效的，但它们有一个隐含的前提假设：**你需要的所有信息都已经在 context 里了，问题只是怎么表述得更好。**

**Context Engineering** 不做这个假设。它要解决的是一组更底层的问题：

- **内容决策** — context 里应该包含哪些信息？不应该包含哪些？
- **来源管理** — 这些信息从哪里获取？（数据库、搜索引擎、上游 API、用户画像……）
- **组装逻辑** — 运行时如何把不同来源的信息拼接成一个 context？拼接顺序是什么？
- **容量管理** — 信息总量超出 context window 时怎么办？保留什么、丢弃什么、压缩什么？
- **质量保障** — 怎么确保注入的信息是准确的、相关的、没有矛盾的？

用一个类比来区分：

| | Prompt Engineering | Context Engineering |
|---|---|---|
| 类比 | 写好一封邮件的措辞 | 决定这封邮件该附带哪些文件、引用哪些数据、抄送给谁 |
| 关注点 | 指令怎么写 | 模型该看到什么 |
| 作用范围 | 静态文本编写 | 运行时信息流的设计与管理 |

### 为什么重要

在 LLM 应用的早期阶段（ChatGPT 刚发布时），大多数使用场景是人对着一个输入框直接写 prompt。那个阶段 Prompt Engineering 就够用了——因为你就是唯一的信息来源，context 里除了你写的 prompt 没有别的东西。

但当你开始构建 LLM 应用产品时，情况发生了根本变化：

- 用户的输入是不可控的，你需要对用户输入做预处理
- 模型需要的知识不在训练数据里，你需要从外部检索并注入
- 一次交互可能涉及多个步骤（Agent），每一步的 context 都不同
- context window 有限，你不能把所有东西都塞进去，必须做取舍

这些问题已经超出了"怎么写好一段 prompt"的范畴。**从"写 prompt"到"工程化管理 context"，这是 LLM 应用从 demo 走向生产的核心转变之一。**

### 设计启示

当你在构建 LLM 应用时，需要关注的设计问题应该从"这个 prompt 怎么写"扩展为：

1. **这个场景下模型需要哪些信息才能给出好的回答？** — 这是内容决策
2. **这些信息从哪来、什么时候获取？** — 这是来源与时机
3. **如何在 token 预算内组织这些信息？** — 这是容量与结构
4. **信息来源不可靠时怎么办？** — 这是质量与安全

Prompt 的措辞优化仍然重要，但它只是这个更大框架中的一个环节。

### 常见误区

**误区："Context Engineering 就是 Prompt Engineering 的新说法。"**

不是改名，是扩展。Prompt Engineering 聚焦于指令文本本身的质量；Context Engineering 聚焦于模型输入的完整信息流的设计。一个是写作问题，一个是架构问题。

**误区："我不做 RAG 也不做 Agent，Context Engineering 跟我没关系。"**

只要你的 LLM 应用存在多轮对话、需要注入用户信息、或者 system prompt 超过几十行，你就已经在做 context 的设计与管理了。Context Engineering 只是给你一个系统性的框架来思考你已经在做的事情。

## 1.3 Context Window 的物理约束

**一句话：** Context window 是 LLM 的"工作台面大小"——所有输入和输出共享这个空间，超出就丢失，中间位置容易被忽视。它是所有 context 设计决策的硬约束。

### 是什么

每个 LLM 都有一个 **context window**（上下文窗口），以 token 数量为单位，定义了模型一次调用中能处理的最大信息量。

几个关键事实：

**1. 输入和输出共享这个窗口。**

如果一个模型的 context window 是 128K tokens，你的输入占了 120K，那模型最多只能生成约 8K tokens 的输出。context 设计时必须预留输出空间。

**2. token ≠ 字符。**

中文大约 1 个汉字 ≈ 1-2 个 token，英文大约 1 个单词 ≈ 1-1.5 个 token。精确的比例取决于具体的分词器（tokenizer）。在做容量规划时需要用实际的 tokenizer 来计算，不能用字符数估算。

**3. "支持 128K"不等于"128K 都好用"。**

这是最容易踩的坑。多项研究和实践表明，LLM 存在显著的**位置偏差**：

- **首部偏好（Primacy Bias）** — 模型对 context 开头的信息给予更高的注意力权重
- **近因效应（Recency Bias）** — 模型对 context 末尾（靠近当前生成位置）的信息也有较高的注意力
- **中间丢失（Lost in the Middle）** — 放在 context 中间位置的信息最容易被忽视

这意味着：即使你的 context 没有超出窗口限制，**信息放在不同位置，模型的处理效果也不同。**

**4. 越长的 context，推理质量越不稳定。**

context 越长，模型需要在更大的信息范围内做注意力分配，关键信息被"淹没"的风险越高。实践中，更短更精准的 context 往往优于更长更全面的 context。

### 为什么重要

场景一：你在做一个文档问答系统，用户问了一个问题，你从文档库检索出了 20 个相关段落，全部塞进 context。模型的回答时好时坏——不是检索出了问题，而是答案所在的段落恰好落在 context 中间位置，被模型忽视了。

场景二：你的 Agent 执行了 15 步操作，每一步的工具调用结果都保留在 context 里。到后面几步，模型开始"忘记"最初的任务目标，行为偏离了用户的原始意图。不是模型变笨了，而是最初的指令在一个很长的 context 里已经被推到了中间位置。

场景三：你以为 context window 有 128K，很富裕，不需要做压缩。但当你算上 system prompt（2K）+ 对话历史（30K）+ 检索结果（20K）+ 工具定义（5K），再预留输出空间（4K），实际可用空间远比你想象的少。

### 设计启示

Context window 的物理约束直接决定了几个设计原则（这些原则在 M3 会详细展开）：

1. **必须做 token 预算规划** — 把 context window 当作固定资源，为不同类型的内容分配预算上限，而不是"装得下就行"
2. **关键信息要放在有利位置** — 最重要的指令和信息放 context 的开头或结尾，不要埋在中间
3. **更少但更精准优于更多但有冗余** — 不是信息越多越好，剔除低相关性内容比增加内容更能提升效果
4. **长 context 需要主动管理** — 多轮对话、长流程 Agent 等场景下，需要设计压缩、摘要、截断策略，而不是被动等到 context 溢出

### 常见误区

**误区："模型支持 128K/1M 的 context window，所以不用担心长度问题。"**

窗口大小是上限，不是最佳工作区间。更大的窗口给了你更多的空间，但不意味着你应该把它填满。实际的工程经验是：**即使窗口足够大，主动管理 context 的长度和结构仍然能显著提升输出质量。** 大窗口是安全网，不是设计目标。

**误区："中间丢失是模型的 bug，以后会修复。"**

位置偏差是 Transformer 注意力机制的结构性特征，不是某个模型的缺陷。不同模型的偏差程度有差异，新模型在逐步改善，但完全消除在当前架构下不现实。在可预见的未来，你的 context 设计需要把位置偏差当作一个持续存在的约束来对待。
