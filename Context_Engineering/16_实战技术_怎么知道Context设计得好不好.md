# 怎么知道 Context 设计得好不好

**一句话：** 没有评估的 Context Engineering 只是在猜——你需要系统性地度量输出质量、溯源问题根因、迭代改进，才能把"感觉还行"变成"确实可靠"。

## 是什么

你已经为鲜果优选客服助手设计了 system prompt、接入了商品知识库、做了对话历史管理、加了输出格式约束。现在的问题是：**它到底好不好？**

"试了几个问题感觉还不错"——这不是评估，这是直觉。直觉在 10 个测试样例上可能是对的，但在 10,000 个真实用户问题面前，你无法靠直觉保证质量。

Context 评估要回答四个核心问题：

```
1. 输出质量：模型的回答是否正确、相关、格式合规？
2. 信息溯源：回答是基于你提供的 context，还是模型自己"编"的？
3. Context 利用率：你注入的 context 模型真的用了吗？有没有被忽略？
4. 效率：你的 token 花得值不值？有没有浪费？
```

这四个维度从不同角度审视你的 context 设计，缺一个就会留下盲区。

### 维度一：输出质量

这是最直观的维度——模型回答得对不对、好不好。

输出质量可以进一步拆解为：

- **正确性**：事实是否准确（草莓保质期说的是 3-5 天还是 3-5 周？）
- **完整性**：是否回答了用户的全部问题（用户问了两个问题，模型只答了一个？）
- **相关性**：回答是否切题（用户问保存方法，模型在介绍种植历史？）
- **格式合规**：是否遵循了预定的输出格式（该输出 JSON 的输出了吗？字段齐全吗？）
- **语气/风格**：是否符合品牌调性（鲜果优选要求亲切专业，模型回答冷冰冰的？）

### 维度二：信息溯源（Grounding / Faithfulness）

模型的回答是"有据可查"还是"凭空捏造"？

```
context 中的信息：草莓冷藏保质期 3-5 天
模型回答：草莓冷藏可以保存 3-5 天        ← Grounded（有据）
模型回答：草莓冷藏可以保存一周左右        ← Hallucinated（幻觉）
```

这个维度在客服场景中尤其重要——如果模型编造了不存在的优惠活动、虚构了退货政策细节，后果比"回答不完美"严重得多。

### 维度三：Context 利用率

你精心准备的 context，模型真的在用吗？

一种常见的情况：你花了大量精力检索并注入了相关的商品 FAQ，但模型完全忽略了它们，转而用自己的预训练知识回答。表面上答案可能也对，但这意味着你的 context 注入管道白做了——而且一旦遇到模型预训练知识覆盖不到的问题，就会翻车。

检测方法：在 context 中刻意放入一些只有 context 中才有的特定信息（比如鲜果优选特有的政策），看模型回答中是否引用了这些信息。

### 维度四：效率

```
效率 = 输出质量 ÷ token 成本
```

两个 context 方案输出质量相同，但一个用 3,000 token，另一个用 1,500 token——后者效率翻倍。

效率评估帮你回答：我的 context 中有没有"占着 token 但没贡献"的部分？

## 为什么重要

### 场景：鲜果优选上线前的窘境

鲜果优选的客服助手开发完毕，团队用手头的 5 个测试问题试了一下：

```
Q: 草莓怎么保存？         → ✅ 回答正确
Q: 你们几点发货？         → ✅ 回答正确
Q: 订单 #12345 到哪了？   → ✅ 回答正确（注入了订单数据时）
Q: 可以退货吗？           → ✅ 回答正确
Q: 你们有车厘子吗？       → ✅ 回答正确
```

5/5 全对，信心满满上线。

上线第一周，投诉涌来：

```
问题 1："用户问'草莓和蓝莓哪个维C高'，模型编了一个数字"
       → 评估盲区：信息溯源（context 里没有营养数据，模型靠幻觉回答）

问题 2："用户连续追问3个问题，模型只回答了最后一个"
       → 评估盲区：完整性（测试时没试过多问题场景）

问题 3："用户用方言问'莓莓咋个保存法'，模型说'抱歉我无法理解'"
       → 评估盲区：鲁棒性（测试问题都是标准普通话）

问题 4："用户问退货政策，模型说'7天无理由退货'"
       → 评估盲区：信息溯源（模型用了电商通用知识而非鲜果优选的特定政策）

问题 5："高峰期每次调用要5秒才返回"
       → 评估盲区：效率（context 太长导致延迟高）
```

5 个测试样例给出了 100% 通过率的假象。如果上线前有一个覆盖更广的评估集和系统性的评估流程，这些问题大部分可以提前发现。

### 评估不是奢侈品

很多人觉得"我只是做个小项目/内部工具/原型验证，不需要正式评估"。但评估的门槛其实很低——**20 个测试问题 + 人工检查就够启动了**。成本是半天工作量，收益是避免上线后的手忙脚乱。

评估的投入回报比通常非常高：花 1 天建评估集，省 1 周修 bug。

为鲜果优选客服助手建立一个最小评估集：

```python
    {

            {"user": "那买两盒打折吗？"},
            {"user": "好，下单了，大概几天到？"}
        ],

```

**覆盖原则：** 不要只测"正常路径"。按照这个比例分配：

```
40% 常见问题（确保基础能力）
30% 政策/规则相关（确保 context 被正确使用）
20% 边界/对抗场景（确保安全性和鲁棒性）
10% 多轮对话（确保上下文管理有效）
```

### 四种评估方法

#### 方法一：人工抽查（Manual Spot-checking）

最简单也最有效的起步方式。

```
操作流程：
1. 跑一遍评估集，收集所有输出
2. 逐个检查每条输出：
   - 关键信息点是否覆盖？（对照 expected_key_points 逐条打勾）
   - 是否有事实错误？
   - 格式是否合规？
   - 有没有幻觉内容？
3. 给每条输出打分（可以简单用 ✅/⚠️/❌）
4. 统计整体通过率，记录失败样例
```

优点：最直观，能发现自动化方法无法发现的微妙问题。
缺点：慢，主观，不适合频繁执行。
适用：早期开发阶段，每次大改后跑一遍。

#### 方法二：参考答案对比（Reference-based Evaluation）

为每个测试问题准备一个"标准答案"或"关键信息点清单"，然后检查模型输出是否覆盖。

```python
def evaluate_response(response, expected_key_points):
    """检查回答中是否包含所有关键信息点"""
    results = []
    for point in expected_key_points:
        # 简单方法：关键词检查
        found = point.lower() in response.lower()
        results.append({"point": point, "found": found})

    coverage = sum(1 for r in results if r["found"]) / len(results)
    return {
        "coverage": coverage,  # 关键点覆盖率
        "details": results,
        "pass": coverage >= 0.8  # 80% 覆盖率算通过
    }

# 示例
response = "草莓建议冷藏保存，一般可以放3-5天。保存前不要清洗，用厨房纸巾吸水效果更好。"
result = evaluate_response(response, ["冷藏", "3-5天", "不要提前清洗"])
# → coverage: 1.0, pass: True
```

优点：可自动化，可频繁执行，结果可复现。
缺点：只能检查"有没有说到"，不能检查"说得对不对""说得好不好"。
适用：回归测试，确保改动没有破坏已有能力。

#### 方法三：LLM 作为评委（LLM-as-Judge）

用另一个 LLM 来评价输出质量。这是近两年快速发展的评估方法。

```python
judge_prompt = """你是一个客服质量评估专家。请评估以下客服回答的质量。

【用户问题】
{question}

【提供给客服的参考信息】
{context}

【客服的回答】
{response}

请从以下维度评分（1-5分）并给出理由：

1. **正确性**：回答中的事实信息是否与参考信息一致？有没有编造内容？
2. **完整性**：是否回答了用户问题的全部要点？
3. **相关性**：回答是否切题，有没有跑偏或包含无关内容？
4. **格式规范**：是否遵循了客服回答的标准格式？
5. **语气得体**：是否亲切专业，符合品牌调性？

请以 JSON 格式输出：
{
  "correctness": {"score": 1-5, "reason": "..."},
  "completeness": {"score": 1-5, "reason": "..."},
  "relevance": {"score": 1-5, "reason": "..."},
  "format": {"score": 1-5, "reason": "..."},
  "tone": {"score": 1-5, "reason": "..."},
  "overall": {"score": 1-5, "reason": "..."},
  "grounding_issues": ["列出回答中没有依据的陈述，如果没有则为空数组"]
}"""
```

优点：能评价自然语言层面的质量，比关键词匹配深入得多。
缺点：评委 LLM 自身也可能出错（评分不稳定、漏判幻觉），需要校准。
适用：中等规模评估（几十到几百条），需要质量评分而不仅仅是 pass/fail。

**校准技巧：** 先用人工对 10-20 条样例打分，然后让 LLM 评委也打分，对比两者的一致性。如果一致性低于 80%，调整评委的 prompt 直到一致性达标。

#### 方法四：自动化指标（Automated Metrics）

借鉴 RAG（Retrieval-Augmented Generation）领域的评估框架，用自动化指标衡量 context 使用效果。

核心指标（RAGAS 风格）：

```
Faithfulness（忠实度）：
  回答中的每条陈述是否都能在 context 中找到依据？
  计算方式：可验证的陈述数 ÷ 总陈述数

Answer Relevance（回答相关性）：
  回答是否切题？
  计算方式：用 LLM 判断回答与问题的相关程度

Context Utilization（上下文利用率）：
  context 中有多少信息被回答实际引用了？
  计算方式：被引用的 context 片段数 ÷ 总注入的 context 片段数

Context Precision（上下文精确度）：
  注入的 context 中有多少是真正相关的？（衡量检索质量）
  计算方式：相关 context 片段数 ÷ 总注入的 context 片段数
```

这些指标不需要参考答案，可以完全自动化运行，适合大规模持续评估。

### 迭代循环：评估 → 诊断 → 修复 → 再评估

评估不是一次性任务，而是一个持续循环：

```
        ┌─────────────────────────────────────────────┐
        │                                             │
        ▼                                             │
   [ 评估 ]  →  发现问题  →  [ 诊断根因 ]  →  [ 修复 ]  →  [ 再评估 ]
        │                                             │
        └── 通过 ──→  ✅ 发布/继续                      │
                                                      │
   诊断方法：逐层剥离（ablation）                        │
   修复对象：context 的某个具体层                         │
        └─────────────────────────────────────────────┘
```

### 诊断根因：当输出有问题时，问题出在哪一层？

模型回答错了，但原因可能来自 context 管道的任何一个环节。诊断的关键是**逐层排查**。

```
问题：用户问"收到坏草莓能退吗"，模型回答"支持7天无理由退货"

排查步骤：

第1步：检查 context 中有没有退货政策
  → 有 ✅ → 说明不是"信息缺失"问题

第2步：检查退货政策的内容是否正确
  → "质量问题24小时内联系客服，非质量问题不支持退货" ✅ → context 内容没问题

第3步：检查模型是否使用了 context 中的政策
  → 没有 ❌ → 模型忽略了 context，用了自己的预训练知识

诊断结果：Context 注意力问题（模型没有"看到"或没有优先使用 context 中的信息）

可能的修复：
- 把退货政策放到 context 中更突出的位置
- 在 system prompt 中强调"只根据提供的政策信息回答，不要使用自己的知识"
- 用 XML 标签包裹政策信息，增加结构化程度
```

**逐层剥离法（Ablation）：**

当你不确定哪个 context 层导致了问题，可以一层一层移除来隔离：

```
完整 context：system prompt + 商品信息 + 退货政策 + 对话历史 + FAQ
                                                            → 输出错误

去掉 FAQ：system prompt + 商品信息 + 退货政策 + 对话历史
                                                            → 输出正确！

结论：FAQ 中有冲突信息干扰了模型判断。
修复：检查 FAQ 内容，移除/修正冲突条目。
```

### 评估的完整实践：鲜果优选客服助手

把上述方法综合起来，一个可落地的评估流程：

```
【一次性准备（半天工作量）】
1. 整理 20 个测试问题（覆盖 4 个类别）
2. 为每个问题标注期望的关键信息点
3. 写一个简单的脚本，批量跑测试并收集输出

【每次改动后执行（30分钟）】
1. 跑评估集，收集输出
2. 用关键词覆盖率做初筛（自动化）
3. 对未通过的案例做人工检查
4. 记录失败模式（哪类问题容易出错？）

【每周回顾（1小时）】
1. 汇总本周的失败模式
2. 识别优先级最高的问题（出现频率 × 严重程度）
3. 针对性修复 context 设计
4. 重新跑评估集验证修复效果

【上线后持续（持续进行）】
1. 从真实用户对话中采样新的测试案例
2. 把线上发现的 bad case 加入评估集
3. 评估集随业务发展持续增长（20 → 50 → 100 → ...）
```

这套流程的关键理念是：**评估集是活的**，它随着你对系统理解的加深而不断扩充和演化。

## 常见误区

**误区一："在 5 个例子上跑通就够了"**

5 个例子只能验证"基本能跑"，不能验证"跑得好"。LLM 的行为具有概率性——同一个问题问两次可能得到不同的回答。而且真实用户的提问方式远比你想象的多样：错别字、方言、一次问多个问题、反复追问、故意挑战……

**最低标准：** 20 个覆盖主要场景的测试用例。这不是一个精确的数字，但它是"从猜测到评估"的最低门槛。

**误区二："评估是给大公司/正式产品做的"**

评估的核心是"检验你的假设"。哪怕你只是做一个个人项目的 prompt，花 10 分钟准备 5 个边界案例测一下，也比"感觉没问题"强十倍。评估的规模可以很小，但习惯不能没有。

一个最小化的评估实践：

```
1. 写下你最担心模型回答错的 3 个问题
2. 跑一遍看结果
3. 如果答错了，调整 context，再跑一遍
```

这就是评估。不需要框架，不需要工具，不需要自动化。先建立习惯，再逐步升级。

**误区三："评估做一次就行了"**

Context 设计是一个迭代过程。以下任何一个变化都可能影响输出质量，需要重新评估：

- 修改了 system prompt
- 更新了知识库内容
- 调整了对话历史管理策略
- 更换了模型版本
- 调整了检索策略
- 修改了输出格式要求

每次改动后跑一遍评估集，是确保"修了 A 没有破坏 B"的唯一方法。这在软件工程中叫**回归测试**——Context Engineering 同样需要。

**误区四："自动化评估可以完全替代人工"**

自动化评估（关键词匹配、LLM 评委、RAGAS 指标）能高效覆盖大量案例，但它们都有盲区：

- 关键词匹配无法判断语义正确性（"3到5天"和"3-5天"匹配不上，但意思一样）
- LLM 评委可能对某些错误"宽容"（因为它自己也会犯同样的错）
- 自动化指标无法感知用户体验（格式正确、信息全面，但读起来很别扭）

最可靠的策略是**自动化做初筛、人工做终审**：

```
自动化：批量跑评估集 → 标记"可能有问题"的案例
人工：只审查被标记的案例 + 随机抽样一些"通过"的案例

这样既有效率，又不会漏掉自动化无法发现的问题。
```

**误区五："Context 出了问题就改 Prompt"**

当评估发现输出质量不佳时，很多人的第一反应是修改 system prompt——加更多指令、加更多约束、加更强的措辞。但问题可能根本不在 prompt 上：

```
问题根因                    正确的修复方向
────────                  ──────────
知识库中缺少相关信息          补充知识库，而不是让 prompt 更"聪明"
检索到了错误的文档            修复检索逻辑，而不是在 prompt 里加"忽略无关信息"
对话历史太长导致遗忘          优化历史管理策略，而不是在 prompt 里重复关键规则
输出格式不一致              加 API 级格式约束，而不是在 prompt 里反复强调格式
```

先诊断根因，再对症下药。Prompt 是 context 的一部分，但不是全部——很多问题的解法在 prompt 之外。
