# 信息衰减与位置效应

**一句话：** 模型对 context 中不同位置的信息关注度并不均匀——开头和结尾的信息更容易被"看到"，中间的信息最容易被忽略，这个现象直接决定了你应该把关键信息放在哪里。

## 是什么

我们在上一篇中知道了 context window 是模型能看到的全部信息。但"能看到"不等于"同等重视"。研究表明，模型对 context 中不同位置的信息有明显的注意力偏差，这种偏差有两个核心现象：

### 首因效应（Primacy Effect）

放在 context **开头** 的信息更容易被模型关注和遵循。模型在生成回答时，开头的指令和信息对后续生成的"锚定"作用最强。

### 近因效应（Recency Effect）

放在 context **末尾**（靠近模型开始生成的位置）的信息也有很高的影响力。模型的注意力机制对刚刚"读到"的内容天然敏感。

### "Lost in the Middle"（中间丢失）

2023 年，Stanford 等机构的研究者发表了一篇影响力很大的论文，标题就叫 *"Lost in the Middle"*。核心发现是：当关键信息被放在长文本的 **中间位置** 时，模型提取和利用该信息的准确率显著下降。

用一张直觉图来理解：

```
模型的注意力分布（示意）：

高 ▓▓▓▓▓▓▓                               ▓▓▓▓▓▓▓▓
   ▓▓▓▓▓▓▓▓                             ▓▓▓▓▓▓▓▓▓
   ▓▓▓▓▓▓▓▓▓                           ▓▓▓▓▓▓▓▓▓▓
   ▓▓▓▓▓▓▓▓▓▓                         ▓▓▓▓▓▓▓▓▓▓▓
   ▓▓▓▓▓▓▓▓▓▓▓░░░░░░░░░░░░░░░░░░░░░▓▓▓▓▓▓▓▓▓▓▓▓
低 ▓▓▓▓▓▓▓▓▓▓▓▓░░░░░░░░░░░░░░░░░░░▓▓▓▓▓▓▓▓▓▓▓▓▓
   ────────────────────────────────────────────────
   Context开头        Context中间         Context末尾
```

这不是说中间的信息完全被忽略——模型仍然会"读到"它——而是在需要**从长文本中定位并提取特定信息**时，中间位置的信息更容易被遗漏或被其他位置的信息覆盖。

需要注意的是：这种效应的强弱取决于多个因素——context 的总长度、任务类型、模型版本。短 context（比如几百 token）中效应不明显；context 越长，效应越显著。新一代模型在缓解这个问题上有改进，但并未完全消除。

## 为什么重要

继续用"鲜果优选"客服助手的场景。假设你把整个 FAQ 文档（包含退货政策、配送范围、会员制度、投诉流程等30个主题）一次性塞进 context：

**场景：退货政策被埋在中间**

```
System prompt：你是鲜果优选客服助手...

FAQ 文档（按字母/拼音排序）：
├── 包装说明...
├── 差评处理...
├── 配送范围...
├── 配送时效...
├── ...（第 3-12 条）
├── 【退货政策：签收后24小时内，需提供开箱视频...】   ← 第15条，正好在中间
├── ...（第 16-25 条）
├── 账号注销...
├── 支付方式...
└── 最新促销活动...

用户：你们能退货吗？需要什么条件？
```

实际测试中会发生什么：模型可能给出一个**大致正确但遗漏关键细节**的回答——比如提到"24小时内可退货"，但漏掉"需提供开箱视频"这个关键条件，或者把"24小时"误说成"48小时"。它不是完全找不到信息，而是从中间位置提取信息时精度下降。

**对比：退货政策放在最前面**

如果你根据用户的问题动态调整 context 结构，把退货政策提到 FAQ 的最前面：

```
System prompt：你是鲜果优选客服助手...

FAQ 文档（按当前问题相关性排序）：
├── 【退货政策：签收后24小时内，需提供开箱视频...】   ← 最前面
├── 退款流程...
├── 换货说明...
├── ...其他主题...
```

这时模型的回答准确率会明显提升——关键条件一个不漏。

这个差异不是因为模型"懒得看中间"，而是 Transformer 注意力机制的统计特性导致的结构性偏差。你不能指望模型"克服"它，只能在设计 context 时适配它。

## 设计启示

- **关键指令放在 system prompt 的开头和结尾。** 如果你有一条绝对不能被违反的规则（比如"不要透露内部系统信息"），把它放在 system prompt 的最开头，并在末尾再重复一次。这是利用首因效应和近因效应的双保险。

- **动态排序比静态排序更有效。** 不要按字母顺序或固定顺序排列 context 中的知识块。根据用户当前的问题，把最相关的信息排到最前面。这意味着同一份 FAQ 文档，面对不同问题时在 context 中的排列顺序应该不同。

- **用结构化标记帮助模型"定位"。** 当 context 较长且无法避免信息出现在中间时，使用清晰的标题、编号、XML 标签等结构化标记，可以在一定程度上帮助模型定位关键信息：
  ```
  <退货政策>
  1. 时限：签收后24小时内
  2. 凭证：必须提供开箱视频
  3. 不支持退货的情况：已切开水果 / 无视频 / 超时
  </退货政策>
  ```
  结构化标记不能完全消除位置效应，但能降低其影响。

- **缩短 context 是最根本的对策。** 位置效应在短 context 中几乎不是问题。如果你能通过精准检索把 context 从 30 条 FAQ 缩减到 3 条最相关的 FAQ，中间丢失的问题自然就消失了——因为根本没有"中间"。

## 常见误区

**误区一："模型能看到整个 context，所以信息放哪里都一样"**
"看到"和"有效利用"是两回事。模型在技术上确实会处理 context 中的每一个 token，但注意力权重的分布不均匀意味着不同位置的信息在实际生成中的影响力不同。这就像你把一本300页的参考书放在考生面前——考生确实"看到"了整本书，但翻到中间某页找到某条关键信息的概率远低于它就放在第一页。

**误区二："按字母顺序排列信息是合理的默认选择"**
对人类来说，字母排序方便查找。但模型不是通过"翻阅查找"来使用 context 的，它通过注意力机制同时处理所有位置的信息——而注意力分布不均匀。更好的默认策略是**按相关性排序**：与当前问题最相关的信息放在最前面，次相关的放后面。

**误区三："新一代模型已经解决了这个问题，不用担心"**
新模型（如支持 128k、200k 甚至 1M context 的模型）确实在长文本处理上有显著改进，在很多基准测试中"Lost in the Middle"效应大幅减弱。但在真实的、复杂的生产场景中，位置效应仍然存在，只是程度降低了。更重要的是，即使未来模型完全消除了位置效应，把相关信息排在前面仍然是好习惯——因为它还能减少 token 消耗（模型更早遇到关键信息，生成更精准，减少无意义的"搜索"行为）。
