# M4 — 工程化模式

> **核心问题：运行时如何组装和管理 context（动态工程实现）**
> **与 M2 的关系：** M2 告诉你有哪些原材料，M4 告诉你怎么在运行时把它们组装起来。
> **学习目标：** 能为一个具体的 LLM 应用设计完整的 context 组装流程。

---

## 4.1 动态 Context 组装（运行时拼接 vs 静态模板）

**一句话：** 生产级 LLM 应用的 context 几乎不会是一段写死的模板，而是在每次请求到来时，根据当前场景动态拼接出来的。

### 是什么

最简单的 LLM 调用方式是写一段固定的 prompt，每次发给模型。比如你写了一个翻译助手，prompt 是"你是一个翻译专家，请将以下内容翻译成英文：{用户输入}"。这就是一个**静态模板**——模板的结构是固定的，只有用户输入那个"槽位"是变化的。

但稍微复杂一点的应用就需要更多：

- 如果用户是 VIP 客户，system prompt 里要追加"对该用户使用更礼貌的语气"
- 如果用户的问题涉及退货政策，要从知识库中检索退货政策的原文并注入
- 如果对话已经进行了 10 轮，要对前面的历史做摘要，否则会超出 context window
- 如果当前是工作时间，要注入"可以推荐用户拨打客服电话"，否则不注入

这意味着最终发给模型的 context，不是从一个模板"填空"出来的，而是在运行时经过一系列**条件判断、数据查询、拼接排列**后"组装"出来的。这就是**动态 Context 组装**。

一个典型的动态组装流程看起来像这样：

1. **基础层**：加载 system prompt 的核心部分（角色定义、基本行为规则）
2. **条件注入层**：根据当前用户属性、请求类型、业务状态，有条件地追加指令片段
3. **知识注入层**：根据用户的具体问题，从外部源检索相关信息并注入（RAG）
4. **历史注入层**：将对话历史（或其摘要）拼接进来
5. **元数据层**：注入当前时间、用户 ID、会话 ID 等结构化信息
6. **容量检查**：计算当前 token 数，如果超出预算则触发压缩或裁剪策略
7. **最终拼接**：按照预设的顺序将各层内容组装成最终的 context

注意，这个流程中的每一步都可能涉及异步的数据查询、复杂的条件分支、甚至对其他模型的调用（比如让一个小模型先做摘要）。这已经不是"写一段 prompt"的问题了，而是一个**工程系统的设计问题**。

### 为什么重要

想象你在做一个企业内部的智能问答系统。用户提问"公司对远程办公的政策是什么？"

如果你用静态模板，context 可能只包含一段固定的 system prompt 加用户的问题。模型只能凭自己的训练数据瞎编一个答案——这在企业场景下是灾难性的。

如果你用动态组装，系统在收到这个问题后：先判断这是"政策类问题"，然后去检索公司内部的政策文档库，取回最相关的 2-3 段原文，注入到 context 中。同时，系统发现这个用户属于海外团队，于是额外注入一段"注意：不同地区的远程办公政策可能有差异，请提醒用户确认所在地区"的指令。最终拼接出来的 context 是高度针对当前请求定制的。

两种方式的差异不在于 prompt 写得好不好，而在于 **context 是否包含了模型回答这个特定问题所需要的特定信息**。动态组装的本质是让 context 成为一个"运行时定制"的产物，而不是一个"提前写好"的模板。

### 设计启示

设计动态组装系统时，有三个核心决策需要做好：

**决策一：哪些内容是"常驻"的，哪些是"条件触发"的。** 常驻内容（比如角色定义、核心规则）在每次请求中都出现；条件触发内容（比如特定领域知识、特殊规则）只在满足条件时注入。这个划分直接影响 context 的 token 预算分配——常驻内容越多，留给动态内容的空间就越少。

**决策二：各层内容的优先级排序。** 当 context 空间不够时，先砍哪部分？通常的优先级是：核心指令 > 直接相关的检索结果 > 对话历史 > 补充性知识 > 元数据。但具体排序要根据业务场景调整。

**决策三：组装逻辑是集中式还是分布式的。** 简单应用可以用一个函数处理所有组装逻辑；复杂应用可能需要一个"context 编排器"，由多个独立模块各自负责自己的部分，编排器负责汇总和排序。

### 常见误区

**误区："动态组装太复杂了，我先用静态模板上线，后面再改。"**

这个思路的问题在于，静态模板和动态组装不仅仅是复杂度的区别，而是**架构模式**的区别。如果你一开始用静态模板，所有逻辑耦合在一段字符串里，后面要改成动态组装，基本等于重写。更好的做法是：即使一开始很简单，也建立"分层拼接"的结构，只是每一层暂时只有一种固定内容。这样后续增加动态逻辑时，只需要替换某一层的实现，而不是推翻整个结构。

**误区："把所有可能用到的信息都塞进去，让模型自己判断用哪些。"**

这是"信息越多越好"的错觉。实际上，无关信息不是"不占地方的免费赠品"，而是会真实地稀释模型的注意力、占用有限的 context window、增加推理成本。动态组装的核心价值恰恰在于**精准地只注入当前需要的信息**。

## 4.2 Context 压缩与摘要策略

**一句话：** 当信息量超过 context window 的容量，或者你需要控制成本和延迟时，你需要在保留关键信息的前提下压缩 context——这不是简单的截断，而是一组需要精心设计的工程策略。

### 是什么

Context window 是有限的。即使当前最大的模型已经支持 100K 甚至 1M token 的窗口，在实际应用中你仍然会遇到"信息装不下"的问题。原因有三：一是有些信息来源确实很大（几百页的 PDF、几千条历史记录）；二是即使装得下，token 越多意味着推理越慢、成本越高；三是研究表明，context 越长，模型对中间位置信息的注意力越弱（所谓的"lost in the middle"现象），并不是装得下就能用好。

Context 压缩就是在这些约束下，用更少的 token 来表达更多的信息。常见的压缩策略可以分为几类：

**1. 截断策略**——最简单粗暴。比如只保留对话历史的最后 N 轮，或者只取检索结果的前 K 条。优点是实现简单、行为可预测；缺点是可能丢掉关键信息。

**2. 摘要策略**——用 LLM 或其他手段将长内容压缩为短摘要。比如将前 10 轮对话生成一段摘要，然后用这段摘要替代原始的 10 轮对话。优点是保留了语义；缺点是摘要过程本身会丢失细节，而且生成摘要需要额外的 LLM 调用。

**3. 选择性保留**——根据相关性评分，只保留与当前问题最相关的内容。比如用户问"退货政策"，那么之前对话中关于"退货"的历史保留，关于"产品推荐"的历史丢弃。

**4. 分层压缩**——对不同类型的内容采用不同的压缩策略。比如：system prompt 不压缩（因为是核心指令），对话历史做摘要，检索结果做截断。

对于长文档处理场景，有两种经典的工程模式：

**Map-Reduce 模式：** 将长文档拆成多个小块，每个小块分别用 LLM 处理（Map 阶段），得到每个块的中间结果（摘要、答案片段等），然后将所有中间结果汇总，再用 LLM 做一次最终处理（Reduce 阶段）。例如：要对一份 200 页的报告做总结，先把它切成 20 个 10 页的块，每个块生成一段摘要，然后把 20 段摘要合在一起再生成最终总结。

**Refine 模式：** 串行地处理文档的每个块。先处理第一个块得到一个初始结果，然后把初始结果和第二个块一起送给模型，让模型"在已有结果基础上，结合新内容，更新结果"。依次处理所有块后，得到最终结果。这种模式的好处是每一步都能看到之前的累积结果，信息传递更连续；缺点是必须串行执行，速度慢。

### 为什么重要

假设你在做一个合同审查助手。用户上传了一份 80 页的合同，问"这份合同里有没有对甲方不利的条款？"

这份合同的原文可能有 15 万个 token，远超大多数模型的 context window。即使勉强塞进一个超大窗口的模型，模型也很可能忽略中间部分的关键条款。你必须设计一个压缩策略。

一种做法是 Map-Reduce：将合同按章节切分，每个章节分别让模型识别"对甲方不利的条款"，然后将各章节的识别结果汇总，最后让模型做一个综合分析。另一种做法是 Refine：从第一章开始逐章分析，每处理完一章就更新"不利条款列表"，到最后一章时你已经有了完整的列表。

如果你不做任何压缩，只是把合同截断到 context window 能装下的前 N 页，那么后面章节里的不利条款就会被完全遗漏——这在法律场景下可能造成严重后果。

### 设计启示

**压缩策略的选择取决于信息的"可丢失程度"。** 有些信息丢了无所谓（比如对话历史中的闲聊），有些信息丢了会直接导致错误输出（比如用户明确提到的约束条件）。在设计压缩策略时，先对 context 中的各部分做一个"关键性分级"，然后对不同级别的内容采用不同的压缩力度。

**摘要不是免费的午餐。** 每次摘要都是一次信息有损压缩。原始内容中的具体数字、精确表述、微妙的限定条件，在摘要后都可能丢失。如果下游任务需要精确细节（比如合同中的具体金额和日期），摘要策略就要特别谨慎，或者采用"摘要 + 关键细节保留"的混合方式。

**预算分配思维。** 将 context window 想象成一个固定大小的预算。在设计系统时，预先为每个部分分配 token 预算：system prompt 占 X%，对话历史占 Y%，检索结果占 Z%，预留给模型输出的空间占 W%。当某个部分超出预算时，触发该部分专属的压缩策略。这比"先塞满，超了再截"要可控得多。

### 常见误区

**误区："context window 越来越大，压缩策略以后就不需要了。"**

即使 context window 无限大，你仍然需要压缩。原因有三：第一，token 越多，推理成本越高，很多应用有严格的成本约束；第二，推理延迟与 token 数量正相关，实时交互场景无法接受几十秒的等待；第三，模型对超长 context 的利用能力并非线性增长——信息越多，注意力越分散，关键信息越容易被淹没。压缩不是为了"装得下"，更是为了"用得好"。

**误区："做摘要就是把原文喂给 LLM 让它总结一下。"**

不指定摘要的目的和保留要点，LLM 做出的摘要未必保留你需要的信息。好的摘要策略应该是**面向任务的**：告诉模型"请保留与退货政策相关的所有细节"或"请保留所有涉及金额和日期的信息"。通用摘要和任务导向摘要，在信息保留上可以有天壤之别。

## 4.3 RAG 管道设计（检索 → 排序 → 注入的完整链路）

**一句话：** RAG 不是"接个向量数据库就行了"，而是一条从用户问题出发，经过查询理解、检索、排序、筛选、格式化，最终将信息注入 context 的完整工程管道。

### 是什么

RAG（Retrieval-Augmented Generation）的基本思路在 M2 中已经介绍过：让模型在回答问题时，先从外部知识源中检索相关信息，然后将检索结果注入 context 供模型参考。M4 关注的是：这条检索到注入的链路，在工程上具体怎么实现。

一条完整的 RAG 管道通常包含以下环节：

**1. 查询理解（Query Understanding）**

用户输入的原始问题未必适合直接用来检索。例如用户说"上次那个问题解决了吗"，这句话里没有任何实质性的检索关键词。查询理解的任务是将用户输入转化为适合检索的形式，常见的做法包括：

- **查询改写**：用 LLM 将口语化的问题改写为更精确的检索查询。比如将"上次那个问题解决了吗"改写为"工单 #12345 的当前状态"（结合对话历史中的上下文）。
- **查询扩展**：生成多个相关查询，扩大检索覆盖面。比如用户问"怎么退货"，同时生成"退货流程"、"退货政策"、"退换货条件"等查询。
- **意图识别**：判断用户的问题类型，决定去哪个知识源检索。比如"产品价格"去产品数据库，"使用方法"去帮助文档，"投诉"去工单系统。

**2. 检索（Retrieval）**

拿着处理好的查询，从一个或多个知识源中获取候选结果。检索方式通常有：

- **向量检索（语义检索）**：将查询和知识库中的文档都转化为向量，用相似度匹配找到语义最接近的内容。优点是能处理同义表达；缺点是对精确匹配（如产品编号、专有名词）不敏感。
- **关键词检索（词汇检索）**：传统的 BM25 等算法，基于词汇重合度匹配。优点是精确匹配能力强；缺点是无法理解同义词和语义关联。
- **混合检索**：同时使用向量检索和关键词检索，将两者的结果合并。这是目前生产系统中最常见的做法，兼顾语义理解和精确匹配。

**3. 排序（Ranking / Reranking）**

检索阶段通常会返回一批候选结果（比如 20-50 条），但你不可能把它们全部注入 context。排序阶段的任务是对候选结果重新排序，筛选出最相关的 Top-K。

常见做法是使用**重排序模型（Reranker）**：这是一种专门训练的交叉编码器模型，它接收"查询-文档对"，输出一个相关性分数。重排序模型比向量检索的相似度分数更准确，因为它能同时"看到"查询和文档的完整内容来判断相关性。

**4. 后处理与注入（Post-processing & Injection）**

经过排序筛选后的结果，不能直接拼接到 context 里就完事。后处理环节包括：

- **去重**：不同检索源可能返回重复或高度相似的内容，需要去重。
- **信息抽取**：检索到的可能是一整篇文档，但模型只需要其中的一个段落。需要从检索结果中提取真正相关的部分。
- **格式化**：将检索结果以模型容易理解的格式组织。比如加上来源标注（"以下信息来自《退货政策 v2.3》"），或者按相关性排列并编号。
- **Token 预算控制**：确保注入的内容不超过分配给检索结果的 token 预算。如果超了，要么减少注入条数，要么对单条结果做截断。

### 为什么重要

假设你在做一个技术文档问答系统，知识库里有 5000 篇技术文档。用户问"在 Kubernetes 中怎么配置水平自动扩缩容"。

如果你只做了最简单的向量检索，返回了 5 条结果，直接拼进 context——你可能会遇到这些问题：结果中有两条讲的是垂直扩缩容（语义相近但答非所问）；有一条是两年前的旧版配置方式（已废弃）；有一条来自社区论坛的非官方回答（可能不准确）。模型综合了这些参差不齐的信息，生成了一个混合了过时配置和错误步骤的回答。

如果你有一条完善的 RAG 管道：查询理解阶段识别出关键词"HPA"（Horizontal Pod Autoscaler）并扩展查询；混合检索同时用语义和关键词匹配；重排序模型筛掉了不太相关的结果；后处理阶段过滤掉了两年前的旧版内容（通过文档时间戳），并标注了每条结果的来源——这时候模型拿到的 context 质量就完全不同了。

RAG 管道的每个环节都在做同一件事：**提高最终注入 context 的信息质量**。任何一个环节的缺失都可能导致低质量的信息污染 context，进而导致模型输出质量下降。

### 设计启示

**将 RAG 管道设计为可独立调试的模块链。** 每个环节（查询理解、检索、排序、后处理）应该是独立的模块，有自己的输入输出接口。这样当系统表现不好时，你可以逐环节排查：是查询改写改错了？还是检索召回不全？还是排序没选对？还是后处理阶段截断了关键信息？如果所有逻辑揉在一起，排查问题会非常困难。

**检索质量是整条管道的瓶颈。** 下游再怎么排序和优化，也无法弥补上游检索阶段的遗漏。如果相关文档根本没被检索到（recall 问题），后面的环节无论多精致都救不回来。所以在资源分配上，检索环节的优化（embedding 模型选择、chunk 策略、索引质量）值得投入最多的精力。

**不要忽视"负面注入"的风险。** 注入不相关或错误的检索结果，比不注入更危险。因为模型倾向于信任 context 中的信息——如果你注入了一段错误的文档，模型会基于这段错误信息生成回答，并且表现得非常自信。设定一个**相关性阈值**，当检索结果的相关性分数低于阈值时，宁可不注入，让模型说"我不确定"，也不要注入低质量信息。

### 常见误区

**误区："RAG 就是向量数据库 + LLM，搭好就能用。"**

这大概是 RAG 领域最普遍的误解。向量数据库只是检索环节的一种实现，而且是最基础的一种。没有查询理解，用户口语化的问题可能根本检索不到相关内容；没有重排序，返回的结果质量参差不齐；没有后处理，注入的信息可能冗余或格式混乱。一个只有向量检索的 RAG 系统，和一条完善的 RAG 管道，在实际效果上可以有巨大的差距。

**误区："chunk 切得越小越好，这样检索越精准。"**

chunk（文档切块）太小，每个 chunk 缺乏上下文，检索到的内容可能片段化到无法理解。太大，则检索粒度粗，可能引入太多无关内容。chunk 的大小需要根据文档类型和检索场景来调整，没有统一最优解。一个常见的工程做法是"小 chunk 检索、大 chunk 注入"——用小粒度的 chunk 做检索匹配，但注入到 context 时返回该 chunk 所在的更大段落，保证上下文的完整性。

## 4.4 Memory 系统设计（短期/长期/工作记忆的分层）

**一句话：** LLM 本身没有记忆，但你可以通过工程手段为它构建一个分层的记忆系统——短期记忆支撑当前对话，工作记忆承载当前任务状态，长期记忆跨会话保存用户偏好和关键事实。

### 是什么

M2 中讲过，LLM 的"记忆"本质上是 context 中包含的历史信息——模型不会自己记住任何东西，所有"记忆"都是你显式注入的。M4 关注的是：在工程上，怎么设计一个系统来管理这些记忆信息的存储、检索和注入。

借鉴认知科学的分类，一个实用的 Memory 系统通常分为三层：

**短期记忆（Short-term Memory）—— 当前对话的原始历史**

就是当前这一次会话中，用户和模型之间的对话记录。在最简单的实现中，短期记忆就是"把之前每一轮的消息都存在一个列表里，每次调用时拼接进 context"。

短期记忆的特点是：完整保留了对话的所有细节，但占用 token 量随对话轮数线性增长。当对话很长时，需要配合 M4.5 中的多轮管理策略来处理。

**工作记忆（Working Memory）—— 当前任务的结构化状态**

工作记忆存放的不是原始对话记录，而是当前任务执行过程中需要持续追踪的**结构化信息**。例如：

- 在一个购物助手中，工作记忆可能是"用户当前的购物车内容、筛选条件、预算上限"
- 在一个代码生成助手中，工作记忆可能是"当前正在编辑的文件名、已经完成的步骤、待处理的需求列表"
- 在一个旅行规划助手中，工作记忆可能是"出发地、目的地、旅行日期、已确定的酒店、待确定的景点"

工作记忆通常以结构化格式（JSON、键值对、列表）存储，在每次调用时注入到 context 中的一个固定位置。它的更新逻辑也是结构化的——比如"用户说了新的预算上限→更新工作记忆中的预算字段"。

工作记忆的关键价值在于：它用远少于原始对话的 token 数，保留了任务的核心状态。10 轮对话可能有 3000 个 token，但从中提取出的工作记忆可能只有 200 个 token，却包含了所有关键决策信息。

**长期记忆（Long-term Memory）—— 跨会话持久化的用户信息**

长期记忆是在多次会话之间持续存在的信息，存储在外部数据库中。典型内容包括：

- 用户偏好（"喜欢简洁的回答风格"、"对辣的食物过敏"）
- 历史交互中的关键事实（"用户是一名前端工程师"、"用户所在城市是上海"）
- 过去的决策记录（"上次推荐了 A 方案，用户选择了 B"）

长期记忆的工程实现通常是：在每次对话结束后（或对话过程中），从对话内容中提取值得长期保存的信息，写入用户级的存储（数据库或向量存储）。在新对话开始时，从长期记忆中检索与当前上下文相关的条目，注入到 context 中。

### 为什么重要

假设你在做一个健康管理助手。用户每天来记录饮食和运动。

没有记忆系统的情况下：每次对话都是全新的开始。用户说"今天跑了 5 公里"，助手说"很好，继续保持"。下一次用户来时，助手完全不知道用户昨天跑了 5 公里，也不知道用户的目标是"月跑量 100 公里"，也不知道用户对乳糖不耐受。

有分层记忆系统的情况下：

- **短期记忆**保证在当前对话中，用户提到的信息不会被"遗忘"——用户说"我今天早上吃了全麦面包"，10 轮对话后助手仍然记得这个信息。
- **工作记忆**维护当前对话的任务状态——"今日已记录：早餐（全麦面包）、午餐（待记录）、运动（跑步 5 公里）"，这样助手能主动提醒"你还没有记录午餐"。
- **长期记忆**让助手跨越会话边界——新对话开始时，助手从长期记忆中加载"用户目标：月跑量 100 公里"、"本月已完成：78 公里"、"用户对乳糖不耐受"，然后能说"你这个月已经跑了 78 公里，还差 22 公里就达到目标了！"

三层记忆各自解决不同时间尺度的问题，缺少任何一层都会让用户体验出现明显的"断裂感"。

### 设计启示

**记忆的"写入"和"读取"是两个独立的设计问题。** 写入：从对话中提取什么信息存入记忆？提取的准确性如何保证？多久提取一次？读取：新对话开始时，从记忆中检索哪些信息注入 context？检索的触发条件是什么？注入多少条？这两个方向都需要独立设计。

**工作记忆是最容易被忽视、但往往最有工程价值的一层。** 很多人直接从"短期记忆"跳到"长期记忆"，忽略了工作记忆。但在多步骤任务中，工作记忆承担的是"在有限 context 中维持任务连贯性"的关键角色。没有工作记忆，要么整个对话历史都塞进 context（昂贵且低效），要么模型会在多轮后"忘记"任务的关键状态。

**长期记忆要做好"遗忘"机制。** 不是所有信息都值得永久保存。用户半年前说"最近在减肥"，半年后可能已经不成立了。需要为长期记忆设计过期、衰减或主动确认的机制，避免过期信息误导模型。

### 常见误区

**误区："把所有对话历史存起来就是长期记忆。"**

原始对话历史不是长期记忆。长期记忆是从对话历史中**提取**出来的、具有持久价值的结构化信息。存储原始对话历史当然有价值（用于审计、分析），但它不适合直接作为长期记忆注入新对话的 context——信息密度太低，噪音太多。

**误区："记忆系统的写入可以完全自动化，不需要用户参与。"**

自动提取的记忆可能有误差。在关键场景下（比如医疗、法律），应该让用户有能力查看、确认、修正系统记录的记忆。完全黑盒的记忆系统一旦记错信息，可能在后续所有对话中持续产生错误输出，而用户完全不知道问题出在哪里。

## 4.5 Multi-turn Context 管理（窗口滑动、截断策略）

**一句话：** 多轮对话中，对话历史会持续膨胀直到撑爆 context window，你需要一套策略来决定保留哪些历史、丢弃哪些历史、以什么形式保留。

### 是什么

当用户和模型进行多轮对话时，每一轮的用户消息和模型回复都会累积。假设平均每轮对话消耗 500 个 token（用户输入 + 模型输出），对话 20 轮后就是 10000 个 token 的历史，40 轮后就是 20000 个 token。这还不算 system prompt、检索结果等其他 context 内容。

多轮 context 管理要解决的核心问题是：**当对话历史的增长速度快于 context window 的容纳能力时，怎么在有限空间内保持对话的连贯性。**

常见的管理策略包括：

**1. 固定窗口截断**

只保留最近 N 轮对话历史，更早的历史直接丢弃。比如保留最近 10 轮。这是最简单的策略，实现起来就是对消息列表做一个切片操作。

优点：实现极简，token 消耗可预测。
缺点：第 11 轮之前的信息完全丢失。如果用户在第 3 轮说了一个关键约束，到第 14 轮时模型已经完全不知道了。

**2. 滑动窗口 + 摘要**

保留最近 N 轮的原始对话，但对更早的历史做摘要。比如：最近 5 轮保持原始消息，第 6-15 轮压缩为一段摘要，第 16 轮之前的全部丢弃。

这相当于把对话历史分成三个区域：
- **清晰区**（最近 N 轮）：完整保留，模型能看到所有细节
- **模糊区**（更早的 M 轮）：压缩为摘要，模型知道大致发生了什么但缺乏细节
- **遗忘区**（更早）：彻底不在 context 中

优点：在有限空间内同时保留了近期细节和远期概要。
缺点：需要额外的 LLM 调用来生成摘要，增加延迟和成本。

**3. 基于 Token 预算的动态管理**

不固定保留几轮，而是为对话历史设定一个 token 预算（比如 4000 token）。每次组装 context 时，从最新的消息开始向前装，装到预算用完为止。

优点：不浪费空间——如果对话轮次少每轮内容多，也能正确处理。
缺点：保留的轮数是浮动的，行为不完全可预测。

**4. 语义选择性保留**

不按时间顺序保留，而是根据每轮对话与当前问题的语义相关性来选择保留哪些历史。比如用户问"刚才说的退货流程的第三步是什么"，即使"退货流程"的讨论发生在 20 轮之前，也应该被保留。

优点：能捕捉到时间上很远但语义上很近的历史信息。
缺点：实现复杂，需要对每轮历史做相关性计算。

在实践中，这些策略往往是组合使用的。比如"最近 5 轮完整保留 + 更早的历史做语义筛选 + 筛选后仍超预算则做摘要"。

### 为什么重要

考虑一个技术支持对话场景。用户在第 1 轮描述了自己的系统环境（操作系统版本、软件版本），第 3 轮描述了错误现象，第 5 轮和客服确认了已经尝试过的解决方案，第 8 轮提供了日志片段。中间穿插了一些闲聊和不太相关的提问。

到了第 15 轮，客服模型需要给出最终的解决建议。这时如果使用简单的"保留最近 5 轮"策略，模型只能看到第 11-15 轮的内容——完全不知道用户的系统环境、错误现象、已尝试的方案。它的建议很可能是"请告诉我你的操作系统版本"——用户会崩溃，因为他在第 1 轮就说过了。

反过来，如果用"滑动窗口 + 摘要"策略，第 1-10 轮被摘要为"用户使用 Windows 11 + 软件 v3.2，遇到启动时崩溃的问题，错误码 0x80070005，已尝试过重装和清除缓存无效，提供了日志显示权限不足"。这段摘要只有几十个 token，但保留了所有关键信息，让第 15 轮的模型能够做出准确的诊断。

### 设计启示

**"保留什么"比"保留多少"更重要。** 在设计截断策略时，不要只考虑轮数和 token 数，还要考虑信息类型。用户提供的事实性信息（环境、约束、偏好）比模型的回复更值得保留；用户的关键决策比中间的探索性提问更值得保留。一个好的多轮管理策略应该具备区分信息重要性的能力。

**摘要的触发时机需要设计。** 不要等到 context 快撑爆时才匆忙做摘要。更好的做法是设定一个阈值（比如对话超过 10 轮时），在不那么紧急的时候提前生成摘要。这样有更多的 token 空间来保留摘要的质量，也不会在用户等待回复时引入额外延迟。

**考虑"钉住"机制。** 允许某些关键消息被"钉住"（pinned），不受截断策略的影响。比如用户在对话开头提出的核心需求描述，可以被标记为"钉住"，无论对话进行到多少轮都始终保留在 context 中。这种机制可以是自动的（由 LLM 判断哪些消息是关键的）或手动的（由用户标记）。

### 常见误区

**误区："对话历史就是完整地按时间顺序保留，超出窗口就从头截断。"**

"从头截断"意味着最早的消息最先被丢弃。但在很多场景下，最早的消息恰恰是最重要的——用户在对话开头通常会描述核心需求和背景。"从头截断"会导致模型丢失对话的"初始设定"。更合理的做法是：第一轮（或前几轮）的核心内容始终保留，截断从中间的不太关键的轮次开始。

**误区："每一轮都要完整保留用户的输入和模型的输出。"**

模型的输出通常比用户的输入更长（尤其是长篇回答），但在对话历史的 context 管理中，模型自己之前的回复往往不如用户输入重要——因为用户输入包含了需求、约束和事实，而模型输出是基于这些信息的推导。在空间紧张时，可以只保留模型输出的摘要甚至只保留一个标记（如"[模型回答了关于退货流程的问题]"），而完整保留用户输入。

## 4.6 多步骤 Agent 的 Context 编排

**一句话：** Agent 在执行多步骤任务时，每一步都会产生新的信息（工具返回值、中间推理、观察结果），context 编排的核心挑战是让 Agent 在步骤推进过程中既不"失忆"又不"信息过载"。

### 是什么

一个 Agent 在执行复杂任务时，通常不会一次调用 LLM 就完成，而是会经历多个步骤。以一个数据分析 Agent 为例，它收到"分析上个月的销售数据并给出优化建议"后，可能经历以下步骤：

1. 调用 SQL 工具查询上月销售数据
2. 拿到查询结果后，分析数据趋势
3. 调用图表工具生成可视化
4. 根据数据趋势和行业知识，提出优化建议
5. 将分析报告和建议整合成最终输出

每一步都涉及一次或多次 LLM 调用，每一步都会产生新的信息（工具的返回结果、模型的中间推理、环境的反馈）。Context 编排要解决的问题是：在第 N 步调用 LLM 时，context 里应该包含哪些来自前 N-1 步的信息。

这里面有一个核心矛盾：

- **如果每一步都带上所有历史步骤的完整信息**，context 会快速膨胀。工具调用的返回结果往往很大（一个 SQL 查询返回的表格可能有几千个 token），几步之后 context 就可能撑爆。
- **如果每一步只看上一步的结果**，Agent 会丢失早期步骤的信息，可能做出与之前推理矛盾的决策，或者重复已经做过的操作。

Agent 的 context 编排就是要在这两极之间找到平衡。常见的编排模式包括：

**全量累积模式：** 每一步的所有信息都累积保留，直到撑不下为止再统一处理。适合步骤少、每步信息量小的简单任务。

**摘要滚动模式：** 维护一个"运行状态摘要"，每完成一步就将该步的关键信息合并到摘要中，下一步的 context 包含摘要 + 当前步骤的详细信息。这样 context 大小基本恒定，不会随步骤增加而无限增长。

**选择性保留模式：** 为每一步的产出标注类型（工具返回值 / 中间推理 / 最终结论 / 错误信息），不同类型有不同的保留策略。比如：工具返回的原始数据在消费后压缩为摘要保留，中间推理只保留结论不保留过程，错误信息完整保留（防止重复犯错）。

**分层 context 模式：** 将 context 分为"任务层"和"步骤层"。任务层包含任务目标、全局约束、整体进度，在整个任务过程中始终存在；步骤层包含当前步骤的详细信息，每一步都会重置。这样任务层保证方向不偏，步骤层保证每一步都有足够的空间放置详细信息。

### 为什么重要

考虑一个研究助手 Agent，用户要求它"调研低代码平台的市场竞争格局，并给出进入策略建议"。Agent 会搜索多个信息源、阅读多篇报告、对比多家竞品。

如果采用"全量累积"，Agent 搜索了 5 篇文章后，每篇文章的原文（几千 token）都在 context 里，已经占了 2 万 token。还没开始分析，context 就快满了。而且模型需要在这 2 万 token 的"原材料"中找到有用的信息，注意力会非常分散。

如果采用"摘要滚动"，Agent 每读完一篇文章就生成一段结构化摘要（"平台名称、核心功能、市场份额、优劣势"），然后只在 context 中保留摘要。5 篇文章的摘要可能只有 1500 token，信息密度极高，模型能清晰地看到所有关键信息的对比。

Agent 的 context 编排直接决定了它在复杂任务中能走多远、走多稳。编排不好的 Agent 会在三五步后就开始"迷失"——忘记自己的任务目标、重复已经做过的操作、被无关信息干扰而偏离方向。

### 设计启示

**为工具返回值设计"消费后压缩"策略。** 工具返回的原始数据（API 响应、数据库查询结果、网页内容）往往体积大但只有部分信息有用。Agent 在"消费"（阅读和提取信息）这些数据后，应该立即将有用的信息提取为紧凑的摘要，丢弃原始数据。这样下一步看到的是"从销售数据中发现：A 产品同比增长 23%、B 产品下降 12%"，而不是整个 SQL 结果表格。

**始终在 context 中保留"任务目标"和"当前进度"。** Agent 在多步骤执行中最常见的失败模式是"目标漂移"——在处理某一步的细节时忘记了整体目标。解决方案很简单：在 context 的固定位置（通常是靠前的位置）始终放置任务目标的描述和当前的执行进度。这就像给 Agent 一个"指南针"，防止它迷失在细节中。

**错误信息的保留优先级应该高于成功信息。** 如果 Agent 在第 3 步尝试了某种方法但失败了，这个失败信息在后续步骤中的价值极高——它防止 Agent 再次尝试同样的失败路径。在压缩 context 时，应该优先保留"什么试过了但不行"，而不是"什么试过了且成功了"（成功的结果通常已经反映在后续步骤的状态中）。

### 常见误区

**误区："让 Agent 自己决定保留什么信息就好。"**

完全依赖 LLM 自己来管理 context 是危险的。LLM 可能判断某个信息"当前不重要"而丢弃，但几步之后突然需要这个信息。LLM 在做信息保留决策时，缺乏对未来步骤的预见能力。更可靠的做法是由工程系统制定明确的保留规则（基于信息类型、大小、步骤距离），而不是每次都让 LLM 来判断。

**误区："Agent 执行步骤越多，能力越强。"**

步骤越多，context 管理的挑战越大，信息丢失的风险越高，错误累积的可能性越大。每一步的 context 编排都可能引入信息损失，多步之后这种损失可能累积到影响最终结果。在设计 Agent 时，应该尽量减少达成目标所需的步骤数，或者将复杂任务拆分为多个子任务，每个子任务独立管理自己的 context。

## 4.7 Context Caching（前缀缓存的工程价值与约束）

**一句话：** Context Caching 是一种由 API 提供方支持的优化手段——如果多次请求的 context 有相同的前缀部分，这部分可以被缓存复用，从而降低成本和延迟。

### 是什么

当你调用 LLM API 时，模型需要先"阅读"你发送的 context（技术上是计算所有 token 的注意力），然后才能生成输出。"阅读"这个过程消耗的计算资源与 context 长度成正比，反映在你的账单上就是按输入 token 数计费。

Context Caching 的基本原理是：如果你连续多次调用模型，而每次的 context 有一段相同的前缀（比如都用同一段 system prompt 开头），那么这段前缀的计算结果可以被缓存起来。后续请求到来时，缓存的部分不需要重新计算，只需要处理缓存之后的新增内容。

具体来说，以 Anthropic 的实现为例：如果你的 context 结构是 [system prompt (2000 token)] + [对话历史 (3000 token)] + [用户最新消息 (200 token)]，并且 system prompt 在多次请求之间不变，那么这 2000 token 的 system prompt 可以被标记为可缓存。第一次请求时正常计算所有 5200 token；第二次请求时，如果前 2000 token 没变，模型可以直接复用第一次的计算结果，只需要处理后面的 3200 token。你为缓存命中的部分支付更低的费用（通常是正常价格的 10%），同时获得更快的响应速度。

使用 Context Caching 有几个关键约束：

**前缀匹配要求：** 缓存只对连续的前缀生效。也就是说，context 的开头部分必须完全一致，不能是中间某段一致或结尾一致。这意味着你需要把不变的内容放在 context 的最前面，把变化的内容放在后面。

**最小长度要求：** 通常需要缓存的前缀达到一定长度（比如 1024 token 以上）才能使用。太短的前缀缓存收益不明显。

**时效性：** 缓存有过期时间（通常是几分钟到几十分钟），过期后需要重新计算。如果请求间隔太长，缓存可能已经失效。

**精确匹配：** 前缀部分必须逐 token 完全一致。即使只改了一个字符，缓存也会失效。这意味着如果你的 system prompt 里包含动态内容（比如"当前时间是 2024-01-15 14:30:00"），每次请求时这个时间都不同，缓存就无法命中。

### 为什么重要

考虑一个文档问答场景：你有一份 50 页的产品手册（约 3 万 token），用户会反复就这份手册提问。每次用户提问，你都把整份手册作为 context 的一部分发给模型。

没有 Context Caching 的情况下：每次请求都要按 3 万 token 的输入收费。用户问了 20 个问题，你支付了 60 万 token 的输入费用。

有 Context Caching 的情况下：第一次请求正常计费 3 万 token。后续 19 次请求中，如果这 3 万 token 的前缀没有变化，每次只需要支付 3 万 × 10% = 3000 token 的缓存费用加上新增消息的正常费用。总共可以节省 80% 以上的输入成本。

除了成本，延迟也会显著降低。不需要重新计算 3 万 token 意味着模型可以更快地开始生成回答。对于实时交互场景，这种延迟的降低非常有价值。

### 设计启示

**设计 context 结构时，把稳定不变的内容放在最前面。** 这不仅仅是为了 Context Caching，也是一个好的结构化习惯。典型的排列顺序是：system prompt（最稳定）→ 长期知识/文档（会话级稳定）→ 对话历史（每轮变化）→ 用户最新消息（每次都变）。这样从前到后，稳定性递减，缓存能命中的前缀最长。

**将"准静态"内容也纳入缓存范围。** 有些内容不是完全不变的，但在一段时间内（比如一个会话内）保持稳定，比如通过 RAG 检索到的背景文档。如果你能把这些内容固定在 context 的前缀位置，在同一会话中就能持续享受缓存带来的收益。

**注意动态内容对缓存的"破坏"效应。** 如果你在 system prompt 里嵌入了时间戳、随机数、或者每次请求都不同的会话元数据，这些看似无关紧要的变化会导致整个前缀的缓存失效。把这类动态信息移到 context 的后部，或者降低其更新频率（比如时间戳精确到小时而不是秒）。

### 常见误区

**误区："Context Caching 是我自己实现的 cache 层。"**

Context Caching 不是你在应用层实现的缓存（比如"相同的问题返回之前缓存的答案"），而是 LLM API 提供方在基础设施层面提供的计算复用机制。你不需要自己实现缓存逻辑，只需要按照 API 的要求标记哪些内容是可缓存的，其余由基础设施处理。当然，应用层的答案缓存是另一种有效的优化手段，但它和 Context Caching 解决的是不同层面的问题。

**误区："有了 Context Caching 就不需要做 context 压缩了。"**

Caching 降低了重复处理相同前缀的成本，但它不解决 context 总量超出 window 限制的问题，也不解决超长 context 导致模型注意力分散的问题。Caching 和压缩是互补的优化手段，不是替代关系。

## 4.8 多 Agent 间的 Context 传递与隔离

**一句话：** 当一个复杂任务由多个 Agent 协作完成时，你需要精确控制每个 Agent 能看到什么信息（隔离）以及 Agent 之间如何传递中间结果（传递），既要保证协作效率又要避免信息泄露和上下文污染。

### 是什么

在复杂的 LLM 应用中，一个任务可能需要多个专门化的 Agent 协作完成。比如一个"客户投诉处理"系统可能有：

- **分类 Agent**：分析投诉内容，判断投诉类型和严重等级
- **检索 Agent**：根据投诉类型从知识库中找到相关的处理流程和历史案例
- **回复 Agent**：基于投诉内容、处理流程和历史案例，生成回复草稿
- **审核 Agent**：检查回复草稿是否合规、语气是否恰当

这四个 Agent 各有各的 system prompt、各有各的工具集、各有各的专长。但它们需要围绕同一个投诉案例协作。这就引出了两个核心问题：

**Context 传递：Agent A 的输出如何成为 Agent B 的输入？**

最直接的做法是把 Agent A 的完整输出（原始回复文本）传给 Agent B。但这往往不是最优的：Agent A 的输出中可能包含大量 Agent B 不需要的信息（比如 Agent A 的推理过程、解释说明），直接全部传递会浪费 Agent B 的 context 空间。

更好的做法是设计**结构化的传递接口**。每个 Agent 不只产出一段自由文本，而是产出一个结构化的结果对象，包含"给下游 Agent 的关键输出"和"给系统记录的详细过程"两部分。下游 Agent 只接收前者。

例如，分类 Agent 的输出接口可能是：
- 给下游：{投诉类型: "物流延迟", 严重等级: "中", 关键实体: ["订单号 xxx", "承诺日期 1月10日"]}
- 给系统记录：完整的分析推理过程

检索 Agent 只需要接收这个结构化结果，而不需要看到分类 Agent 是怎么一步步推理出来的。

**Context 隔离：Agent A 不应该看到什么？**

隔离有两个层面的需求：

第一是**安全隔离**——某些信息不应该被特定 Agent 看到。比如审核 Agent 不应该看到用户的详细个人信息（只需要看到投诉内容和回复草稿），以避免不必要的隐私数据暴露。

第二是**认知隔离**——给 Agent 过多无关信息会干扰其判断。分类 Agent 的任务是分类，如果你把知识库的检索结果也塞给它，它可能会被检索结果中的信息干扰，做出不纯粹的分类判断。每个 Agent 应该只看到**完成自己任务所需的最小信息集**。

### 为什么重要

设想一个"合同审查"的多 Agent 系统：

- Agent A 负责提取合同中的关键条款
- Agent B 负责将条款与法律法规对比
- Agent C 负责生成审查意见

如果没有设计好传递机制，Agent A 把整份合同的原文（几万 token）传给 Agent B。Agent B 的 context 中大部分空间被合同原文占据，留给法律法规参考的空间就很少了。而实际上 Agent B 只需要 Agent A 提取出的关键条款（可能只有几百 token），不需要合同原文。

如果没有设计好隔离机制，Agent C 不仅能看到 Agent A 和 Agent B 的输出，还能看到它们的完整推理过程和中间调试信息。这些信息不仅浪费空间，还可能让 Agent C 产生"过度自信"——它看到 Agent A 和 Agent B 已经做了详细分析，可能就不再独立思考，而是直接搬运前面的结论，失去了作为"最后一道关"的审核价值。

### 设计启示

**定义清晰的"Agent 接口协议"。** 每个 Agent 应该有明确的输入输出规范，就像软件工程中的 API 契约一样。输入规范定义了这个 Agent 需要什么信息才能工作；输出规范定义了它会产出什么结构化结果。Agent 之间通过这些规范进行对接，而不是直接传递原始的 context 内容。

**使用"编排层"来管理跨 Agent 的 context 流转。** 不要让 Agent 之间直接通信，而是引入一个编排器（Orchestrator），负责调度 Agent 的执行顺序、从上游 Agent 的输出中提取下游 Agent 需要的信息、组装下游 Agent 的 context。这样 context 的流转逻辑集中在编排层，每个 Agent 可以专注于自己的任务。

**区分"共享 context"和"私有 context"。** 有些信息所有 Agent 都需要（比如任务目标、全局约束），可以放在共享 context 中；有些信息只有特定 Agent 需要（比如某个 Agent 的专有工具返回值），应该放在私有 context 中。这种区分既有效控制了每个 Agent 的 context 大小，又确保了关键信息的全局可见性。

### 常见误区

**误区："所有 Agent 共享同一个 context 最简单。"**

把所有 Agent 的 system prompt、工具调用结果、中间推理都堆在同一个 context 里，会导致三个问题：一是 context 爆炸性增长；二是每个 Agent 被大量无关信息干扰；三是安全隔离完全不存在。多 Agent 系统的 context 管理之所以复杂，恰恰是因为"全共享"不可行。分层、分域的 context 管理是必须的。

**误区："Agent 之间的信息传递越多越好，多总比少好。"**

过度传递和传递不足一样有害。过度传递会占用下游 Agent 的 context 空间、引入噪音、增加误判风险。正确的做法是"必要且充分"——传递下游 Agent 完成任务所必需的最小信息集。如果不确定某个信息是否需要传递，可以先不传，等发现下游 Agent 表现不好时再按需增加。

## 4.9 多租户场景下的 Context 隔离与复用

**一句话：** 在 SaaS 产品中，多个租户（客户）共享同一套 LLM 应用，你需要确保租户之间的数据完全隔离（A 客户的信息不能出现在 B 客户的 context 中），同时尽可能复用公共资源以控制成本。

### 是什么

当你构建一个面向多个企业客户的 LLM 产品时（比如一个智能客服 SaaS），每个客户（租户）都有自己的知识库、业务规则、用户数据和对话历史。多租户 context 管理要解决的核心问题是：如何在运行时组装 context 时，确保每个租户只看到属于自己的数据，同时在工程上尽可能高效。

**隔离维度：**

1. **知识库隔离**：租户 A 的产品文档不能出现在租户 B 的 RAG 检索结果中。这通常通过在向量数据库中为每个租户设置独立的命名空间（namespace）或在检索时添加租户 ID 过滤条件来实现。

2. **对话历史隔离**：租户 A 的用户与模型的对话记录不能泄露给租户 B。这看似显而易见，但在实现中容易出问题的地方是：缓存层、日志系统、few-shot 示例池。如果系统从历史对话中自动选择 few-shot 示例注入 context，必须确保选取的示例来自同一租户。

3. **配置隔离**：每个租户可能有不同的 system prompt、不同的业务规则、不同的品牌语气。这些配置在 context 组装时必须按租户加载。

4. **记忆隔离**：如果系统有长期记忆能力（M4.4），不同租户的用户记忆必须严格隔离。

**复用维度：**

完全的隔离意味着完全没有复用，这在成本上可能不可接受。可以安全复用的部分通常包括：

1. **基础模型和基础设施**：所有租户共享同一个 LLM API，同一套检索基础设施。

2. **通用 system prompt 的公共部分**：比如"你是一个客服助手"这种通用指令，所有租户都一样，可以作为公共前缀。租户特有的配置（品牌名称、业务规则）追加在公共部分之后。结合 Context Caching（M4.7），公共前缀可以被缓存复用。

3. **通用知识**：比如法律法规、行业通用知识等不属于任何特定租户的内容，可以作为公共知识库供所有租户使用。

4. **匿名化后的数据**：在某些场景下，脱敏后的交互数据可以用于改善所有租户的体验（比如优化检索排序模型），但这需要严格的隐私合规审查。

### 为什么重要

假设你在做一个面向电商企业的智能客服 SaaS。客户 A 是卖服装的，客户 B 是卖电子产品的。

一个真实的安全事故场景：你的 RAG 系统在检索时没有正确过滤租户 ID。客户 B 的一个终端用户问"你们的退货政策是什么"，系统去向量数据库检索"退货政策"，因为没有租户过滤，检索到了客户 A 的退货政策文档（恰好相关性分数更高）。于是模型基于客户 A 的退货政策回答了客户 B 的用户。轻则用户收到错误信息，重则泄露了客户 A 的商业敏感信息。

另一个场景：你的系统使用 few-shot 示例来提升回答质量，示例是从历史对话中自动选取的。由于没有做好租户隔离，客户 B 的 context 中可能出现客户 A 的对话示例，其中包含客户 A 的产品信息、价格策略甚至客户隐私数据。这不仅是体验问题，更是严重的数据安全和合规问题。

### 设计启示

**在 context 组装管道的最早期就注入租户标识。** 不要等到检索阶段才考虑租户隔离。从 context 组装的第一步开始，就应该知道当前请求属于哪个租户，并且在每一个涉及数据查询的环节（知识库检索、对话历史加载、配置加载、记忆读取）都强制附加租户过滤条件。这种"默认隔离"的思路比"事后检查"更安全。

**把 context 的层次结构与租户隔离边界对齐。** 将 context 分为"平台公共层"、"租户配置层"和"用户交互层"。公共层所有租户共享（可缓存），配置层按租户加载，交互层按用户+租户加载。这种分层与 M4.1 的动态组装架构天然吻合，在每一层的注入逻辑中加上隔离检查即可。

**建立"context 审计"机制。** 在生产环境中，定期对组装好的 context 做审计抽查——检查 context 中是否出现了不属于当前租户的信息。这种审计可以是自动化的：用规则检查（context 中出现的企业名称是否与当前租户一致）或者用模型检查（让一个审计模型判断 context 中是否有跨租户信息泄露的迹象）。

### 常见误区

**误区："我的向量数据库支持命名空间，所以隔离问题已经解决了。"**

向量数据库的命名空间只解决了检索层的隔离。但 context 中的信息来源不止检索——还有 system prompt 中的配置片段、对话历史、长期记忆、few-shot 示例、工具调用结果等等。每一个信息来源都需要独立考虑隔离，遗漏任何一个都可能导致跨租户的信息泄露。

**误区："每个租户部署一套独立系统就不需要考虑隔离了。"**

对于大型客户这是可行的（所谓的"单租户部署"），但成本极高。大多数 SaaS 产品的经济模型依赖于多租户共享基础设施来分摊成本。而且即使是单租户部署，也可能有"租户内部的隔离"需求——比如一个企业的不同部门之间可能也有数据隔离要求。隔离是一个需要在架构层面系统性解决的问题，不能只靠"物理隔离"这一种手段。
