# 二、语义理解类

> **本类共性：** 模型"读懂了"你的 Prompt，也确实在执行，但输出方向偏了——因为它理解的意思和你心里想的意思之间有一道缝隙。这类问题的根源不在指令结构，而在**语义精度和上下文补全**上。你说的每个词，模型都认识，但它脑补的含义和你默认的含义不一样。

---

## 2.1 意图错位

> **快速识别：** 模型做了，但方向完全不对——它理解了字面意思，但没 get 到你真正想要什么

**典型症状**

- 输出和你的要求"沾边"，但明显不在点上
- 模型做了一个泛泛的通用版本，没有命中你真正关心的改进维度
- 你会有一种"我不是这个意思"的感觉，但又说不清模型哪里理解错了

**为什么会发生**

你说了"做什么"，但没说"为什么做"和"做完要达到什么效果"。模型只能从字面意思出发，按训练数据中最常见的理解方向去执行。比如你说"润色"，模型默认改语法和措辞；但你真正想要的可能是"让文案更有说服力"。字面意思和真实意图之间那道缝隙，就是意图错位的来源。

**怎么修**

- **描述目标效果，而非操作动作**：不要只说"润色""优化""改改"，要说清楚改完之后应该是什么样的。
- **说明使用场景和受众**：告诉模型这段内容给谁看、在哪里用、用来干什么，帮它校准方向。
- **提供对标示例**：给一个"我想要的大概是这种感觉"的参考，比文字描述更精准。

**示例**

### 示例 1：文案润色

用户原始提示词：
```
帮我润色这段话，让它更好。
```

失效输出：
模型只做了通用的语法修正和措辞替换（把"很好"改成"出色"，把"做了"改成"完成了"），没有触及真正的改进维度——用户实际想让文字在 B2B 销售场景中更有说服力。

改进版提示词：
```
帮我修改这段话，使其在 B2B 销售邮件中更有说服力。

目标效果：读起来更自信、更简洁，突出对客户的价值而非产品功能。
受众：企业 IT 采购决策者。
参考风格：Stripe 和 Datadog 的官方产品介绍页。

原文：[粘贴原文]
```

改进后预期效果：
输出从"通用润色"变为"针对 B2B 采购决策者的价值导向文案"，语言自信简洁，重点从罗列功能转向客户收益，风格对标专业 SaaS 产品介绍。

---

### 示例 2：SQL 优化

用户原始提示词：
```
帮我优化这段 SQL。
```

失效输出：
模型对 SQL 做了格式美化（加了缩进、换了关键词大小写），但没有触及实际的性能问题——真正需要优化的是缺少索引和全表扫描。

改进版提示词：
```
帮我优化这段 SQL 的查询性能。当前在 500 万行的表上执行需要 12 秒。

优化目标：将查询时间降至 1 秒以内。
请从以下维度分析：索引策略、查询计划、JOIN 方式、子查询改写。
不需要调整 SQL 格式或代码风格。

[粘贴 SQL]
```

---

### 示例 3：简历修改

用户原始提示词：
```
帮我改改简历。
```

失效输出：
模型修改了简历中的语法错误和格式排列，但没有触及核心问题——用户真正需要的是让工作经历的描述更突出量化成果，以匹配正在申请的高级工程师职位。

改进版提示词：
```
帮我修改简历中的"工作经历"部分。

目标职位：高级后端工程师（3 年以上经验要求）
核心改进方向：
- 每段工作经历要突出可量化的成果（如性能提升百分比、处理的数据量、节省的成本）
- 使用 STAR 格式（情境→任务→行动→结果）重写关键项目描述
- 去掉泛泛的职责描述（如"负责后端开发"），替换为具体贡献

[粘贴简历内容]
```

改进后预期效果：
工作经历从"负责后端开发"这类泛泛描述，变为"将订单查询接口响应时间从 800ms 优化至 120ms，日均处理请求量 500 万次"这类可量化的成果陈述，直接对标高级工程师岗位的要求。

## 2.2 歧义词陷阱

> **快速识别：** 你说"优化"，模型改了代码风格；你其实想要的是性能优化

**典型症状**

- 你用了"专业""简洁""优化""详细"这类词，模型的理解和你的理解完全不同
- 输出方向偏了，但模型不会告诉你"你这个词我是这样理解的"
- 同一个词在不同场景下，模型每次的解读可能都不一样

**为什么会发生**

"专业""简洁""优化""详细"这些词在不同语境下含义差异很大。模型会按训练数据中最常见的解读去执行——比如"优化代码"最常见的意思是改格式和命名，而不是提升性能。你脑子里有明确的定义，但模型只能猜，而且不会主动来问你。

**怎么修**

- **把抽象词拆解成具体操作**：不说"写得更专业"，而说"每个论点后加数据支撑，使用行业术语，去掉口语化表达"。
- **给正反示例**：告诉模型"像这样是对的"和"像这样是不对的"，用实例消歧。
- **先定义再使用**：如果一定要用抽象词，先在 Prompt 里给它下个明确的定义。

**示例**

### 示例 1："专业"的歧义

用户原始提示词：
```
帮我把这份报告写得更专业。
```

失效输出：
模型只是替换了一些词汇让文字更"书面化"（"很多"→"大量"、"搞了一个"→"开展了一项"），但没有触及用户真正期望的：加入数据支撑、引用权威来源、使用行业术语。

改进版提示词：
```
帮我提升这份报告的专业性。

在本次任务中，"专业"具体定义为：
1. 每个核心论点后附数据支撑或来源引用
2. 使用信息安全行业的标准术语（如 SIEM、SOAR、零信任）
3. 去掉所有口语化表达（如"搞了一个""弄好了"）
4. 使用"调研发现""数据表明"等正式表述引入论据

[粘贴报告原文]
```

改进后预期效果：
报告从"换了几个书面词"升级为真正有专业深度的文档——论点有数据支撑，术语使用准确，表述正式规范，读起来像行业分析师写的而不是学生作业换了个措辞。

---

### 示例 2："优化"的歧义

用户原始提示词：
```
帮我优化这段代码。
```

失效输出：
模型对代码做了风格优化（改了变量命名、加了注释、调整了缩进），但用户真正想要的是性能优化——这段代码在处理大数据量时出现超时。

改进版提示词：
```
帮我优化这段代码的执行性能。

当前问题：处理 10 万条记录时耗时超过 30 秒，需要降到 5 秒以内。
优化方向：算法复杂度、内存使用、I/O 效率。
不需要修改：代码风格、命名规范、注释（保持现状即可）。

[粘贴代码]
```

---

### 示例 3："详细"的歧义

用户原始提示词：
```
请详细说明 Kubernetes 的架构。
```

失效输出：
模型输出了一篇 3000 字的长文，从容器化的历史讲到每个组件的源码层实现，远远超出了用户的需求——用户只是想要一份适合向非技术领导汇报的、覆盖核心组件和交互关系的概述。

改进版提示词：
```
请说明 Kubernetes 的架构。

"详细"的定义：
- 覆盖所有核心组件（Control Plane 和 Worker Node 各模块）
- 说明组件之间的交互关系
- 每个组件用 2-3 句话解释其职责
- 总篇幅控制在 800 字以内
- 目标读者：有基础 IT 知识但不了解 K8s 的技术经理
```

改进后预期效果：
输出是一份结构清晰的架构概述，覆盖了 API Server、etcd、Scheduler 等核心组件及其交互关系，每个组件解释简洁到位，技术经理能在 3 分钟内读完并理解整体架构，不会被源码级细节淹没。

## 2.3 隐含假设缺失

> **快速识别：** 模型给了一个"通用版"答案，因为你没说出那些"理所当然"的前提

**典型症状**

- 输出是"正确的废话"——技术上没错，但完全不适用于你的具体情况
- 模型推荐了一个你根本用不了的方案（太重、太贵、不符合你的技术栈）
- 你觉得"这不是显而易见的吗"，但模型确实不知道

**为什么会发生**

你心里有一堆"理所当然"的前提——团队规模、技术栈、预算、适用法律、文件格式等等——但这些从没写进 Prompt 里。模型没有这些背景信息，只能按最通用的方式生成。你觉得"不用说也知道"的事情，对模型来说就是不存在的。

**怎么修**

- **主动把隐含前提写出来**：写 Prompt 时问自己一句："我觉得理所当然、但换个人可能不知道的背景是什么？"然后写进去。
- **用角色定义注入行业背景**：给模型一个角色（如"你是一个熟悉中国大陆法规的法务顾问"），帮它加载对应的默认假设。
- **把环境约束单独列一段**：团队规模、预算、技术栈、数据量等关键约束，单独拉出来写清楚。

**示例**

### 示例 1：法律文书

用户原始提示词：
```
帮我写一份用户协议。
```

失效输出：
模型输出了一份美国法律体系下的英文用户协议模板，引用了 GDPR 和 CCPA。用户的隐含假设是：面向中国大陆用户的 SaaS 产品，需要符合《个人信息保护法》。

改进版提示词：
```
帮我写一份用户协议。

背景：
- 产品类型：面向中国大陆企业用户的 SaaS 电信费用管理系统
- 适用法规：《个人信息保护法》《网络安全法》《数据安全法》
- 用户群体：企业客户（非个人消费者，不涉及未成年人）
- 语言：中文
- 需要包含的条款：数据收集范围、数据存储位置、第三方共享规则、用户权利（查阅/删除/导出）
```

改进后预期效果：
输出的用户协议基于中国大陆三大数据法规，条款结构符合国内 SaaS 行业惯例，语言为中文，不再出现 GDPR/CCPA 等不相关的法规引用。

---

### 示例 2：技术方案设计

用户原始提示词：
```
帮我设计一个日志收集系统。
```

失效输出：
模型推荐了 ELK Stack（Elasticsearch + Logstash + Kibana）的标准方案，但用户的实际情况是：3 人小团队、预算有限、运维能力弱，需要的是轻量级解决方案而非重型基础设施。

改进版提示词：
```
帮我设计一个日志收集系统。

环境约束：
- 团队规模：3 人，无专职运维
- 服务器：2 台 4C8G 的 VM（Proxmox VE 环境），不能再加机器
- 日志量：每天约 5GB
- 预算：只用开源工具，不购买商业服务
- 保留期：90 天
- 核心需求：能按关键词搜索、按时间范围过滤、设置错误告警

请在以上约束下推荐最轻量的方案。
```

---

### 示例 3：数据处理脚本

用户原始提示词：
```
帮我写一个 Python 脚本处理 CSV 文件。
```

失效输出：
模型写了一个用 pandas 读取 UTF-8 编码 CSV 的标准脚本，但用户的 CSV 实际上是从中国银行系统导出的 GBK 编码文件、分隔符是制表符、前 3 行是系统元数据不是表头。脚本一运行就报编码错误。

改进版提示词：
```
帮我写一个 Python 脚本处理 CSV 文件。

文件特征：
- 编码：GBK（中国银行系统导出）
- 分隔符：制表符（Tab）而非逗号
- 前 3 行是系统元数据（需跳过），第 4 行是真正的表头
- 字段中可能包含逗号和换行符
- 文件大小约 200MB，需要考虑内存效率

请使用 pandas，注意处理以上所有边界情况。
```

改进后预期效果：
脚本使用 `encoding='gbk'`、`sep='\t'`、`skiprows=3` 等参数正确读取文件，对字段内的特殊字符做了处理，并用 `chunksize` 分块读取控制内存，一次跑通不报错。

## 2.4 负空间盲区

> **快速识别：** 模型在你没限制的方向上自由发挥，加了你不想要的内容

**典型症状**

- 输出里多了你完全没要求的内容（无中生有的承诺、未经授权的推荐、不请自来的替代方案）
- 模型在你没约束的维度上"创造性发挥"，而这些发挥往往踩雷
- 你说了"要什么"，但没说"不要什么"，模型在空白地带自由填充

**为什么会发生**

模型的生成逻辑是"补全"——它会尽量让输出完整、全面。如果你只说了"要什么"，模型会在所有你没提到的维度上按自己的判断去补充。这些补充可能是合理的，但也可能是你不想要甚至有害的（比如随口给客户承诺了折扣）。你没画的边界，模型不会自动守住。

**怎么修**

- **正向要求 + 负向约束配套写**：每条"要什么"后面跟一条"不要什么"。
- **单独列一段"禁止包含"清单**：把你能想到的不该出现的内容集中列出来。
- **逐维度设置边界**：内容范围、语气风格、推荐范围、承诺权限——每个维度都想一下要不要画线。

**示例**

### 示例 1：产品介绍

用户原始提示词：
```
帮我写一段产品介绍，重点突出我们产品的速度优势。
```

失效输出：
模型在强调速度优势的同时，还顺带贬低了竞争对手（"远超同类产品 XX 的龟速表现"），加入了无法核实的数据（"比行业平均快 300%"），还用了夸大其词的表述（"颠覆性的速度革命"）。

改进版提示词：
```
帮我写一段产品介绍，重点突出我们产品的速度优势。

要求：
- 突出速度优势，使用我们自己的基准测试数据：[提供数据]
- 不要提及或贬低任何竞争对手
- 不要使用无法验证的数据或百分比
- 不要使用"颠覆""革命""碾压"等夸张表述
- 语气自信但克制，面向专业技术决策者
```

改进后预期效果：
产品介绍基于真实基准测试数据描述速度优势，语气专业克制，没有竞品攻击、虚假数据或夸张措辞，可以直接放进面向技术决策者的产品文档中。

---

### 示例 2：邮件起草

用户原始提示词：
```
帮我写一封邮件，告诉客户项目会延期两周。
```

失效输出：
邮件中模型自行加入了"我们将为您提供 15% 的费用减免作为补偿"——用户从未授权过任何补偿承诺，模型自行创造了一个商业承诺。

改进版提示词：
```
帮我写一封邮件，告诉客户项目会延期两周。

内容要求：
- 说明延期原因（第三方供应商交付延迟）
- 给出新的预计交付日期
- 表达歉意

禁止包含：
- 任何形式的补偿承诺（折扣、退款、免费服务等）
- 承认我方过错的措辞（延期原因是第三方，不是我们）
- 对未来项目的时间线承诺
```

改进后预期效果：
邮件准确传达了延期信息和原因，表达了歉意，没有任何未经授权的补偿承诺或归责措辞，可以直接发送给客户而不会造成商业风险。

---

### 示例 3：技术博客

用户原始提示词：
```
帮我写一篇关于 Docker 容器化部署的技术博客。
```

失效输出：
模型写了一篇不错的博客，但在末尾加入了"如果你觉得 Docker 太复杂，也可以考虑 Podman 或直接使用云服务商的容器平台"这样的替代方案推荐——偏离了推广 Docker 的写作目的。

改进版提示词：
```
帮我写一篇关于 Docker 容器化部署的技术博客。

内容范围：
- 聚焦 Docker 本身，涵盖基础概念、Dockerfile 编写、docker-compose 实战
- 面向有 Linux 基础但没有容器化经验的开发者

禁止包含：
- 不要推荐或讨论 Docker 的替代方案（Podman、containerd、云服务等）
- 不要讨论 Docker 的缺点或局限性
- 不要加入与容器编排（Kubernetes）相关的内容
```

## 本类自检清单

写完 Prompt 后，逐条检查：

- [ ] **意图透传**：有没有只说了"做什么"没说"为了什么效果"？→ 补上目标效果、使用场景和受众
- [ ] **歧义词检查**：Prompt 里有没有"优化""专业""详细""简洁"这类模糊词？→ 拆解成具体操作，或先下定义再使用
- [ ] **隐含前提**：有没有你觉得"不用说也知道"的背景信息？→ 写出来，特别是技术栈、法规、预算、团队规模等约束
- [ ] **负空间封堵**：有没有只说了"要什么"没说"不要什么"？→ 加一段"禁止包含"清单，逐维度画线
- [ ] **对标示例**：模型可能理解偏的地方，有没有给一个参考示例帮它校准？→ 一个好示例胜过三段描述
