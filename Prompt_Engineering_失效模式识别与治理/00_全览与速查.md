# Prompt Engineering 失效模式：全览与速查

> **本文档定位：** 实战故障排查入口。遇到 Prompt 输出异常时，通过本文快速定位失效类型，再按需跳转到对应子模块查看详细解释和示例。

## 目标读者

有实战经验的开发人员——能写代码、能跑通项目，但 Prompt Engineering 知识是碎片化习得的。

**本文档帮你建立三种能力：**
- **识别能力**：看到输出异常，能说出是哪类失效
- **归因能力**：知道这个失效为什么发生
- **处置能力**：有 2~3 个可立即尝试的修复动作

**使用方法：** 建议先将全文通读几遍，在大脑中建立 27 种失效模式的知识地图。实战中遇到输出异常时，凭记忆快速诊断；若模式之间混淆拿不准，再回来查速查表和辨析文档。

## 为什么要学失效模式

本质上，你在学一套**"调试 AI 输出"的方法论**。

就像开发者需要懂常见 bug 类型（空指针、竞态条件、内存泄漏）才能高效 debug 一样，学失效模式就是建立一套针对 LLM 输出的故障排查框架。

### 1. 从"盲目试错"变成"精准定位"

- **没学之前：** 输出不对 → 换个说法再试 → 还不对 → 再换 → 反复碰运气
- **学了之后：** 输出不对 → 识别出是"指令遗漏" → 知道加编号 + 末尾核查 → 一次修好

省的是每次和 AI 来回拉扯的时间。

### 2. Prompt 一次写对的概率大幅提升

当你脑中有 27 种失效模式的索引，写 Prompt 时会自动规避已知陷阱。比如：
- 写到 5 条并列指令时，你会自然地加编号和核查步骤
- 用到"优化""专业"这类词时，你会意识到要先定义它
- 给文档让模型回答时，你会加上"只基于文档，不要引入外部知识"

这是一种**前置防御能力**，而非事后修补。

### 3. 在 AI 协作中建立可靠的工程直觉

碎片化的 PE 技巧（"加 few-shot""用 CoT"）是零散招式，失效模式框架是内功心法——你理解的是**模型为什么会这样行为**，而不只是背诵某个技巧。

### 4. 对使用 Agent/工具链的开发者尤其关键

当 LLM 从"聊天工具"变成"系统组件"（Agent、RAG、自动化流水线），一个 Prompt 失效可能导致：
- Agent 越权执行了不该做的操作
- RAG 系统混入了文档外的幻觉信息
- 多轮对话中捏造了用户没说过的需求

这时候失效模式识别不再是"提效"，而是**防事故**。

> **一句话总结：** 学失效模式 = 把"和 AI 对话"从碰运气变成工程化可控的过程。对开发者来说，这是当前 AI 协作中投入产出比最高的元技能之一。

> **注意：** 本文档描述的是模型的**系统性弱点和风险倾向**，而非每次必现的 bug。能力越强的模型在越简单的场景下越不容易触发——但在真实的复杂业务场景中，这些失效模式依然是高频出现的。

## 诊断流程图

遇到 Prompt 输出异常时，按以下路径快速定位：

```
你的 Prompt 输出哪里不对？
│
├─ 指令没被好好执行
│   ├─ 有的要求被漏掉了 ─────────────────── → 指令遗漏（01）
│   ├─ 字面上做了，但结果不是我想要的 ────── → 字面服从（01）
│   ├─ 两个要求互相打架，模型默默放弃一个 ── → 指令冲突（01）
│   └─ 最重要的反而最敷衍 ─────────────────── → 优先级混乱（01）
│
├─ 模型理解偏了
│   ├─ 做了，但方向完全不对 ────────────────── → 意图错位（02）
│   ├─ "优化""专业"这类词被误解了 ─────────── → 歧义词陷阱（02）
│   ├─ 给了个"通用版"，没考虑我的具体情况 ── → 隐含假设缺失（02）
│   └─ 在我没限制的方向上自由发挥 ──────────── → 负空间盲区（02）
│
├─ 内容/事实有问题
│   ├─ 自信地给了错误的数据/名称/日期 ─────── → 事实幻觉（03）
│   ├─ 回答了文档里根本没有的信息 ──────────── → 推断幻觉（03）
│   ├─ 信息是过时的，但没有任何提示 ─────────── → 时效幻觉（03）
│   └─ 给了一堆"正确的废话" ──────────────── → 过度泛化（03）
│
├─ 格式/结构乱了
│   ├─ 前几条格式好，越往后越随意 ──────────── → 格式漂移（04）
│   ├─ JSON/YAML 写到一半变成自然语言 ─────── → 结构坍塌（04）
│   └─ 嵌套层级关系错了或被压平了 ──────────── → 层级混乱（04）
│
├─ 之前说的被忘了
│   ├─ 开头设的规则，后半段不管用了 ─────────── → 长程遗忘（05）
│   ├─ 放在文档中间的关键信息被漏掉 ─────────── → 位置偏置（05）
│   ├─ 回答混入了文档中没有的"常识" ─────────── → 来源混淆（05）
│   └─ 模型说"你之前提到过"但我没说过 ─────── → 多轮记忆捏造（05）
│
├─ 推理有漏洞
│   ├─ 结论看着对，但中间推理链断了 ─────────── → 推理跳步（06）
│   ├─ 一开始就判错方向，后面全在错误前提上自圆其说 → 路径锁定（06）
│   ├─ 顺着我问题的暗示方向说了 ──────────── → 确认偏误（06）
│   └─ 一串"因此""所以"但每步都站不住 ────── → 伪逻辑链（06）
│
└─ 模型行为本身不对
    ├─ 角色扮演突然蹦出"作为一个 AI" ─────── → 角色出戏（07）
    ├─ 正常问题被当成有害请求拒绝了 ─────────── → 过度拒绝（07）
    ├─ 不知道答案但不敢说，编了一个 ──────────── → 角色压力服从（07）
    └─ 只让查数据，它顺手建了仪表盘还发了邮件 ── → 越权执行（07）
```

括号中的数字对应子模块文件编号。

## 失效模式速查表

| 类别 | 失效模式 | 典型症状 | 核心触发条件 | 首选对策 |
|------|---------|---------|-------------|---------|
| **指令遵循** | 指令遗漏 | 给了 5 条要求，只满足了前 2-3 条 | 多条并列指令 | 编号 + 末尾核查步骤 |
| **指令遵循** | 字面服从 | 技术上满足了要求，但结果完全不是想要的 | 只描述操作不描述意图 | 补充意图 + 提供负例 |
| **指令遵循** | 指令冲突 | 两个要求互相矛盾，模型默默放弃了一个 | 隐性矛盾需求并列 | 自洽性检查 + 优先级声明 |
| **指令遵循** | 优先级混乱 | 最重要的需求反而最敷衍 | 多指令无主次 | 标注优先级 + 核心前置 |
| **语义理解** | 意图错位 | 做了，但方向偏离了真正的需求 | 目标描述不明确 | 描述目标而非操作 |
| **语义理解** | 歧义词陷阱 | "优化""专业"等词被按默认含义理解 | 使用模糊形容词 | 拆解为具体操作描述 |
| **语义理解** | 隐含假设缺失 | 给了"通用版"答案，忽略了你的具体背景 | 未写出"理所当然"的前提 | 显式化所有隐含前提 |
| **语义理解** | 负空间盲区 | 在你没限制的方向上加了不想要的内容 | 只定义要什么 | 同时定义不要什么 |
| **知识事实** | 事实幻觉 | 自信地给出具体数字/名称，查证后是编的 | 低频冷僻知识 | RAG + 要求标注不确定性 |
| **知识事实** | 推断幻觉 | 回答了给定文档里根本没有的信息 | 文档未覆盖的内容 | 范围锁定 + 要求引用原文 |
| **知识事实** | 时效幻觉 | 给的信息过时了，但语气毫无不确定性 | 知识截止日期后的变化 | 注入当前日期 + 实时搜索 |
| **知识事实** | 过度泛化 | 回答全是"正确的废话"，毫无针对性 | 缺乏具体上下文 | 提供充足背景 + 缩小问题 |
| **格式结构** | 格式漂移 | 前几条格式完美，越往后越随意 | 长输出任务 | Few-shot 示例 + 分批生成 |
| **格式结构** | 结构坍塌 | JSON/YAML 写到一半突然变成自然语言 | 复杂结构化输出 | 禁止回退 + Schema 约束 |
| **格式结构** | 层级混乱 | 嵌套结构的层级关系错了或被压平 | 多层嵌套结构 | 提供完整 Schema 示例 |
| **上下文管理** | 长程遗忘 | 开头设定的规则，到后半段就不管用了 | 超长上下文 + 早期指令 | 关键指令首尾重复 |
| **上下文管理** | 位置偏置 | 放在文档中间的关键信息被遗漏 | 关键信息在中间位置 | 重要内容前置或后置 |
| **上下文管理** | 来源混淆 | 回答混入了文档中没有的"常识" | 文档与模型内置知识混合 | 明确划定知识边界 |
| **上下文管理** | 多轮记忆捏造 | 模型说"你之前提到过"，但你没说过 | 长对话 + 模糊信息 | 应用层维护事实状态 |
| **推理逻辑** | 推理跳步 | 结论听着有理，中间推理链条断了 | 复杂因果分析 | Chain-of-Thought 提示 |
| **推理逻辑** | 路径锁定 | 一开始判错方向，后续全在错误前提上展开 | 初期假设错误 | 要求多假设并行分析 |
| **推理逻辑** | 确认偏误 | 顺着问题的暗示方向回答 | 问题措辞暗示答案 | 中性措辞 + 平衡视角 |
| **推理逻辑** | 伪逻辑链 | 一串"因此""所以"看着流畅但站不住 | 推理无需举证 | 要求量化或举证 |
| **角色边界** | 角色出戏 | 角色扮演突然蹦出"作为一个 AI" | 触发元认知 | 系统提示强化角色约束 |
| **角色边界** | 过度拒绝 | 正常问题被当成有害请求拒绝 | 安全机制过度泛化 | 明确声明意图和上下文 |
| **角色边界** | 角色压力服从 | 不知道答案但不敢说，编了一个 | 不知道但要"有帮助" | 显式授权"不知道" |
| **角色边界** | 越权执行 | 只让查数据，它还生成报告发了邮件 | 任务边界未明确 | 明确边界 + 先报告后执行 |
