# RAG 的 Context 注入

**一句话：** RAG（检索增强生成）的本质是"先找资料，再回答问题"——通过检索外部知识并注入到 context 中，让模型基于真实数据生成回答，而非凭记忆编造。

## 是什么

RAG 的全称是 Retrieval-Augmented Generation（检索增强生成）。拆开来看：

- **Retrieval（检索）**：根据用户的问题，从外部知识库中找到相关的信息片段
- **Augmented（增强）**：把找到的信息注入到发给模型的 context 中
- **Generation（生成）**：模型基于注入的信息来生成回答

为什么需要 RAG？因为 LLM 有两个根本性的局限：

1. **知识有截止日期**：模型的训练数据不包含最新信息。你问它"鲜果优选现在芒果多少钱"，它不可能知道。
2. **知识不包含你的私有数据**：你的产品目录、订单系统、库存状态——这些模型从未见过。

RAG 的解决思路很直接：既然模型不知道，那就把它需要的信息找出来，塞进 context 里，让它"开卷考试"。

### 完整 Pipeline

一个 RAG 系统从用户提问到最终回答，经历以下步骤：

```
用户提问
  ↓
① Query 处理（理解用户到底在问什么）
  ↓
② Embedding（把问题转成向量表示）
  ↓
③ Vector Search（在向量数据库中搜索相似内容）
  ↓
④ Ranking / Reranking（对搜索结果排序，筛选最相关的）
  ↓
⑤ Format（把检索到的内容格式化为结构化文本）
  ↓
⑥ Inject（注入到 context 的合适位置）
  ↓
⑦ 模型生成回答
```

下面用「鲜果优选」客服助手的真实场景，逐步拆解每一个环节。

### Step ① Query 处理

用户的原始提问往往不能直接用来检索。需要做预处理：

```
用户原始问题："你们那个芒果怎么这么贵啊，上次买才 29 块"

处理后的检索 query：
- 主查询："芒果 价格"
- 扩展查询："芒果 促销活动"、"芒果 历史价格"
```

为什么需要处理？因为用户的口语化表达包含情绪（"怎么这么贵啊"）、时间引用（"上次"）、隐含意图（想知道为什么涨价 / 有没有优惠）。直接拿原始问题去做 embedding 检索，可能匹配到的是关于"贵"的内容，而不是芒果的价格信息。

常见的 Query 处理策略：

```python
def process_query(user_message: str) -> list[str]:
    """将用户消息处理为适合检索的 query 列表"""
    queries = []

    # 1. 提取核心实体和意图
    # "芒果怎么这么贵" → 实体: 芒果, 意图: 价格查询
    queries.append("芒果 价格 规格")

    # 2. 扩展相关查询（覆盖用户可能的隐含需求）
    queries.append("芒果 促销 优惠活动")

    # 3. 如果有时间引用，加入时间相关查询
    queries.append("芒果 价格变动 调价")

    return queries
```

### Step ② Embedding（向量化）

把处理后的 query 转换成数学向量（一组数字），这样才能和知识库中的内容做相似度比较。

```python
from openai import OpenAI

client = OpenAI()

def embed_query(query: str) -> list[float]:
    response = client.embeddings.create(
        model="text-embedding-3-small",
        input=query
    )
    return response.data[0].embedding  # 返回一个 1536 维的向量
```

需要注意：知识库中的文档在入库时就已经做过 embedding 了。检索时只需要对 query 做 embedding，然后和已有的向量做比较。

### Step ③ Vector Search（向量搜索）

在向量数据库中，找到和 query 向量最相似的文档片段：

```python
def search_knowledge_base(query_embedding: list[float], top_k: int = 5):
    """在向量数据库中搜索最相似的文档片段"""
    results = vector_db.query(
        vector=query_embedding,
        top_k=top_k,
        include_metadata=True
    )
    return results

# 返回示例：
# [
#   {"text": "越南金枕芒果，规格500g/盒，售价39.9元...", "score": 0.92, "source": "product_catalog"},
#   {"text": "芒果类产品2024年3月调价通知...", "score": 0.87, "source": "price_notices"},
#   {"text": "当季水果促销：芒果买二送一...", "score": 0.85, "source": "promotions"},
#   {"text": "芒果保存方法：常温放置2-3天...", "score": 0.71, "source": "product_faq"},
#   {"text": "热带水果运输损耗率约15%...", "score": 0.65, "source": "logistics_docs"},
# ]
```

### Step ④ Ranking / Reranking（重排序）

向量相似度只是"语义接近"，不等于"对回答这个问题最有用"。Reranking 用更精细的模型重新评估每个结果的相关性：

```python
def rerank_results(query: str, search_results: list, top_k: int = 3):
    """用 reranker 模型重新评估相关性"""
    reranked = reranker.rank(
        query=query,
        documents=[r["text"] for r in search_results],
    )

    # 过滤低相关性结果
    filtered = [r for r in reranked if r["relevance_score"] > 0.5]

    return filtered[:top_k]

# Reranking 后的结果（注意顺序可能变化）：
# 1. "越南金枕芒果，规格500g/盒，售价39.9元..."  (最直接回答价格问题)
# 2. "当季水果促销：芒果买二送一..."              (回应"贵"的隐含诉求)
# 3. "芒果类产品2024年3月调价通知..."             (解释为什么和上次不一样)
#
# 被过滤掉的：
# - "芒果保存方法" → 和价格问题无关
# - "热带水果运输损耗率" → 太泛，不是用户需要的
```

为什么 reranking 很重要？一个真实的例子：用户问"芒果过敏能吃吗"，embedding 搜索可能返回大量芒果产品信息（因为都包含"芒果"这个词，语义上接近），但 reranker 能识别出用户关心的是"过敏"，优先返回健康提示类文档。

### Step ⑤ Format（格式化检索结果）

这一步是 Context Engineering 的关键——同样的信息，格式不同，模型的理解和使用效果天差地别。

**糟糕的做法：把检索结果原样扔进去**

```
以下是检索到的相关信息：
越南金枕芒果，规格500g/盒，售价39.9元，产地越南，保质期常温3天冷藏7天，
含糖量约15%，属于热带水果，当前库存充足，SKU编号MG2024031500，入库时间
2024-03-15，供应商编号SUP-VN-003...当季水果促销：芒果买二送一，活动时间
3月15日至3月31日，适用SKU：MG2024031500、MG2024031501...
```

**好的做法：结构化、有标记、有来源**

```
## 检索到的相关信息

### 产品信息 [来源: 产品目录, 更新时间: 2024-03-15]
- 产品名：越南金枕芒果
- 规格：500g/盒
- 当前售价：39.9 元
- 库存状态：充足

### 当前促销活动 [来源: 促销系统, 活动截止: 2024-03-31]
- 活动内容：芒果买二送一
- 适用产品：越南金枕芒果（500g/盒）

### 价格变动记录 [来源: 调价通知, 发布时间: 2024-03-01]
- 2024年3月起，芒果类产品因进口成本上涨，价格上调约15%
- 调价前售价：34.9 元 → 调价后售价：39.9 元
```

两者的区别：

| 维度 | 糟糕版本 | 好的版本 |
|------|---------|---------|
| 结构 | 一大段文字混在一起 | 按类型分块、用 Markdown 结构化 |
| 噪音 | 包含 SKU、供应商编号等内部信息 | 只保留回答问题需要的字段 |
| 来源 | 不知道信息从哪来 | 每块信息标注来源和时间 |
| 可用性 | 模型需要自己从中提取有用信息 | 模型可以直接引用 |

格式化的代码实现：

```python
def format_retrieved_context(results: list[dict]) -> str:
    """将检索结果格式化为结构化的 context 文本"""
    sections = []

    for result in results:
        source = result.get("source", "unknown")
        updated = result.get("updated_at", "未知")
        content = result["formatted_content"]  # 已经提取了关键字段

        section = f"### {result['title']} [来源: {source}, 更新时间: {updated}]\n{content}"
        sections.append(section)

    header = "## 检索到的相关信息（以下信息来自知识库，请基于这些信息回答用户问题）\n"
    return header + "\n\n".join(sections)
```

### Step ⑥ Inject（注入到 Context）

检索到的信息应该放在 context 的什么位置？这直接影响模型的回答质量。

一个完整的 context 结构：

```
┌─────────────────────────────────────────┐
│ System Prompt                           │  ← 身份定义、行为规则
│ "你是鲜果优选的客服助手..."              │
├─────────────────────────────────────────┤
│ 检索到的相关信息（RAG 注入区域）          │  ← 检索结果放在这里
│ ### 产品信息 [来源: ...]                │
│ ### 促销活动 [来源: ...]                │
│ ### 价格变动 [来源: ...]                │
├─────────────────────────────────────────┤
│ 对话历史                                │  ← 之前的对话轮次
│ User: 上次在你们这买了芒果挺好吃的        │
│ Assistant: 谢谢您的好评！...             │
├─────────────────────────────────────────┤
│ 当前用户消息                             │  ← 最新的问题
│ User: 你们那个芒果怎么这么贵啊，          │
│       上次买才29块                       │
└─────────────────────────────────────────┘
```

**为什么放在 system prompt 之后、对话历史之前？**

- 放在 system prompt 中：会和角色指令混在一起，模型可能把检索信息当成"永久指令"
- 放在对话历史之后、用户消息之前：对于较长的对话历史，检索信息会被"推远"，模型注意力下降（参考 M1 中关于位置效应的讨论）
- 放在 system prompt 之后、对话之前：模型把它当作"参考资料"来用，既不会和指令混淆，又离最终回答足够近

完整的 messages 构造：

```python
def build_messages(system_prompt: str, retrieved_context: str,
                   chat_history: list[dict], user_message: str) -> list[dict]:
    """构建完整的 messages 列表"""

    messages = [
        {
            "role": "system",
            "content": system_prompt
        },
        {
            # 将检索结果作为一条 system 消息注入
            "role": "system",
            "content": retrieved_context
        },
    ]

    # 添加对话历史
    messages.extend(chat_history)

    # 添加当前用户消息
    messages.append({
        "role": "user",
        "content": user_message
    })

    return messages
```

有些实现会把检索结果直接拼接到 user message 中：

```python
# 另一种做法：把检索结果和用户消息放在一起
user_content = f"""以下是和用户问题相关的参考信息：

{retrieved_context}

---
用户的问题是：{user_message}

请基于以上参考信息回答用户的问题。如果参考信息中没有相关内容，请如实告知。"""
```

两种做法各有优劣。放在 system 中更"干净"，模型把它当参考资料。放在 user 中更"显眼"，模型不容易忽略。实际开发中需要根据具体模型和场景测试。

## 为什么重要

### 场景一：没有 RAG 的客服助手

用户问："你们的越南金枕芒果现在多少钱？"

没有 RAG 时模型只能这样回答：
> "抱歉，我无法查看实时价格。建议您查看我们的APP或小程序获取最新价格信息。"

或者更糟糕——模型编造一个价格：
> "越南金枕芒果目前售价 25.9 元/500g。"（完全是虚构的）

有 RAG 时，模型的 context 中已经有了产品目录信息，可以准确回答：
> "越南金枕芒果目前售价 39.9 元/500g 盒装。现在正好有买二送一的活动，到3月31日截止，相当于每盒约 26.6 元，比较划算哦！"

### 场景二：Chunk Size 的影响

假设知识库中有一篇产品文档：

```
越南金枕芒果

品种介绍：
金枕芒果原产于越南，果肉金黄色，口感细腻，甜度高。
我们精选越南南部种植基地的优质芒果，从采摘到上架不超过72小时。

规格与价格：
- 精选装：500g/盒，售价39.9元
- 家庭装：1kg/盒，售价69.9元
- 派对装：2kg/盒，售价129.9元

储存方法：
未成熟的芒果可常温放置2-3天催熟。已成熟的芒果建议冷藏保存，
保质期约5-7天。切开后请在24小时内食用。

过敏提示：
芒果含有漆酚类物质，部分人群可能出现接触性过敏反应。
如果您对芒果过敏，建议避免购买。食用前可先少量尝试。
```

**Chunk 太小（每 50 字一个 chunk）：**

```
chunk_1: "越南金枕芒果\n\n品种介绍：金枕芒果原产于越南，果肉金黄色"
chunk_2: "口感细腻，甜度高。我们精选越南南部种植基地的优质芒果"
chunk_3: "从采摘到上架不超过72小时。\n\n规格与价格：\n- 精选装"
chunk_4: "500g/盒，售价39.9元\n- 家庭装：1kg/盒，售价69.9元"
```

用户问"芒果多少钱有哪些规格"，检索到 chunk_4，但它缺少产品名称的上下文——模型可能不确定这到底是哪种芒果的价格。而且 chunk_3 和 chunk_4 被人为割裂了。

**Chunk 太大（整篇文档一个 chunk）：**

如果你有 500 篇产品文档，每篇都是一个完整 chunk，那么检索到的 3 个结果就是 3 篇完整文档。大量内容注入到 context 中，其中大部分是噪音（比如用户问价格，但储存方法和过敏提示也一起进来了），既浪费 token，又可能干扰模型注意力。

**合适的 Chunk Size（按语义段落分割）：**

```
chunk_A: "越南金枕芒果 - 品种介绍：金枕芒果原产于越南，果肉金黄色，
          口感细腻，甜度高。精选越南南部种植基地的优质芒果，从采摘到
          上架不超过72小时。"

chunk_B: "越南金枕芒果 - 规格与价格：精选装500g/盒售价39.9元；
          家庭装1kg/盒售价69.9元；派对装2kg/盒售价129.9元。"

chunk_C: "越南金枕芒果 - 储存方法：未成熟常温放置2-3天催熟；
          已成熟冷藏保存，保质期约5-7天；切开后24小时内食用。"

chunk_D: "越南金枕芒果 - 过敏提示：含漆酚类物质，部分人群可能
          接触性过敏。过敏者建议避免购买，食用前可先少量尝试。"
```

注意每个 chunk 都包含了产品名称（"越南金枕芒果"）作为上下文锚点，这样即使单独检索到某一个 chunk，也能明确知道它在说什么产品。

**Chunk Size 的经验参考值：**

| 内容类型 | 建议 Chunk 大小 | 原因 |
|---------|---------------|------|
| 产品信息 | 按字段分块（名称+某一类属性） | 字段间相对独立 |
| FAQ | 一个问答对一个 chunk | 问答对是天然的语义单元 |
| 政策文档 | 按段落/条款分块（200-500 字） | 条款通常自包含 |
| 操作手册 | 按步骤/小节分块 | 步骤是天然边界 |

## 设计启示

### 1. RAG 不只是"搜索+拼接"，每一步都是设计决策

RAG 管线中的每一步都涉及 context engineering 的考量：

- **Query 处理**决定了你能不能找到正确的信息
- **Chunk 策略**决定了找到的信息质量
- **Reranking**决定了注入的信息是否真正相关
- **Format**决定了模型能否有效理解和使用信息
- **Position**决定了模型有多大概率"注意到"这些信息

最大的杠杆通常在 **Format** 和 **Chunk 策略**这两步——它们直接决定了 context 的质量。

### 2. 检索结果需要"加工"，不能直接使用

从数据库或搜索引擎拿到的原始数据，几乎从来不适合直接注入 context。你需要：

```python
# 原始数据（来自数据库）
raw_product = {
    "sku": "MG2024031500",
    "name": "越南金枕芒果",
    "spec": "500g/盒",
    "price_cents": 3990,
    "category_id": 203,
    "supplier_id": "SUP-VN-003",
    "warehouse_zone": "B-3-07",
    "created_at": "2024-03-15T08:30:00Z",
    "stock_count": 342,
    "min_stock_alert": 50,
}

# 加工后的 context（给模型看的）
formatted = """
### 产品：越南金枕芒果
- 规格：500g/盒
- 售价：39.9 元
- 库存状态：充足
"""
```

原则：只保留对回答问题有用的字段，用人类可读的格式呈现，隐藏内部编号。

### 3. 告诉模型如何使用检索到的信息

不要仅仅注入信息，还要给模型使用指引：

```
## 参考信息使用规则
- 请基于以下检索到的信息回答用户问题
- 如果检索信息不足以回答，请如实告诉用户，不要编造
- 引用价格、规格等具体数据时，以检索信息为准
- 如果检索信息之间存在矛盾，以更新时间较新的为准
```

### 4. 设计"检索失败"的兜底方案

检索不到相关信息是常态，需要在 context 中提前规划：

```
## 当检索信息不足时
如果以下参考信息无法回答用户的问题，请回复：
"关于这个问题，我需要帮您查一下，请稍等。"
然后调用 escalate_to_human() 工具转接人工客服。

不要基于自己的训练知识猜测产品价格、库存、促销信息。
```

## 常见误区

### 误区一："RAG 能解决幻觉问题"

**错误理解：** 用了 RAG，模型就不会编造信息了。

**实际情况：** RAG 只是给模型提供了正确信息的"机会"。模型仍然可能：
- **忽略检索结果，使用自己的训练知识**（尤其是当检索结果和训练知识冲突时）
- **对检索结果做错误推理**（信息是对的，但推导出了错误结论）
- **在检索信息的"缝隙"中编造细节**（检索到了价格但没有配送信息，模型可能编造配送时间）

正确的做法：RAG + 明确的指令约束（"只基于以下信息回答，不确定就说不知道"）+ 输出验证。

### 误区二："把搜索结果直接塞进 prompt 就行"

**错误理解：** 检索出来什么就放什么，模型自己会挑有用的。

**实际情况：** 未加工的检索结果常常包含：
- 无关的字段（SKU、内部编号、数据库时间戳）
- 格式混乱的文本（HTML 残留、特殊字符、编码问题）
- 重复的信息（多个 chunk 说的是同一件事）
- 和问题无关的内容（相似度分数刚好过阈值的噪音结果）

这些噪音会消耗 token 预算、分散模型注意力、甚至误导模型。格式化和筛选这一步不是可选的，是必须的。

### 误区三："Embedding 相似度高就是相关性高"

**错误理解：** 向量搜索返回的 top 结果就是最相关的。

**实际情况：** Embedding 相似度衡量的是语义距离，不是"对回答问题的有用程度"。例如：

```
用户问："芒果过敏怎么办？"

Embedding 搜索结果：
1. "越南金枕芒果，果肉金黄色..." (相似度 0.89) — 包含"芒果"，但说的是产品介绍
2. "芒果千层蛋糕，含芒果果肉..." (相似度 0.86) — 也包含"芒果"，完全不相关
3. "芒果含漆酚类物质，过敏者..." (相似度 0.84) — 真正相关的内容

Reranking 后：
1. "芒果含漆酚类物质，过敏者..." — 直接回答问题 ✓
2. "越南金枕芒果..." — 可能有参考价值
3. "芒果千层蛋糕..." — 过滤掉 ✗
```

这就是为什么 reranking 步骤很重要——它弥补了 embedding 搜索"只看像不像，不看有没有用"的局限。

### 误区四："Chunk 越大信息越完整，效果越好"

**错误理解：** 为了避免信息丢失，应该用大 chunk，甚至把整篇文档作为一个 chunk。

**实际情况：** Chunk 过大带来的问题：
- **检索精度下降**：一个 2000 字的 chunk 中可能只有一句话和问题相关，但整个 chunk 的 embedding 被其他内容"稀释"了
- **注入噪音增多**：大量无关内容占用 context 空间
- **多样性降低**：top-3 结果可能来自同一篇文档的不同部分，而不是三个不同来源的信息

正确的做法是按语义边界分割，每个 chunk 是一个自包含的信息单元，并在 chunk 中保留足够的上下文标识（比如产品名称、所属章节）。
