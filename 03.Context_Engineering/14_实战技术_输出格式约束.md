# 输出格式约束

**一句话：** 在 context 中明确规定模型的输出格式，是控制模型行为最直接、最可靠的手段之一——你定义了输出的"形状"，模型就会在这个形状里填充内容。

## 是什么

当你调用 LLM 时，模型默认会用"自由散文"的方式回答——想说多少说多少，想用什么格式就用什么格式。这在闲聊中没问题，但在工程场景中，你几乎总是需要模型按照特定格式输出。

**输出格式约束**就是在 context（通常是 system prompt 或指令部分）中，明确告诉模型"你的回答必须长什么样"。这不是一个单独的技术，而是 Context Engineering 的核心操作之一——通过塑造 context 来塑造输出。

格式约束的常见类型，从松到紧排列：

```
松                                                               紧
│                                                                 │
▼                                                                 ▼
自然语言提示     模板/步骤要求     Markdown结构     XML标签     JSON Schema
"简洁回答"     "先问候再回答"    "用表格呈现"    "<answer>"   {"type":"object",...}
```

### 1. 自然语言提示（最松）

```
请用一句话回答，不要超过50字。
```

这是最弱的约束——模型大概率会遵守，但没有结构化保证。

### 2. 模板/步骤要求

```
请按以下步骤回答：
1. 先确认用户的问题
2. 给出直接答案
3. 补充一个相关建议
```

指定了输出的"段落结构"，但每个段落内部仍然自由。

### 3. Markdown 结构

```
请用以下格式回答：

## 问题确认
（复述用户问题）

## 解答
（直接回答）

## 相关建议
（一条补充建议）
```

有明确的标题层级，便于前端解析和展示。

### 4. XML 标签

```
请用以下格式回答：

<confirmation>复述用户问题</confirmation>
<answer>直接回答</answer>
<suggestion>一条补充建议</suggestion>
```

XML 标签在 LLM 中有两个独特优势：第一，标签名本身就是语义提示，模型看到 `<answer>` 就知道这里该放答案；第二，标签具有明确的开闭结构，便于程序解析提取。Claude 系列模型对 XML 格式的遵循度尤其好。

### 5. JSON Schema（最紧）

```
请以 JSON 格式回答，严格遵循以下结构：

{
  "confirmation": "复述用户问题",
  "answer": "直接回答",
  "suggestion": "一条补充建议",
  "confidence": "high/medium/low"
}
```

最严格的约束——输出必须是合法的 JSON，每个字段名和类型都预先定义。这种格式最适合需要被下游程序解析的场景。

### 格式约束的底层原理

为什么在 context 中写一段格式要求，模型就会遵循？因为 LLM 的生成过程是"续写"——模型根据前文预测下一个 token。当你在 system prompt 中写了 `请以 JSON 格式输出`，并给了一个模板，模型在生成第一个 token 时就会强烈倾向于输出 `{`，因为这和前文的指令最连贯。格式约束本质上是在**利用模型的续写机制，把输出引导到特定的 token 序列模式上**。

约束越具体、越结构化，模型"跑偏"的可能性就越小——因为每一步可选的 token 都被前文的模式大幅缩窄了。

## 为什么重要

### 场景一：客服回答一致性（鲜果优选）

假设鲜果优选的客服助手没有格式约束，面对同一个问题"草莓能放多久"，不同时刻可能产生完全不同风格的回答：

```
回答 A："草莓一般冰箱冷藏能放3-5天。"
回答 B："您好！关于草莓的保存问题，让我为您详细解答。草莓是一种非常娇嫩的水果……（300字长文）"
回答 C："3-5天冷藏，1个月冷冻。注意不要洗了再放。"
```

三种回答都"对"，但风格差异巨大。用户体验不一致，品牌感全无。

加入格式约束后：

```
你是鲜果优选的客服助手。回答用户问题时，严格按以下结构：

1. **问候**：一句话打招呼（不超过10字）
2. **解答**：直接回答用户问题（2-3句话）
3. **保养贴士**：给一条实用的补充建议
4. **追问**：问用户是否还有其他问题
```

现在每次回答都是同样的节奏：

```
您好！🍓

草莓冷藏保存一般可以放 3-5 天。建议不要提前清洗，用厨房纸巾垫在容器底部吸收多余水分，能有效延长保鲜时间。

小贴士：如果买得多一时吃不完，可以洗净去蒂后冷冻保存，做果昔时直接取用。

请问还有其他想了解的吗？
```

### 场景二：下游系统集成

鲜果优选的客服系统不只是给用户看文字——它还需要把一些结构化数据传给后端系统。比如，当模型判断用户需要人工客服时，需要把这个判断传给路由系统。

没有格式约束的输出：

```
"这个问题比较复杂，我建议您联系人工客服处理。"
```

后端系统怎么知道这是"需要转人工"？只能靠关键词匹配——脆弱、不可靠。

加入 JSON 格式约束：

```
你是鲜果优选的客服助手。每次回答必须输出以下 JSON：

{
  "reply_text": "给用户看的回复文本",
  "intent": "问题分类，如：product_info/order_status/complaint/other",
  "needs_human": true/false,
  "confidence": "high/medium/low"
}
```

现在输出变成：

```json
{
  "reply_text": "非常抱歉给您带来不好的体验。您描述的商品变质问题我们非常重视，我正在为您转接专属客服，会优先为您处理退款和补偿。",
  "intent": "complaint",
  "needs_human": true,
  "confidence": "high"
}
```

后端直接读 `needs_human` 字段就能做路由决策，`intent` 字段可以分配给对应类目的专家客服。这就是格式约束的工程价值——**把模型的输出变成程序可消费的结构化数据**。

### 场景三：格式约束减少歧义

模糊的输出指令会产生歧义。比如"列出草莓的营养成分"，模型可能输出一段散文，也可能输出一个列表，还可能输出一个表格。如果你的前端只能渲染 Markdown 表格，散文和列表都会显示成一坨。

明确的格式约束消除了这种不确定性：

```
请以 Markdown 表格格式列出，包含以下列：
| 营养成分 | 每100g含量 | 每日推荐占比 |
```

## 设计启示

### 根据消费者选择格式强度

格式约束的强度应该取决于**谁来消费这个输出**：

```
输出消费者            推荐格式              理由
──────────          ──────────           ──────────
用户直接阅读          自然语言 + 步骤模板    自然、易读
前端渲染展示          Markdown / XML        可解析、可排版
后端系统处理          JSON                  可直接 parse
多方同时消费          JSON（含 text 字段）   程序读 JSON，用户读 text 字段
```

### Prompt 级 vs API 级格式约束

有两种实施格式约束的方式：

**方式一：Prompt 级约束**——在 system prompt 或用户指令中写格式要求。

```python
messages = [
    {"role": "system", "content": "你必须以 JSON 格式回答..."},
    {"role": "user", "content": "草莓怎么保存？"}
]
response = client.chat.completions.create(
    model="gpt-4",
    messages=messages
)
```

优点：灵活，任何模型都支持。缺点：模型可能不遵守，输出可能不是合法 JSON（比如多了一句解释文字，或者漏了一个逗号）。

**方式二：API 级约束（Structured Output / JSON Mode）**——通过 API 参数强制输出格式。

```python
# OpenAI 的 JSON Mode
response = client.chat.completions.create(
    model="gpt-4o",
    messages=messages,
    response_format={"type": "json_object"}
)

# OpenAI 的 Structured Outputs（更严格）
response = client.chat.completions.create(
    model="gpt-4o",
    messages=messages,
    response_format={
        "type": "json_schema",
        "json_schema": {
            "name": "customer_service_response",
            "schema": {
                "type": "object",
                "properties": {
                    "reply_text": {"type": "string"},
                    "intent": {"type": "string", "enum": ["product_info", "order_status", "complaint", "other"]},
                    "needs_human": {"type": "boolean"},
                    "confidence": {"type": "string", "enum": ["high", "medium", "low"]}
                },
                "required": ["reply_text", "intent", "needs_human", "confidence"]
            }
        }
    }
)
```

API 级约束通过 **constrained decoding（约束解码）** 在模型生成 token 时就禁止非法 token，从根本上保证输出格式合法。

**选择建议：**

- 输出给人看 → prompt 级约束就够了（自然语言 + 模板）
- 输出给程序读 → 尽量用 API 级约束（保证格式合法）
- API 不支持结构化输出 → prompt 级约束 + 输出后校验 + 失败重试

### 严格格式与自然度的权衡

格式越严格，输出越可预测，但也越"机械"。

```
高自然度，低可控性：
"亲，草莓放冰箱冷藏区就行，别洗别盖保鲜膜太紧，一般能放三五天没问题~"

高可控性，低自然度：
{
  "reply_text": "草莓冷藏保存3-5天。保存时不要提前清洗，使用透气容器存放。",
  "storage_days": 4,
  "storage_method": "refrigeration"
}
```

这不是非此即彼的选择。实际工程中的常见做法是**双层设计**：

```json
{
  "reply_text": "亲，草莓放冰箱冷藏区就行，别洗别盖保鲜膜太紧，一般能放三五天没问题~",
  "metadata": {
    "intent": "product_info",
    "confidence": "high",
    "needs_human": false
  }
}
```

`reply_text` 里是给用户的自然语言（可以活泼、口语化），`metadata` 里是给程序的结构化数据（严格、可解析）。两者在同一个 JSON 中共存。

### 格式约束 + 少样本示例 = 最佳效果

单纯的格式说明有时不够——模型可能理解了格式结构，但对内容风格把握不准。最有效的做法是**格式说明 + 1-2 个完整示例**：

```
你是鲜果优选的客服助手。回答格式如下：

<greeting>简短问候</greeting>
<answer>直接解答</answer>
<tip>实用小贴士</tip>

示例：
用户：芒果怎么催熟？
<greeting>您好！</greeting>
<answer>将芒果和苹果一起放入纸袋中，密封后置于室温环境，一般1-2天即可催熟。苹果释放的乙烯气体能加速芒果成熟。</answer>
<tip>催熟后的芒果请尽快食用或转入冰箱冷藏，室温下成熟芒果只能保存1-2天。</tip>
```

示例同时传达了格式和风格——模型不仅知道要用 XML 标签，还知道 `<greeting>` 要简短、`<answer>` 要有具体方法和原理、`<tip>` 要实用可操作。

## 常见误区

**误区一："格式指令只是锦上添花的装饰"**

很多人认为格式约束只影响输出的"外观"，不影响内容质量。事实恰恰相反——格式约束会深刻影响模型的推理过程。当你要求模型先输出 `<thinking>` 再输出 `<answer>` 时，模型被迫在回答前先"想一想"，这会显著提升回答质量。当你要求模型输出结构化的 JSON 时，模型需要为每个字段独立思考，而不是在一段散文中模糊带过。**格式不仅是输出的容器，还是思维的脚手架。**

**误区二："模型总会遵守格式要求"**

即使你在 prompt 中写了详细的格式规范，模型仍然可能不遵守。常见的违规情况：

- 在 JSON 前后加上解释性文字（"好的，这是您要的 JSON：\n{...}"）
- 在 JSON 中使用注释或省略号
- 漏掉必需字段
- 字段值类型错误（该用布尔值的地方用了字符串 "true"）

对于关键场景，你需要：（1）使用 API 级强制格式；（2）对输出做 schema 校验；（3）校验失败时自动重试。不要假设模型会100%遵守格式。

**误区三："JSON 输出总是比自然语言好"**

JSON 格式有明确的适用场景——当输出需要被程序解析时。但如果输出直接给人看，强制 JSON 会有几个问题：

- 消耗更多 token（JSON 的花括号、引号、字段名都占 token）
- 降低回答的自然度和温度
- 增加系统复杂度（前端需要 parse JSON 再渲染）

对于"给人看的回答 + 给程序读的元数据"这种混合需求，用前面提到的双层设计。对于纯粹给人看的内容，Markdown 或自然语言模板往往是更好的选择。

**误区四："格式约束只需要写在 system prompt 里"**

如果对话很长，模型可能会"遗忘"system prompt 中的格式要求（尤其是格式要求在 system prompt 中不够突出时）。对于关键的格式约束，可以：

- 在 system prompt 中定义格式
- 在用户消息末尾再次提醒（"请按规定的 JSON 格式输出"）
- 使用少样本示例强化格式记忆
- 利用 API 级约束作为最终保障
