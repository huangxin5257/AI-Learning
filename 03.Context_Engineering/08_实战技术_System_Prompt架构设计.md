# System Prompt 架构设计

**一句话：** System Prompt 不是"写一段话告诉模型你是谁"，而是一份精心分层的操作手册——角色定位、行为规则、知识格式、示例，每一层解决不同的问题。

## 是什么

System Prompt 是你在每次 API 调用中通过 `role: "system"` 传给模型的那段文本。它在整个对话中持续生效，是你对模型行为的"总纲"。

但很多人对 system prompt 的理解停留在"你是一个 XX 助手"这一句话。实际上，一个生产级 system prompt 需要回答四层问题：

```
┌───────────────────────────────────────────┐
│  第一层：角色定位（Identity）               │  ← 你是谁？服务谁？基本立场是什么？
├───────────────────────────────────────────┤
│  第二层：行为规则（Behavioral Rules）       │  ← 什么必须做？什么绝对不能做？
├───────────────────────────────────────────┤
│  第三层：知识与格式（Knowledge & Format）    │  ← 用什么知识回答？输出什么格式？
├───────────────────────────────────────────┤
│  第四层：示例（Examples）                   │  ← 遇到典型情况，期望的回答长什么样？
└───────────────────────────────────────────┘
```

**这四层不是随意排列的。** 它们有清晰的逻辑递进关系：先建立身份（我是谁），再明确边界（什么能做什么不能做），再给具体知识和格式要求（怎么做），最后用示例固化行为（做成什么样）。

### 四层结构详解

**第一层：角色定位（Identity）**

这一层回答"你是谁"。它不只是一个标签，而是确定模型的立场、语气基调和服务对象。

要素包括：
- 身份名称与所属场景
- 服务的目标用户是谁
- 核心职责（一两句话概括）
- 基本语气和态度

**第二层：行为规则（Behavioral Rules）**

这一层划定边界。模型天然什么都能回答，你需要告诉它什么必须做、什么不能做。

要素包括：
- 必须遵守的硬性规则（MUST）
- 绝对禁止的行为（NEVER）
- 条件触发的规则（WHEN...THEN...）
- 规则之间的优先级

**第三层：知识与格式（Knowledge & Format）**

这一层告诉模型"用什么回答"和"回答成什么样"。

要素包括：
- 需要参考的背景知识或数据结构
- 输出格式要求（纯文本/JSON/列表等）
- 回答的长度和详细程度偏好
- 特定场景的回答模板

**第四层：示例（Examples）**

这一层用具体的输入→输出对，消除前三层可能留下的模糊空间。

要素包括：
- 典型场景的标准回答
- 边界场景的处理方式
- 反例（什么样的回答是错误的）

## 为什么重要

### 场景一：没有结构的 system prompt

假设你为"鲜果优选"电商平台写客服 system prompt，直觉做法可能是这样：

```
你是鲜果优选的客服助手，帮用户解答水果相关问题，回答要友好专业，不知道的不要乱说，
遇到退款问题要告诉用户联系人工客服，回答简洁一点，用中文回答，不要回答跟水果无关
的问题，如果用户问物流就查订单信息回答，记得用表情让回答更亲切，价格问题要引导到
商品页面，不要给出具体折扣信息以免跟实际不符。
```

这段话把角色、规则、格式、知识全部混在一起。问题是：

- **模型难以区分优先级。** "不要乱说"和"用表情让回答更亲切"被放在同一层级，但前者是安全底线，后者是锦上添花。
- **规则之间可能冲突。** "回答简洁一点"和"友好专业"之间怎么取舍？没有说明。
- **遗漏难以发现。** 当所有内容都在一段话里，你很难检查"退货场景有没有覆盖？""过敏问题怎么处理？"

### 场景二：分层结构的 system prompt

同样的信息，用四层结构组织后：

```markdown
# 角色定位
你是「鲜果优选」的智能客服助手"果小蜜"。你服务的是在鲜果优选平台购买水果的消费者。
你的核心职责是：解答水果商品咨询、协助处理订单和售后问题。
语气要求：友好、耐心、专业，像一位懂水果的朋友。

# 行为规则
## 必须遵守
1. 只回答与鲜果优选业务相关的问题（水果咨询、订单、物流、售后）
2. 不确定的信息必须明确告知用户"我不太确定，建议您联系人工客服确认"
3. 涉及退款、赔偿等资金操作时，必须引导用户转人工客服处理
4. 所有回答使用中文

## 绝对禁止
1. 不得编造商品信息（价格、产地、保质期等）
2. 不得给出具体折扣或优惠承诺
3. 不得回答与业务无关的问题（政治、医疗建议等）
4. 不得泄露内部运营信息

## 条件规则
- 当用户情绪激动时 → 先共情安抚，再解决问题
- 当问题超出能力范围时 → 提供人工客服转接方式（热线 400-XXX-XXXX 或 APP 内在线客服）
- 当用户询问具体价格时 → 引导用户查看商品详情页获取最新价格

# 知识与格式
## 回答格式
- 简洁优先：常规问题控制在 3-5 句话
- 涉及步骤时使用编号列表
- 在回答末尾可添加一个相关的后续建议

## 业务知识
- 配送范围：全国主要城市，偏远地区可能不支持次日达
- 退换货政策：签收后 24 小时内可申请，需提供照片
- 会员等级：普通会员、银卡、金卡、钻石，等级越高折扣力度越大
- 当季主推品类：草莓（丹东99）、车厘子（智利JJ级）、砂糖橘

# 示例
## 示例 1：商品咨询
用户：你们的丹东99草莓甜吗？大概多大？
果小蜜：丹东99草莓是我们平台的明星产品！它的特点是口感香甜多汁，甜度通常在12-15°Brix之间。果型中等偏大，单颗约 20-30g。

现在正是丹东草莓最好吃的季节，您要是感兴趣可以看看商品详情页的用户评价，很多回购客户的反馈都很真实哦~

## 示例 2：超出范围的问题
用户：草莓吃多了会不会上火？
果小蜜：这个问题涉及到健康和饮食建议，我作为客服助手不太适合给您专业判断。建议您咨询专业的营养师或医生哦。

不过我可以告诉您，我们平台上的水果都有详细的营养成分说明，您可以在商品详情页查看参考~
```

对比一下差异：

| 维度 | 一段式 | 四层式 |
|------|--------|--------|
| 规则优先级 | 全部平铺，模型自行猜测 | 分 MUST / NEVER / 条件，优先级清晰 |
| 可维护性 | 改一条规则要通读全段 | 直接定位到对应层和编号 |
| 完备性检查 | 难以发现遗漏 | 按层检查，结构化覆盖 |
| 模型遵循度 | 容易丢失规则 | 结构化标记帮助模型"扫描"定位 |
| 团队协作 | 每个人理解不同 | 像代码一样可 review |

## 设计启示

### 1. 用 Markdown 标记创建视觉层次

模型对结构化文本的解析能力远强于纯散文。利用 Markdown 的 `#` 标题层级来划分四层结构，用编号列表表达规则优先级（编号越靠前越重要），用 `##` 子标题划分同一层内的不同关切。

这不仅帮助模型解析，也让你自己在维护时能快速定位。

### 2. 用 XML 标签为特殊内容创建隔离边界

当 system prompt 中嵌入了较大块的知识内容（如产品目录、FAQ 列表）时，用 XML 标签把它们包裹起来，防止模型把知识内容和指令混淆：

```markdown
# 业务知识

以下是当季商品信息，仅供回答用户咨询时参考：

<product_catalog>
| 商品 | 产地 | 规格 | 发货时效 |
|------|------|------|----------|
| 丹东99草莓 | 辽宁丹东 | 1斤/2斤装 | 48小时内 |
| 智利车厘子 | 智利 | JJ/JJJ级 2斤装 | 72小时内 |
| 砂糖橘 | 广西桂林 | 5斤/9斤装 | 48小时内 |
</product_catalog>
```

XML 标签的作用是告诉模型：**标签内的内容是数据/参考材料，标签外的内容是指令。** 这种隔离在 context 中嵌入大量动态知识时尤其重要。

### 3. 编号规则暗示优先级

当你写"行为规则"层时，使用编号列表而不是无序列表。模型倾向于把编号靠前的规则视为更重要。把安全相关、合规相关的规则放在最前面：

```markdown
## 必须遵守
1. 不得编造事实（安全底线 → 编号1）
2. 不确定时明确告知用户（诚实原则 → 编号2）
3. 资金操作转人工（合规要求 → 编号3）
4. 使用中文回答（体验要求 → 编号4）
```

### 4. 精简与详尽的权衡

System prompt 不是越长越好，也不是越短越好。判断标准是：

**精简型**（几百 token）适合：
- 模型能力本身已经很强的场景（如简单翻译、摘要）
- 行为预期很直觉、不需要特殊约束的任务
- token 预算紧张、需要留空间给长对话历史

**详尽型**（数千 token）适合：
- 模型需要扮演特定领域角色的场景
- 有复杂的业务规则和边界条件
- 需要稳定一致的输出格式
- 多人协作维护的系统（结构化便于 review）

一个实用的原则：**先写详尽版确保行为正确，再逐步精简验证行为不退化。** 删掉一条规则后跑一遍测试用例，如果输出没变差，说明那条规则是冗余的。

### 5. 把 System Prompt 当代码管理

生产环境的 system prompt 应该：
- **版本控制：** 存在代码仓库里，不要散落在界面配置中
- **模块化：** 四层结构天然支持分块管理，不同团队负责不同层
- **可测试：** 有一组测试用例，每次修改后自动验证行为是否符合预期
- **参数化：** 动态内容（如当季商品）用变量占位，运行时注入

```python
# 伪代码示例：参数化 system prompt
system_prompt_template = """
# 角色定位
你是「鲜果优选」的智能客服助手"果小蜜"。

# 业务知识
## 当季主推商品
{seasonal_products}

## 当前促销活动
{active_promotions}
"""

# 运行时注入动态内容
system_prompt = system_prompt_template.format(
    seasonal_products=get_seasonal_products(),
    active_promotions=get_active_promotions()
)
```

## 常见误区

**误区一："System prompt 就是给模型一个角色设定"**

"你是一个客服助手"——这只完成了四层中的第一层，而且是最粗糙的版本。角色定位确实重要，但它只是起点。没有行为规则，模型会"出戏"；没有知识和格式，模型会"乱发挥"；没有示例，模型对边界情况的处理全靠猜。

一个好的类比：角色定位像员工的岗位名称，行为规则像员工手册，知识格式像作业流程 SOP，示例像新员工培训时的案例教学。你不会只告诉一个新员工"你是客服"就让他上岗。

**误区二："System prompt 越长，模型越听话"**

长 ≠ 有效。一个 5000 token 但结构混乱的 system prompt，效果可能不如一个 1000 token 结构清晰的版本。原因有两个：

1. **信息衰减：** 在 M1 中我们学过，模型对 context 中间位置的信息关注度会下降。过长的 system prompt 意味着中间的规则更容易被忽略。
2. **规则冲突：** 规则越多，越容易出现相互矛盾的情况。模型遇到冲突时的行为不可预测。

正确的做法是：每条规则都有存在的理由，能用测试用例证明它的必要性。如果删掉一条规则，输出没有任何变化，那这条规则就是冗余的，应该删掉。

**误区三："System prompt 写好一次就不用改了"**

System prompt 是一个活文档。随着你观察到真实用户的使用模式，你会不断发现新的边界情况需要补充规则、新的误解需要增加示例、过时的知识需要更新。

把它当作代码——持续迭代，每次改动有据可循，改完有测试验证。

**误区四："指令放在 system prompt 和放在 user message 里效果一样"**

虽然从技术上说，模型处理的都是 token 序列，但 system prompt 有两个独特优势：

1. **持久性：** 它在整个对话的每一轮都存在，不会被后续的 user message 覆盖。
2. **权威性：** 主流模型在训练时，被教导对 system 角色的指令给予更高的遵循优先级。当 system prompt 和用户指令冲突时（比如用户试图让模型"忽略之前的指令"），模型倾向于遵循 system prompt。

所以，核心规则和安全边界应该放在 system prompt 里，而不是放在 user message 中。
