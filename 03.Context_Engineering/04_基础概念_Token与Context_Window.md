# Token 与 Context Window

**一句话：** Token 是模型处理文本的最小单位，Context Window 是一次调用中模型能"看到"的 token 总量上限——理解这两个概念，你才能做出合理的 context 预算决策。

## 是什么

### Token：模型眼中的"字"

人类读文字的单位是"字"或"词"，但模型不是。模型把文本切成一种叫 **token** 的片段来处理——这个切法既不是按字符，也不是按单词，而是介于两者之间的 **子词单元（subword unit）**。

具体来说，模型使用一种预训练好的分词器（tokenizer），把文本拆解成它"认识"的片段：

```
英文示例：
"I love strawberries" → ["I", " love", " straw", "berries"]    共 4 个 token

中文示例：
"鲜果优选的退货政策" → ["鲜", "果", "优", "选", "的", "退", "货", "政策"]    约 8-9 个 token
```

几个直觉：
- 英文中，一个常见单词通常是 1 个 token，罕见长词会被拆成多个。粗略估算：**1 token ≈ 0.75 个英文单词**。
- 中文中，一个汉字通常是 1-2 个 token。粗略估算：**1 个汉字 ≈ 1-2 个 token**。
- 空格、标点、换行符都会消耗 token。
- 代码、JSON、XML 等结构化文本因为包含大量符号，token 消耗比你想象的要多。

为什么模型不直接按字符或按词处理？因为子词分词是效率和覆盖率的平衡——既不会像字符级那样序列过长，又不会像词级那样遇到生词就无法处理。

### Context Window：模型的"工作桌面"

**Context Window（上下文窗口）** 是一次 API 调用中，模型能处理的 token 总量上限。你可以把它想象成模型的工作桌面——桌面就这么大，放进去的所有材料（输入 + 输出）不能超过这个面积。

一些模型的 context window 大小：

```
模型/规格              Context Window
──────────────         ──────────────
GPT-3.5 (早期)         4,096 tokens     ≈ 3,000 字 / 6 页
GPT-4 (标准)           8,192 tokens     ≈ 6,000 字 / 12 页
GPT-4 Turbo           128,000 tokens    ≈ 96,000 字 / 200 页
Claude 3.5 Sonnet     200,000 tokens    ≈ 150,000 字 / 300 页
Claude (扩展)          1,000,000 tokens  ≈ 750,000 字 / 一本长篇小说
```

**一个关键细节：输入和输出共享同一个 window。** 如果你的 context window 是 8,192 个 token，你的输入占了 7,000，那模型最多只能输出 1,192 个 token。输入塞得越满，留给模型回答的空间就越小。

## 为什么重要

回到我们的"鲜果优选"客服助手。假设你用一个 8k token 的模型，来看两种设计方案的对比：

**方案 A：把所有商品信息塞进 context**

鲜果优选有 500 种商品，每种商品描述约 100 字（约 100-200 token）。全部塞进去：

```
500 商品 × ~150 token/商品 = ~75,000 token
```

直接超出 8k 窗口。即使你用 128k 的模型勉强塞下，问题是：用户问"草莓怎么保存"，模型需要在 75,000 token 的商品信息里找到草莓那一条——其余 499 条全是噪音。

**方案 B：只检索相关商品信息**

用户问草莓相关问题时，只把草莓的商品描述、保存建议、常见问题注入 context：

```
System prompt（身份+规则）：     ~300 token
草莓商品信息：                   ~150 token
草莓相关 FAQ（3条）：            ~300 token
对话历史（最近3轮）：            ~500 token
──────────────────────────────
总计：                          ~1,250 token
留给模型输出：                   ~6,750 token     充裕！
```

同样的 8k 模型，方案 B 不仅装得下，而且信噪比极高（越高越好）——模型看到的每一条信息都和问题相关。

**成本方面：** API 调用按 token 计费。方案 A 即使技术上可行，每次调用消耗的 token 也是方案 B 的 60 倍。如果客服助手每天处理 10,000 次对话，这个差距就是实实在在的账单数字。

## 设计启示

- **做 token 预算，像做财务预算一样。** 在设计 context 系统前，先算清楚：system prompt 占多少、历史对话占多少、检索知识占多少、留给输出多少。每一部分都有上限，加起来不能超支。
- **不要被大 window 冲昏头脑。** 128k 甚至 1M 的 context window 听起来很大，但"能装下"和"用得好"是两回事。窗口越大，成本越高，延迟越大，而且模型对超长文本的处理质量并不均匀（这个问题在下一篇"信息衰减与位置效应"中详细讨论）。
- **选择性注入 > 全量灌入。** Context Engineering 的核心工作之一，就是构建一条管道，把"当前这个问题最需要的信息"精准地注入 context，而不是把所有可能相关的信息一股脑倒进去。
- **为输出留足空间。** 如果你需要模型生成一篇详细的回答或一段长代码，要提前规划输出所需的 token 数，从输入预算中扣除。

## 常见误区

**误区一："Token 就是单词"或"token 就是字符"**
都不是。Token 是子词单元，分词方式由 tokenizer 决定。英文中 "unhappiness" 可能被切成 ["un", "happiness"] 两个 token，中文中"人工智能"可能是 3-4 个 token。不同模型的 tokenizer 不同，同一段文字在不同模型下的 token 数可能不一样。如果你需要精确计算，应该用对应模型的 tokenizer 工具实测，而不是靠估算。

**误区二："128k 的 context window 意味着我可以把所有东西都塞进去"**
技术上也许装得下，但这样做有三个代价：第一，成本暴涨（按 token 计费）；第二，延迟增加（模型处理更多 token 需要更多时间）；第三，质量下降（大量无关信息会稀释关键信息的注意力权重，导致模型"找不到重点"）。Context window 是上限，不是目标——你不应该努力填满它。

**误区三："输出不占 context window"**
占的。输入和输出共享同一个 window。很多开发者只算输入的 token 数，忘了给输出留空间，结果模型回答到一半就被截断。设计时要把输出的预期长度也算进预算。
