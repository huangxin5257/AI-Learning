# Context 是什么

**一句话：** Context 就是你每次调用模型时，模型能"看到"的全部信息——它是模型生成回答的唯一依据。

## 是什么

当你和 ChatGPT 对话、调用 API、或使用任何 LLM 应用时，模型并不"知道"任何关于你的事情。它没有记忆，没有后台数据库，没有互联网连接。它唯一能依赖的，就是这一次请求中你发给它的全部文本。

这个"全部文本"就是 **Context**。

具体来说，一次典型的 API 调用中，context 包含：

```
┌─────────────────────────────────────┐
│  System Prompt                      │  ← 你预设的身份、规则、背景知识
│  ─────────────────────────────────  │
│  用户消息 1                          │  ← 用户说了什么
│  模型回复 1                          │  ← 模型之前回答了什么
│  用户消息 2                          │
│  模型回复 2                          │
│  ...                                │
│  用户最新消息                        │  ← 当前这轮的输入
└─────────────────────────────────────┘
         ↑
    模型能看到的全部内容 = Context
```

一个关键的直觉：**模型不会"回忆"上一次对话，也不会"主动查找"信息。** 如果某条信息不在 context 里，对模型来说它就不存在。

这就像一场开卷考试——考生（模型）的表现完全取决于你（出题者）在卷子上放了什么参考资料。放对了资料，学生答得准；放错了或没放，学生只能靠猜。

## 为什么重要

想象你在用 LLM 构建一个电商客服助手。用户问："我的订单到哪了？"

**场景 A：context 里没有订单信息**
模型只能回答："请提供您的订单号，我帮您查询。"——或者更糟，编造一个虚假的物流状态。

**场景 B：context 里注入了该用户的订单数据**
```
用户订单 #12345：已发货，顺丰快递 SF1234567890，预计3月25日送达
```
模型现在可以准确回答："您的订单 #12345 已通过顺丰快递发出，单号 SF1234567890，预计明天（3月25日）送达。"

两次调用用的是同一个模型、同一个问题。唯一的区别是 context 不同。这就是 context 的决定性作用——**你给模型什么，模型就用什么。**

## 设计启示

- **Context 是你能控制的最重要杠杆。** 你不能改变模型的参数（除非微调），但你可以完全控制每次给模型什么 context。
- **"模型回答得不好"往往不是模型的问题，而是 context 的问题。** 当你对输出不满意时，第一反应应该是检查 context：是不是缺了关键信息？是不是塞了太多干扰信息？是不是信息的表达方式模型难以理解？
- **Context 设计是一个工程问题。** 在简单场景下随手写 prompt 就够了，但当系统变复杂（多轮对话、外部知识、工具调用），如何组织和管理 context 就需要系统性的方法。

## 常见误区

**误区一："模型有记忆力"**
很多 ChatGPT 用户以为模型"记得"之前聊过什么。实际上，每次 API 调用都是独立的。你看到的"对话延续"是因为应用把历史消息拼接进了 context，而不是模型自己记住了。当对话太长被截断时，模型就会"忘记"早期内容——因为那些内容被从 context 中移除了。

**误区二："给模型越多信息越好"**
Context 有容量限制（context window），而且模型对超长文本的处理能力会下降。塞入大量不相关信息不仅浪费 token 预算，还会稀释真正重要的信息，导致模型"找不到重点"。Context 设计的核心挑战之一就是在有限空间里放入最相关的信息。

**误区三："Context 就是 prompt"**
Prompt 通常指用户直接输入的那段文字。但 context 远不止于此——它包括 system prompt、对话历史、工具调用结果、检索到的外部知识等。把 context 等同于 prompt，会让你忽略那些对模型输出有巨大影响的"隐形"组成部分。
