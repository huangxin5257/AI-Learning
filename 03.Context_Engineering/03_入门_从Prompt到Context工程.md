# 从 Prompt 到 Context 工程

**一句话：** Prompt Engineering 关注"这句话怎么写效果好"，Context Engineering 关注"整个输入系统怎么设计才可靠"——这是从局部技巧到整体系统能力的跃迁。

## 是什么

**Prompt Engineering（PE）** 聚焦于单次交互中的措辞优化：用什么指令词、怎么组织句子、加不加 "think step by step"。它解决的是"怎么跟模型说话"。

**Context Engineering（CE）** 聚焦于模型每次调用时接收到的全部信息的系统化设计：什么信息该出现、什么该排除、怎么组织、怎么动态组装、怎么随对话演进而更新。它解决的是"怎么构建模型的信息环境"。

用一个类比：

> PE 像是学习"怎么向厨师清楚地点菜"——用词准确、描述清晰。
> CE 像是设计"整个厨房的供应链系统"——确保食材新鲜到位、配方手册随时可查、库存实时更新、不同菜品的备料互不干扰。

二者的关系不是非此即彼，而是尺度不同：

```
  Prompt Engineering                    Context Engineering
  ──────────────                        ───────────────────
  一句 prompt 怎么写                     整个 context 怎么设计
  手动、静态                             系统化、动态
  关注措辞和技巧                          关注架构和流程
  "我写了一段很好的 prompt"               "我设计了一套 context 组装管道"
  适合简单场景                            适合复杂系统
```

PE 是 CE 的子集。写好每一句 prompt 是必要的，但当系统复杂到一定程度，光靠"写好 prompt"已经不够。

## 为什么重要

回到我们的客服助手案例。假设你已经写好了一个不错的 prompt，在测试中表现良好。但上线后你会立刻遇到这些问题：

**问题 1：知识不可能写死在 prompt 里**
你有500个商品、30条政策、10种促销活动。用户问到哪个，就需要哪个的信息。你不可能把所有知识塞进一个 prompt——塞不下，也会互相干扰。你需要**动态检索**相关信息并注入 context。

**问题 2：对话越来越长**
用户第1轮问退货政策，第5轮问物流进度，第10轮回头说"刚才那个退货的事"。对话历史不断膨胀，怎么决定保留什么、压缩什么、丢弃什么？这是 **context 管理**问题。

**问题 3：需要调用外部工具**
查订单要调物流 API，查库存要调仓储系统。工具返回的 JSON 怎么格式化后注入 context、让模型能理解并基于它回答？这是 **Tool use context 设计**问题。

**问题 4：质量要稳定可控**
不能今天回答得好明天就乱说。需要系统化地评估 context 质量，发现问题时能定位到是哪一层 context 出了问题。这是 **context 可观测性**问题。

这些问题单靠"把 prompt 写好一点"是解决不了的。它们需要**工程化的方法**——这就是 Context Engineering 的领域。

## 设计启示

- **当你的应用超过"单轮问答"，就该开始用 CE 的视角思考了。** 只要涉及多轮对话、外部知识、工具调用中的任何一个，你就需要设计 context 系统，而不只是打磨 prompt 措辞。
- **CE 的核心思维是"管道设计"。** 把 context 组装想象成一条管道：原始信息从不同来源流入 → 经过筛选、排序、格式化 → 组装成最终的 context → 送入模型。每个环节都是可设计、可测试、可优化的。
- **PE 的技巧在 CE 中仍然有用。** 比如 system prompt 里怎么描述角色、few-shot 示例怎么写，这些仍然是 PE 层面的技巧。CE 不是否定 PE，而是把 PE 放入更大的系统框架中。

## 常见误区

**误区一："Context Engineering 只是 Prompt Engineering 的新名字"**
不是改名，是升维。PE 关注的是文本层面的优化，CE 关注的是系统层面的架构。就像"写好一个 SQL 查询"和"设计整个数据管道"是不同层次的问题。当有人说"你只需要写好 prompt 就行了"，他可能在处理简单场景；当系统变复杂，你会发现 prompt 只是 context 的一小部分。

**误区二："CE 很复杂，小项目用不上"**
CE 的思维方式在任何规模都有用。即使是一个简单的翻译脚本，问自己"这次调用的 context 里有什么、缺什么"也能帮你发现问题。CE 不一定意味着复杂的系统，它首先是一种思考方式——从"我写了什么"转向"模型看到了什么"。

**误区三："学了 CE 就不需要学 PE 了"**
恰恰相反。CE 需要 PE 作为基础。如果你连一句清晰的指令都写不好，再精妙的 context 架构也没用。建议路径是：先掌握 PE 的基本技巧（清晰指令、角色设定、few-shot 示例），然后用 CE 的视角把这些技巧组织进更大的系统中。
