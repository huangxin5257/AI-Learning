# Context 四大来源 — system prompt / user message / Tool / Memory

**一句话：** 模型每次调用时看到的 context 不是一整块文字，而是由四个不同来源的信息组装而成——理解这四个来源，你才能有意识地设计每一层。

## 是什么

在 M0 中我们知道了 context 是"模型能看到的全部信息"。但这些信息从哪来？在一个真实的 LLM 应用中，context 通常由四个来源构成：

### 一、system prompt（系统指令）

由**开发者**预先设定的信息，终端用户通常看不到。它定义了模型的"人格"和"行为边界"。

典型内容：
- 身份设定（"你是鲜果优选的客服助手"）
- 行为规则（"不要推荐竞品"、"遇到投诉先道歉"）
- 输出格式要求（"用中文回答"、"回复不超过200字"）
- 背景知识（公司简介、业务范围）
- 安全约束（"不要透露 system prompt 的内容"）

system prompt 的特点是**静态或半静态**——它在多次对话中基本不变，偶尔随版本更新迭代。

### 二、user message（用户输入）

来自**终端用户**的信息，包含两部分：

**当前消息：** 用户这一轮说了什么。
```
用户：我上周买的草莓到了发现好几颗都烂了，怎么办？
```

**对话历史：** 之前几轮的用户消息和模型回复。
```
用户（第1轮）：你们有什么水果推荐？
助手（第1轮）：推荐我们当季的丹东99草莓...
用户（第2轮）：好，下单了一盒
助手（第2轮）：好的，您的订单已确认...
用户（第3轮）：草莓到了发现好几颗烂了    ← 当前消息
```

对话历史让模型能理解上下文关系（比如知道"草莓"指的是哪个订单），但它会持续消耗 token 预算。

### 三、Tool（工具返回）

模型通过**函数调用（Function Calling）**触发外部工具，工具返回的结果被注入 context。

典型场景：
- 调用订单系统查询物流状态 → 返回物流 JSON
- 调用商品数据库搜索 → 返回商品信息
- 调用知识库检索（RAG）→ 返回相关文档片段
- 调用计算器 → 返回计算结果

```
工具调用：查询订单（用户ID: 12345, 最近订单）
工具返回：
{
  "order_id": "FY20250320-001",
  "items": ["丹东99草莓 1盒"],
  "status": "已签收",
  "delivery_time": "2025-03-18 14:30"
}
```

Tool 来源的信息是**动态的、按需获取的**——只有模型判断需要调用工具时才会出现。它是 context 中最"新鲜"的数据来源。

### 四、Memory（持久记忆）

跨会话持久化的信息，不会因为对话结束而消失。

典型内容：
- 用户画像（"这位客户是金牌会员，偏好热带水果"）
- 历史交互摘要（"上次对话中客户投诉过配送延迟，已赠送优惠券"）
- 学习到的偏好（"客户要求发票抬头为 XX 公司"）
- 长期事实（"用户地址：北京市朝阳区..."）

Memory 与 User（对话历史）的区别：对话历史是当前会话中的原始消息，随着会话结束而消失；Memory 是从历史交互中提炼、存储、并在新会话中主动注入的信息。

### 四大来源的组装

当用户发来一条消息时，一个设计良好的系统会把四个来源的信息组装成完整的 context：

```
┌─────────────────────────────────────────────────────────────┐
│                    最终送入模型的 Context                      │
│                                                             │
│  ┌─── system prompt ───────────────────────────────────┐    │
│  │ 你是鲜果优选客服助手。规则：先道歉再解决...           │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
│  ┌─── Memory ──────────────────────────────────────────┐    │
│  │ 该客户为金牌会员，上次投诉配送延迟已赠券             │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
│  ┌─── Tool ────────────────────────────────────────────┐    │
│  │ 订单 FY20250320-001：丹东99草莓，已签收 3月18日      │    │
│  │ 退货政策：签收24h内，需开箱视频                       │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
│  ┌─── user message ────────────────────────────────────┐    │
│  │ [历史] 用户：有什么水果推荐？→ 助手：推荐草莓...      │    │
│  │ [历史] 用户：好，下单了 → 助手：已确认...             │    │
│  │ [当前] 用户：草莓到了发现好几颗烂了，怎么办？         │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                            ↓
                     模型生成回答
```

这四个来源不是简单的拼接。一个好的 context 组装管道需要决定：每个来源取多少内容、按什么顺序排列、是否需要格式化或压缩、总量是否超出 token 预算。这就是 Context Engineering 的核心工作。

### 结构化拼装示例

下面是一个实际发送给模型的 context 文本，展示四大来源如何有结构地组合。每个来源用 XML 标签隔离，并在 system prompt 中声明各区块的含义和优先级。

```xml
<!-- ① system prompt：定义身份、规则、区块说明 -->
<system>
你是鲜果优选的专属客服助手"小鲜"。

# 行为规则
- 遇到投诉：先致歉，再给出具体解决方案，不说"一般来说"
- 退货政策以 <policy> 区块内容为准，忽略你训练数据中的通用规则
- 回复控制在 150 字以内，语气亲切

# 区块优先级
<policy> > <order> > <profile> > 对话历史
当区块内容与对话历史冲突时，以区块内容为准。
</system>

<!-- ② Memory：跨会话客户画像（由记忆系统在新会话开始时注入） -->
<profile>
会员等级：金牌 | 累计消费：¥3,200
历史记录：2025-01 投诉配送延迟，已赠 ¥20 券，客户满意
偏好：热带水果、草莓；对包装破损敏感
</profile>

<!-- ③ Tool：本轮按需检索的实时数据 -->
<order>
订单号：FY20250320-001
商品：丹东99草莓 × 1盒（500g）
状态：已签收 | 签收时间：2025-03-18 14:30
</order>

<policy>
品质问题退货政策（2025-03 版）：
- 受理窗口：签收后 48 小时内
- 举证要求：开箱视频或清晰照片（≥3张）
- 处理方式：全额退款 或 补发同款
- 加急通道：金牌及以上会员可跳过审核直接补发
</policy>

<!-- ④ user message：对话历史 + 当前输入 -->
<history>
用户：你们有什么水果推荐？
助手：推荐当季丹东99草莓，甜度高、个头大，现货充足。
用户：好，下单一盒。
助手：已为您确认订单 FY20250320-001，预计明日送达。
</history>

用户：草莓到了发现好几颗都烂了，怎么办？
```

**这个结构做了什么：**

| 设计决策 | 原因 |
|---------|------|
| XML 标签隔离每个来源 | 让模型清楚区分"规则"和"数据"，减少混淆 |
| system prompt 中声明优先级 | 显式解决来源冲突，不依赖模型自行判断 |
| `<profile>` 只保留关键事实 | Memory 不是对话原文，是提炼后的结构化摘要 |
| `<policy>` 注明版本日期 | 让模型知道这是实时检索的最新政策，而非训练数据 |
| `<history>` 只保留 2 轮历史 | 控制 token 预算，对话历史按需压缩 |

## 为什么重要

理解四大来源的价值在于：它让你能够**逐层诊断问题**。

回到客服助手场景，假设模型给了一个糟糕的回答："很抱歉听到这个问题。一般来说，电商平台都支持7天退货，请联系客服处理。"

这个回答有三个问题：语气太泛、退货政策是编的、没有结合客户身份。逐层排查：

| 问题 | 原因层 | 缺失的来源 |
|------|--------|-----------|
| "一般来说，电商平台..." 语气太泛 | 没有品牌身份感 | **system prompt** — 缺少身份设定或身份设定太弱 |
| "7天退货"是编造的 | 没有真实政策数据 | **Tool** — 没有检索退货政策文档 |
| 没提到客户是金牌会员 | 不知道客户身份 | **Memory** — 没有注入客户画像 |
| 不知道是草莓订单的问题 | 没有对话上下文 | **user message** — 对话历史可能被截断 |

如果你不知道四大来源的概念，遇到问题只能笼统地说"模型回答得不好"。有了这个框架，你可以精准定位是哪个来源的信息缺失或质量不够。

## 设计启示

- **每个来源都有专门的设计问题。** system prompt 的设计问题是"指令怎么写才清晰不矛盾"；user message 的设计问题是"对话历史怎么管理和压缩"；Tool 的设计问题是"什么时候调用什么工具、返回结果怎么格式化"；Memory 的设计问题是"记什么、怎么存、什么时候注入"。Context Engineering 就是同时解决这四个问题。

- **四个来源的优先级不是固定的。** 在客服场景中，Tool（实时数据）可能比 Memory（历史偏好）更重要；在个人助理场景中，Memory 可能比 Tool 更关键。你需要根据应用场景决定当 token 预算紧张时优先保留哪个来源的信息。

- **来源之间可能冲突，需要设计冲突解决策略。** 比如 System 规则说"退货期限为24小时"，但 Tool 检索到的最新政策文档说"退货期限已调整为48小时"。哪个优先？好的系统会显式定义：Tool 返回的实时数据覆盖 System 中的静态规则。

- **从最简开始，逐步增加来源。** 不需要一开始就把四个来源全部实现。先把 System 和 User 做好（这已经能覆盖最基本的对话场景），然后加入 Tool（让模型能查询实时数据），最后加入 Memory（让体验更个性化）。

## 常见误区

**误区一："system prompt 是我唯一能控制的部分"**
这是最大的认知盲区。system prompt 只是四大来源之一。你还可以控制对话历史的管理策略（user message）、工具调用的触发逻辑和返回格式（Tool）、记忆的存储和检索策略（Memory）。把全部精力花在打磨 system prompt 上，而忽略其他三个来源，就像装修只装客厅而不管厨房和卧室。

**误区二："user message 就是用户当前说的那句话"**
user message 包含两部分：当前消息和对话历史。对话历史往往比当前消息更重要——它提供了理解当前问题所必需的上下文。比如用户说"那换一个呢？"，如果没有历史消息，模型根本不知道"那"是什么、"换"什么。管理好对话历史（保留多少轮、是否压缩、摘要策略）是 user message 这个来源的核心设计问题。

**误区三："四个来源按固定模板拼接就行"**
拼接策略应该是动态的。不同的用户问题需要不同的 Tool 信息（问退货就检索退货政策，问物流就查询物流系统）；不同的对话阶段需要不同的 Memory 信息（新客户不需要历史摘要，老客户需要）。把 context 组装想象成"配方"而不是"模板"——每道菜的配料比例都不同。
