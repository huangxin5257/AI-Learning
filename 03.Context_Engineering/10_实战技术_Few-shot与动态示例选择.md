# Few-shot 与动态示例选择

**一句话：** Few-shot 是通过在 context 中放入"输入→输出"示例来教模型正确行为的技术——静态示例解决格式和语气问题，动态示例解决知识和边界问题。

## 是什么

Few-shot learning（少样本学习）在 LLM 场景中的含义很具体：**在 prompt 中放几个"输入→输出"的示例，让模型从示例中学习你期望的行为模式，然后把这个模式应用到新的输入上。**

最简单的理解方式是看一个对比：

**Zero-shot（零示例）—— 只给指令：**
```
请把用户的评价分类为"正面"、"负面"或"中性"。

用户评价：草莓很新鲜，就是有两颗有点软
分类：
```

模型需要靠自己理解"正面/负面/中性"的分类标准。它可能把"有两颗有点软"判为负面，也可能判为中性——标准不明确。

**Few-shot（少样本）—— 给指令 + 示例：**
```
请把用户的评价分类为"正面"、"负面"或"中性"。

示例 1：
用户评价：包装特别好，水果没有破损，味道也很甜！
分类：正面

示例 2：
用户评价：收到的时候已经烂了三四个，物流太慢了
分类：负面

示例 3：
用户评价：大小还行，甜度一般般，还能接受吧
分类：中性

---
用户评价：草莓很新鲜，就是有两颗有点软
分类：
```

现在模型不仅知道要分类，还能从示例中看到你的分类标准：主要问题导向的是负面，小瑕疵但整体满意的是中性/正面。它对"草莓很新鲜，就是有两颗有点软"的判断会更符合你的预期（正面偏中性）。

### 静态 Few-shot vs 动态 Few-shot

**静态 Few-shot：** 示例写死在 prompt 中，每次请求都用同样的示例。

适合用来固定：
- 输出格式（回答应该长什么样）
- 语气和风格（专业 vs 口语化）
- 通用的处理模式（比如"先确认问题→再给方案→最后询问是否解决"）

**动态 Few-shot：** 根据当前用户输入，从一个示例库中实时检索最相关的几个示例放入 context。

适合用来处理：
- 垂直领域知识（根据用户问的商品类型，选取该类商品的历史 QA）
- 边界情况（根据用户问题的特征，选取类似的棘手案例）
- 个性化场景（根据用户画像，选取同类用户的交互示例）

两者的关系不是替代，而是互补——你可以同时使用静态示例固定格式，用动态示例提供领域知识。

### 对比示例（Contrastive Examples）

一种特别有效的 few-shot 技巧是同时展示正确和错误的示例，帮助模型理解"边界在哪里"。

```markdown
## 回答示例

### 正确示例
用户：你们的草莓打农药吗？
✅ 正确回答：我们的丹东99草莓来自合作基地，采用绿色种植标准。具体的检测报告您可以在商品详情页的"品质认证"板块查看。如果您对食品安全有任何顾虑，我可以帮您联系客服主管详细了解。

### 错误示例
用户：你们的草莓打农药吗？
❌ 错误回答：绝对没有！我们的草莓是纯天然有机的，不含任何农药残留，您可以放心食用。
❌ 错误原因：做出了无法验证的绝对承诺（"绝对没有""纯天然有机"），可能与事实不符，存在法律风险。
```

对比示例的价值在于：指令告诉模型"不要做绝对承诺"，但模型不一定理解"绝对承诺"的边界在哪里。一个正面+反面的示例对比，直接展示了边界的具体位置。

## 为什么重要

### 场景一：指令说不清的问题，示例一看就懂

你在为鲜果优选客服写 system prompt，想要求模型用"温暖但专业"的语气回答。

你可以写指令：
```
语气要求：温暖但专业。不要过于正式（不像客服机器人），也不要过于随意（不像朋友聊天）。
要有温度感，但同时保持信息准确和条理清晰。
```

这段指令本身没错，但"温暖但专业"是一个连人类都很难精确定义的概念。不同的模型版本、不同的 context 长度，可能对这段话产生不同的理解。

换成 few-shot 示例：

```markdown
## 语气示例

用户：我的订单怎么还没到啊，都等了三天了！
果小蜜：理解您着急的心情！我帮您查了一下，订单 #12345 目前显示在配送途中，预计明天上午送达。最近因为天气原因部分地区物流有延迟，给您带来不便真的很抱歉。我会持续帮您关注物流动态，有更新第一时间告诉您~

用户：你们这个车厘子是什么等级的？
果小蜜：我们在售的智利车厘子是 JJ 级和 JJJ 级两种规格。JJ 级果径约 28-30mm，JJJ 级约 30-32mm，都是当季空运直达的。您可以根据预算和用途选择——自己吃 JJ 级性价比很高，送人的话 JJJ 级卖相更好。需要我帮您看看哪款有货吗？
```

模型读完这两个示例后，"温暖但专业"就从一个抽象概念变成了具体的行为模式：有共情表达（"理解您着急的心情"）但不夸张，有专业信息（果径、等级）但不生硬，句尾用"~"和反问句保持亲切感。

**这就是 few-shot 最核心的价值：用具体示例消除抽象指令的模糊性。**

### 场景二：动态示例大幅提升专业回答质量

鲜果优选有上百种水果，每种水果的保存方法、食用建议、常见问题都不同。你不可能在 system prompt 的静态示例里覆盖所有品类。

当用户问"山竹怎么挑？"时，如果 context 里的 few-shot 示例都是草莓和车厘子的，模型只能靠通用知识回答——准确性和专业度都会下降。

但如果你有一个示例库，里面存了历史上优秀客服对各品类问题的标准回答，然后用语义检索挑出最相关的：

```python
# 用户问："山竹怎么挑？买回来能放几天？"

# 从示例库中检索与"山竹 挑选 保存"相关的优质问答
dynamic_examples = example_store.search("山竹 挑选 保存", top_k=2)

# 检索到的示例：
# 示例 A：
#   用户：怎么判断山竹新不新鲜？
#   回答：挑山竹有个简单方法——用手轻捏果壳，能微微按下去说明新鲜多汁，
#         按不动说明太老了。另外看蒂头的叶子，绿色的比枯黄的更新鲜...
#
# 示例 B：
#   用户：山竹买多了吃不完怎么保存？
#   回答：山竹不耐储存，常温下大概只能放2-3天。建议用保鲜袋包好放冰箱
#         冷藏，可以延长到5-7天。不过还是建议尽快食用口感最好...
```

这两个动态检索到的示例直接教会模型"回答山竹问题应该包含哪些具体知识"，比任何通用指令都有效。

## 设计启示

### 1. 格式化示例：清晰的分隔符是关键

模型需要清楚区分"哪里是示例的输入""哪里是示例的输出""哪里是示例的结束"。混乱的格式会导致模型把示例的一部分当成输出，或者把新输入的一部分当成示例。

推荐的格式：

```markdown
<examples>

<example>
用户：我买的芒果表面有黑点，是坏了吗？
客服：芒果表面出现少量黑点是正常的自然现象，不影响食用。这通常是果皮被阳光灼伤或运输中的轻微碰压造成的。您切开看看果肉的情况——如果果肉颜色正常、没有异味，就可以放心吃。如果果肉也有发黑变软的情况，可能确实有损坏，您拍张照片我帮您走售后流程。
</example>

<example>
用户：你们有没有适合老人吃的水果推荐？
客服：给老人选水果，建议优先考虑质地软、易消化的品种。我推荐您看看这几款：
1. 香蕉——软糯好消化，我们的云南高山香蕉甜度很稳定
2. 猕猴桃——维C含量高，催熟后质地很软
3. 木瓜——对肠胃友好，口感也好

这几款目前都有货，需要我帮您看看哪个有优惠活动吗？
</example>

</examples>
```

关键要点：
- 用 XML 标签（`<examples>`、`<example>`）或分隔线（`---`）明确划分示例边界
- 输入和输出之间用一致的标记区分（如"用户："和"客服："）
- 所有示例保持统一的格式结构

### 2. 示例数量：遵循收益递减规律

Few-shot 的效果随示例数量增加而递减：

```
示例数量   效果增量    适用场景
  0 → 1   ████████   巨大提升——模型终于理解你要什么格式
  1 → 2   █████      显著提升——模型看到不同情况的处理方式
  2 → 3   ███        中等提升——模型对模式更加确信
  3 → 5   ██         边际提升——额外示例主要覆盖边界情况
  5 → 10  █          微小提升——通常是浪费 token
```

**实用建议：**

- **格式/语气固化：** 2-3 个静态示例通常足够
- **分类任务：** 每个类别至少 1 个示例，总数 3-5 个
- **复杂推理/边界判断：** 3-5 个示例，包含正常情况和边界情况
- **超过 5 个：** 除非你在做非常细粒度的行为控制，否则考虑把额外示例转为指令规则

记住：每个示例都占用 context 空间。5 个详细示例可能消耗 500-1000 token，这些空间本可以用来放更多的用户对话历史或检索到的知识。

### 3. 动态示例选择的实现路径

从静态到动态的演进路径：

**阶段一：硬编码静态示例**
```python
# 最简单，直接写在 prompt 里
system_prompt = """
...（角色、规则）...

## 回答示例
{static_example_1}
{static_example_2}
"""
```

**阶段二：基于规则的示例选择**
```python
# 根据简单规则选取示例组
def select_examples(user_query):
    if "退" in user_query or "换" in user_query:
        return REFUND_EXAMPLES
    elif "推荐" in user_query or "什么好" in user_query:
        return RECOMMENDATION_EXAMPLES
    elif "物流" in user_query or "快递" in user_query:
        return DELIVERY_EXAMPLES
    else:
        return DEFAULT_EXAMPLES
```

**阶段三：基于语义相似度的动态检索**
```python
# 使用向量相似度从示例库中检索
from sentence_transformers import SentenceTransformer

model = SentenceTransformer('text2vec-large-chinese')

# 预先将所有示例编码为向量存入向量数据库
# example_store 中存储了数百个历史优质问答对

def select_examples(user_query, top_k=3):
    query_embedding = model.encode(user_query)
    results = example_store.search(query_embedding, top_k=top_k)
    return results

# 用户问："你们的榴莲是猫山王吗？泰国的还是马来西亚的？"
# 检索到最相似的 3 个历史问答：
#   1. 关于榴莲品种区别的问答 (相似度 0.92)
#   2. 关于榴莲产地和口感的问答 (相似度 0.87)
#   3. 关于榴莲保存方法的问答 (相似度 0.78)
```

建议从阶段一开始，验证 few-shot 对你的场景确实有效后，再根据需要升级到阶段二或三。不要跳过验证直接上向量检索——有时候 2 个写得好的静态示例效果就足够了。

### 4. 示例质量远比数量重要

一个错误的示例对模型的影响远大于一个正确的示例。模型会忠实地学习示例中的模式——包括错误的模式。

```markdown
## 坏示例（隐含了错误模式）
用户：你们的草莓是有机的吗？
客服：是的，我们的草莓都是有机种植的，完全不使用农药！

问题：如果这不是事实，模型会学到"可以做出未经验证的承诺"这个模式。
```

检查示例质量的清单：
- **事实准确性：** 示例中的信息是否属实？
- **风格一致性：** 所有示例的语气是否一致？（不要一个很正式一个很口语化）
- **规则符合性：** 示例是否违反了你在 system prompt 中设定的任何规则？
- **格式统一性：** 所有示例的输出格式是否相同？
- **代表性：** 示例是否覆盖了不同类型的场景？（不要 3 个示例都是价格问题）

### 5. 示例与指令的关系：互补而非替代

指令和示例各有所长：

| 维度 | 指令 | 示例 |
|------|------|------|
| 擅长 | 表达明确的规则（"不要做X"） | 展示隐性的模式（语气、风格、判断标准） |
| 覆盖范围 | 广泛（一条规则覆盖所有情况） | 具体（每个示例只覆盖一种情况） |
| Token 效率 | 高（几句话就能表达一条规则） | 低（一个完整示例可能要上百 token） |
| 模型遵循度 | 中等（模型可能误解抽象指令） | 高（具体示例很难误解） |

最有效的做法是**指令定方向，示例定标准**：

```markdown
# 行为规则（指令定方向）
- 回答要简洁，控制在 3-5 句话
- 遇到无法确认的信息不要编造

# 示例（示例定标准）
## 示例 1：简洁回答的标准是什么样
用户：你们支持当日达吗？
客服：部分城市支持当日达服务。下单时您在地址页面可以看到预计送达时间，
标注"今日达"的就支持。您在哪个城市？我帮您确认一下。

## 示例 2：遇到不确定信息时怎么处理
用户：你们的蓝莓是哪里产的？
客服：我们平台目前有云南和智利两个产地的蓝莓在售，具体到您看的那款，
建议您在商品详情页查看产地信息，那里的信息最准确。需要我发个链接给您吗？
```

指令说"简洁"和"不要编造"，示例展示"简洁"具体是 3 句话的程度、"不要编造"具体是怎么表述的。两者缺一不可。

## 常见误区

**误区一："示例越多越好"**

这是最常见的误区。增加示例的真实成本是 context 空间——每多一个示例，就少了放其他有用信息（对话历史、检索知识）的空间。而且示例越多，模型可能越倾向于"模仿"示例而不是"理解"规则，导致在示例没覆盖到的场景表现反而退化。

正确的做法是：从 2 个示例开始，逐步增加并测量效果。当增加新示例不再带来明显提升时，就停下来。

**误区二："示例和指令可以互相替代"**

有人认为"给够示例就不需要指令了"，或者"指令写清楚就不需要示例了"。两种观点都有问题：

- 只有示例没有指令：模型能学到格式，但可能学错规则。比如你的 3 个示例碰巧都是回答产品问题的，模型可能不知道遇到投诉该怎么处理。
- 只有指令没有示例：模型能理解规则，但可能在执行细节上偏离你的预期。"友好专业"到底是什么程度？没有示例就全靠模型自己理解。

指令覆盖面广但不够具体，示例具体但覆盖面窄。两者配合才能同时保证广度和精度。

**误区三："随便什么示例都比没有好"**

错误的示例比没有示例更糟。一个包含虚假信息的示例会教模型"可以编造信息"。一个格式不一致的示例会让模型对输出格式感到困惑。一个与当前场景无关的示例会占用 token 但不带来任何价值。

尤其在动态 few-shot 场景中，如果检索质量差、选出来的示例和用户问题不相关，这些"噪声示例"会误导模型，效果可能还不如不放示例。所以动态 few-shot 系统必须有相似度阈值——**低于阈值的宁可不放。**

**误区四："Few-shot 是通用的万能方案"**

Few-shot 最适合的场景是：任务有明确的输入→输出模式、你能提供高质量示例、而且模式可以从少数示例中泛化。

它不太适合的场景包括：
- **需要精确遵循复杂逻辑的任务**（比如多步计算），这时候结构化的指令链（chain of thought）比示例更有效
- **输出高度依赖输入细节的任务**（比如法律文书生成），这时候示例的泛化价值有限
- **模型已经做得很好的任务**（比如简单翻译），加示例可能是浪费空间

在引入 few-shot 之前，先用 zero-shot 测试一下效果。如果 zero-shot 已经够好，就不需要 few-shot。如果 zero-shot 有明确的不足（格式不对、语气不对、判断标准不一致），再考虑用 few-shot 来解决这些具体问题。
