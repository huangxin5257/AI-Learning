# Context 质量三要素 — 相关性、密度、时序

**一句话：** 好的 context 不只是"信息正确"，还需要满足三个质量维度：与当前任务相关（相关性）、单位 token 承载的有效信息多（密度）、信息按正确的时间或逻辑顺序排列（时序）。

## 是什么

前面几篇我们讨论了 context 的容量限制（token 与 context window）、位置效应（信息衰减）、以及信息来源（四大来源）。但即使你在这些维度上都做得不错——token 没超、关键信息放在前面、四个来源都有——context 的质量仍然可能不合格。

Context 质量可以用三个要素来衡量：

### 一、相关性（Relevance）

**context 中的信息是否与当前任务/问题相关。**

相关性是最直觉的维度：用户问退货政策，你放了退货政策就是相关的，放了公司创业史就是不相关的。但实际操作中，相关性的判断比想象的微妙：

- **直接相关：** 用户问"草莓能退吗"，退货政策文档直接相关。
- **间接相关：** 用户问"草莓能退吗"，该用户的会员等级间接相关（金牌会员可能有特殊退货权益）。
- **看似相关实际干扰：** 用户问"草莓能退吗"，你放入所有水果的保鲜指南——看起来都是"草莓相关信息"，但对回答退货问题毫无帮助，反而占用 token 预算。

### 二、密度（Density）

**单位 token 中承载了多少有效信息，即信噪比。**

同样的信息可以用不同方式表达，消耗的 token 量可能差几倍：

**低密度版本（约 120 token）：**
```
关于我们的退货政策，首先非常感谢您选择鲜果优选。我们非常重视每一位客户的购物体验。
如果您收到的商品有任何问题，我们提供了完善的售后保障。具体来说，您可以在签收之后
的24个小时之内提出退货申请。不过需要注意的是，为了保障双方的权益，我们需要您提供
一段开箱视频作为凭证。如果商品已经被切开了，或者您没有提供开箱视频，又或者已经
超过了24小时的时限，那么很遗憾我们无法为您办理退货。
```

**高密度版本（约 40 token）：**
```
退货政策：
- 时限：签收后24小时内
- 必要凭证：开箱视频
- 不支持：已切开 / 无视频 / 超时
- 退款：审核通过后3个工作日原路返回
```

两个版本传达的核心信息完全相同，但高密度版本只用了约三分之一的 token。在 context window 有限的情况下，省下的 token 可以用来放其他有价值的信息。

### 三、时序（Temporal Ordering）

**信息是否按正确的时间顺序或逻辑顺序排列。**

很多人忽略这个维度，但它对模型理解因果关系和做出正确判断至关重要：

**时间顺序：** 对话历史、操作日志、事件流需要按时间排列。如果你把"用户申请退货"放在"用户收到商品"前面，模型可能理解为用户还没收到货就要退货。

**逻辑顺序：** 规则和例外的先后关系。应该先说规则再说例外（"24小时内可退 → 但以下情况除外"），而不是先说例外再说规则（"切开的水果不能退 → 不过一般来说24小时内可以退"）。

**优先级顺序：** 当多条信息可能冲突时，后出现的信息往往被模型理解为"覆盖"或"修正"前面的信息——利用这个特性可以把最新政策放在旧政策后面。

## 为什么重要

用"鲜果优选"客服助手的场景，展示三个要素分别出问题时的后果：

### 相关性出问题：高相关 vs 低相关

**低相关 context：** 用户问"我的草莓坏了怎么办"，context 中塞了全部30条 FAQ：

```
System：你是鲜果优选客服...
FAQ 1：配送范围覆盖全国300个城市...
FAQ 2：会员积分规则...
FAQ 3：如何开发票...
...
FAQ 15：退货政策：签收后24小时内...
...
FAQ 30：企业团购优惠...
```

模型回答："您好，很抱歉给您带来不好的体验。您可以申请退货，一般在签收后一定时间内..."——回答含糊，因为关键信息被30条无关 FAQ 稀释了。

**高相关 context：** 只注入与问题相关的信息：

```
System：你是鲜果优选客服...
退货政策：签收后24小时内，需开箱视频...
该用户订单：草莓1盒，3月18日签收
当前时间：3月19日10:00（距签收不到24小时）
```

模型回答："很抱歉您收到的草莓有质量问题！您的订单是昨天下午签收的，目前还在24小时退货时限内。请您提供开箱视频，我立即为您处理退货退款。"——精准、可执行。

### 密度出问题：高密度 vs 低密度

假设 token 预算只剩 200 token 可以放知识信息。

**低密度方案：** 200 token 只够放退货政策的"散文版"，退款流程放不下了。

**高密度方案：** 200 token 够放退货政策（结构化列表）+ 退款流程 + 投诉升级规则，三条信息全覆盖。

结果：低密度方案下，模型只能回答退货条件，但说不清退款怎么到账；高密度方案下，模型可以一次性完整回答。**同样的 token 预算，密度不同，能覆盖的信息量不同，输出质量也不同。**

### 时序出问题：正确时序 vs 错乱时序

用户与客服的对话历史被错误排列注入了 context：

**错乱时序：**
```
助手：已为您办理退货        ← 第3轮的回复
用户：草莓坏了要退货        ← 第1轮的消息
助手：请提供开箱视频        ← 第2轮的回复
用户：视频已上传            ← 第3轮的消息
用户：退款什么时候到账？     ← 第4轮（当前）
```

模型可能感到混乱：退货已经办了吗？视频上传了没有？当前是什么状态？它可能给出矛盾的回答，或者重复已经完成的步骤。

**正确时序：**
```
用户（第1轮）：草莓坏了要退货
助手（第1轮）：请提供开箱视频
用户（第2轮）：视频已上传
助手（第2轮）：已为您办理退货
用户（第3轮）：退款什么时候到账？    ← 当前
```

模型清楚地理解：退货已办理完成，用户现在想知道退款时间。回答："您的退货申请已通过审核，退款将在3个工作日内原路返回您的支付账户。"

### 三要素的交互

三个要素不是独立的，它们会互相影响：

- **高相关但低密度 = 臃肿的 context。** 信息都是相关的，但表达太啰嗦，token 预算被浪费，挤不进其他同样重要的信息。
- **高密度但低相关 = 高效的垃圾。** 每个 token 都承载了信息，但这些信息跟当前问题无关，模型照样找不到重点。
- **高相关高密度但时序错乱 = 容易出错的好材料。** 信息既相关又精炼，但顺序乱了，模型可能理解错因果关系，给出逻辑上矛盾的回答。

三个要素需要同时满足，context 质量才能真正达标。

## 设计启示

- **相关性靠检索解决。** 不要试图手动判断什么信息相关——用 RAG（检索增强生成）等技术，根据用户的当前问题自动检索最相关的知识片段。检索质量直接决定相关性质量。

- **密度靠格式优化解决。** 把 context 中的信息从"自然语言叙述"转化为"结构化列表/表格/标签"。在写 system prompt 和组织知识文档时，刻意追求精炼。每多一个没有信息量的词（"首先"、"需要注意的是"、"具体来说"），就少一个 token 的有效信息空间。

- **时序靠管道设计解决。** 在 context 组装管道中，为每类信息定义明确的排序规则：对话历史按时间升序、知识片段按相关性降序、规则按优先级排列。不要依赖信息"碰巧"以正确顺序出现。

- **三个要素可以作为 context 的质量检查清单。** 当模型输出不理想时，按顺序自问：相关信息是否到位了？信息表达是否够精炼？信息的排列顺序对不对？

## 常见误区

**误区一："越多相关信息越好"**
相关性有边际递减效应。给模型3条高度相关的 FAQ 通常比给30条"可能相关"的 FAQ 效果好——后者虽然覆盖面更广，但大量中等相关性的信息会分散注意力、消耗 token 预算、引发位置效应。**相关性的目标是精准，不是全面。**

**误区二："信息密度越高越好，压缩到极致"**
过度压缩会损害可读性，模型也可能无法正确解析。比如把退货政策压缩成 "退:24h/需视频/不退:切开+无视频+超时" 虽然极致精炼，但模型理解起来可能不如正常的结构化列表。密度优化要找平衡点——**去掉冗余的废话，但保留必要的结构和清晰度。**

**误区三："事实性信息不需要考虑时序"**
即使是看似"静态"的事实信息，呈现顺序也会影响模型的理解。比如"该商品原价99元，促销价69元"和"该商品促销价69元，原价99元"传达的语义微妙不同——前者强调"在打折"，后者强调"这就是价格"。在退货政策中，先说"可退货的条件"再说"不可退货的例外"比反过来更容易让模型正确判断边界情况。信息的呈现顺序隐含了优先级和逻辑关系，不能随意排列。
