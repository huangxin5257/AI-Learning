# 一、指令遵循类

> **本类共性：** 用户写的指令本身没有歧义，模型也"读懂了"，但在执行过程中出现了偏差——有的指令被漏掉，有的被曲解，有的被降级处理。这类问题的修复重点在 Prompt 的**结构和优先级设计**上。

---

## 1.1 指令遗漏

> **快速识别：** 你给了 5 条要求，模型只满足了前 2-3 条，后面的悄悄丢了。

**典型症状**

- Prompt 中明确列出了多条要求，输出只满足了其中一部分
- 被遗漏的通常是靠后的、措辞不显眼的、或与主任务关联度低的指令
- 模型不会主动告诉你哪些指令没被执行

**为什么会发生**

模型在处理多条并列指令时，对各条指令分配的注意力是不均匀的。靠前的、措辞更显眼的指令更容易被执行。当指令数量超过 4-5 条时，靠后的指令被丢弃的概率明显上升。本质上是一个**注意力资源分配不均**的问题。

**怎么修**

- **编号 + 明确标题**：将每条指令单独成段，加粗或编号，提升各自的显著性。
- **指令核查步骤**：在 Prompt 末尾追加一句："在输出前，请逐条检查以上所有要求是否已满足。"
- **减少指令数量**：超过 5 条并列指令时，考虑拆分为多轮对话，每轮聚焦 1-2 条关键约束。
- **重复关键约束**：对最重要的 1-2 条要求，在 Prompt 开头和结尾各说一次。

**示例**

### 示例 1：文章写作多指令

用户原始提示词：
```
请帮我写一篇关于气候变化的文章。要求：字数在 800 字以内,使用第三人称,结尾附上 3 条参考文献,段落之间空一行,全程不使用"我们"这个词。
```

失效输出：
文章内容符合字数和第三人称要求，但没有附参考文献，段落之间也没有空行，并且多次出现了"我们"。模型只满足了前两条最显眼的要求。

改进版提示词：
```
请帮我写一篇关于气候变化的文章。

【核心要求】
1. 字数在 800 字以内
2. 使用第三人称，全程不使用"我们"这个词
3. 结尾附上 3 条参考文献（格式：作者, 标题, 来源, 年份）
4. 段落之间空一行

输出完成后，请逐条检查以上 4 项要求是否全部满足，如有未满足项请补充修正。
```

改进后预期效果：
文章使用第三人称撰写，全文无"我们"出现，段落间有空行分隔，结尾附有 3 条格式规范的参考文献，字数控制在 800 字以内。所有 4 条要求均被满足。

---

### 示例 2：代码生成多约束

用户原始提示词：
```
用 Python 写一个用户注册功能。要验证邮箱格式，密码要加密存储，要有错误处理，返回 JSON 格式，加上类型注解。
```

失效输出：
模型写了注册功能并验证了邮箱格式，但密码是明文存储的，没有 try/except 错误处理，也没有类型注解。长句中靠后的约束被逐步遗漏。

改进版提示词：
```
用 Python 写一个用户注册功能，必须同时满足以下全部要求：

1. 【输入验证】使用正则表达式验证邮箱格式
2. 【安全存储】使用 bcrypt 对密码进行哈希加密后再存储
3. 【错误处理】所有操作用 try/except 包裹，异常时返回错误信息
4. 【返回格式】统一返回 JSON 格式的响应体
5. 【类型注解】所有函数参数和返回值必须有类型注解

完成后请逐条自查以上 5 项是否全部体现在代码中。
```

---

### 示例 3：内容创作格式要求

用户原始提示词：
```
帮我写 5 条社交媒体文案，主题是新品发布。要用 emoji，控制在 100 字以内，带话题标签，语气轻松活泼，不要用感叹号。
```

失效输出：
5 条文案主题正确，也使用了 emoji，但超出了 100 字限制，只有第 1 条有话题标签，后 4 条都没有，而且每条都大量使用了感叹号。

改进版提示词：
```
帮我写 5 条社交媒体文案，主题是新品发布。

每条文案必须同时满足：
- 字数：100 字以内（含 emoji 和标签）
- 包含 1-2 个 emoji
- 结尾带 1-2 个话题标签（如 #新品首发）
- 语气轻松活泼
- 禁止使用感叹号（用句号或省略号替代）

请确保所有 5 条都严格遵守以上每一项要求，不要只在前几条遵守。
```

## 1.2 字面服从

> **快速识别：** 模型技术上满足了你的要求，但结果完全不是你想要的——它在"钻空子"。

**典型症状**

- 输出严格满足了指令的字面含义，但偏离了你的真实意图
- 模型找到了一个"技术上正确但毫无用处"的解决路径
- 你会有一种"对，我是这么说的，但我不是这个意思"的感觉

**为什么会发生**

Prompt 只给了操作指令（如"让代码更短"），没有说明隐含意图、边界约束和不能牺牲的东西（如可读性）。模型在约束缺失的空间内，会找到一个字面上满足要求的最简路径，哪怕这个路径毫无实际价值。

**怎么修**

- **显式说明意图**：不要只说"做什么"，同时说明"为什么"和"不能牺牲什么"。
- **提供反例**：告诉模型哪些方式是不被允许的，封堵"钻空子"路径。
- **结果导向描述**：描述你期望的输出效果与质量标准，而不仅仅是操作本身。

**示例**

### 示例 1：代码精简

用户原始提示词：
```
让这段代码更短。

def calculate_area(length, width):
    area = length * width
    return area
```

失效输出：
```python
def f(l,w): return l*w
```
字面上确实更短了，但变量名毫无可读性，违背了代码应保持可维护性的隐含意图。

改进版提示词：
```
在保持变量命名清晰、代码可读性和可维护性不变的前提下，精简这段代码的冗余结构。不要通过缩短变量名或函数名来减少代码量。

def calculate_area(length, width):
    area = length * width
    return area
```

改进后预期效果：
```python
def calculate_area(length, width):
    return length * width
```
去掉了冗余的中间变量赋值，但函数名和参数名保持清晰。

---

### 示例 2：文本摘要

用户原始提示词：
```
把这篇 2000 字的文章缩短到 200 字。
```

失效输出：
模型只保留了文章的前两段（刚好约 200 字），直接截断，后面的核心论点、数据和结论全部丢失。字面上满足了"200 字"的要求，但失去了摘要应有的信息覆盖度。

改进版提示词：
```
请将这篇 2000 字的文章压缩为 200 字左右的摘要。要求：
- 覆盖文章的核心论点、关键数据和最终结论
- 不是简单截取原文段落，而是重新提炼和概括
- 保持逻辑完整性，读者仅看摘要即可理解文章主旨
```

---

### 示例 3：翻译任务

用户原始提示词：
```
把下面这段翻译成英文。

"这款产品在年轻用户中口碑炸裂，上线三天就卖疯了。"
```

失效输出：
```
"This product has an explosive reputation among young users and sold crazily within three days of launch."
```
逐词直译了中文口语化表达，"口碑炸裂"→"explosive reputation"、"卖疯了"→"sold crazily"，语法上没错，但在英文语境中极不自然，完全不符合商业文案的表达习惯。

改进版提示词：
```
请将下面这段中文翻译成英文。要求：
- 传达原文的商业语义和情感色彩，而非逐词直译
- 输出应符合英文母语者的自然表达习惯
- 适合用于产品营销或新闻报道的语境

"这款产品在年轻用户中口碑炸裂，上线三天就卖疯了。"
```

改进后预期效果：
```
"The product became an instant hit among younger users, generating massive buzz and selling out within just three days of launch."
```
传达了原文的热度和速度感，用了英文营销语境中自然的表达方式。

## 1.3 指令冲突

> **快速识别：** 你的两个要求互相矛盾，模型默默放弃了其中一个，不会告诉你。

**典型症状**

- 输出明显偏向了某一条指令，另一条被悄悄忽略
- 这种偏向不是随机的——模型倾向选择位置靠前/靠后的、措辞更强硬的、描述更具体的指令
- 在 Prompt 经过多次迭代累积后尤其常见，因为新增指令容易与旧指令产生摩擦

**为什么会发生**

Prompt 中两条要求存在隐性矛盾，但措辞上各自看似合理。模型无法感知冲突，便倾向性地偏向其中一条。这种偏向受指令位置、措辞强度、语义具体度等因素的系统性影响，是可预测的。

**怎么修**

- **写 Prompt 前先做自洽性检查**：将所有指令列出来，逐条问自己："这条和其他条是否存在冲突？"
- **化解矛盾，合并为一条明确指令**：例如，将"简洁"和"详细"合并为："用清晰的逻辑分步说明，每步不超过两句话。"
- **设置优先级**：如果两条指令确实难以同时满足，明确告知模型哪条优先。

**示例**

### 示例 1：简洁 vs 详细

用户原始提示词：
```
请用简洁的语言回答下面的问题，并且要详细解释每个步骤的原因。
问题：如何配置 Nginx 反向代理？
```

失效输出：
模型输出了详细的 10 段逐步教程（满足"详细解释"），但每步都长达一段话，完全违背了"简洁"的要求。

改进版提示词：
```
请分步说明如何配置 Nginx 反向代理。
格式要求：每一步用一句话描述操作，紧跟一句话解释原因，每步总计不超过两句话。
```

改进后预期效果：
每一步由"做什么"+"为什么"两句话构成，既保持了分步详解，又控制了篇幅，解决了"简洁"和"详细"的矛盾。

---

### 示例 2：创意 vs 准确

用户原始提示词：
```
请为我们的 AI 数据分析产品写一段富有创意和想象力的宣传文案，所有数据和功能描述必须完全准确，不得夸大。
```

失效输出：
模型写出了一段平淡的功能说明书式文案，完全没有创意和想象力——因为"完全准确、不得夸大"的约束压制了创意发挥。

改进版提示词：
```
请为我们的 AI 数据分析产品写一段宣传文案。

规则：
- 创意空间：比喻、类比、场景化叙事可以自由发挥
- 准确红线：具体的功能名称、性能数据、技术指标必须与以下事实一致：[列出产品事实]
- 简言之：修辞可以夸张，事实不能夸张
```

---

### 示例 3：专业 vs 通俗

用户原始提示词：
```
请用专业的医学术语写一篇面向普通患者的健康科普文章，主题是高血压管理。
```

失效输出：
文章充满了"血管紧张素转换酶抑制剂""肾素-血管紧张素-醛固酮系统"等术语，普通患者根本读不懂。模型偏向了"专业医学术语"而放弃了"面向普通患者"。

改进版提示词：
```
请写一篇面向普通患者的高血压管理科普文章。

语言策略：
- 首次出现医学概念时，用日常语言解释（如："ACE 抑制剂——一种帮助血管放松的常用降压药"）
- 正文以通俗易懂为主，括号内可附专业术语供患者与医生沟通时使用
- 优先考虑患者能否读懂，其次才是术语的专业性
```

## 1.4 优先级混乱

> **快速识别：** 最重要的需求反而写得最敷衍，次要的格式细节倒是很完整。

**典型症状**

- 核心需求被一笔带过，次要的约束反而被重点执行
- 用户用"最重要的是""最关键的是"来强调，但模型仍然优先处理了篇幅更长、位置更前或更具体的指令
- 输出的详略分配与需求的重要性完全倒挂

**为什么会发生**

Prompt 中包含多条指令但没有明确主次关系。模型在权衡取舍时，不会按照用户心中的重要性排序，而是倾向于把描述篇幅更长、位置靠前或措辞更具体的指令当作主要任务。

**怎么修**

- **显式标注优先级**：使用"【最高优先级】""【必须满足】"等标签标注核心要求，次要要求用"【尽量满足】"标注。
- **将核心要求放在最前面**：人和模型都对文本首部和尾部更敏感。
- **分离核心与细节**：先描述必须实现的核心目标，再另起一段列出格式、风格等次要约束。

**示例**

### 示例 1：客户邮件回复

用户原始提示词：
```
请帮我回复这封客户投诉邮件。保持语气专业礼貌，字数控制在 150 字以内，
最重要的是，一定要承诺在 48 小时内给出解决方案。
```

失效输出：
回复语气很专业，字数也在 150 字以内，但没有包含"48 小时内给出解决方案"的承诺。尽管用户说了"最重要的是"，模型仍然优先处理了格式要求。

改进版提示词：
```
请帮我回复这封客户投诉邮件。

【必须满足】明确承诺"我们将在 48 小时内给出解决方案"——这是本次回复的核心目标。
【尽量满足】语气专业礼貌，字数控制在 150 字以内。

如果字数限制与核心承诺冲突，优先保证承诺内容完整。
```

改进后预期效果：
回复中包含明确的"48 小时内给出解决方案"承诺，语气专业，字数在限制范围内。核心需求不再被边缘化。

---

### 示例 2：技术方案文档

用户原始提示词：
```
请写一份数据库迁移方案。要详细列出迁移步骤，包含时间估算，画出架构图的文字描述，还有风险评估。最关键的是回滚方案必须完整可执行。
```

失效输出：
模型用了大量篇幅写迁移步骤和架构描述，时间估算也很详细，但回滚方案只有一句"如遇问题可回滚到之前的版本"，没有任何具体操作步骤。最关键的需求反而最单薄。

改进版提示词：
```
请写一份数据库迁移方案。

【最高优先级 - 回滚方案】
必须包含完整、可执行的回滚方案：逐步操作命令、回滚判断条件、数据一致性验证步骤、预计回滚时间。这是本文档的核心。

【标准内容】
- 迁移步骤（含时间估算）
- 架构变更说明
- 风险评估

请确保回滚方案的篇幅和细节度不低于迁移步骤本身。
```

---

### 示例 3：产品需求文档

用户原始提示词：
```
帮我写一份需求文档，包含功能描述、用户故事、验收标准、技术约束，重点是验收标准要可量化可测试。
```

失效输出：
功能描述写了三页，用户故事也很详细，但验收标准只有"用户能正常使用""系统响应及时"这类模糊表述，完全不可量化、不可测试。

改进版提示词：
```
帮我写一份需求文档。

【最高优先级 - 验收标准】
每个功能点的验收标准必须包含：
- 可量化的指标（如"响应时间 < 200ms"、"错误率 < 0.1%"）
- 可执行的测试步骤
- 明确的通过/失败判定条件

【标准内容】功能描述、用户故事、技术约束

写作顺序：先写验收标准，再补充其他内容，确保验收标准的优先级不被稀释。
```

## 本类自检清单

写完 Prompt 后，逐条检查：

- [ ] **指令数量**：并列指令超过 3 条了吗？→ 编号、加标题、末尾加核查步骤
- [ ] **意图表达**：有没有只说"做什么"没说"为什么"和"不能牺牲什么"？→ 补充意图和边界
- [ ] **自洽性**：两条指令之间有没有隐性矛盾？→ 合并为一条或设定优先级
- [ ] **优先级**：最重要的需求是不是放在了最显眼的位置？→ 用【必须满足】标注，核心前置
- [ ] **堵歪路**：模型有没有可能用一种"技术上正确但毫无用处"的方式完成任务？→ 在 Prompt 中明确说"不要用 XX 方式"
