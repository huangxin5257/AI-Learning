# 四、格式与结构类

> **本类共性：** 模型理解了你要什么内容，但在输出的格式和结构上出了问题——格式越写越乱、结构化输出中途崩溃为自然语言、或嵌套层级关系错位。这类问题的修复重点在**格式模板的强化和输出约束**上。

---

## 4.1 格式漂移

> **快速识别：** 前几条格式完美，越往后越随意，最后完全跑偏了

**典型症状**

- 前几个条目严格遵守了指定格式，后面逐步开始省略字段、丢分隔符、改用自然段落
- 条目越多，漂移越严重——10 条内容里前 4-5 条没问题，后面就散了
- 模型不会提醒你格式已经变了，它自己也"没意识到"

**为什么会发生**

模型在生成长内容时，对格式约束的注意力会逐步衰减。越往后生成，前面 Prompt 中格式模板的影响力越弱，模型会慢慢"滑"回自己最自然的输出方式。条目越多、格式越复杂，漂移就越容易出现。

**怎么修**

- **首尾双重锚定**：在 Prompt 开头定义格式模板，末尾再重复一次"请确保所有条目严格遵守上述格式"。
- **给 2-3 个 Few-shot 示例**：直接在 Prompt 里展示完整的格式范例，让模型有具体参照物。
- **分批生成**：超过 5 个条目时，拆成多次请求，每次只生成 2-3 个，降低后半段漂移风险。
- **末尾加格式核查指令**："输出完成后，请检查所有条目是否都严格遵循格式模板。"

**示例**

### 示例 1：产品说明文档

用户原始提示词：
```
请生成一份包含 10 个产品的说明文档，每个产品的格式如下：
**产品名称**：xxx
**功能描述**：xxx
**适用场景**：xxx
---
```

失效输出：
前 4-5 个产品严格遵守了格式，从第 6 个开始"---"分隔符消失了，第 8 个开始改用自然段落描述，到第 10 个已经完全没有了结构标记。

改进版提示词：
```
请生成一份包含 10 个产品的说明文档。

每个产品必须严格使用以下模板，不得在任何一个条目中偏离：

**产品名称**：xxx
**功能描述**：xxx（限 1-2 句话）
**适用场景**：xxx（限 1-2 句话）
---

以下是前 2 个的示例：

**产品名称**：CloudSync Pro
**功能描述**：实时多端文件同步，支持增量备份和版本回溯。
**适用场景**：需要在多台设备间保持文件一致的远程办公团队。
---
**产品名称**：DataGuard
**功能描述**：自动化数据脱敏工具，支持正则和 AI 混合识别。
**适用场景**：处理包含个人隐私信息的测试数据集。
---

请按此格式生成剩余 8 个产品。完成后请检查所有 10 个条目是否都包含三个加粗字段和分隔符。
```

改进后预期效果：
全部 10 个产品条目格式一致，每条都包含加粗的"产品名称""功能描述""适用场景"三个字段，条目之间均有"---"分隔符，后半段不再出现格式退化。

### 示例 2：术语表生成

用户原始提示词：
```
请生成一份包含 20 个机器学习术语的中英对照术语表，格式为：| 英文 | 中文 | 简要定义 |
```

失效输出：
前 10 个术语严格使用了表格格式，从第 11 个开始变成了"11. Overfitting（过拟合）：指模型在训练集上表现好……"的列表格式，表格结构完全丢失。

改进版提示词：
```
请生成一份包含 20 个机器学习术语的中英对照术语表。

格式规则（必须对所有 20 行一致遵守）：

| 英文术语 | 中文术语 | 简要定义（不超过 20 字） |
|----------|----------|------------------------|
| Overfitting | 过拟合 | 模型在训练集上表现好但泛化差 |

请以 Markdown 表格输出全部 20 行，不要在中途切换为列表或段落格式。完成后检查是否所有行都在表格内。
```

### 示例 3：多条评审意见

用户原始提示词：
```
请对这篇论文给出 8 条审稿意见，每条格式为：
[类型：Major/Minor] 意见内容
引用原文位置：xxx
```

失效输出：
前 4 条意见格式正确，第 5 条开始省略了"引用原文位置"，第 7 条开始连"类型"标签也消失了，变成了纯自然语言的评论。

改进版提示词：
```
请对这篇论文给出 8 条审稿意见。

每一条必须严格使用以下格式（无例外）：

---
**[类型：Major/Minor]**
**意见内容**：具体描述问题和建议
**引用原文位置**：指出论文中对应的章节或段落
---

以下是格式示例：

---
**[类型：Major]**
**意见内容**：实验部分缺少消融实验，无法判断各模块的独立贡献。建议补充。
**引用原文位置**：第 4.2 节，实验设置段落
---

请确保第 1 条到第 8 条格式完全一致，不要在后半部分省略任何字段。
```

## 4.2 结构坍塌

> **快速识别：** 让模型输出 JSON/YAML，写到一半突然变成了自然语言

**典型症状**

- 前半段是合法的 JSON/YAML/XML，到某个节点突然插入自然语言解释
- 模型在内容复杂或重复时容易"偷懒"，用"与上面类似"之类的文字代替结构化数据
- 输出看着像结构化格式，但实际上已经无法被程序解析

**为什么会发生**

当结构化内容变长或出现重复模式时，模型会"觉得"用自然语言描述更高效、更清楚，于是自作主张切换输出方式。嵌套越深、条目越多，坍塌概率越高。本质上是模型的自然语言倾向压过了结构约束。

**怎么修**

- **明确禁止自然语言回退**："只输出合法的 JSON，不要添加任何解释性文字、注释或 Markdown 代码块标记。"
- **用 system prompt 设角色**：在 system prompt 中设定角色为"JSON 生成器"，强化结构化输出预期。
- **程序侧校验 + 自动重试**：在应用层对输出做 JSON 解析，解析失败则带错误信息重新请求。
- **用 Structured Output API**：如果平台支持强制 JSON schema 输出，直接用，从根本上杜绝坍塌。

**示例**

### 示例 1：员工信息 JSON

用户原始提示词：
```
请以 JSON 格式输出以下 5 名员工的信息：姓名、部门、入职年份。
```

失效输出：
```json
[
  {"name": "张三", "department": "研发", "year": 2020},
  {"name": "李四", "department": "市场", "year": 2019},
  // 注意：以下员工信息不完整，仅供参考
  以下是剩余三名员工的信息：王五在财务部，2021年入职……
]
```

改进版提示词：
```
请以 JSON 数组格式输出以下 5 名员工的信息。

输出要求：
- 只输出合法的 JSON，不要添加任何注释、解释性文字或 Markdown 代码块标记
- 每个对象包含三个字段：name (string), department (string), year (integer)
- 即使信息不完整，也必须保持 JSON 结构，在不确定的字段中填入 null
- 输出必须能通过 JSON.parse() 解析

员工信息：张三/研发/2020，李四/市场/2019，王五/财务/2021，赵六/人事/2022，孙七/研发/2023
```

改进后预期效果：
输出为合法的 JSON 数组，包含 5 个对象，每个对象都有 name、department、year 三个字段，无注释、无自然语言插入，可直接通过 JSON.parse() 解析。

### 示例 2：多层配置文件

用户原始提示词：
```
请生成一个 Nginx 配置的 YAML 表示，包含 3 个 server block，每个有不同的 upstream 和 location 规则。
```

失效输出：
前两个 server block 用了正确的 YAML 格式，到第三个时模型写道："第三个 server block 的配置与第二个类似，只需将 upstream 地址改为 192.168.1.3:8082 即可。"——YAML 结构中途崩塌为自然语言。

改进版提示词：
```
请生成一个 Nginx 配置的 YAML 表示，包含 3 个 server block。

严格规则：
- 输出必须是完整、合法的 YAML 格式，从第一行到最后一行
- 不得在 YAML 中插入任何自然语言描述、注释或"与上面类似"的省略说明
- 即使 server block 之间配置相似，也必须完整写出每一个
- 不要添加 Markdown 代码块标记（不需要 ```yaml）
```

---

### 示例 3：复杂数据结构

用户原始提示词：
```
请用 JSON 生成一份课程表，包含周一到周五，每天 4 节课，每节课有课程名、老师、教室。
```

失效输出：
周一到周三的数据是正确的 JSON 格式，到周四模型写道："周四和周五的课表安排如下：上午有数学课（张老师，A301）和英语课……"——JSON 结构完全坍塌为自然语言。

改进版提示词：
```
请用 JSON 生成一份课程表。

数据要求：周一到周五，每天 4 节课，每节课包含 course_name, teacher, classroom 三个字段。

输出规则：
- 顶层结构为对象，key 为 "monday" 到 "friday"
- 每天的 value 为包含 4 个对象的数组
- 所有 5 天必须完整输出，禁止用文字描述代替 JSON 结构
- 禁止使用"同上""与周一类似"等省略表达
- 只输出 JSON，不要有任何其他内容
```

## 4.3 层级混乱

> **快速识别：** 要求三层嵌套结构，模型给出来的层级关系是错的或被压平了

**典型症状**

- 要求多层嵌套，但模型把子节点提升到了父级，层级被压扁
- 节点归属错误——本该属于 A 的子项被挂到了 B 下面
- 用 path 字符串或扁平列表"伪装"嵌套，JSON 结构本身其实只有一层

**为什么会发生**

模型对层级关系的理解依赖 Prompt 中的描述方式。纯文字描述（如"公司 > 部门 > 员工"）的层级信号不够强，模型容易把它简化为扁平结构。嵌套越深、每层的属性越多，出错概率越高。

**怎么修**

- **直接给结构示例**：不要只用文字描述层级，直接给出目标结构的完整 JSON/YAML 模板，让模型照着填。
- **分层生成**：先让模型生成顶层结构，确认无误后再逐层填充子结构。
- **用 JSON Schema 约束**：通过 API 的 function calling 或 structured output 功能传入精确的 schema，从根源上防止层级错误。

**示例**

### 示例 1：公司组织结构 JSON

用户原始提示词：
```
请生成如下 JSON 结构：公司 > 部门 > 员工（每个员工有姓名和职级）。
```

失效输出：
```json
{
  "company": "ABC Corp",
  "departments": ["研发", "市场"],
  "employees": [
    {"name": "张三", "department": "研发", "level": "P6"}
  ]
}
```
员工没有嵌套在部门内部，而是被提升为公司级别的平铺列表，层级关系丢失。

改进版提示词：
```
请按以下精确结构生成 JSON（2 个部门，每个部门 2 名员工）：

{
  "company": "公司名",
  "departments": [
    {
      "name": "部门名",
      "employees": [
        {"name": "姓名", "level": "职级"}
      ]
    }
  ]
}

注意：employees 必须嵌套在对应 department 对象内部，不得提升为顶层数组。
```

改进后预期效果：
输出的 JSON 中 employees 数组正确嵌套在各自 department 对象内部，形成 company → departments → employees 的三层结构，不存在层级提升或压扁的问题。

### 示例 2：多级目录大纲

用户原始提示词：
```
请生成一份三级目录大纲：大章节 > 小节 > 知识点，主题是"机器学习入门"。
```

失效输出：
模型生成了一个两级列表——大章节和知识点直接并列，"小节"层级被压缩消失了。例如："1. 监督学习 → 1.1 线性回归 → 1.2 逻辑回归"，其中线性回归和逻辑回归应该是"回归方法"小节下的知识点，而非直接挂在大章节下。

改进版提示词：
```
请生成一份"机器学习入门"的三级目录大纲。

格式要求（严格三级，不得压缩）：

1. [大章节名称]
   1.1 [小节名称]
       - [知识点 A]
       - [知识点 B]
   1.2 [小节名称]
       - [知识点 C]

要求：
- 每个大章节包含 2-3 个小节
- 每个小节包含 2-3 个知识点
- 知识点必须嵌套在小节下，不得直接挂在大章节下
```

---

### 示例 3：文件系统 JSON

用户原始提示词：
```
用 JSON 表示一个文件系统目录树，3 层深度：根目录 > 子文件夹 > 文件。
```

失效输出：
所有文件夹和文件被平铺为一个列表，每个对象用 path 字段（如"/root/docs/readme.md"）来表示层级关系，但 JSON 结构本身只有一层——丧失了嵌套的目录树结构。

改进版提示词：
```
用 JSON 表示一个文件系统目录树。

精确结构定义：
{
  "name": "root",
  "type": "directory",
  "children": [
    {
      "name": "docs",
      "type": "directory",
      "children": [
        {"name": "readme.md", "type": "file", "size": "2KB"}
      ]
    }
  ]
}

要求：
- 使用 children 数组表示嵌套关系，不要用 path 字符串平铺
- 生成 3 个一级子目录，每个包含 2-3 个文件或子目录
- type 字段区分 "directory" 和 "file"
```

## 本类自检清单

写完 Prompt 后，逐条检查：

- [ ] **格式锚定**：条目超过 5 个了吗？→ 给 Few-shot 示例，首尾重复格式要求，或分批生成
- [ ] **结构化约束**：要求输出 JSON/YAML 了吗？→ 明确禁止自然语言插入，指定"只输出合法的 XXX"
- [ ] **防偷懒**：内容有重复模式吗？→ 明确禁止"同上""与前面类似"等省略表达
- [ ] **层级示例**：需要嵌套结构吗？→ 直接给完整的结构模板，不要只用文字描述层级关系
- [ ] **可解析性**：输出需要程序处理吗？→ 加上"输出必须能通过 JSON.parse() 解析"等可验证的硬约束
