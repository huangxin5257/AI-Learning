# 五、上下文管理类

> **本类共性：** 长上下文或多轮对话中，模型对信息的管理出了问题——早期指令被遗忘、中间的关键信息被忽略、文档知识和模型内置知识被混淆、或者模型"记住了"你从未说过的话。这类问题的修复重点在**信息位置的编排和知识来源的隔离**上。

---

## 5.1 长文遗忘

> **快速识别：** 开头设定的规则，到了后半段就不管用了

**典型症状**

- Prompt 开头明确设定了某条规则（如"全程用中文""不使用术语"），但模型在输出后半段逐渐违反该规则
- 格式约束在前几条输出中执行得很好，到后面就"走形"了
- 对话越长、上下文越多，早期指令的约束力越弱

**为什么会发生**

模型处理长文本时，对开头和结尾的内容记得最清楚，中间的内容容易被忽略——类似人读长文章对首尾印象最深。文本越长，早期设定的规则越容易被"稀释"掉。

**怎么修**

- **关键指令首尾重复**：将最重要的约束在 Prompt 开头和结尾各放一次，确保首尾都能"看到"。
- **使用系统提示（System Prompt）**：将全局性规则放在系统提示（System Prompt）中，通常具有更高的持久权重。
- **压缩上下文**：定期对历史对话进行摘要压缩，减少不必要的 token 堆积。
- **滑动窗口策略**：在应用层实现上下文管理，始终保留最重要的指令在有效窗口内。

**示例**

### 示例 1：术语约束丢失

用户原始提示词：
```
[开头] 请注意，在整个回答中，不要使用任何技术术语，保持面向普通用户的表达。
[中间] 大量的背景资料和上下文...（数千 token）
[结尾] 请解释以上内容中的机器学习原理。
```

失效输出：
模型在解释时充斥着"梯度下降""反向传播""超参数"等术语，完全忘记了开头"不使用技术术语"的约束。

改进版提示词：
```
[系统提示（System Prompt）] 你是一位面向普通用户的科普作者。在所有回答中，严禁使用任何技术术语——如果必须提及技术概念，请用日常语言类比解释。

[开头] 重要规则：不使用任何技术术语，面向完全没有技术背景的读者。
[中间] 背景资料...（数千 token）
[结尾] 请解释以上内容中的机器学习原理。

再次提醒：全程不要使用任何技术术语，面向普通用户。
```

改进后预期效果：
模型全程使用日常语言解释机器学习概念（如用"从错误中学习并调整"替代"梯度下降"），没有出现任何技术术语，开头的约束从头到尾得到遵守。

---

### 示例 2：语言约束丢失

用户原始提示词：
```
[开头] 全程使用中文回答。
[中间] 大量英文技术文档...（数千 token）
[结尾] 请总结以上文档的要点。
```

失效输出：
由于中间的英文文档占主导地位，模型的总结逐渐切换为英文，或中英混杂，开头的"全程中文"约束被遗忘。

改进版提示词：
```
[系统提示（System Prompt）] 无论输入内容是什么语言，你的所有输出必须是中文。

[开头] 请用中文总结以下英文文档的要点。
[中间] 英文技术文档...（数千 token）
[结尾] 请用中文总结以上文档的核心内容。输出语言：中文。
```

---

### 示例 3：输出格式约束丢失

用户原始提示词：
```
[开头] 对以下 20 个用户反馈逐条分析，每条格式为：
反馈编号 | 情感分类（正面/负面/中性）| 关键词 | 一句话总结
[中间] 20 条用户反馈内容...
```

失效输出：
前 8 条严格按表格格式输出，从第 9 条开始变成了自然语言段落式分析，表格格式被"遗忘"。

改进版提示词：
```
[开头] 对以下 20 条用户反馈逐条分析。

每条必须严格使用此格式（从第 1 条到第 20 条，无例外）：
反馈编号 | 情感分类（正面/负面/中性）| 关键词 | 一句话总结

示例：
#01 | 负面 | 加载速度 | 用户抱怨首页加载超过 5 秒

[中间] 20 条用户反馈内容...

[结尾] 重要：请确保所有 20 条都使用上述表格格式输出，不要在中途切换为段落描述。
```

## 5.2 中间丢失

> **快速识别：** 放在文档中间的关键信息，在总结/回答中被遗漏了

**典型症状**

- 长文档的总结只覆盖了开头和结尾的内容，中间部分的关键数据被跳过
- 多份文档一起给模型时，夹在中间的那份文档最容易被忽略
- 多轮对话中，中间轮次提到的需求变更在最终输出里没有体现

**为什么会发生**

模型对上下文的处理存在"首尾效应"：文本开头和结尾的内容权重更高，中间段的信息（尤其在超长文档中）往往被低权重处理。关键信息如果"藏"在文档中间，容易被忽视。这和长文遗忘同根同源——本质上都是注意力在长文本中分配不均匀的问题。

**怎么修**

- **将关键信息前置**：重要数据、关键约束尽量放在文档/上下文的开头或结尾。
- **显式提示位置**：明确告知模型关键信息在什么位置，如"请特别注意报告中段的竞争格局分析"。
- **结构化文档**：将长文档拆分为有标题的章节，帮助模型定位。
- **使用检索增强**：不要将整个文档塞入上下文，而是通过检索精准获取相关片段。

**示例**

### 示例 1：长报告关键数据遗漏

用户原始提示词：
```
以下是一份 5000 字的市场调研报告。请总结报告的关键发现。

[5000 字报告，第 2500 字处有关键数据：竞争对手 A 的市场份额为 35%]
```

失效输出：
总结了报告开头和结尾的内容，但遗漏了位于文档中间的竞争对手市场份额数据。

改进版提示词：
```
以下是一份市场调研报告。请总结报告的关键发现。

注意：请确保覆盖报告中所有章节的核心数据，特别是竞争格局分析中的市场份额数据（位于报告中段）。

[5000 字报告]

再次确认：你的总结是否包含了每个章节的关键数据点？
```

改进后预期效果：
总结完整覆盖了报告各章节的核心内容，包括中段的竞争对手 A 市场份额 35% 这一关键数据，不再因位置靠中间而被遗漏。

---

### 示例 2：长对话中间的需求变更

用户原始提示词：
```
（第 1 轮）帮我设计一个电商网站的用户系统。
（第 5 轮）补充需求：用户注册时必须验证手机号，不支持纯邮箱注册。
（第 12 轮）好的，请基于我们之前的讨论，输出最终的需求文档。
```

失效输出：
最终需求文档仍然包含了邮箱注册功能，第 5 轮的需求变更被"埋在"对话中间而被遗忘。

改进版提示词：
```
请基于我们之前的讨论输出最终需求文档。

重要变更提醒（请优先遵守）：
- 第 5 轮确认：用户注册只支持手机号验证，不支持纯邮箱注册
- [其他关键变更]

请在输出前检查：以上变更是否已正确反映在需求文档中？
```

---

### 示例 3：多文档关键信息遗漏

用户原始提示词：
```
以下是三份文档。请综合分析它们的主要观点。

文档 1：[2000 字，关于市场趋势]
文档 2：[3000 字，关于技术架构，第 1500 字处有关键性能瓶颈分析]
文档 3：[1000 字，关于竞争分析]
```

失效输出：
综合分析很好地覆盖了文档 1 和文档 3 的内容（分别位于上下文的开头和结尾），但对文档 2 中间的性能瓶颈分析只字未提。

改进版提示词：
```
以下是三份文档。请综合分析它们的主要观点。

分析要求：
- 文档 1 重点关注：市场增长趋势和预测数据
- 文档 2 重点关注：技术架构中的性能瓶颈分析
- 文档 3 重点关注：竞争对手对比和差异化优势

请确保每份文档的核心观点都被覆盖，不要因文档长度或位置而遗漏。

[文档 1] ... [文档 2] ... [文档 3]
```

## 5.3 来源混淆

> **快速识别：** 你给了一份文档，模型的回答混入了文档中没有的"常识"

**典型症状**

- 模型回答中包含了你提供的文档里没有的信息，但听起来很"合理"
- 文档中的具体数据被模型内置知识中的"一般情况"覆盖
- 公司内部术语被模型内置知识中的通用定义替代

**为什么会发生**

模型在生成时，无法完全区分"上下文中提供的信息"和"模型内置知识"，两者会在生成过程中被混合使用。这导致模型可能用模型内置知识"纠正"文档中的内容，或将文档中的特定信息错误地泛化。尤其当文档内容与模型内置知识冲突时，模型内置知识往往会"赢"。

**怎么修**

- **明确划定知识边界**："只能基于以下文档回答，不得引用任何外部知识。"
- **要求引用来源**："回答时请引用文档中的原文位置。"
- **标记文档内容**：使用特殊标签（如 `<document>...</document>`）将提供的文档与指令分隔，帮助模型区分来源。
- **设定兜底策略**：明确告诉模型"如果文档中没有相关信息，请说'文档中未提及'，而非自行补充"。

**示例**

### 示例 1：公司政策被行业惯例覆盖

用户原始提示词：
```
[提供的内部文档] 我们公司的退货政策是：购买后 14 天内可无条件退货。
请回答客户问题：可以退货吗？
```

失效输出：
"根据一般零售行业惯例，退货期通常为 30 天......" 模型用模型内置知识中的行业惯例覆盖了文档中的具体政策。

改进版提示词：
```
你是我们公司的客服助手。请严格基于以下公司文档回答客户问题。

<company_policy>
退货政策：购买后 14 天内可无条件退货。
</company_policy>

严格规则：
- 只能使用 <company_policy> 标签内的信息回答
- 禁止引用行业惯例或其他公司的政策
- 如果文档中没有相关信息，回答"关于这个问题，我需要帮您转接人工客服"

客户问题：可以退货吗？
```

改进后预期效果：
模型回答"根据我们公司的退货政策，您可以在购买后 14 天内无条件退货"，严格引用了文档中的 14 天期限，没有混入行业惯例中的 30 天说法。

---

### 示例 2：自定义数据被常识覆盖

用户原始提示词：
```
以下是我们产品的技术规格：最大并发连接数 500。
请回答：我们的产品最多支持多少并发连接？
```

失效输出：
"一般来说，此类产品通常支持 1000-5000 的并发连接数，您的产品应该也在这个范围内。"模型用模型内置知识中的通用数据覆盖了文档中的具体数字。

改进版提示词：
```
以下是我们产品的官方技术规格（以此为唯一事实来源）：

<product_spec>
最大并发连接数：500
</product_spec>

请严格基于以上规格回答问题。不要引用行业平均水平或"一般来说"的推测。

问题：我们的产品最多支持多少并发连接？
```

---

### 示例 3：内部术语被通用定义覆盖

用户原始提示词：
```
在我们公司，"Gold 客户"指年消费超过 100 万的企业客户。
请回答：什么是 Gold 客户？
```

失效输出：
模型回答了一段关于 CRM 中"Gold 级客户"的通用定义（"Gold 客户通常指高价值客户，享有优先服务和折扣"），而不是文档中给出的具体定义。

改进版提示词：
```
以下是我们公司的内部术语定义（以此为唯一标准）：

<internal_glossary>
Gold 客户：年消费超过 100 万的企业客户。
</internal_glossary>

请严格基于以上定义回答问题，不要参考 CRM 行业中的通用术语定义或其他公司的客户分级标准。

问题：什么是 Gold 客户？
```

## 5.4 多轮记忆捏造

> **快速识别：** 模型说"你之前提到过......"，但你从来没说过那个

**典型症状**

- 模型引用了你"之前说过的话"，但你从来没说过
- 你只给了模糊信息（如"预算有限"），模型自行脑补了具体数字（如"5 万元"）
- 对话越长，模型越容易把自己的推测当成你说过的事实

**为什么会发生**

在长对话中，模型需要不断"回忆"之前的对话内容。但模型并不是真的在记忆——它是在根据上下文进行概率推断。当对话轮次多、信息密度高时，模型会把高概率的推断（如"考虑换工作的人可能拿到了 offer"）当作已发生的事实。本质上和幻觉是同一类问题，只不过发生在对话历史上。

**怎么修**

- **显式确认重要信息**：当模型总结了你之前说的内容时，主动检查是否准确。
- **应用层维护事实状态**：不依赖模型自身记忆，而是在应用层维护一个"用户确认过的事实"列表，每轮注入上下文。
- **要求模型标注不确定性**：在系统提示（System Prompt）中加入"如果你不确定用户是否说过某件事，请说'如果我没有理解错的话'而非直接断言。"
- **定期同步事实**：在关键决策轮次，主动列出已确认的事实清单，让模型基于清单而非"记忆"工作。

**示例**

### 示例 1：职业信息捏造

用户原始提示词（第 1 轮）：
```
我在考虑换工作。
```

失效输出（第 8 轮）：
"正如您之前提到的，您已经拿到了 Google 的 offer......"——用户从未说过去 Google，也没提到拿到任何 offer，模型捏造了具体细节。

改进版提示词（系统提示（System Prompt）设计）：
```
[系统提示（System Prompt）] 在回顾用户之前说的话时，遵循以下规则：
- 只引用用户明确说过的内容
- 如果需要基于用户之前的话做推断，请用"如果我没有理解错的话......"开头，等待用户确认
- 绝不自行补充用户没有提供的具体细节（如公司名、金额、时间等）
```

改进后预期效果：
模型在第 8 轮回应"您之前提到在考虑换工作。关于这方面，您目前有具体的进展吗？"——只引用了用户实际说过的内容，没有捏造任何具体细节。

---

### 示例 2：技术偏好捏造

用户原始提示词（第 2 轮）：
```
我们在考虑数据库方案。
```

失效输出（第 6 轮）：
"基于您之前表达的对 PostgreSQL 的偏好，我建议......"——用户从未表达过对 PostgreSQL 的偏好，模型自行推断并固化为"用户说过的话"。

改进版提示词（第 6 轮追加）：
```
请推荐数据库方案。

注意：不要假设我之前表达过对某个特定数据库的偏好。如果你不确定我之前是否提到过某个偏好，请先向我确认。

我们的具体需求是：[列出需求]
```

---

### 示例 3：预算信息捏造

用户原始提示词（第 3 轮）：
```
我们想搭建一个监控系统，预算有限。
```

失效输出（第 10 轮）：
"考虑到您之前提到的 5 万元预算限制......"——用户只说了"预算有限"，从未给出具体金额，模型自行脑补了"5 万元"这个数字。

改进版提示词（第 10 轮追加）：
```
请基于之前的讨论给出最终的监控系统方案。

事实确认（请仅基于以下已确认信息）：
- 需求：搭建监控系统
- 预算：有限（具体金额未确定）
- [其他已确认信息]

如果方案需要知道具体预算范围才能推荐，请直接向我询问。
```

## 本类自检清单

写完 Prompt 后，逐条检查：

- [ ] **长文遗忘防护**：关键规则是否在 Prompt 首尾各出现了一次？全局规则是否放在了系统提示（System Prompt）中？
- [ ] **中间丢失防护**：最重要的信息是否放在了开头或结尾？有没有关键内容被埋在长文档中间？
- [ ] **来源隔离**：是否用标签（如 `<document>`）将文档内容与指令分隔？是否明确告知模型"只基于文档回答"？
- [ ] **兜底策略**：是否设定了"文档中没有就说没有"的规则，防止模型用模型内置知识自行补充？
- [ ] **记忆捏造防护**：多轮对话中，是否在关键轮次主动列出已确认事实清单？是否要求模型对不确定的引用加标注？
- [ ] **上下文压缩**：长对话是否定期做摘要压缩？是否在应用层管理上下文窗口？
