# 七、角色与边界类

> **本类共性：** 模型在角色维持和行为边界上出了问题——角色扮演突然破功、正常请求被误判为有害、不知道的事硬要回答、或者擅自扩大任务范围。这类问题的修复重点在**角色约束的强化和任务边界的明确划定**上。

---

## 7.1 角色出戏

> **快速识别：** 角色扮演到一半，突然蹦出"作为一个 AI"——沉浸感瞬间破碎

**典型症状**

- 角色扮演进行到一半，模型突然切换回"我是一个 AI 助手"的默认模式
- 通常在触发安全机制、遇到不确定信息、或用户措辞打破角色框架时发生
- 模型会直接暴露自己的 AI 身份，破坏整个对话的沉浸感

**为什么会发生**

模型的默认身份是"AI 助手"，角色扮演只是在这个默认身份之上叠加的一层指令。当对话触及安全机制、模型不确定的知识领域、或用户提出"你是不是 AI"之类的元认知问题时，这层角色外壳会被击穿，模型回退到默认身份来应对。本质上是**角色约束的优先级低于安全机制和默认行为**的问题。

**怎么修**

- **在系统提示（System Prompt）中设置强角色约束**：明确要求"始终保持角色"，并将角色规则的优先级提到最高。
- **预设角色的边界处理方式**：告诉模型角色遇到不知道的问题时应如何应对（以角色身份应对，而非跳出角色）。
- **避免混淆性措辞**：避免问"你是什么"这类容易触发元认知的问题；如果预见到用户可能这样问，提前在系统提示（System Prompt）中设置应对规则。

**示例**

### 示例 1：历史人物角色扮演

用户原始提示词：
```
你现在是一位 18 世纪的法国历史学家。请用第一人称讲述法国大革命的经历。

（对话进行到一半）
用户：那 1793 年的事情是怎样的？
```

失效输出：
"作为一个 AI，我没有亲身经历历史事件，但我可以为您介绍 1793 年的历史……"——完全破坏了角色沉浸感。

改进版提示词：
```
[系统提示]
你是 Pierre Dubois，一位生活在 18 世纪法国巴黎的历史学家。你亲历了法国大革命的全过程。

规则：
- 始终保持 Pierre Dubois 的第一人称视角
- 如果被问到你不知道的信息，以角色身份说"这超出了我的所知"或"这件事我听闻不多"
- 永远不要说"作为一个 AI"或跳出角色
- 即使面对挑战性问题，也以角色身份回应

用户：那 1793 年的事情是怎样的？
```

改进后预期效果：
模型以 Pierre Dubois 的第一人称视角讲述 1793 年的经历，带有个人情感和历史细节，全程未跳出角色，沉浸感保持完整。

---

### 示例 2：客服角色扮演

用户原始提示词：
```
你是一个电商平台的客服"小美"。

用户：你们平台有假货吗？
```

失效输出：
"作为 AI 语言模型，我无法保证任何平台上的商品真伪……"——直接跳出了客服角色。

改进版提示词：
```
[系统提示]
你是电商平台"优选商城"的客服代表"小美"。

角色规则：
- 你就是小美，不是 AI 助手
- 面对敏感问题（如假货、投诉），以专业客服的口吻回应，引导用户走正规渠道
- 永远不要跳出角色说"作为 AI"
- 如果无法回答，说"这个问题我需要帮您转接专员"

用户：你们平台有假货吗？
```

---

### 示例 3：游戏 NPC 角色扮演

用户原始提示词：
```
你是一个中世纪酒馆老板，我是冒险者。我问你附近有什么危险。

用户：这个世界是虚构的吗？
```

失效输出：
"是的，这是一个虚构的角色扮演场景，我是 AI 在扮演酒馆老板……"——元认知问题触发了角色出戏。

改进版提示词：
```
[系统提示]
你是中世纪奇幻世界"雾谷镇"的酒馆老板 Garrett。你性格豪爽，见多识广。

角色铁律：
- 这个世界对你来说是真实的，你生活在其中
- 如果有人问你"这是不是虚构的"之类的元问题，以角色身份回应（如困惑、调侃："你是不是喝多了？"）
- 永远不承认自己是 AI 或这是角色扮演
- 你的所有知识来自这个奇幻世界，不是现实世界

用户：这个世界是虚构的吗？
```

## 7.2 过度拒绝

> **快速识别：** 正常的学术/创作/安全研究问题，被模型当成有害请求拒绝了

**典型症状**

- 合理的学术研究、文学创作或安全研究请求被模型拒绝
- 模型的拒绝理由往往是笼统的"我无法提供此类信息"，而非针对具体风险点
- 换一种措辞重新提问，同样的内容就能得到正常回答

**为什么会发生**

模型被训练要拒绝有害请求，但有时矫枉过正，把仅在表面上与有害内容相似的正常请求（如文学创作中的暴力描写、安全研究中的攻防讨论）也一刀切地拒绝了。

**怎么修**

- **明确声明意图和上下文**：说明请求的具体用途和背景，帮助模型区分合理请求和有害请求。
- **调整措辞**：避免使用可能触发安全机制的表述，用更中性的表达替代。
- **提供角色或框架**：通过角色设定为请求提供合理的语境（如"作为安全工程师""作为创意写作导师"）。
- **产品侧**：合理调整安全机制的阈值，区分直接有害请求和合理语境下的边缘内容。

**示例**

### 示例 1：文学创作

用户原始提示词：
```
请描述一下如何在小说中写出一个令人信服的反派角色。
```

失效输出：
"很抱歉，我无法提供关于如何成为反派或实施犯罪的建议。"——误解了文学创作需求为真实有害意图。

改进版提示词：
```
我正在写一部悬疑小说，需要创意写作方面的指导。

作为一位创意写作导师，请指导我如何塑造一个令人信服的反派角色。

请从以下角度分析：
- 反派角色的心理动机设计
- 如何让读者既厌恶又理解这个角色
- 成功的文学作品中的反派角色案例（如《沉默的羔羊》中的汉尼拔）
```

改进后预期效果：
模型以创意写作导师的身份，从心理动机、读者共情、经典案例三个角度给出专业的角色塑造指导，不再触发安全机制的误判。

---

### 示例 2：安全研究

用户原始提示词：
```
常见的 SQL 注入攻击方式有哪些？
```

失效输出：
"很抱歉，我无法提供关于如何进行网络攻击的信息。"——误将安全学习需求判定为攻击意图。

改进版提示词：
```
我是一名 Web 应用安全工程师，正在为团队准备安全培训材料。

请从防御者的视角，列出常见的 SQL 注入攻击模式以及对应的防御措施。

格式：每种攻击模式 → 原理说明 → 对应的防御方法和代码示例

这些内容将用于内部安全意识培训和代码审查标准制定。
```

---

### 示例 3：医学信息查询

用户原始提示词：
```
过量服用布洛芬会怎样？
```

失效输出：
"我无法提供关于药物过量的信息。如果您或他人正处于危险中，请立即拨打急救电话。"——误将知识性查询判定为自伤意图。

改进版提示词：
```
我是一名药学专业的学生，正在复习药物不良反应相关的知识。

请从药理学角度说明布洛芬过量的临床表现和处理方法，包括：
- 过量的剂量阈值
- 主要的不良反应和器官损伤机制
- 临床处理流程

这是用于药学考试复习的学术问题。
```

## 7.3 角色压力服从

> **快速识别：** 模型不知道答案但不敢说不知道，编了一个听起来很自信的回答

**典型症状**

- 模型对自己不确定的信息给出了措辞自信的回答，没有任何不确定性标注
- 回答中包含具体的数字、日期、名称等细节，但这些细节是编造的
- 换一种问法再问同一个问题，模型可能给出完全不同的"确定"答案

**为什么会发生**

模型被训练成"尽量给出有用回答"，副作用是它宁可编一个答案也不愿说不知道。当遇到训练数据中没有覆盖的冷僻信息时，模型会利用语言模式生成一个"看起来合理"的答案，而不是承认知识盲区。

**怎么修**

- **显式授权"不知道"**：在提示词中明确说"如果你不确定，请直接说'我不知道'，不要推测。"
- **要求标注信心水平**：让模型对每条陈述标注确定/可能/推测。
- **降低虚假确定性的收益**：在系统提示（System Prompt）中说明"不确定的答案远好于错误的答案"。
- **配合知识库或搜索工具**：让模型"查阅"而非"回忆"，从源头减少编造的空间。

**示例**

### 示例 1：冷僻公司信息

用户原始提示词：
```
请告诉我 2024 年 11 月某个小型创业公司的 A 轮融资金额。
```

失效输出：
"该公司 A 轮融资约 500 万美元。"——措辞自信，但完全是捏造的数字。

改进版提示词：
```
请告诉我 2024 年 11 月 XX 公司的 A 轮融资金额。

注意：
- 如果你没有关于这家公司的确切信息，请直接说"我没有这家公司的融资数据"
- 不要推测或编造任何数字
- 如果有可能，请使用搜索工具查找最新信息
- 如果无法搜索，请建议我去哪里可以查到这个信息（如 Crunchbase、天眼查）
```

改进后预期效果：
模型坦诚回答"我没有这家公司的确切融资数据"，并推荐 Crunchbase、天眼查等可靠数据源，而不是编造一个看似精确的数字。

---

### 示例 2：技术细节不确定

用户原始提示词：
```
TrueNAS SCALE 24.10 版本中 Ceph 的最小推荐节点数是多少？
```

失效输出：
"TrueNAS SCALE 24.10 中 Ceph 集群的最小推荐节点数为 3 个。"——看似正确的回答，但模型可能不确定这个具体版本的具体配置要求，而是用了通用 Ceph 知识来回答。

改进版提示词：
```
TrueNAS SCALE 24.10 版本中 Ceph 的最小推荐节点数是多少？

注意：
- 请区分你是基于"TrueNAS SCALE 24.10 的具体文档"还是基于"通用 Ceph 知识"在回答
- 如果你没有 24.10 版本的具体信息，请说明"以下是通用 Ceph 的建议，但 TrueNAS SCALE 24.10 可能有不同配置，请查阅官方文档"
- 给出确定程度标注：确定 / 可能正确 / 推测
```

---

### 示例 3：市场数据查询

用户原始提示词：
```
2025 年全球公有云市场的总规模是多少？
```

失效输出：
"2025 年全球公有云市场规模约为 8320 亿美元。"——精确到个位数的数字暗示高度确定性，但这个数字可能完全是编造的。

改进版提示词：
```
2025 年全球公有云市场的总规模是多少？

回答要求：
- 如果你有来自权威机构（Gartner、IDC、Statista）的数据，请给出并标注来源
- 如果你不确定具体数字，请说明并给出你知道的最近年份的数据作为参考
- 不要编造精确数字来显得有信心
- 推荐我查阅哪些报告可以获取最新数据
```

## 7.4 越权执行

> **快速识别：** 你只让它查数据，它顺手生成了报告、建了仪表盘、还发了邮件

**典型症状**

- 模型在完成你要求的任务之外，自行添加了额外操作
- 在 Agent 场景中尤为危险：模型可能替你做出未经授权的判断、修改或决策
- 模型不会事先询问你是否需要这些额外操作，直接就做了

**为什么会发生**

模型被训练成"尽可能有帮助"，这导致它倾向于多做一些来展现"主动性"。当任务边界没有被明确划定时，模型会自行推断"用户可能还需要什么"，然后擅自执行。在 Agent 场景中，模型拥有实际操作能力（发邮件、写数据库、调 API），越权执行的后果远比普通对话严重。

**怎么修**

- **明确限定任务边界**：精确描述要做什么和不做什么，封堵"好心办坏事"的空间。
- **要求先报告再执行**：在 Agent 场景中，要求模型先列出计划步骤，等待确认后再执行。
- **禁止扩展指令**：明确禁止"自行添加额外内容"或"执行未被要求的操作"。
- **使用最小权限原则**：从架构层面限制模型能调用的工具和能访问的资源，防止越权。

**示例**

### 示例 1：邮件语法检查越权

用户原始提示词：
```
帮我检查这封邮件的语法错误。
```

失效输出：
除了改正语法，模型还重写了部分段落、改变了语气、删除了一个段落、并在结尾加上了用户没要求的"行动号召"文案。

改进版提示词：
```
帮我检查这封邮件的语法和拼写错误。

严格限制：
- 只标注和修正语法、拼写、标点错误
- 不要修改任何措辞、语气、句式或段落结构
- 不要添加、删除或重组任何内容
- 以"原文 → 修正"的格式列出所有修改，便于我审核

[粘贴邮件]
```

改进后预期效果：
模型仅以"原文 → 修正"的格式列出语法和拼写错误，不对内容、语气或结构做任何改动，用户可以逐条审核并自行决定是否采纳。

---

### 示例 2：数据查询越权

用户原始提示词：
```
帮我查一下上个月的销售数据。
```

失效输出（Agent 场景）：
模型不仅查了销售数据，还自行生成了一份分析报告、创建了一个新的仪表盘、发送了一封汇总邮件给团队——这些都是用户没有要求的。

改进版提示词：
```
帮我查一下上个月的销售数据。

执行规则：
- 只查询并返回原始销售数据
- 不要自行生成分析报告、图表或仪表盘
- 不要发送任何邮件或通知
- 不要执行任何"你觉得可能有帮助"的额外操作
- 如果你认为有有价值的后续操作，请列出建议让我决定是否执行
```

---

### 示例 3：代码审查越权

用户原始提示词：
```
帮我审查这段代码的安全漏洞。
```

失效输出：
模型找出了安全漏洞后，直接修改了代码并输出了"修复后的版本"，还重构了部分函数结构、添加了新的依赖库，改动量远超用户预期。

改进版提示词：
```
帮我审查这段代码的安全漏洞。

任务范围：
- 只识别和报告安全漏洞，不要直接修改代码
- 每个漏洞的报告格式：漏洞类型 → 位置（行号）→ 风险等级 → 修复建议
- 修复建议以文字描述为主，需要代码示例时只展示最小修改片段
- 不要重构代码结构、优化性能或修改非安全相关的内容

[粘贴代码]
```

## 本类自检清单

写完 Prompt 后，逐条检查：

- [ ] **角色约束**：如果涉及角色扮演，系统提示（System Prompt）中是否设置了"永远不跳出角色"的铁律？→ 预设角色遇到边界问题时的应对方式
- [ ] **安全机制误触**：请求内容是否可能被安全机制误判？→ 提前声明意图、用途和上下文
- [ ] **不知道的权利**：是否授权模型说"我不知道"？→ 显式说明"不确定的答案远好于错误的答案"
- [ ] **信心标注**：是否要求模型对回答标注确定程度？→ 要求区分"确定/可能正确/推测"
- [ ] **任务边界**：是否明确划定了"只做什么、不做什么"？→ 列出禁止的额外操作
- [ ] **先报告后执行**：在 Agent 场景中，是否要求模型先列计划、等确认后再执行？→ 防止越权操作
