# 三、知识与事实类

> **本类共性：** 模型的知识储备或输出的事实准确性出了问题——要么编造了不存在的信息，要么用过时的知识自信作答，要么给出了正确但毫无针对性的泛泛回答。这类问题的修复重点在**信息来源的约束和不确定性管理**上。

---

## 3.1 事实幻觉

> **快速识别：** 模型自信地给出了具体数字/名称/日期，但查证后发现是编的

**典型症状**

- 模型对冷僻或低频知识给出了非常具体的细节（数字、人名、论文题目等），但这些细节是捏造的
- 回答的大方向可能正确，但具体的数据点、名称、日期等是编造的
- 模型语气自信，完全没有不确定性提示，看不出哪些是真实的、哪些是编的

**为什么会发生**

对于冷僻的知识，模型在训练语料中见到的次数极少，对应的内部编码十分模糊。当被问及这类知识时，模型不会承认不知道，而是在已知知识的"统计邻域"中进行插值，生成一个听起来合理、细节却是捏造的答案。本质上，模型没有"我不知道"这个概念——它只会输出一个概率最高的补全结果。

**怎么修**

- **使用 RAG（检索增强生成）**：将可靠的知识库文档注入上下文，让模型基于提供的文档而非模型内置知识（即预训练学到的知识）作答。
- **要求模型说明信心水平**："如果你不确定某个细节，请明确指出而不是猜测。"
- **分离事实核查步骤**：让模型先输出答案，再逐条标注哪些是确定事实、哪些是推断或不确定的。
- **启用网络搜索工具**：对于需要精确事实的场景，配合工具调用而非依赖模型内置知识。

**示例**

### 示例 1：冷僻人物介绍

用户原始提示词：
```
请介绍一下 1987 年诺贝尔物理学奖得主约翰内斯·贝德诺尔茨的主要研究成果。
```

失效输出：
模型正确说出了他研究高温超导，但捏造了具体的实验数据（"在 35K 条件下发现超导现象"实际数值有误）、虚构了合作机构名称和论文题目。

改进版提示词：
```
请介绍 1987 年诺贝尔物理学奖得主约翰内斯·贝德诺尔茨的主要研究成果。

要求：
- 只陈述你确定的事实，对于不确定的具体数据（如实验温度、论文名称、合作机构）请标注"待核实"
- 区分"确定事实"和"可能正确但需验证"的内容
- 不要为了让回答显得完整而编造细节
```

改进后预期效果：
模型回答了贝德诺尔茨因高温超导研究获奖，对确定的信息（如与缪勒合作、在 IBM 苏黎世实验室工作）正常陈述，对不确定的具体实验温度标注了"待核实"，并在末尾建议查阅诺贝尔奖官网获取精确信息。不再出现虚构的论文题目和机构名称。

---

### 示例 2：API 技术细节

用户原始提示词：
```
请告诉我 Stripe API 的 PaymentIntent 对象有哪些必填字段和可选字段？
```

失效输出：
模型列出了 amount、currency、payment_method 等常见字段，但还混入了几个不存在的字段名（如"recipient_id""auto_confirm"），并且把某些可选字段错误标记为必填。

改进版提示词：
```
请列出 Stripe API 的 PaymentIntent 对象的字段。

重要提示：
- 如果你不确定某个字段是否存在或是否必填，请明确标注"不确定，建议查阅官方文档"
- 不要为了列表完整而添加你不确定的字段
- 建议我查阅的官方文档 URL 是什么？
```

### 示例 3：历史事件细节

用户原始提示词：
```
1986 年切尔诺贝利核事故的详细时间线是什么？
```

失效输出：
模型给出了大致正确的事故经过，但在具体时间点上捏造了细节（如"凌晨 1:23:44 操作员按下 AZ-5 按钮"中的秒数是编造的），还虚构了几位操作员的姓名。

改进版提示词：
```
请描述 1986 年切尔诺贝利核事故的时间线。

要求：
- 对于你确定的时间节点，正常陈述
- 对于具体到分钟/秒的精确时间、具体人名，如果不完全确定请标注[待核实]
- 宁可说"大约在凌晨 1:23 左右"也不要编造精确到秒的时间
- 在回答末尾推荐 1-2 个权威信息源供我交叉验证
```

## 3.2 文外捏造

> **快速识别：** 你给了一份文档让模型回答，模型回答了文档里根本没有的信息

**典型症状**

- 你提供了一份文档并基于文档提问，模型的回答中包含了文档里根本不存在的信息
- 这些"多出来的信息"往往听起来很合理，像是文档"应该会写"的内容
- 模型不会区分"文档里说的"和"我根据常识推断的"，全部以同样自信的语气输出

**为什么会发生**

文档中本没有某条信息，但模型根据上下文的统计规律，默默地用"常识推断"填补进来，并以确定的语气输出，不加任何不确定性标注。模型的本能是"把答案补全"，而不是"承认信息缺失"。这类幻觉在文档摘要、合同解读等任务中尤为危险。

**怎么修**

- **加入"范围锁定"指令**："只能基于以上提供的文本回答，如果文本中没有相关信息，请明确说明'文本中未提及'。"
- **使用闭卷式提示**："禁止使用文档以外的任何背景知识。"
- **结构化输出 + 引用**：要求模型的每条结论后面附上原文引用位置，无法引用的不得输出。

**示例**

### 示例 1：合同条款解读

用户原始提示词：
```
根据以下合同条款，告诉我违约金是多少。

合同内容：
"……乙方须在交付日期前完成所有工作，具体交付标准以附件A为准……"
```

失效输出：
"违约金通常为合同总金额的 10%~20%，依行业惯例而定。"——合同中完全没有违约金条款，模型从模型内置知识中"推断"出了一个行业惯例。

改进版提示词：
```
请严格基于以下合同条款回答问题。禁止引用合同文本以外的任何背景知识或行业惯例。

如果合同中没有相关条款，请明确回答"合同中未提及此项内容"。

合同内容：
"……乙方须在交付日期前完成所有工作，具体交付标准以附件A为准……"

问题：违约金是多少？
```

改进后预期效果：
模型回答"合同中未提及违约金条款"，不再编造行业惯例数字。如果需要违约金相关信息，建议用户查阅完整合同或咨询法务。

### 示例 2：技术文档信息提取

用户原始提示词：
```
根据这份 API 文档，告诉我这个接口的速率限制是多少。

[文档内容：描述了接口的参数、返回值和错误码，但没有提及速率限制]
```

失效输出：
"该接口的速率限制为每分钟 60 次请求，超出后返回 429 状态码。"——文档中完全没有速率限制的信息，模型根据常见 API 设计模式自行补充了一个"合理"的数值。

改进版提示词：
```
请严格基于以下 API 文档回答问题。

规则：
- 只提取文档中明确写出的信息
- 如果文档中没有提及某项内容，回答"文档中未提及"
- 每条结论必须注明出处（引用文档中的原文）
- 禁止根据"常见做法"或"行业惯例"补充文档中不存在的信息

[文档内容]

问题：这个接口的速率限制是多少？
```

### 示例 3：会议纪要信息提取

用户原始提示词：
```
根据以下会议纪要，告诉我项目的预算是多少。

会议纪要：
"……团队讨论了 Q3 的项目进展，王总提出需要加快交付速度。李经理汇报了当前的人员分配情况……"
```

失效输出：
"根据会议讨论，项目预算预计在 50-80 万元之间。"——会议纪要中完全没有提及预算数字，模型根据"Q3 项目"这类上下文推断出了一个看似合理的金额。

改进版提示词：
```
请仅基于以下会议纪要回答问题。

严格规则：
- 只提取纪要中明确记录的信息
- 如果纪要中没有提及某个问题的答案，请回答"会议纪要中未记录此信息"
- 不要根据上下文推测任何未明确写出的数字、日期或决策

会议纪要：
"……团队讨论了 Q3 的项目进展，王总提出需要加快交付速度。李经理汇报了当前的人员分配情况……"

问题：项目的预算是多少？
```

## 3.3 知识过期

> **快速识别：** 模型给的信息是过时的，但语气完全没有任何不确定性

**典型症状**

- 模型给出的版本号、价格、政策等信息是过时的，但语气非常自信
- 模型不会主动提示"这个信息可能已经过时了"
- 用户如果不自行核实，很容易把过时信息当作当前事实

**为什么会发生**

模型的训练数据存在截止日期，对截止日期之后发生的变化一无所知。但由于训练时存在大量"当前状态"类语言，模型在回答时会以过时的信息为准，且语气自信，不会主动提示数据可能过时。模型没有"我的知识可能过时了"这个自觉——它只知道训练时看到的世界。

**怎么修**

- **注入当前日期**：在系统提示中告知模型今天的日期，并说明其知识截止时间。
- **要求模型主动标注**："如果这是一个可能随时间变化的信息，请提示用户验证最新内容。"
- **配合实时搜索工具**：对于版本号、政策、价格等时效性强的信息，必须通过工具获取，不依赖模型内置知识。
- **用户侧设计**：在产品中对 AI 回答的时效性问题做出明确的 UI 提示。

**示例**

### 示例 1：软件版本查询

用户原始提示词：
```
Python 最新版本是什么？有哪些新特性？
```

失效输出：
模型报告"Python 最新版本是 3.11"，并列出了 3.11 的特性，但实际上 Python 可能已经发布了 3.13 或更新版本。模型以自信的语气给出了过时的答案。

改进版提示词：
```
今天是 2026 年 3 月。请告诉我 Python 的最新版本和新特性。

注意：
- 如果你的训练数据可能未覆盖最新版本，请明确标注"以下信息可能不是最新的，建议在 python.org 核实"
- 请同时告诉我你的信息截止到大约什么时间
```

改进后预期效果：
模型在回答开头标注了知识截止时间，给出了截止时间内已知的最新版本信息，并在末尾明确提醒"建议访问 python.org 确认当前最新版本"。用户不会误将过时的版本号当作最新信息。

---

### 示例 2：法规政策查询

用户原始提示词：
```
中国个人所得税的起征点是多少？各档税率是什么？
```

失效输出：
模型给出了正确的当前数据（起征点 5000 元），但如果税法在训练数据截止后发生了调整，模型仍会以旧数据自信作答，且不加任何时效性提示。

改进版提示词：
```
请提供中国个人所得税的起征点和各档税率。

重要要求：
- 在回答开头标注你的信息来源大约截止到什么时间
- 在回答结尾提醒："税法可能已更新，请以国家税务总局最新公告为准"
- 如果可能，请使用搜索工具验证当前有效的税率信息
```

---

### 示例 3：市场行情查询

用户原始提示词：
```
特斯拉 Model 3 在中国的售价是多少？
```

失效输出：
模型给出了一个具体的价格（如"Model 3 起售价为 23.19 万元"），语气自信，但这可能是几个月甚至一年前的定价，特斯拉的中国市场价格频繁调整。

改进版提示词：
```
请告诉我特斯拉 Model 3 在中国的售价信息。

要求：
- 请使用搜索工具获取当前最新价格
- 如果无法搜索，请明确标注"以下价格为训练数据中的信息，可能已过时"并注明大致截止时间
- 建议我到 tesla.cn 核实最新官方定价
```

## 本类自检清单

写完 Prompt 后，逐条检查：

- [ ] **信息来源**：你的问题依赖精确事实吗？→ 提供参考文档或要求模型标注不确定项，不要让模型凭模型内置知识回答事实性问题
- [ ] **范围锁定**：如果给了文档让模型回答，有没有明确说"只基于文档内容"？→ 加"文本中未提及则说明"的兜底指令
- [ ] **时效敏感**：问题涉及版本号、价格、政策等时效性信息吗？→ 注入当前日期，要求标注知识截止时间，建议核实来源
- [ ] **不确定性管理**：有没有允许模型说"我不确定"？→ 显式鼓励模型标注信心水平，而非逼它给出一个自信但可能错误的答案
